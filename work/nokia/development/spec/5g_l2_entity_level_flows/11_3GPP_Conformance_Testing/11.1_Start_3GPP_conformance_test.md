[image-0][image\_desc]The image is a sequence diagram illustrating the process of a 3GPP conformance test. The diagram depicts interactions between different components: 5G-L1-UL, 5G-L1-DL, 5G-L2-PS, and 5G-CP.
The test starts with a WebUI triggering a transceiver or receiver performance test in a specific cell, followed by 5G-CP triggering the test setup in 5G-L2-PS.
The diagram then branches based on whether the cell group and cell already exist. If not, 5G-CP creates them. Cell and Subcell Setup occurs, with 5G-CP skipping cell setup in 5G-L2-LO. A `PsTm_StartConformanceTestReq` is sent, followed by a `PsTm_StartConformanceTestResp`.
The diagram further differentiates based on the "Test type".
**Test type is Transmitter:**
This section has two main branches based on the test model and load type.
\* **Test model is TM1.1\_ETSI and loadType is flatLoad and loadLevelFlat is low (5G\_L2\_7098, 5G\_L2\_7067):**
\* For each SSB slot, `DIData_SsBlockSendReq` is sent.
\* For each SIB1 slot, `DIData_PdschSendReq` is sent, followed by `DIData_PdschPayloadTbSendReq`.
\* **Other cases:**
\* For each DL transmission slot, `DIData_Pdcc hSendReq (DCI)` is sent, followed by `DIData_PdschSendReq`.
\* There are optional steps for "Two UEs per subcell per slot for some test models (5G\_L2\_1749)", involving `DIData_PdschSendReq` and `DIData_PdschPayloadTbSendReq`.
**Test type is Receiver:**
This section details the receiver testing process.
\* For each UL transmission slot, there's an optional step for `[CB009514-I]If eCPRI 7.2e is enabled`, involving `UIData_PuschReceiveReqL1ru`, `UIData_PuschReceiveReq`, and `UIData_PuschReceiveRespPs`.
\* A loop handles HARQ retransmissions. If `CRC NOK` and `numOfTx < maxUIHarqTx`, it waits for HARQ retransmission. If `CRC NOK` and `numOfTx = maxUIHarqTx`, HARQ is released. If `CRC OK`, `UIData_PuschReceiveResplo` is sent.
**Test type is Control channel testing for PUCCH:**
\* For each UL transmission slot, `UIData_PucchReceiveReq` is sent, followed by `UIData_PucchReceiveRespPs`.
\* There's an optional step for `UIData_Pucch ReceiveRespHarqD`.
**Test type is Control channel testing for PRACH:**
\* For each UL transmission slot, `UIData_PrachReceiveReq` is sent, followed by `UIData_PrachReceiveInd`.
Finally, there are two optional steps at the end:
\* For FR1 TDD and FR2 only (not applicable to FR1 FDD), `PsTM_BeamPatternChangeReq` is sent, followed by `PsTM_BeamPatternChangeResp`.
\* Only for UL receiver testing and UL control channel testing, `PsTM_MeasurementReportind` (Periodical report) is sent.
