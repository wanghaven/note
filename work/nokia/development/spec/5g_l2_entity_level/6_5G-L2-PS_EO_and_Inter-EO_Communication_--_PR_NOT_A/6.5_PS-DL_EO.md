Note: from formal EFS specification perspective, only the highlighted functionalities are due to be covered by chapter 2.2 DL Scheduler.

Here is a list of the PS-DL EO functionalities with brief description:

Message Handling

Processing of messages / events targeted to the Event Queue (EQ) served by PS-DL EO, both external messages / events (i.e. messages / events sent by an external entity or component outside of L2-PS, e.g. Xp interface messages from CRM client, PUCCH/PUSCH receive response event from L1-UL, etc) and internal events (i.e. events either sent within the same EO or sent from other EO of the same L2-PS scheduler instance) are involved.

Note: from message receiving / dispatching perspective, there is no difference between SlotSynchroInd event (or, itf::l2::common::SlotSynchroInd with namespace included) event and other events that are specified further in 5G_L2_6503 Internal Event Reception sub-chapter. SlotSynchroInd is only special that it is used to trigger the DL Scheduling procedure of each slot (so it must be sent out periodically at a fixed time point which is preconfigurable and predictable).

Refer to 5G_L2_6503 “Internal Event Reception” for detail.

Slot Preparation

The functionalities to determine which slot to schedule next.

DL scheduling is started after the reception of SlotSynchroInd event which is sent from platform after slot synchronization service is started (5G_L1L2_FLOW_669_replaced([11177443](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BeTEwPvREe-AqvopbP1qhQ))). Usually SlotSynchroInd is received at a fixed time point as illustrated in 5G_L1L2_FLOW_672_replaced([11177470](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BeTrwvvREe-AqvopbP1qhQ)) “5G-L2-RT slot synchronisation timing”, but in some special situation, e.g. in the situation when L2-PS load is high or overloaded, the reception of SlotSynchroInd may be delayed. In some extremely bad situation, e.g. when L2-PS is too overloaded, the reception of SlotSynchroInd may be delayed too long that L2-PS has to skip the scheduling of one or more slots, in order to assure the L1-DL message receive window restriction mentioned in 5G_L2_350 “Slot Timing” can still be guaranteed for the new slot. See 5G_L2_1851 for more about slot skipping triggering.

One (major) functionality of Slot Preparation is to measure the SlotSynchroInd event reception delay (i.e. slotDelayn of 5G_L2_1853) and trigger slot skipping when needed.

Refer to 5G_L2_6677 for detail.

Schedule Preparation

The resource preparation functionalities and routines that are essential for the DL signal and/or DL data scheduling of the slot selected in Slot Preparation procedure. We’ll use term ‘current slot’ or ‘slot N’ to indicate this slot being selected for DL scheduling. There may be one or multiple slots skipped before slot N being selected, Schedule Preparation only covers the functionalities and routines after slot skipping.

Refer to 5G_L2_6678 for detail.

CSI-RS Schedule

The scheduling of CSI-RS resources.

Following CSI-RS resources are involved:

\- CSI-RS resource(s) for tracking

\- CSI-RS resource(s) for beam measurement

\- CSI-RS resource(s) for channel status measurement

Implementation note: CSI-RS Schedule functionality / procedure is covered by l2ps::dl::sch::bfgroup::Scheduler::scheduleCsiRs(const utils::Xsfn& xsfnOnAir) function from coding perspective.

Refer to 5G_L2_6679 for detail.

SS Burst Schedule

The scheduling of SS signal and PBCH data.

Implementation note: SS Burst Schedule functionality / procedure is covered by l2ps::dl::sch::bfgroup::Scheduler::scheduleSsBurst(itf::l2::SFN sfnOnAir, itf::l2::Slot slotNoOnAir) function from coding perspective.

Refer to 5G_L2_6680 for detail.

Pre-Schedule

The scheduling of the DL Pre-Scheduling functionalities specified in 5G_UP_ALG_4183_replaced([11174302](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuat9vvSEe-AqvopbP1qhQ)) and/or the Pre-Scheduler functionalities specified in 5G_L2_221.

Implementation note: Pre-Schedule functionality / procedure is covered by l2ps::dl::sch::pre::Scheduler::schedule() function from coding perspective.

Refer to 5G_L2_6495 for detail.

TD Schedule

The scheduling of DL Time Domain Scheduling functionalities specified in 5G_UP_ALG_4228_replaced([11175135](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kug0mPvSEe-AqvopbP1qhQ)) and/or the TD Scheduler functionalities specified in 5G_L2_231.

Note: SD Schedule and FDM schedule are both carried out in TD Scheduling phase, as the result, SD Schedule and FDM schedule could be seen as essential parts of TD Schedule.

After TD scheduling, PS-DL EO may send out one or multiple PsInternal_FdSchedulReq to PS-FD EO(s) for FD scheduling.

Implementation note: TD Schedule functionality / procedure is covered by l2ps::dl::sch::td::Scheduler::schedule(const l2ps::utils::Xhfn& xhfnOnAir) function from coding perspective.

Refer to 5G_L2_6496 for detail.

SD Schedule

The scheduling of the DL MU-MIMO for mMIMO TDD FR1 (Beam Based) functionalities specified in 5G_UP_ALG_4358_replaced([11141887](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kr17FfvSEe-AqvopbP1qhQ)) and/or the SD Scheduler functionalities specified in 5G_L2_251.

Note: SD Schedule is carried out within TD Scheduling phase.

Implementation note: the MU pairing functionality of SD Schedule is covered by l2ps::dl::sch::td::muMimo::SdScheduler::schedule(const l2ps::db::SubcellIndices& availableSubCellIndexes, const uint32_t maxNumAvailableSubCells, const bool isMumimoAllowed) function from coding perspective.

Refer to 5G_L2_6497 for detail.

FDM Schedule

The PDCCH/PDSCH/PUCCH resource allocation or pre-allocation functionalities carried out in TD Scheduling phase when FDM is enabled (actFdmScheduling == TRUE).

Note: FDM Schedule is carried out within TD Scheduling phase.

Implementation note: FDM Schedule functionality / procedure is covered by l2ps::pscommon::sch::fdm::Scheduler::schedule(Cs2& cs2, const utils::Xsfn& xsfnOnAir, bool isFdmParellelizedByMuMimo) from coding perspective.

Refer to 5G_L2_6824 for detail.

SR / CSI-Report Reception

The scheduling of SR / (periodic) CSI-Report receptions.

Refer to 5G_L2_7945 Schedule PUCCH procedure for SR / CSI-Report reception when multiplexed with HARQ A/N feedback.

Refer to 5G_L2_7949 SR/CSI-Report Alone Schedule for SR / CSI-Report reception when not multiplexed with HARQ A/N feedback.

Post TD Schedule

All the functionalities that are carried out after the execution of TD Schedule / SD Schedule / FDM Schedule functionalities / procedures and before or overlaps with the execution of FD Schedule functionalities / procedures.

\- If PS-FD EO runs on the same core as PS-DL EO, FD Schedule functionalities / procedures are all executed after Post TD Schedule functionalities / procedures.

\- If PS-FD EO runs on a different core than PS-DL EO, the execution of FD Schedule functionalities / procedures may overlap with the execution of Post TD Schedule functionalities / procedures.

Refer to 5G_L2_7449 Post TD Schedule for detail.

FD Schedule

Frequency domain scheduling of the UEs which is the output of TD Scheduling phase, more precisely, the handling of PsInternal_FdSchedulReq event within PS-FD EO.

After FD Scheduling, or more precisely, after the handling of each of PsInternal_FdSchedulReq event, PS-FD EO sends out a PsInternal_FdSchedulResp event to PS-DL EO(s) for further processing.

Implementation note: FD Schedule functionality / procedure is covered by l2ps::fd::sch::MainComponent::processFdScheduleReq(const itf::l2::ps::internal::FdScheduleReq& fdScheduleReqMsg) from coding perspective.

Refer to 5G_L2_6500 FD Scheduler for detail.

Post Schedule (Critical Path)

The time critical functionalities that are carried out after all PsInternal_FdSchedulResp messages are received from PS-FD EO(s), or the time critical functionalities that are carried out after TD Scheduling if no PsInternal_FdSchedulReq being sent out (i.e. FD Scheduling is not triggered). Note: time critical functionalities refer to the functionalities and/or procedures must be carried out respecting the L1-DL reception window timing restrictions (5G_L2_350).

Refer to 5G_L2_6501 for detail.

Post Schedule (Non-Critical Path)

The functionalities that are carried out after all time critical functionalities are finished execution (so they are not time critical) and are essential for the DL scheduling of the coming slot(s) after slot N (so they are not categorized into Periodical Routine Triggering category).

Refer to 5G_L2_6502 for detail.

Periodical Routine Triggering

All the functionalities or routines which are also triggered for execution within PS-DL EO after SlotSynchroInd event reception, but are not suitable to be categorized into any categorizations mentioned above as they are not essential for DL scheduling.

Note: When to check and trigger the execution of Periodical Routine Triggering functionalities is rather a decision of SW implementation, and there is no need to document it in DOORS.

#### 6.5.3.2.0-3 DSS Activation Model Configure FDD Duty Cycle and Number of DL Symbols for DSS for EIRP (ID: 11245099)

#### 6.5.10.1.0-3 Post TD Schedule Model Schedule PUCCH (ID: 11245317)

**6.5.10.1.0-3.0-1**  (ID: `11245320`)

Post TD Schedule Model If bfSchedulerHasTriggeredScheduling (5G_L2_6676) is set to FALSE, carried out the PUCCH resource scheduling functionalities / procedures that mentioned below. Note: If bfSchedulerHasTriggeredScheduling is set to TRUE, the PUCCH resource scheduling functionalities / procedures that mentioned below are triggered for execution from Post Schedule (Critical Path) phase (5G_L2_6501).

  1. Prepare and send PsInternal_DlHarqFeedbackReq to PS-UL EO.

  2. If current slot is the last slot of HARQ A/N bundling window, prepare UlData_PucchReceiveReq for all the UEs whose DL TD/FD scheduling are finished (so the PUCCH resource allocation for HARQ A/N feedback within the whole HARQ A/N bundling window, which is possibly multiplexed with periodic CSI report and/or SR report, is determined). Especially, prepare following information per PUCCH resource allocation result:




1) total number HQRQ A/N bits used in UlData_PucchReceiveReq sending

2) beam ID to be used for PUCCH reception. Either sector beam or UE specific beam is selected per beamforming control for PUCCH reception.

3) In case of UE specific beam is selected for PUCCH reception, prepare PsInternal_BeamSelectionResultInd with the selected UE specific beam.

Please also refer to DL Scheduler Digams.vsd (5G_L2_6856), adRunPucchScheduler page for relevant activity diagram arranged according to L2-PS SW implementation.

Informatively, a brief summary of HARQ A/N feedback multiplex with periodic CSI report and SR:

\- If HARQ A/N feedback PUCCH slot is also pre-configured to receive periodic CSI report for CQI/RI and/or periodic CSI report for beam report, HARQ A/N feedback bits could be multiplexed with periodic CSI report bits using same PUCCH resource which is pre-configured for HARQ A/N feedback report. PUCCH format 2 resource is used in case of short PUCCH and PUCCH format 3 resource is used in case of long PUCCH.

\- In case of PUCCH format 3 being used for HARQ A/N feedback, no matter multiplexed with periodic CSI report or not, if HARQ A/N feedback PUCCH slot is also pre-configured to receive SR report, HARQ A/N feedback bits could be multiplexed with SR report bit on a PUCCH format 3 resource which is pre-configured for HARQ A/N feedback report.

\- In case of PUCCH format 1 is being used to report HARQ A/N feedback, if HARQ A/N feedback PUCCH slot is also pre-configured to receive SR report, HARQ A/N feedback bits could be multiplexed with SR report bit on the same PUCCH format 1 resource which is pre-configured to receive HARQ A/N feedback. #### 6.5.13.2 Message / Event Dispatching

##### 6.5.13.2.0-1 Message / Event Dispatching The messages / events received by PS-DL EO are either dispatched from EO level (i.e. L2-PS scheduler instance level) or dispatched from serving cell level. Rational: There is one and only one PS-DL EO deployed for each L2-PS scheduler instance, and one and only one cell group is served by each PS-DL EO. Considering one cell group contains only one cell in FR1 and contains up to 8 cells in FR2, each PS-DL EO or L2-PS scheduler instance serves one cell in FR1 and up to 8 cells in in FR2. For FR1, although only one cell is served by each PS-DL EO, the message dispatching of it is also carried out firstly from EO level then from cell level as FR2 does. From PS-DL EO SW implementation PoV, EO level messages / events dispatching is controlled via l2ps::pscommon::dispatcherFsm::QueueDispatcherFsmImpl structure and cell level messages / events dispatching is controlled via l2ps::dl::em::QueueFsmImpl structure. Accordingly, for PS-DL EO, we use “QueueDispatcherFsmImpl Statechart diagram” (5G_L2_7125) to document the state chart of EO level messages / events dispatching and use “QueueFsmImpl Statechart diagram” (5G_L2_7127) to document the state chart of cell level messages / events dispatching. (ID: 11245551)

###### 6.5.13.2.1.0-8 EO level messages / events dispatching 200 dispatcherWaitFdSchedRespState (ID: 11245654)

**6.5.13.2.1.0-8.0-1**  (ID: `11245659`)

EO level messages / events dispatching The DL scheduling state of a certain slot.

Usually only PsInternal_FdScheduleResp event(s) is expected to be processed within this state. For all other messages and/or events:

\- If it is a itf::l2::common::SlotSynchroInd event and the number of skipped itf::l2::common::SlotSynchroInd events is less than rdMaxSkippedSlotsWaitingForFdResponse (default: 50), L2-PS skips it silently;

\- If it is a itf::l2::common::SlotSynchroInd event and the number of skipped itf::l2::common::SlotSynchroInd events reaches rdMaxSkippedSlotsWaitingForFdResponse, current dispatcherWaitFdSchedRespState is aborted and a new slot scheduling begins then after.

\- If it is an event other than PsInternal_FdScheduleResp or itf::l2::common::SlotSynchroInd, the processing of it is delayed out of this dispatcherWaitFdSchedRespState, in other words, the processing of it is carried out after the DL scheduling of current slot.

Before exiting dispatcherWaitFdSchedRespState, L2-PS checks to see whether the processing of one or more messages / events is delayed. If yes, L2-PS takes event priority into consideration for later message dispatching, where higher priority messages / events are processed firstly. For messages / events belong to same priority, dispatching is always FIFO based. Refer to 5G_L2_7215 for detail about message / event dispatching priority and the list of messages / events of each message / event dispatching priority.

\- Please do not obfuscate the priority based message dispatching here mentioned and EQ based event dispatching which is also (EQ’s) priority based: for a certain message / event, after it is sent to a dedicated EQ served by PS-DL EO, it is firstly dispatched to PS-DL EO according to the general EM message dispatching mechanism based on EQ priority; after it reaches PS-DL EO, priority based message dispatching here mentioned may kick in.

\- It is possible that the pending message / event that triggers priority based message dispatching after dispatcherWaitFdSchedRespState is not received during current dispatcherWaitFdSchedRespState state, but is received several slots ago. That is, depending on CPU /memory load of DL scheduler, the processing of a low priority message / event, e.g. the events from C-Plane, may be delayed for more than one slots.

##### 6.5.13.3.1 Slot Synchronization Procedures

###### 6.5.13.3.1.1 itf::l2::common::SlotSynchroInd

###### 6.5.13.3.1.1.0-1 itf::l2::common::SlotSynchroInd SlotSynchroInd is used to trigger the DL Scheduling procedure of each slot (so it must be sent out periodically at a fixed time point which is preconfigurable and predictable). When SlotSynchroInd is received firstly after PS-DL EO start up or when SlotSynchroInd is received after the DL Scheduling procedure of previous slot is completed, that is, when SlotSynchroInd is received in dispatcherDefaultState (5G_L2_7125), the DL Scheduling procedure is started for next slot as illustrated in 5G_L2_6684. Implementation note: l2ps::dl::sch::MainComponent::handle(const itf::l2::common::SlotSynchroInd&amp; / unused /) is called by the Message / Event Dispatching frame work of PS-DL EO to start the DL Scheduling procedure of one slot. When SlotSynchroInd is received during the DL Scheduling procedure, that is, when SlotSynchroInd is received in dispatcherWaitFdSchedRespState (5G_L2_7125), L2-PS skips it silently to assure the ongoing DL scheduling procedure are still carried out smoothly. Up to rdMaxSkippedSlotsWaitingForFdResponse (default: 50) SlotSynchroInd events could be skipped. After rdMaxSkippedSlotsWaitingForFdResponse SlotSynchroInd events are skipped, L2-PS aborts the ongoing DL scheduling procedure and start the DL scheduling procedure of a new slot. Please refer to 5G_L2_6036 also for the SlotSynchroInd skipping here mentioned. (ID: 11245865)
