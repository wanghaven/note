# 1.1 Deployment

## 1.1.0-1 Deployment L2-RT EO cores deployment discovering (ID: 11140216)

**1.1.0-1.0-2**  (ID: `11140228`)

Deployment
until now L2-RT discovers EO cores deployment thanks to boardConfigId.

L2-RT can get the boardconfig using following function and API messages

1. Get slotId using DDAL API

int ddal\_id\_get\_slot\_id (int slotId)

SlotId:

1, 2, 3, 5, 6, 7 for ABIO

1. Get boardconfig using UP HWAPI

API2\_STARTUP\_GET\_BOARD\_CONFIG\_REQ\_MSG (slotId)

API2\_STARTUP\_GET\_BOARD\_CONFIG\_RESP\_MSG(slotID, boardconfig number, [DeviceConfig number for device(s)])

Note: In order to reduce number of boardConfigId hardcoded in 5G-L2-PS, at run time 5G-L2-PS may use em\_core\_count API instead of boardConfigId value to retrieve container relative cores id allocated. It is to prepare future work where boardConfigId impact on 5G-L2-PS deployment will be changed and make 5G-L2-PS independent of it. Up to CP3 team to decide to switch to em\_core\_count API or not from now. Note that 5G-L2-LO has switched to em\_core\_count API for EO cores deployment purpose. [CB007431-A] When feature is delivered it is not expected to support previous Sw EO deployment, but if at CP3 it is decided to support backward compatibility on master, special attention shall be done with fact that actual 5G-L2-LO deployment is same inconsistently of the boardConfigId used. There may be conflict between actual 5G-L2-LO implementation and 5G-L2-PS EO deployment if 5G-L2-PS maintained backward compatibility and so still depend on boardConfigId. In hypothesis of last scenario is adopted some CP3 adaptations on 5G-L2-LO are required. [End CB007431-A]

## 1.1.0-2 Deployment L2-RT EO cores deployment based on flavor ID (ID: 11140235)

**1.1.0-2.0-3**  (ID: `12745691`)

Deployment
CB007696  introduces a new framework  for software deployment on snowfish and Xeon based boards. This is also available on Marlin **[CB014419-SR-EB] and Nemo**[End CB014419-SR-EB] based boards.\*\*\*\*

1. 5G-L2-PS and 5G-L2-LO extracts FLAVOR\_ID, CONTAINER\_PRIVATE\_PARAMETER and RT\_CORE\_AFFNITIES  from OS environment variables using std::get\_env api during startup.
2. 5G-L2-PS and 5G-L2-LO can use values extracted in step 1 above to compute start core and number of cores.
3. if 5G-L2-PS and 5G-L2-LO cannot retrieve env variables mentioned in step 1, it will fall back to usage of earlier release way(refer 5G\_L2\_2839\_replaced([11140216](https://dn-prod.ext.net.nokia.com/rm/resources/BI_23hy8vvTEe-AqvopbP1qhQ))) of computing startcore and number of cores using ddal APIs.

Following is the General information related to new framework introduced by CB007696  and environment variables  mentioned in requirement above, the information is gathered from various sections of CB007696 CFAM to give an overview and context to user plane impacts. For further details refer to CB007696 CFAM

* Framework supports software deployments of linux containers for a board(including all snowfish and marlin **[CB014419-SR-EB] and Nemo**[End CB014419-SR-EB] based deployments supported until CB007696) type and is stored in a centralized file that will be part of the software package.\*\*\*\*
* Framework allows  flexible UP container configuration, information to compute start core and number of cores is provided when starting the container.
* **RTCCSConfig.xml and CCSEarlyConfig.xml configuration files are stored.(refer to requirement 5G\_UP\_2342\_replaced([11386314](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ufBnPfvkEe-AqvopbP1qhQ)))**
* On Marlin **[CB014419-SR-EB] and Nemo**[End CB014419-SR-EB] based boards, em-odp.conf and odp.conf configuration files are stored as well.\*\*\*\*
* FlavorID shall have a number that represents specialization of container, Corelation between FlavorId, no of RT cores, containerprivate parameter, associated boardconfig is detailed at following link(centralized information of all FlavorID per board type, Reference: 5G\_UP\_2342\_replaced([11386314](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ufBnPfvkEe-AqvopbP1qhQ))).

Link : <https://nokia.sharepoint.com/:x:/r/sites/ASCOP/PA/_layouts/15/Doc.aspx?sourcedoc=%7B143666A9-E899-4972-85BD-0777E8D7EFF8%7D&file=BTS_Software_Deployments.xlsm&wdLOR=c6396DBCD-D8B7-4F07-A5CA-627C85E7A098&action=default&mobileredirect=true>

UPHWAPI gets container specific attributes like FlavorID,ContainerPrivateParameter and RTCoreAffinity from BOAM in API5 \_VIRT\_NODE\_START\_REQ\_MSG and sets them as OS supported environment variables before starting L2RT containers

ContainerPrivate parameter represented as privateparam in column l(excel in link above) represent no of L2LO cores

RTCoreaffinity represents total no of RT cores and represented in tab ListFlavorID(excel in link above)

Note: BOAM decides if the new framework to be activated via a BOAM RnD parameter. UP containers get the info of usage of new framework only if new set of environment variables mentioned above are set by UPHWAPI. Refer to chapter 1.3.1.8 Feature activation in CB007696 CFAM.

## 1.1.0-3 Deployment EqId on 2nd VF for XpIf communication (ID: 11140255)

**1.1.0-3.0-2**  (ID: `11140270`)

Deployment
5G-L2-PS UL/DL shall use the 2nd Virtual Function (VF) to communicate via the backplane with the XpIf component. This is used for the two datapaths

* between the XpIf component and 5G-L2-PS DL
* between the XpIf component and 5G-L2-PS UL

[Before CB008140-AC]A new EqId shall be allocated to map the 2nd VF and the associated queues such that 5G-L2-PS can receive EN DSS messages from XpIf. 5G-L2-PS will provide these  EqIds in the message PsCell\_DssCellCreateCrmClientResp (5G\_L2\_IF\_251\_replaced([10971820](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2Uqf2_u6Ee-AqvopbP1qhQ))) to the 5G-CP-RT.

[CB008140-AC]New EqIds (separate for DSS and PsPeerCtrl messages) shall be allocated to map the 2nd VF and the associated queues such that 5G-L2-PS can receive:

• EN DSS messages from XpIf. 5G-L2-PS will provide these  EqIds in the message PsCell\_DssCellCreateCrmClientResp (5G\_L2\_IF\_251\_replaced([10971820](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2Uqf2_u6Ee-AqvopbP1qhQ))) to the 5G-CP-RT.

• PsPeerCtrl messages from XpIf. 5G-L2-PS will provide these EqIds in the message [Before PR656432]PsCell\_IgNbCaCellCreateResp[PR656432]PsCell\_IgnbCaCellCreateResp[End PR656432] (5G\_L2\_IF\_2511\_replaced([10973529](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2U1e_vu6Ee-AqvopbP1qhQ))) to the 5G-CP-RT.[End CB008140-AC]

## 1.1.0-4 Deployment L2-RT deployment for positioning functionality (ID: 11140277)

**1.1.0-4.0-1**  (ID: `11140282`)

Deployment
For the SRS positioning functionality, a L2-RT pool with a lightweight 5G-L2-PS is configured. In this case, the 5G-L2-PS is not configured with regular cell contexts but shall support the configuration of positioning cell contexts (refer to 5G\_UP\_11750\_replaced([11369761](https://dn-prod.ext.net.nokia.com/rm/resources/BI_udsKdfvkEe-AqvopbP1qhQ))).

Please note some major implementation consequences:

* 5G-L2-PS UL/DL scheduler EOs are not created
* 5G-L2-PS creates dedicated PosSch scheduler EOs specifically for positioning cells
* 5G-L2-LO exists and occupies some cores but is not configured/does not operate

## 1.1.0-5 Deployment L2-RT container library sharing (ID: 12801145)

**1.1.0-5.0-2**  (ID: `13409922`)

Deployment
Since CB010827-A, container library sharing is supported by platform, that is, a single 5G-L2-PS / 5G-L2-LO library copy could be shared among multiple L2-RT instances. 5G-L2-PS / 5G-L2-LO performance could benefit from container library sharing. To resolve PR848272 which is a KPI decreasing issue with new HW deployment, it is decided to deploy L2RT container library sharing on ABIP and ABIQ capacity board for a better performance.

Future development consideration: LC or SC EFS module (like this 5G L2 Entity Level module) may not be the most suitable place to specify inter container library sharing, and group agreement among all stakeholders, including SFS, EFS, SW Arch, etc, is needed to find out an ideal manner to specify inter container things.
