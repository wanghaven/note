## 4.9.0-1 Traceability Document Owner Maciej 2. Lewandowski (Nokia) (ID: 11228848)

## 4.9.1 Sherpa Trace

### 4.9.1.0-2 Sherpa Trace [Before CNI-115810] L2 Sherpa trace functionality provides insight of L2 interfaces. For 5G-L2-HI this means mainly RLC and PDCP user plane protocol from data transmission and data reception messages and flow control related messages. Sherpa trace supports only streaming mode. Streaming mode shall stream trace information to external server by IP/UDP messages. Streaming mode shall be activated and deactivated by user request according to 5G_L1L2_FLOW_769_replaced( 11205620 ). 5G-L2-HI shall support Sherpa traces independently from L2 trace (5G_L2_2695_replaced( 11230027 )). L2 Sherpa trace shall have fixed content on item level. Filtering happens on object level, I.e. traced object can be selected. [End CNI-115810] (ID: 11228866)

### 4.9.1.1 Trace items

#### 4.9.1.1.0-2 Trace items [Before CNI-115810] Sherpa trace shall support several  trace types in RLC entity of L2-HI and in PDCP entity of L2-HI. details of the report for each trace type is detailled in next chapters. [End CNI-115810] (ID: 11228883)

### 4.9.1.2 Trace filtering

### 4.9.1.2-1 Trace filtering

#### 4.9.1.2-1.0-2 Trace filtering [Before CNI-115810] All sub-components shall send separate stream messages. If there are CPU several cores for same sub-component they also shall send independent stream messages. Receiver can separate these messages by l2SourceNid in the message header. 5G-L2-HI (DU) and 5G-L2-HI (CU) shall send a stream message by two triggers. When message gets full (size reaches 1400 bytes) Timer timeout. Timer shall be started when a trace streaming starts and the timeout value shall be 500ms At the streaming trace stop all stream messages shall be flushed out. [End CNI-115810] (ID: 11230022)

#### 4.9.1.2.0-2 Trace filtering [Before CNI-115810] Sherpa traces are filtered on object level. This means that if trace is activated for an object then full trace content shall be collected and reported for that object. Filtering object is cell and it shall identified by list of parameter physCellId in trace start request L2LogDu_StreamActivationtReq. 5G-L2-HI shall support up to eight cell filter list. Filter object list modification is not supported. In order to modify filter list the ongoing measurement must be stopped and new one started. [End CNI-115810] (ID: 11230003)

## 4.9.2 L2 Trace

### 4.9.2.1 Description

#### 4.9.2.1.0-3 Description L2 trace functionality provides insight of L2 internal functionality and L2 interfaces. For 5G-L2-HI this means mainly PDCP and RLC user plane protocols. L2 trace shall support Streaming and Snapshot -modes. Streaming mode shall stream trace information to external server by IP/UDP messages. Streaming mode shall be activated and deactivated by user request according to 11205620 . Snapshot mode shall store trace information into internal file system. Snapshot mode shall be active continuously and it shall store information in a ring buffer manner. I.e data shall be saved into limited storage and when storage is full the old information shall be discarder to make room for new. Snapshot mode shall follow the functionality define in 11205475 . L2 trace shall have dynamic content on item level. This means that a filtering parameter shall be set to define the trace items that are included (into Stream message or Snapshot file). The trace items shall have fixed size and structure. [CNI-148276] Array type and optional type parameters in trace items structure can be allocated statically or dynamically based on users' preference. [End CNI-148276] Trace items no longer maintained in 5G-L2-HI EFS. Content is available on the Software Architecture Document (ID: 13444085)

### 4.9.2.3 Trace bitrate control

#### 4.9.2.3.0-1 Trace bitrate control Trace bitrate is not limited or controlled by the application. Instead that is left for tester to ativate trace items in such manner that resulting snapshot time lenght is not too short / stream bitrate is not too big. Resulting trace bitrate can be estimated from table in 5G_L2_2699. (ID: 11231290)

### 4.9.2.4 Trace filtering

#### 4.9.2.4.0-2 Trace filtering L2 trace shall support filtering on object level and  on item level. Filtering object is cell and it shall identified by list of parameter physCellId in trace start request L2LogDu_StreamActivationtReq [Before PR755442] /L2LogCu_StreamActivationReq [End PR755442]. 5G-L2-HI shall support up to eight cell filter list. Filter object list modification is not supported. In order to modify filter list the ongoing measurement must be stopped and new one started. Trace items shall be activated / deactivated by R&amp;D parameters that defines the filtering bitmap. See 5G_L2_2756_replaced( 11232476 ) and 5G_L2_2757_replaced( 11232485 ), separate parameters for Tx (items that are related to DL) and Rx (items that are related to UL). Default filter state (ACT/DEACT)  is stated in the item table in 5G_L2_2698_replaced( 11230039 ). Note that filtering R&amp;D parameters are common to both stream and snapshot. Filtering parameters shall be read at the streaming start. Snapshot filtering shall be updated dynamically run time as it is collected all the time. Note that modified trace item filter is active until next component restart. Filtering bitmap mapping to trace items left for implementation. I.e. the exact bit to item mapping must be checked from source code. (ID: 11231307)

### 4.9.2.5 Streaming mode

#### 4.9.2.5.0-1 Streaming mode At the streaming trace start 5G-L2-HI shall clear all stream messages. Stream messages shall not exceed standard Ethernet frame size (1500 bytes). 5G-L2-HI shall transmit stream message immediately when it gets full. Stream messages shall be flushed out at latest 50ms after the first item addition if the message did not get full before that. 5G-L2-HI shall not send empty stream messages that do not contain any items. At the streaming trace stop 5G-L2-HI shall flush all stream messages out. All sub-components shall send separate stream messages. If there are CPU several cores for same sub-component (e.g. DU-L2HI) they also shall send independent stream messages. (ID: 11231317)

### 4.9.2.6 Snapshot mode

#### 4.9.2.6.0-2 Snapshot mode 5G-L2-HI shall activate snapshot at the component start and it shall be active all the time.[Before CNI-106544] Except when streaming trace is activated. For the duration of streaming trace  5G-L2-HI shall pause writing items into the snapshot file. After streaming trace is stopped 5G-L2-HI shall continue writing snapshot automatically.[End CNI-106544] Items shall be written to snapshot based on the filter R&amp;D parameter (see 5G_L2_2719_replaced( 11231297 )). Snapshot shall behave in ring buffer manner so that most recent information is included and oldest items are discarded as file gets full. See 5G_L1L2_FLOW_696_replaced( 11204637 ) for snapshot buffer size. During Snapshot collection phase it is allowed to lose new trace data (i.e. the data that is generated during the collection). [Before CNI-106544]Note, due CPU load reasons the snapshot is paused for the duration of streaming trace. [End CNI-106544] (ID: 11231333)

#### 4.9.2.7.0-1 Parsing trace information Parser tool for reading saved streaming trace files and snapshot files shall be implemented by L2 implementation team. Tool shall convert the trace data into human reaable format. (ID: 11231346)

#### 4.9.4.1.0-3 Bearer Monitoring via EMIL in 5G-L2-HI (CU) Maintaining of monitored bearers counter (ID: 11231899)

**4.9.4.1.0-3.0-1**  (ID: `11231905`)

Bearer Monitoring via EMIL in 5G-L2-HI (CU) When monitoring session is active, 5G-L2-HI (CU) shall maintain numOfTracedBearers internal counter per bearer group and qiType/qiValue (see Note 1).

Upon creation of new bearer (reception of HiUserCu_BearerSetupReq) numOfTracedBearers is incremented by 1 and bearer is added to the monitored pool.

In case numOfTracedBearers < numBearers (see Note 1), 5G-L2-HI shall add new bearer to the monitored pool.

Upon removal of monitored bearer (reception of HiUserCu_BearerDeleteReq, see Rationale) numOfTracedBearers is decremented by 1 (see Note 2).

Note:

  1. qiType, qiValue and numBearers are received via L2LogCu_BearerMonitoringReq under TraceFilter (5G_L2_IF_5182_replaced([10994173](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2XDFfPu6Ee-AqvopbP1qhQ))) structure.

  2. It is possible that in case new monitoring session is started in short time since the last one, bearers from previous session will be still monitored as they were not yet deleted. This might lead to:

  3. decreased number of new bearers monitored with same qiType/qiValue when new filter applies same quality indicator,

  4. presence of not yet deleted bearers from previous session in EMIL tool.



Rationale:

It was decided that numOfTracedBearers will be decremented during bearer delete procedure. It will avoid going through all the bearers in all bearer groups to clear monitoring flag which might have significant performance impact.
