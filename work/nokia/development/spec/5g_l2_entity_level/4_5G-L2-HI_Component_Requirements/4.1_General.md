### 4.1.4.2 Timers

#### 4.1.4.2.0-1 Timers RLC and PDCP layer timer accuracy (ID: 11220839)

**4.1.4.2.0-1.0-2**  (ID: `12809296`)

Timers RLC and PDCP protocol timers are handled by 5G-L2-HI and that processing is not tied to 5GTF-subframe or slot boundaries at any way. Timeout value for these vary from 1ms to 500ms

Timers for RLC:

  * [Before CNI-136896-D] t-PollRetransmit [CNI-136896-D] timerPollRetransmit [End CNI-136896-D]
  * t-reassembly
  * t-StatusProhibit



Timers for PDCP:

  * t-Reordering
  * tMaxPollingResp
  * discardTimer (see: [11223415](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_qbQfvTEe-AqvopbP1qhQ))



Minimum 1ms accuracy shall be maintained for these timers.

**Rationale:**

These timers do not have direct impact on RTT times or throughput. Microsecond level accuracy is not relevant for the purpose. Accuracy of one TTI would be desirable but with current timer service 1ms is best we can achieve with reasonable effort.

**4.1.4.3.0-1.0-1.0-1**  (ID: `11220870`)

IF support 5G-L2-HI shall support Optimized F1-U interface

If the PDCP and RLC entities of 5G-L2-HI are co-located, the data communication between PDCP and RLC shall be an internal event and will not go through TRSW. 

Note:

  1. There is no F1-U interface in Classical BTS, but to keep the processing flow similar between cloud and classical solutions, an “Optimized F1-U” internal interface shall be implemented for data transfer between PDCP and RLC.



In this solution, the following points shall be applied.

· Optimized F1-U shall not use GTP and Transport.

· Optimized F1-U shall use F1-U headers and TEID for the internal interface.

  1. Optimized F1-U shall keep the similar processing flow for cloud and classical product so that some code reuse is possible in the future.

  2. If the remote IP address is 0.0.0.0 then 5G-L2-HI knows that the PDCP and RLC are co-located.




Currently PDCP and RLC are co-located for classical BTS only.

**4.1.5.0-2.0-2**  (ID: `11220986`)

Procedures TCP MSS Clamping

---

**4.1.5.0-2.0-2.0-1**  (ID: `11220993`)

Procedures The PDCP TCP MSS (TCP Maximum Segment Size) Clamping mechanism is used to avoid a fragmentation and to improves the performance. It’s applied for DL and UL direction and is set up for each new SA and NSA bearer.

The TCP MSS clamping replace the original MSS in the TCP SYN or TCP SYN/ACK packet when the newly calculated MSS is lower than the original MSS.

If 5G-L2-HI receives UdpMaxDatagramSize IE in HiUserCu_BearerSetupReq message (5G_L2_IF_3395_replaced([10989245](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WkkUvu6Ee-AqvopbP1qhQ))) and rdTcpMssClampingEnabled is set to true (5G_L2_9159_replaced([11232183](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AhW4PvTEe-AqvopbP1qhQ))), then 5G-L2-HI calculates TCP MSS value as below:

  * MSSvalue = udpMaxDatagramSize - deductions



Deductions:

  * IP Header Length - 20 bytes for IPv4, 40 bytes for IPv6
  * TCP Header Length - 20 bytes
  * GTP-U Header Length - 20 bytes
  * PDCP Header length - 7 bytes (3 bytes for 18bit SN + 4 bytes for MAC-I), IF Integrity protection is configured.
  * PDCP Header length - 3 bytes (3 bytes for 18bit SN ), IF Integrity protection is configured.
  * SDAP Header Length - 1 byte (if SDAP header is configured)



The detailed calculation of MSS value is available in 5G_UP_ALG_13094_replaced([11224828](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJ-cPvSEe-AqvopbP1qhQ)).

The detailed TCP MSS Clamping procedure is describe in 5G_UP_ALG_13096_replaced([11224870](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJ-dvvSEe-AqvopbP1qhQ)).

Notes:

  1. The TCP MSS Clamping is applied even if integrity protection is disabled, then in deduction the value of PDCP Header length is different (5G_UP_ALG_13094_replaced([11224828](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJ-cPvSEe-AqvopbP1qhQ))).

  2. For SA bearer, the TCP MSS Clamping is applied even if data PDU is without SDAP header, then there is no additional deduction for it. (5G_UP_ALG_13094_replaced([11224828](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJ-cPvSEe-AqvopbP1qhQ))).




Exceptions:

  1. This procedure is not applied for IPv4 packets with header options and IPv6 packets with extension headers present (5G_UP_ALG_13803_replaced([11224845](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJ-cvvSEe-AqvopbP1qhQ))).

  2. This procedure is not applied when CPU overload actions have been initiated (5G_UP_ALG_13514_replaced([11229095](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kymDVvvSEe-AqvopbP1qhQ))) and rdCpuNonCriticalLoadThreshold is exceeded (5G_UP_ALG_13433_replaced([11229157](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kymDX_vSEe-AqvopbP1qhQ))).

  3. This procedure is not applied when the MSS value is not exist in the IP packets with options.

**4.1.5.0-2.0-3.0-1**  (ID: `11221005`)

Procedures PDCP PDU duplicate transmission

---

**4.1.5.0-2.0-3.0-1.0-2**  (ID: `11221018`)

Procedures Feature 5GC001354-B introduces support of PDCP PDU duplicate transmission for NSA 3x Bearer.

If data is not sent over LTE leg for a long time, it may lead to TA timer expiry in LTE MeNB and eventually may lead to SgNB release. The scope was to prevent TA timer expiry at MeNB for NSA 3x DRB (with dlDataSplitMode = [Before CB008490-DIF] dlOverF1UXxU [CB008490-DIF] overF1UXxU [End CB008490-DIF]) and mentioned TA timer expiry can be avoided by sending duplicate PDCP PDU periodically towards LTE MeNB over X2-U link.

Assumptions:

  1. Duplicate PDCP PDU should be sent over X2-U link periodically when following conditions are satisfied:  
X2-U RL outage is not indicated AND  
Data transmission is not paused over X2-U AND  
Data transmission is not suspended over X2-U AND  
eNB indicates no data for transmission at RLC (detected by X2-U DDDS)

  2. Sending of duplicate PDCP PDU should be allowed using flow control mechanism in the following cases as a special handling for duplicate PDU transmission:  
\- X2-U link is not available based on throughput (5G_UP_ALG_5427_replaced([11218845](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxmk3fvSEe-AqvopbP1qhQ)))  
\- Data split is disabled due to CPU load based flow control action (5G_UP_ALG_5430_replaced([11229191](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kymqZPvSEe-AqvopbP1qhQ)))

  3. Handling of DDDS received over X2-U and F1-U consisting of the same “Highest transmitted NR PDCP Sequence Number” and/or same “Highest successfully delivered NR PDCP Sequence Number” due to duplicate transmission is required.

  4. The periodicity of sending duplicate PDCP PDU should be less than TA timer value. Periodicity for sending duplicate PDCP PDU shall be a configurable parameter and there shall be an option to disable sending of duplicate PDCP PDU.

  5. Data on flight for X2-U link should be incremented during the duplicate PDCP PDU transmission.

  6. Retransmission of duplicated PDU will not be supported during intra gNB mobility.

  7. If the duplicate PDCP PDU sent over X2-U is lost:  
\- Feedback may never be received and the call may be released by eNB in worst case.  
\- Duplicate PDCP PDU should be sent even if feedback is not received to avoid call release by eNB.

  8. Handling of “Highest successfully delivered NR PDCP Sequence Number” for duplicate PDCP PDU




\- PDCP PDU buffer should be released after receiving the first ACK over either X2-U or F1-U interface.

\- Special handling is required to mark the PDCP PDU as duplicated over X2-U link.

\- Data on flight should be updated for the other interface for which ACK is not yet received.

\- PDCP PDU discard indication may be sent over the other interface to release PDCP PDU (i.e. RLC SDU) buffered at RLC if it is not already transmitted.

  1. Support sending of discard indication when ACK is received over the other link. (NOTE: Discarding of PDCP PDU sent with Retransmission flag = 1 will not be supported). PDCP PDU discarding is done only at L2-HI RLC. (Rationale: Discarding at L2-LO RLC cannot be done since RLC SNs are already assigned and discarding PDUs with SN will lead to SN gaps and eventually RLC retransmission failures).

**4.1.5.0-2.0-3.0-2**  (ID: `11221025`)

Procedures Fast retransmission of lost PDCP PDUs

---

**4.1.5.0-2.0-3.0-2.0-1**  (ID: `11221032`)

Procedures Feature 5GC001354-A introduces support of fast re-transmission of lost PDCP PDUs.

The intention was to Support fast retransmission of lost PDCP PDUs to improve the end to end throughput.

In this subfeature there are introduced:

  * new fields in the DDDS frame:



Retransmitted NR PDCP SN Ind

Delivered Retransmitted NR PDCP SN Ind

Retransmitted NR PDCP Sequence Number

Successfully delivered retransmitted NR PDCP Sequence Number

  * new fields in the DUD frame:



Retransmission flag

Assumptions for gNB-DU:

  1. At gNB-DU RLC, priority of PDU transmission should be in following order starting from highest priority:  
1) RLC control PDUs,  
2) RLC retransmission PDUs,  
3) PDCP retransmission PDUs (new RLC SDUs)  
4) PDCP new PDUs (new RLC SDUs)



Assumptions for gNB-CU:

  1. The gNB-DU/eNB shall report in DDDS with PDCP PDU that are lost over F1-U/X2-U.

  2. Relevant flow control parameters (eg: Data on flight, data outflow to MAC etc., ) shall be updated after retransmission of PDCP PDUs.

  3. The gNB-CU shall retransmit these PDCP PDU with retransmission bit in the NR-U header set to '1' with following options:  
\- Retransmission over the same link where lost packet report was received  
\- Retransmission over F1-U or X2-U based on the flow control decision




Note that NR-U header set to '1' is only for sent packets over the interface for which lastSentPdcpSN>PdcpSn of current sending packet. In the other case there is no setting of NR-U header.

  1. Intra gNB Mobility scenario for NSA 3x DRB - Data recovery case: Inter DU/Intra DU mobility without MeNB change - gNB-CU retransmits over X2-U, the PDCP PDUs which were not acked over the source F1-U tunnel. The retransmission bit in the NR-U header shall be set to TRUE. Compared to legacy, where there were retransmissions after F1-U resume and packets were sent only over F1-U, here retransmissions over X2-U are already happening during suspend F1-U.

Procedures PDCP t-Reordering timer expiry prevention

---

**4.1.5.0-2.0-3.0-3.0-1**  (ID: `11221045`)

Procedures Feature 5GC001354-C introduces support of prevent PDCP t-Reordering timer expiry at UE for NSA 3x DRB.

For NSA 3x Bearer, the PDCP PDU is sent over F1-U and X2-U. In case one of the link becomes degraded, PDCP PDU sent over that link is delayed. This can lead to PDCP reordering timer expiry at UE which leads to data loss. To avoid such cases, following procedure shall be performed:

  * Delay Threshold 1:

  * It is used to monitor the transmission delay of the PDCP PDU. The transmission delay may occur due to the delay or loss of the PDCP PDU over the F1-U/X2-U interfaces which may lead to t-Reordering timer expiry at the UE.  
2\. Start Threshold 1 monitoring for the PDCP PDU with SN x based on the NIDD parameter NRBTS/NRDRB_PDCP.pdcpDelayThresh1TrigType value:  
0 - transmittedPdcpSnTrig: Delay Threshold 1 monitoring is triggered based on the Highest Transmitted PDCP SN in DDDS received over the other interface link.  
1 - pdcpPduSendTrig: Delay Threshold 1 monitoring is triggered after sending the PDCP PDU.

  * gNB-CU shall retransmit the PDCP PDU over the other link if the Threshold 1 is exceeded. (NOTE: PDCP PDU retransmission due to exceeding of threshold 1 is done only once over the other link). 

  * Threshold 1 monitoring shall be stopped upon receiving DDDS indicating that the PDCP PDU was transmitted or retransmitted.

  * Delay Threshold 2:

  * It is used to monitor the delivery delay of the PDCP PDU. The delivery delay may occur due to air link degradation/ RLC retransmissions leading to t-Reordering timer expiry at the UE. 

  * Start Threshold 2 monitoring after the PDCP PDU is indicated as transmitted via DDDS message. 
  * gNB-CU shall retransmit the PDCP PDU over the other link if Threshold 2 is exceeded. (NOTE: PDCP PDU retransmission due to retransmission timer expiry is done only once over the other link). 
  * Threshold 2 monitoring shall be stopped upon receiving DDDS indicating that the PDCP PDU was successfully delivered to UE.



Assumptions:

  1. DL flush discard procedure shall be supported for specific events like RLO, uni-directional link failure etc

  2. If the total number of PDCP PDUs retransmitted but not yet acknowledged for the DRB exceeds pdcpMaxNumRetxPdu (NIDD parameter), PDCP shall not perform retransmission of PDCP PDUs until acknowledgment is obtained for the retransmitted PDUs that has already been transmitted.

  3. If the PDCP PDU is indicated as successfully delivered to UE, 

  4. If the PDCP PDU was also sent over the other interface but not yet indicated as transmitted via DDDS, send discard indication over the other interface. PDCP PDU discarding is done only at L2-HI RLC. (Rationale: Discarding at L2-LO RLC cannot be done since RLC SNs are already assigned and discarding PDUs with SN will lead to SN gaps and eventually RLC retransmission failures).

  5. Delay Threshold 1 should be much lesser than PDCP t-Reordering timer

  6. < 50% at max, it should be set ~40% of NRDRB_PDCP.tReorderingUl

  7. Optimal value needs to be found from simulation.

  8. Delay Threshold 2 Should be much lesser than PDCP t-Reordering timer but greater than RLC status prohibit timer/SR periodicity considering the delay to receive the RLC Status PDU.

  9. 70% - 80% max based on the exact value of the NRDRB_PDCP.tReorderingUl

  10. Optimal value needs to be found from simulation.

**4.1.5.0-2.0-5.0-1.0-1**  (ID: `11221114`)

Procedures DL UE AMBR stands for Downlink UE Aggregated Maximum Bit Rate and means that the gNB enforces maximum downlink bit rate as sum of all non-GBR DRBs established to the UE [PR797600] except 5QI5 established without AMBR params [End PR797600].

All the non-GBR DRBs [PR797600] with AMBR activated (AmbrParamsStatic IE is present in HiUserCu_BearerSetupReq) [End PR797600] of the UE should be allocated to the same L2-HI instance and one bearer group.

When non-GBR DBR is setup (5G_L1L2_FLOW_5534_replaced([11185606](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Be797fvREe-AqvopbP1qhQ))) OR released (5G_L1L2_FLOW_5539_replaced([11185676](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Be8k_vvREe-AqvopbP1qhQ))) OR DL UE AMBR is modified, the DL UE AMBR limit provided to gNB is split statically for each PDU session based on the scheduling weights ratio of DRBs of each PDU session. Then PDU Session limit is distributed for each non-GBR DRB based on scheduling weights of DBRs according to PDU Session limit.

The dynamic split of the UE-AMBR limit is performed periodically based on dlAmbrMonitorPeriod.

After feature is activated, 5G-L2-HI shall enforce the downlink bit rate of all non-GBR DRBs established per UE based on

 ambrParamsStatic IE (5G_L2_IF_6166_replaced([10987786](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WY-Lfu6Ee-AqvopbP1qhQ))) provided in HiUserCu_BearerSetupReq (5G_L2_IF_1020_replaced([10987413](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WWh6Pu6Ee-AqvopbP1qhQ))) AND

 ambrUeParamsModifiable (5G_L2_IF_6218_replaced([10988219](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WcohPu6Ee-AqvopbP1qhQ))) provided in HiUserCu_BearerModifyReq (5G_L2_IF_1033_replaced([10988005](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WazWvu6Ee-AqvopbP1qhQ)))

The 5G-L2-HI shall limit the sum of amount of DL data sent towards the gNB-DU (RLC) via F1-U for the UE to be within the DL UE AMBR limit as computed and provided by C-plane in HiUserCu_BearerModifyReq with cause AmbrUpdate (5G_L2_17993_replaced([11225069](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_0zUPvTEe-AqvopbP1qhQ))) with dlUeAmbrLimit present (5G_L2_IF_6218_replaced([10988219](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WcohPu6Ee-AqvopbP1qhQ))).

Amount of DL data sent by each non-GBR DRB shall be below or equal to the limit assigned to each DRB, the sum of DL data sent by all non-GBR DRBs of a PDU Session sent over F1-U shall be below or equal the PDU session limit assigned to this PDU Session and over all DL data sent across multiple PDU sessions shall be below or equal to the DL UE AMBR limit. AMBR limit is calculated before packet is sent and does not take the current packet size into account so AMBR limit can be crossed.

Detailed distribution of DL UE AMBR per each PDU Session is under "Dynamic distribution of DL UE AMBR to the PDU Sessions" (5G_L2_18438_replaced([11228306](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AHHN_vTEe-AqvopbP1qhQ))).

Notes:

  1. [PR797600] Validation of 5QI5 with NRBTS.NRDRB.resType = nonGBR_Signaling is in the Cplane responsibility. [End PR797600]



Exceptions:  
1\. GBR DRBs of the UE shall not be considered for UE AMBR enforcement algorithm.  
2\. For Intra-gNB NR-DC, DL UE AMBR limitation is taken care for the split bearer. In this case PDCP is a single entity for MCG and SCG and DL AMBR limitation is allocated at PDCP level, the limit for a single non-GBR bearer (for MCG and SCG) is taken as a share of total UE AMBR limit.  
3\. For Inter-gNB NR-DC AMBR is applied only at the non-GBR bearers located on MCG of the UE since PDCP entity is different for MCG and SCG. In this situation the overall throughput for all the non-GBR bearers of the UE (the sum of MCG and SCG) can be higher than UE AMBR limit.

  1. DL UE AMBR is applied only for:

  2. typeOfBearer = SA (5G_L2_IF_1020_replaced([10987413](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WWh6Pu6Ee-AqvopbP1qhQ))) AND

**4.1.5.0-4.0-1.0-1.0-2.0-2**  (ID: `11221230`)

Procedures Bearer Inactivity detection shall be started first time directly after the bearer setup followed by the first bearer modification.

The subsequent detections of bearer Inactivity are started directly after handling of UL or DL data packet i.e. when buffers are emptied due to sending or discarding.

For both cases mentioned above, the bearer is in ACTIVE state.

A bearer keeps the ACTIVE state until the bearer Inactivity (of both UL and DL link) will be detected.

Starting the Inactivity timer for the first round of Inactivity detection

At the beginning of the first round of Inactivity detection i.e. upon receiving HiUserCu_BearerModifyReq message (5G_UP_ALG_5292_replaced([11217645](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxf3JvvSEe-AqvopbP1qhQ))) which transits bearer to ACTIVE state, 5G-L2-HI shall start the Inactivity timer enlarged by the value of rdBearerInitializationDelay (defined in 5G_L2_2735_replaced([11232111](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3Agv0vvTEe-AqvopbP1qhQ))).

The Inactivity timer is defined by bearerInactivityTimer (NIDD parameter) or rdBearerMinimumInactivityTimer (see 5G_L2_2162_replaced([11221191](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_c_6_vTEe-AqvopbP1qhQ)) Note 2).

Starting the Inactivity timer for the second and next Inactivity detection rounds

At the beginning of 2nd and all the subsequent Inactivity detection rounds, 5G-L2-HI shall start the Inactivity timer just after a data packet was handled and if the buffer is empty.

Bearer Inactivity detection in UL and DL links

The Inactivity of the bearer’s UL or DL link is detected if the Inactivity Timer expired.

Note, that for DL link, if the buffer becomes empty, the Inactivity Timer is started. Upon expiration of the timer, the INACTIVE state is set for this bearer.

For UL link, if the SDU queue becomes empty, the Inactivity Timer is started. Upon the timer’s expiration, the INACTIVE state is set for this bearer.

Bearer Inactivity Indication to C-Plane

If both links UL and DL have the INACTIVE state, the 5G-L2-HI shall send the HiUserCu_BearerInactivityInd message to C-Plane (CP-NRT) [04155861] and may receive HiUserCu_BearerDeteleReq. In the meantime, L2-Hi may detect bearer activity and informs C-plane about it using HiUserCu_BearerActivityInd. If L2-HI receives HiUserCu_BearerDeteleReq for this bearer, L2-HI replies to C-plane HiUserCu_BearerDeteleResp and may fill inSession IE to True as long as the conditions for "in session" are met in relevant perion of time (this race condition is due to the way in which C-plane handles comunnication with L2-HI) [End 04155861]

There shall be no Inactivity indication message repetition sent to C-Plane until Activity is detected and reported (see 5G_L2_2169_replaced([11221263](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_dm-fvTEe-AqvopbP1qhQ))).

**4.1.5.0-5.0-1.0-6.0-2**  (ID: `11221772`)

Procedures Delay based Flow Control Split

---

**4.1.5.0-5.0-1.0-6.0-2.0-3**  (ID: `11221792`)

Procedures Filtering out available 5G-L2-LO instances based on PDU delivery delay and 5G-L2-LO buffer flow is shown in 5G_L2_2600_replaced([11227623](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AC1zPvTEe-AqvopbP1qhQ)).

1\. Selection of the fastest data path

The PDUs split on Data Paths is based on delivery delays on these paths. The fastest data path is preferable. The algorithm checks all the available candidate paths in search of the one with the shortest delivery delay.

2\. Example implementation

For the path to be selected, the flow control algorithm:

\- checks the maximum sending capacity and the minimum buffer capacity of the candidate path

\- calculates its estimated PDU delivery delay

\- compares calculated delay with the stored minimum delay

Thus, the algorithm accepts only the shortest delay, so the fastest data path can be selected.

2.1. Checking the conditions allowing PDUs sending

As a prerequisite, the flow control algorithm checks (and updates) the maximum sending capacity and the minimum buffer capacity.

Sending PDUs to the path is allowed if both of the following conditions are met:

\- maxPduSizeAllowed > 0 (maximum sending capacity)

\- minBufferCapacity > 0 (minimum buffer capacity)

In other words, L2-HI keeps sending out PDUs till maximum sending capacity or minimum buffer capacity is bigger than zero. After sending a RLC PDU to L2-LO the mentioned parameters (i.e. maxPduSizeAllowed and minBufferCapacity) are decreased by the PDU size.

2.2. Calculating estimated delivery delay

The estimated PDU delivery delay for the path is the simple sum:

\- estimatedPduDeliveryDelay = currentPduDelay + packetDeliveryTime

The packet delivery time depends on the PDU size and can be obtained by calculating the following ratio:

\- packetDeliveryTime = pduSize / eTh

where:

\- pduSize - size of RLC SDU + size of RLC header

\- eTh - estimated throughput of the data path (reported from secondary 5G-L2-LO instance[PR724241], refer to 5G_UP_ALG_5736_replaced([11221148](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx0ANvvSEe-AqvopbP1qhQ))[End PR724241])

2.3. Comparing calculated and stored delays

The newly calculated delay (estimatedPduDeliveryDelay) for the candidate path is compared to the previous stored minimum delay and if estimated PDU delivery delay is shorter this path is selected (and minimum delay is now updated to the shorter delay). Otherwise the next candidate path is processed.

[CNI-80040]In case of RLC PDU which was selected to be retransmitted due to Proactive RLC Retransmission for Carrier aggregation (refer 5G_L2_12915_replaced([11227736](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3ADc4PvTEe-AqvopbP1qhQ))) additional caPduReSendingDelayMargin is added for the given path delay.[End CNI-80040]

**4.1.5.0-5.0-1.0-6.0-3.0-1**  (ID: `11221805`)

Procedures The main goal of this algorithm is to check estimatedSchedulingDelay per each active 5G-L2-LO instance and compare it to t-reassemblyIBCA value, which is threshold NIDD value delivered by C-Plane and by default set to 40ms.

Filtering out available 5G-L2-LO instances based on scheduling delay is shown in 5G_L2_2844_replaced([11227673](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3ADc1vvTEe-AqvopbP1qhQ)).

There are few possibilities which takes part:

1\. If all available legs have estimatedSchedulingDelay lesser than threshold:

All instances are mark as available for data sending.

2\. If all available legs have estimatedSchedulingDelay greater than threshold:

Calculate how much data we could send on each leg within one second, which actually calculates as follows:

dataBytesPerSec = (rlcPduDrainRate * 1000) / estimatedSchedulingDelay

*rlcPduDrainRate indicates how much data was submitted to MAC in last valid DDDs report period.

5G-L2-LO instance with highest dataBytesPerSec should be marked as available, and rest of them are marked as unavailable.

3\. If one of the 5G-L2-LO instances has estimatedSchedulingDelay lesser than threshold and at least one L2LO instance has estimatedSchedulingDelay above threshold:

Calculate how much data could be sent on each leg within one second (based on equation from point 2) and mark one as available with highest dataBytesPerSec.

4\. If more than one L2LO instance has estimatedSchedulingDelay lesser than threshold:

5G-L2-LO instances above threshold shall be marked as unavailable and those within threshold are marked as available.

If any of 5G-L2-LO instance is set as unavailable due to above points, it is held in this state for time equal rdSchedulingDelayDeactiveTime (usually set to 250ms).

IF actOptimizedFlowControl is set to nonOptimized:

After that time 5G-L2-LO instance is set as available and one PDU is forced to be sent on that instance to:

\- initialize schedulingDelay to 0,

\- update latest eTh.

ELSE IF actOptimizedFlowControl is set to probingPduDisabled:

5G-L2-HI shall rely on last received 5G-L2-LO DDDs report to:

\- calculate scheduling delay value and check if it is less than dlTReassemblyIBCA,

\- check if [Before CB008140-L] l2loBufferEmpty is set to True [CB008140-L] bufferThresholdLevlReached = BufferEmpty[End CB008140-L],

for given 5G-L2-LO instance.

If one of above conditions is fulfilled then:

\- mark related 5G-L2-LO instance as available,

\- reset timer based on rdSchedulingDelayDeactiveTime.

END

When freshly available 5G-L2-LO instance again returns estimatedSchedulingDelay above threshold, then process is repeated over and over again unless 5G-L2-HI DU buffer becomes empty, or one of the following algorithms comes into action:

\- Data Path availability due to PDU sending rate (5G_L2_6806_replaced([11221811](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_hRWPvTEe-AqvopbP1qhQ))),

  1. rdSchedulingDelayDeactiveTime is defined in 5G_UP_ALG_5902_replaced([11222985](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx-YSvvSEe-AqvopbP1qhQ)).

**4.1.5.0-5.0-1.0-6.0-5.0-2**  (ID: `12492331`)

Procedures The main purpose of this algorithm is deprioretization for data path when there are not enough data. Algorithm is used when there are multiple [Before CB009103-B] secondary [End CB009103-B] 5G-L2-LO instances and the amount of data in the RLC buffer is too small (it hapens usually when terminating a call or a low throughput call).

Using higher number of 5G-L2-LO instances for data split is not necessary to achieve the required delivery time, because it is inefficient when the target latency can be achieved by splitting the data into fewer 5G-L2-LO instances.

**Exception:**

5G-L2-LO instance(s) for which "UlBufferSplitStatus=True" was received in the last LoData_UeDataDeliveryStatusInd message (UL CA is active) are excluded from algorithm.

The main condition for activation of the Bufer Based algorithm is checking if RlcBufferSize is lower than bufferThesholdScellDePrioritize (default value is 200kB) for a duration of ScellDePrioritizeEvalPeriod (default value is 50ms).

The algorithm is based on reducing the number of secondary 5G-L2-LO instances by rejecting the path with the lowest eTh and is performed in several steps. If the number of instances is greater than 1, the algorithm performs some important calculations. It takes into account several key parameters such as:

\- BufferThesholdScellDePrioritize and ScellDePrioritizeEvalPeriod are NIDD parameters received from C-Plane,

\- eTh - The current estimated throughput which is defined in 5G_UP_ALG_5736_replaced([11221148](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx0ANvvSEe-AqvopbP1qhQ)),

\- PduDelay - current estimated PDU delay,

\- DeliveryDelayWithDataSplit - delay shown in ms calculated by dividing the total number of PDUs currently in the RLC buffer by the Sum eTh of all 5G-L2-LO instances available for sending data

\- DeliveryDelayThWithDataSplit - the NIDD parameter sent by C-Plane. It comes in the BeareModifyReq message and its values are set from 0-200 but default is 5.

\- RlcBufferSize - Total number of bytes in RLC buffer

Note: eTh and PduDelay are received in LoData_UeDataDeliveryStatusInd or DU DL DATA DELIVERY STATUS message.

If the sum of DeliveryDelayWithDataSplit and TotalPduDelay is lesser than the NIDD value of deliveryDelayThWithDataSplit multiplied by the number of 5G-L2-LO instances the algorithm checks the path with the smallest eTh and deactivates it.

This operation is repeated until all paths with the lowest eTh are eliminated. The algorithm is stopped when the above condition is not met.

Examples can be checked in [11221451](https://dn-prod.ext.net.nokia.com/rm/resources/TX_58C-wPvREe-AqvopbP1qhQ) and [11221463](https://dn-prod.ext.net.nokia.com/rm/resources/TX_5_8nUPvREe-AqvopbP1qhQ).
