Perform the steps described in 5G_L2_2360_replaced([11224400](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_wh4_vTEe-AqvopbP1qhQ)) if typeOfBearer = NSA.

  2. for the UL PDCU PDU(s) received over the new F1-U and X2-U tunnel, the previous security key shall be used for deciphering.

  3. for the UL PDCU PDU(s) received over the old F1-U and X2-U tunnel, the security key provided in the HiUserCu_BearerModify with cause = Suspend and with securityParams is filled shall be used for deciphering.

  4. If 5G-L2-HI detects loss of UL PDCP SDU and in HiUserCu_BearerSetupReq (5G_L2_IF_1020_replaced([10987413](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WWh6Pu6Ee-AqvopbP1qhQ))):




• outOfOrderDeliveryUl in is set to true, 5G-L2-HI does not store PDCP SDUs in buffer and delivers them to upper layer (5G_UP_IF_3547_replaced([11531227](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tWgQOfv1Ee-AqvopbP1qhQ))) regardless of the state of PDCP t-Reordering reordering timer (5G_L2_IF_1020_replaced([10987413](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WWh6Pu6Ee-AqvopbP1qhQ)))

• outOfOrderDeliveryUl is set to false, 5G-L2-HI stores PDCP SDUs in buffer. The number of PDCP SDUs in reordering queue shall be limited to value of rdUlPdcpReorderingElementCount (5G_L2_2746_replaced([11232300](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3Ah99vvTEe-AqvopbP1qhQ))). If new SDU would exceed this limit then 5G-L2-HI shall follow requirement 5G_UP_ALG_5917_replaced([11223864](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyD32PvSEe-AqvopbP1qhQ)).

  1. 5G-L2-HI shall perform the TCP MSS Clamping if udpMaxDatagramSize is filled in HiUserCu_BearerSetupReq (5G_L2_9145_replaced([11220986](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_bxyfvTEe-AqvopbP1qhQ))).

  2. Construct GTP-U G-PDUs from PDCP SDUs. One PDCP SDU shall be mapped to one GTP-U G-PDU. GTP-U header shall be filled as defined in 5G_UP_IF_4878_replaced([11545478](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tYqzYfv1Ee-AqvopbP1qhQ)).

  3. Send constructed GTP-U G-PDU to TRSW using TPUP/TrswData interface (5G_L2_IF_1516_replaced([10995238](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2XJzK_u6Ee-AqvopbP1qhQ))).




5G-L2-HI shall also provide the IP version, gNB source IP address, S-GW destination IP address, source UDP port, destination UDP port and the DSCP. Destination UDP port shall be set to 2152. Other information including IP version, gNB source IP address, S-GW destination IP address, source UDP port and the DSCP are obtained during bearer setup. [CB012369-A]5G-L2-HI CU shall use the latest dscp if it is received in HiUserCu_BearerModifyReq(Refer 5G_L2_IF_1026_replaced([10987931](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WaMS_u6Ee-AqvopbP1qhQ))).[End CB012369-A]

The determination whether the radio bearer is for signaling or user data shall be done based on setup parameters.

Note:

  1. PDCP in L2-HI does not support control PDU handling.



Rationale:

  1. Instead of using fixed PDCP windows size from 3GPP, the size is defined by R&D parameter RdUlPdcpReorderingElementCount (see: 5G_L2_2746_replaced([11232300](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3Ah99vvTEe-AqvopbP1qhQ))). This is due limited buffering capability. There is not enough room for full PDCP window (worst case due in-sequence-delivery functionality), even for one bearer.

  * 5G-L2-HI shall perform the TCP MSS Clamping if udpMaxDatagramSize is filled in HiUserCu_BearerSetupReq (5G_L2_9145_replaced([11220986](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_bxyfvTEe-AqvopbP1qhQ))).

  * Construct GTP-U G-PDUs from SDAP SDUs. One SDAP SDU shall be mapped to one GTP-U G-PDU. GTP-U header shall be filled as defined in 5G_UP_IF_4878_replaced([11545478](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tYqzYfv1Ee-AqvopbP1qhQ)).




5G-L2-HI shall fill the GTP-U extension header “PDU Session Container” using the format defined in the "UL PDU SESSION INFORMATION"

 RQI = 0

 QFI

o When SDAP header is present, QFI present in the SDAP header

o When SDAP header is not present, QFI associated with the DRB 

"PDU Session Container” is defined in 5G_UP_IF_5049_replaced([11547359](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tY75J_v1Ee-AqvopbP1qhQ)).

"UL PDU SESSION INFORMATION" format is defined in 5G_UP_IF_4293_replaced([11539223](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tXoRl_v1Ee-AqvopbP1qhQ)).

  2. 5G-L2-HI shall also provide the IP version, source IP address, destination IP address, source UDP port, destination UDP port and the DSCP.

  3. DSCP is derived based on the QFI using the "QFI to DSCP mapping table" . Incase the QFI is not present in the QFI to DSCP mapping table, DSCP shall be set based on the "Default DSCP".
  4. IP version, source IP address, destination IP address, source UDP port, destination UDP port, QFI to DSCP mapping table and Default DSCP are provided during PDU Session setup

  5. Send constructed GTP-U G-PDU to TRSW with higher priority value (PCP = 2) for 5QI-5 [Before CB008118-A]and 5QI-1[CB008118-A], 5QI-69 and GBR [End CB008118-A] bearer based on new API of em_send_with_priority , but keep default priority value (PCP = 1) for the other bearers.




Rationale:

  1. Instead of using fixed PDCP windows size from 3GPP, the size is defined by R&D parameter. This is due limited buffering capability. There is not enough buffer for full PDCP window (worst case due in-sequence-delivery functionality), even for one bearer.

  2. In UL, when UE does not find mapping for a QFI, it will use default DRB to send the PDU. From gNB pov to be able to provide the QFI to UPF, it needs to get the SDAP Header from the UE on the default DRB. Hence SDAP Header in the UL is needed for default DRB.

  3. For NG-U interface, the DSCP value is based on the QFI present in the received PDU. QFI to DSCP mapping is provided by CPlane during the PDU session setup.

  4. PDU Session Setup is performed after Bearer Setup.




Exceptions:

Exc.3a. If PDU Session (i.e. SDAP protocol) configuration was not received yet, then L2-HI shall discard the SDAP SDU and log an error.

OR

### 4.4.2.0-2 Data processing Handling of UL AM RLC PDU from 5G-L2-LO (ID: 11224414)

**4.4.2.0-2.0-3**  (ID: `11224432`)

Data processing When the AMD PDUs or byte segments of AMD PDUs are received from 5G-L2-LO, 5G-L2-HI shall 

  * Discard received AMD PDUs or byte segments of AMD PDUs if they fall outside of the reassembly window. For definition of reassembly window, see 5G_UP_IF_3260_replaced([11528915](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tWDkSvv1Ee-AqvopbP1qhQ))
  * Check if the received AMD PDUs or byte segments of AMD PDUs are already received then discard the duplicated AMD PDUs or byte segments of AMD PDUs.
  * If the AMD PDU or byte segments of the AMD PDUs are not discarded in the above steps, then
  * If the byte segments of the AMD PDU is received and the corresponding SN was not received before and total number of RLC PDU waiting for re-assembly > rdUlRlcReassemblyContextCount (for DRBs) discard the received byte segments.
  * else place the received RLC PDU in the receive buffer and follow the actions in 5G_UP_IF_3270_replaced([11529001](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tWEyZfv1Ee-AqvopbP1qhQ)) [CB013684-E]and update the PM counters related to the number of successfully received RLC SDUs as mentioned in 5G_L2_21724_replaced([11224480](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_xI8PvTEe-AqvopbP1qhQ)) [End CB013684-E]

  * The 5G-L2-HI shall detect the loss of the AMD PDUs at the lower layer and request retransmissions to its peer AM RLC entity. [CB013684-E]5G-L2-HI DU shall update the PM counters related to the requested retransmission of the UL SDUs as mentioned in 5G_L2_21749_replaced([11224492](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_xI8vvTEe-AqvopbP1qhQ)) [End CB013684-E]




See 5G_UP_IF_3273_replaced([11529038](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tWFZcPv1Ee-AqvopbP1qhQ)) for flow chart of the process.

Rationale:

In worst case, the number of RLC PDU waiting for re-assembly is based on the AM_Window_Size. Since the value is quite large when compared to the buffering capability, we need to limit the number of RLC PDU waiting for re-assembly. The value of rdUlRlcReassemblyContextCount shall be defined based on buffering capability.

Note : Discarding of RLC PDU segment inside the receive window due to buffer limitation will cause that in the RLC STATUS PDU, these discarded PDU are not marked as NACKED. This causes that the UE RLC TX window does not move forward causing UE RLC Tx window to stall.

**4.4.3.1.0-3.0-5**  (ID: `11224632`)

PDCP HFN Desynchronization Detection for non-ciphered PDCP PDUs

---

**4.4.3.1.0-3.0-5.0-2**  (ID: `12678900`)

PDCP  On receipt of the UL PDCP PDU when ciphering is disabled, 5G-L2-HI (CU) shall validate received UL PDCP PDU SN by checking if the received PDU’s SN are in sequence with respect to previous PDU received.

 If the PDPC SN difference between previous PDCP SN and the current PDCP SN is greater than or equal to half of “SN space” then 5G-L2-HI (CU) shall increment HFN_SN_DESYNC_DETECT_DRB counter and send HiUserCu_BearerErrorInd with cause "HfnDesyncDetected" to C-plane.

**"half of SN space" Computation:**

5G-L2-HI (CU) shall compute "half of SN space" using configured HiUserCu_BearerSetupReq.PdcpParams.snFieldLength.

"half of SN space" = 2 (HiUserCu_BearerSetupReq.PdcpParams.snFieldLength - 1)

For example, if the configured value is 18 bit, and if the PDPC SN difference between previous PDCP SN and the current PDCP SN is greater than or equal to 2 (18-1) = 2 17 = 131072, then HFN Desynchronization is detected.

**Note:**

1\. This method is applicable for both VoNR and non-VoNR PDCP PDUs.

2\. In this method, legacy counters M55320C07001 ... 20, M55320C08001 ... 20[CB010980-D], M55352C00017, M55352C00018 [End CB010980-D] are not updated.

3\. This procedure is applicable for EN-DC, NR-DC as well.

4\. 5G-L2-HI (CU) considers that ciphering is disabled for DRB when HiUserCu_BearerSetupReq.PdcpParams.SecurityParams.encryptionAlgorithm = NULL.
