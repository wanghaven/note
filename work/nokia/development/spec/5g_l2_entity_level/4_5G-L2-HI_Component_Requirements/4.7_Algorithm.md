**4.7.1.0-1.0-1.0-3**  (ID: `12826791`)

Flow Control on F1-U and X2-U or Xn-U DDDS shall be generated according to [11222264](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx6G3_vSEe-AqvopbP1qhQ) (Refer Note 1). This subchapter aims to provide entity level algorithms for handling F1-U Downlink Data Delivery Status Indication message parameters.

**Note:**

1) For UM bearers where flow control is disabled, DDDS shall be generated only when rdDuAllowPartialFlowControl ([11232037](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AgIw_vTEe-AqvopbP1qhQ)) is set as True according to [11222311](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx6t5PvSEe-AqvopbP1qhQ). rdDuAllowPartialFlowControl is an R&D parameter used for internal testing purpose. This shall have same value as rdCuAllowPartialFlowControl defined at CU. The default value shall be set as TRUE.

2) Periodic sending of DDDS indication shall be done according to [11222311](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx6t5PvSEe-AqvopbP1qhQ). In case actDdddReduction is set to TRUE via HiUserDu_BearerSetupReq ([10989936](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2Wo1w_u6Ee-AqvopbP1qhQ)), both L2Hi DU (where RLC SDUs + unacked PDUs for AM mode are buffered) and L2Lo (where RLC PDUs are buffered) buffers shall be empty to trigger DDDS skipping due to whole RLC buffer (L2Hi DU and L2Lo together) empty. Buffer empty at L2Lo is indicated from L2Lo via internal DDDS and is determined based on [Before CB015504-A] **LoData_UeDataDeliveryStatusInd:drbStatusInfo:bufferThresholdLevlReached (5G_L2_IF_2375_replaced([10986071](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WMw4Pu6Ee-AqvopbP1qhQ))) [CB015504-A] **LoData_UeDataDeliveryStatusInd:drbStatusInfo:drbInfoDetails:bufferThresholdLevlReached ([12825071](https://dn-prod.ext.net.nokia.com/rm/resources/TX_CIXo8as3EfCyQasEtWkQ4A)) [End CB015504-A] set to 0 on all available DL data paths for the DRB (when CA is configured).****

a. If yes, go to point 2

b. If no, go to point 3

  1. Check if incoming highest transmitted PDCP SN fits into the range (previous highest transmitted PDCP SN, highest sent PDCP SN]:



a. If yes, go to point 3

b. If no, go to point 4.

  1. Incoming highest transmitted PDCP SN should be stored as highest transmitted PDCP SN for given L2-LO instance.

  2. End the procedure.

**4.7.1.0-2.0-5.0-2**  (ID: `11226997`)

Flow Control on F1-U and X2-U or Xn-U DDDS validation for ReTx IEs in case of NR-U SN wraparound and PDCP SN wraparound on retransmission IEs

---

**4.7.1.0-2.0-5.0-2.0-1**  (ID: `11227003`)

Flow Control on F1-U and X2-U or Xn-U 5G-L2-Hi CU shall use the following parameters:

retransmittedNruSnPrev (in case of F1-U) or retransmittedNruSnPrevXx (in case of X2-U) AND

previous Retransmitted NR PDCP SN AND

previous Successfully delivered retransmitted NR PDCP SN parameters AND

check if there is any NR-U retransmission pending on specific path,

to decide if needs to validate or completely ignore what is inside retransmission IEs in DDDS for the following wraparound cases:

 NR-U SN wraparound - that means after successful mapping of PDCP SN to NR-U SN when validating DDDS where previous values can be bigger than current one and as result invalidate DDDS.

 PDCP SN wraparound on retransmission IEs with same SNs as previous HFN – that had to be mapped into NR-U SN.

To protect against unnecessary retx IEs handling in case of NR-U SN wraparound or PDCP SN wraparound on retransmission IEs with same SNs as previous HFN when previous Retransmitted NR PDCP SN is another value than INVALID, 5G-L2-Hi CU shall

 reset retransmission SNs stored for validation purpose: retransmittedNruSnPrev/retransmittedNruSnPrevXx and previous Retransmitted NR PDCP Sequence Number and previous Successfully delivered retransmitted NR PDCP Sequence Number IEs from DDDS if all retransmitted PDUs are acknowledged.

 ignore retransmission IEs from subsequent DDDS until the next retransmission is done over the interface if retransmittedNruSnPrev/ retransmittedNruSnPrevXx (5G_UP_ALG_5448_replaced([11218917](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxnL6PvSEe-AqvopbP1qhQ)), 5G_UP_ALG_5451_replaced([11218932](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxnL6vvSEe-AqvopbP1qhQ)),5G_UP_ALG_5480_replaced([11219139](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxoaCvvSEe-AqvopbP1qhQ)) and 5G_UP_ALG_5544_replaced([11219548](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxq2S_vSEe-AqvopbP1qhQ))) AND previous Retransmitted NR PDCP SN AND previous Successfully delivered retransmitted NR PDCP SN are all INVALID. All of them are invalid when no retransmission ever happened OR retransmissions happened, but DDDS marked them as retransmitted.

Detailed handling in case of both wraparounds can be found under the following link: <https://nokia.sharepoint.com/:f:/r/sites/5Gdoc/AS_Tech_Spec/Drawings%20and%20diagrams%20for%205G/User%20Plane>

Notes:

1\. In case of 5G-L2-HI CU ignore retransmission IEs, rest of DDDS should still be processed as usual.

2\. retransmittedNruSnPrev/ retransmittedNruSnPrevXx represent the oldest retransmitted NR-U SN, on which L2Hi has not yet received Retransmitted NR PDCP Sequence Number (DDDS has not arrived).

**4.7.1.0-2.0-6**  (ID: `11227010`)

Flow Control on F1-U and X2-U or Xn-U Protection logic to delete dangling packets marked for retransmission due to threshold

---

**4.7.1.0-2.0-6.0-1**  (ID: `11227015`)

Flow Control on F1-U and X2-U or Xn-U If the following scenario would happen:

a) PDU is sent over F1-U/X2-U and reaches eNB after 60ms (rare case of large delay over the link)

b) Theshold1 is exceeded (threshold delay is lower value than it takes for lost report to come from other path) and PDU is marked for retransmission over X2-U/F1-U (is marked for retx on the other link than was in the previous point) due to delay threshold monitoring 

c) Later, the lost packet report is received over F1-U/X2-U (from the link was sent first) but will be ignored since PDU is already marked for retransmission in step b

d) The PDU retransmission is delayed more than pdcpRetxPduSendMaxDelay, the marking for retransmission will be removed for the PDU

e) Now the PDU is neither sent over X2-U nor will be retransmitted over F1-U so this PDU is never released, 

so, then 5G-L2-Hi shall delete such dangling packets.

### 4.7.1.0-4 Flow Control on F1-U and X2-U or Xn-U Random Selection Of DRB when dl data split is disabled due to cpu overload handling (ID: 11227067)

**4.7.1.0-4.0-1**  (ID: `11227074`)

Flow Control on F1-U and X2-U or Xn-U The last digit of drb index is in range {0-9}, 10 numbers in total.

Select randomly a certain number of digit according to maxPercentageDrbOverF1UOvld.

If the last digit of the drb index of a certain drb is one of randomly selected digits, the F1U interface is chosed for this drb. 

Otherwise, X2U interface is chosed for this drb.

### 4.7.1.0-9 Flow Control on F1-U and X2-U or Xn-U Handling of PDCP PDU states on interfaces (ID: 11227415)

**4.7.1.0-9.0-1**  (ID: `11227421`)

Flow Control on F1-U and X2-U or Xn-U 5G-L2-Hi shall remember information about the PDCP PDU packet from each interface (context) that packet was sent on.

Definition:

Context – all packet on interface related parameters (i.e. NR-U SNs) where PDCP PDU was transmitted/retransmitted, until mobility or lost happen (i.e. reset of the entire link or reduction of DoF on a given link).

Rationale:

Duplicated, transmitted and retransmitted states are considered as single states that are not related to any interface. Apart from the state, each PDCP PDU packet can have context – one on F1-U interface and the second on X2-U interface and is referring to all parameters connected with the interface. The PDCP PDU packet can be in only one state, but it can have more than one context at a time. Some procedures need to take packets transmitted through specific interface, e.g., F1-U context or X2-U context. Context of a packet can be deleted due to packet lost or due to mobility. If the packet is retransmitted once, it will always be treated as in this state, unless mobility changes its state.

Interpretation:

 Example 1

PDCP PDU is transmitted over X2-U, then is retransmitted over F1-U due to delay.

1) After retransmission:

PDCP PDU has context over X2-U (PDCP PDU is transmitted) and as well over F1-U (PDCP PDU is retransmitted).

PDCP PDU state is retransmitted (not related to any interface).

E.g. 1: “Select PDCP PDUs which are (transmitted/retransmitted) over X2-U.”

Is the PDCP PDU considered? Yes, it is.

E.g. 2: “select PDCP PDUs which are last transmitted/retransmitted over X2-U.”

Is the PDCP PDU considered? No, it is not.

2) For mobility data recovery:

PDCP PDU keeps retransmitted state and has context over X2-U. Context F1-U is deleted.

3) For mobility PDCP reestablishment:

PDCP PDU reset state from retransmitted to transmitted when the packet is sent over F1-U/X2-U, but before both (F1-U and X2-U) contexts are deleted (refer to 5G_UP_ALG_8576_replaced([11220310](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxvH4vvSEe-AqvopbP1qhQ))).

 Example 2

1) PDCP PDU is transmitted over X2-U.

PDCP PDU has context over X2-U (packet is transmitted).

PDCP PDU state is transmitted.

2) This PDCP PDU is lost over X2-U (1st time).

There is NO context, packet is marked for retx and still transmitted.

So PDCP PDU state is transmitted.

3) This PDCP PDU is retransmitted over X2-U (1st time)

PDCP PDU has context over X2-U.

PDCP PDU state is retransmitted.

4) PDCP PDU is lost again over X2-U (2nd time).

PDCP PDU has NO context, packet is marked for retx.

PDCP PDU state is retransmitted.

5) Retransmit the PDCP PDU over F1-U - due to loss over X2-U.

PDCP PDU has NO X2-U context, has only F1-U context.

PDCP PDU state is retransmitted.

 Example 3

1) PDCP PDU is sent over F1-U.

PDCP PDU has context over F1-U.

PDCP PDU state is transmitted.

2) PDCP PDU retx over X2-U due to exceeding the thresholds (for pdcpPduRetxDelayThresh1 and pdcpPduRetxDelayThresh2 refer to 5G_UP_ALG_7509_replaced([11220415](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxvuy_vSEe-AqvopbP1qhQ)) and 5G_L2_IF_1020_replaced([10987413](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WWh6Pu6Ee-AqvopbP1qhQ))).

PDCP PDU has context over F1-U and X2-U.

PDCP PDU state is retransmitted.

3) PDCP PDU is lost over X2-U.

PDCP PDU has context over F1-U. There is no context over X2-U anymore.

PDCP PDU state is retransmitted.

In case when amountOfSentData >= maxBurstLimitPerBearerPerMsX2 during the last ms cycle AND current time - time since the last packet sent on X2-U [CB008490-DT] /Xn-U [End CB008490-DT] interface < 100 microsecond, 5G-L2-HI shall set XXAvaliable = False (see Rationale).

### 4.7.2.0-3 Flow Control on LoData Interface Handling of deltaTime (ID: 11227651)

**4.7.2.0-3.0-1**  (ID: `11227656`)

Flow Control on LoData Interface In case when the RLC buffer is empty OR when the currentPduDelay = 0,

The delta time is considered as 0 (Rationale 1).

When the RLC buffer becomes Non-Empty (refer Note), the delta time shall be computed as the time difference between - the time when the RLC buffer became Non-Empty and the current time.

[PR623521]

For the deltaTime calculation, the result of 't-t0' shall be rounded down to an integer.

[End PR623521]

t0 = startTransmissionTimestamp shall be maintained per 5G-L2-LO instance.

For each 5G-L2-LO instance modification of startTransmissionTimestamp is done when LoDataDeliveryStatus message is received and currentPduDelay != 0 and RLC SDU buffer is non-empty.

If RLC SDU buffer is empty upon reception of LoDataDeliveryStatus message – startTransmissionTimestamp is set when RLC SDU buffer becomes non-empty.

startTransmissionTimestamp is reset to 0 when:

\- currentPduDelay = 0

\- RLC SDU buffer is empty or becomes empty

Rationale:

1\. When there is no data at the 5G-L2-LO instance there is nothing to be sent out.

In such case, the delta time – which is used to offset the decay in 5G-L2-LO buffer size (i.e. due to the DDDS time interval) has no relevance.

Note:

RLC buffer Non-Empty means that there are RLC SDUs for initial transmission to be sent or RLC PDUs to be retransmitted.
