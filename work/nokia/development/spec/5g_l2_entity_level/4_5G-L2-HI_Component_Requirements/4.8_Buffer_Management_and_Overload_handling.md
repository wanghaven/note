## 4.8.2 Buffer management for S1 and NG-U GTP-U packets

### 4.8.2.0-1 Buffer management for S1 and NG-U GTP-U packets G-PDU packets discarding due to buffer usage (ID: 11228509)

**4.8.2.0-1.0-3**  (ID: `11228528`)

Buffer management for S1 and NG-U GTP-U packets
If free storage in buffer drops below certain threshold 5G-L2-Hi discards G-PDU packet and logs this every 10000 discarded packets.

## 4.8.3 RLC buffer management for F1-U, optimized F1-U and Xn-U GTP-U packets

**4.8.3.0-1**  (ID: `11228540`)

RLC buffer management for F1-U, optimized F1-U and Xn-U GTP-U packets
If the free storage in RLC buffer drops below the bipPoolCriticalMargin, then 5G-L2-HI(DU) shall discard the G-PDU and shall log this for every 10000 discarded packets.

Increment the PM counter DL\_SDU\_DISC\_BOP\_DU(M55349C03003) for every dropped RLC SDU. Ref : 5G\_L2\_17426

Note : bipPoolCriticalMargin shall be decided during the implementation.

Ref : 5G\_UP\_ALG\_26226\_replaced([11225687](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyO29vvSEe-AqvopbP1qhQ))

## 4.8.4 Overload protection for streaming NR L2 logs

### 4.8.4.0-1 Overload protection for streaming NR L2 logs Description (ID: 11228553)

**4.8.4.0-1.0-3**  (ID: `11228572`)

Overload protection for streaming NR L2 logs
CPU OVL state, further referred in this requirement as OVL.

For Cloud deployment L2-HI-DU checks for OVL independently from L2-HI-CU, for Classical deployment L2-HI-DU doesn't have own OVL control mechanism and L2-HI-CU handle it for both parts.

If "actTraceOvlProt" is set to "TRUE" (received in L2LogCu\_OvlProtectionForStreamingConfigReq on 5G-L2-HI-CU or Cloud: L2LogDu\_OvlProtectionForStreamingConfigReq on 5G-L2-HI-DU) and at least one streaming is enabled (component streaming [Before CNI-115810] or Sherpa [End CNI-115810] ) CPU load is compared with R&D flag rdAvgL2HICPULoadThreshStream and if is above, then OVL is detected.

Action taken when OVL is detected.

OAM is informed about OVL with L2LogCU\_OvlStatusInd on 5G-L2-HI-CU or Cloud: L2LogDu\_OvlStatusInd on 5G-L2-HI-DU, L2LogCu\_SuspendStreamingReq and L2LogDu\_SuspendStreamingReq are expected to be received in order to suspend all ongoing streaming.

Guard Timer is started, of duration indicated by R&D parameter rdOvlProtStreamingGuardTimerInterval.

Guard Timer expiration and actions taken.

After Guard Timer expiration CPU load is compared with R&D flag rdAvgL2HICPULoadThreshStream.

If OVL is still present then Guard Timer is restarted,

otherwise OAM is informed about end of OVL with L2LogCU\_OvlStatusInd on L2-HI-CU or Cloud: L2LogDu\_OvlStatusInd on 5G-L2-HI-DU.

If L2LogCU\_OvlStatusAck on L2-HI-CU or Cloud: L2LogDu\_OvlStatusAck on 5G-L2-HI-DU, is not received, then message is sent again.

Notes:

Reception of L2LogCu\_StreamActivationtReq on L2-HI-CU or Cloud: L2LogDu\_StreamActivationtReq on 5G-L2-HI-DU, while OVL is ongoing is treated as end of OVL.

Streams suspended with L2LogCu\_SuspendStreamingReq or L2LogDu\_SuspendStreamingReq have to be activated again with L2LogCu\_StreamActivationtReq or L2LogDu\_StreamActivationtReq.

R&D parameter is defined in 5G\_L2\_14404\_replaced([11232435](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AjMEfvTEe-AqvopbP1qhQ)).
