# 2.11 R&amp;D CPU Time Measurements

## 2.11.1 Definition of time measurement

**2.11.1.0-1**  (ID: `11200178`)

Definition of time measurement
Three recording types:

* Recording of the measurement of the CPU time needed for the 5G-L2-PS RT core treatments.

list of treatment is defined by the requirement 5G\_L1L2\_FLOW\_751\_replaced([11205385](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Bgk8rPvREe-AqvopbP1qhQ))

* Recording of timeStamp of the events sent by L2RT EO.

list of events is defined by the requirement 5G\_L1L2\_FLOW\_751\_replaced([11205385](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Bgk8rPvREe-AqvopbP1qhQ))

* Recording of SFN/slot counter when they are updated at L2 level.

## 2.11.2 Recording mode and shared memories

**2.11.2.0-1**  (ID: `11200191`)

Recording mode and shared memories
The L2-RT logger is the part running in each 5G-L2-RT core, which has to record the measurements in shared memories.

The L2-NRT collector is the part running in 5G-L2-RT NRT core, running in NRT core. It has to get all the measurements present in shared memories.

L2-NRT and L2-RT have to share memory. There is one L2-NRT collector per Xeon running a L2-RT instance.

There is one shared memory per L2-RT logger. This memory is allocated and initialized by the  L2-RT logger.

There is one global shared memory, allocated and initialized by L2-NRT collector.

There are two modes: recording and no-recording.

This mode is stored in the global shared memory.

During the recording mode, the L2-RT loggers perform read/write access in shared memories.

During the not-recording mode, only the L2-NRT collector performs read access to the shared memories of L2-RT-collectors, few TTI delay after the switch between recording to not-recording mode.

During the recording mode, the L2-RT loggers performs read access to the global shared memory, to get the current mode, only one time per TTI. No access is done by  L2-NRT collector.

L2-NRT collector only performs read/write access to the global shared memory when the mode is switched.

## 2.11.3 Recording of time measurement

**2.11.3.0-1**  (ID: `11200200`)

Recording of time measurement
During the recording mode, each 5G-L2-RT logger records the timemarker in the circular buffer in its shared memory.

When the no-recording mode is activated, the 5G-L2-RT logger has to be able to provide the total number of recorded timeMarkers to the L2-NRT collector. This is calculated from the pointer to the next free timeMarker in the circular buffer, and from an indicator set to true when the buffer is full. The pointer and the indicator are reset at the beginning of the recording mode.

That also allows L2-NRT collector to get all the timeMarkers in the chronological sort.

## 2.11.4 Time Marker

**2.11.4.0-1**  (ID: `11200211`)

Time Marker
Content of the timeMarker:

The timeMarker contains 64 bits. there are several timeMarker types. According the timeMarker type, the content of the 64 bits is not the same.

Common part for any marker (40bits, MSB part):  from MSB to LSB:

* 28bits for the timestamp, which is a counter of multiple of CPU tick number (CPU tick number/16)
* 2 bits for the timeMarker type identifier (Entry=0, Exit=1, Event=2, reserved value=3)
* 10bits for the probe identifier

Specific part depending of marker type (24bits, LSB part): from MSB to LSB:

> Entry and Exit L2 marker type:

-  8 bits spare

* 16 bits = BIP message identifier

Entry and Exit markers allows to provide the CPU time needed for a treatment.

They contain the timestamp of the entry and the exit of the 5G-L2-PS treatment.

> Event L2 marker type:

* 8 bits for subcellid
* 16 bits spare (debug info)

> exception for probeId=2 for the SFN marker, which is a specific event L2 marker:

* 8 bits for slot counter
* 12 bits for SFN counter
* 4 bits spare (debug info)

SFN marker allows to provide the values SFN and slot for the timeLine analyser.

> exception for probeId=3 for the performance marker, which is a specific entry/exit L2 marker:

* 8 bits spare
* 16 bits spare.

Entry and exit performance marker is recorded in sequence, without any code between entry and exit, in order to check the time needed to record a probe.

This performance marker time does not provide a very high precision, it can be impacted by compiler optimizations (at building time) and CPU cache refresh (at running time).

Specific part of the reserved marker is not defined.

Event marker and SFN marker are punctual events.

For coherence reason with the analyser, a probe identifier has to be used either in an event marker or an entry/exit markers, but never both.

## 2.11.5 structure of the generated file

**2.11.5.0-1**  (ID: `11200224`)

structure of the generated file
Content of the file created by the collector

The file uses the JSON format for all the data, allowing to read them directly by a basic editor.

There is one file per L2RT instance.

The Json format does not include any constraint for the order of the Json objects in the file.

The description below does not provide any constraint for the order of the object.

The file contains

* one "snapshotData" object, containing global information about the snapshot,
* one dictionary object, containing names and values of probes, and name and values of the messageIds
* one "swCompnentData" object containing an array of elements, one by SW component, containing all the data provided by this component.

The snapshotData object contains:

- a field "title", set to a constant string, as "timeMarker log file", to identify the content of the file

- a field "fileVersion" set to 1 for the first version of structure of the file, which may evolve later.

- a field "moduleName" set to "5G-L2-RT" for feature I58 and I77

- a field "snapshotStatus", set to 0 if the file is correct, or another value corresponding to an errorId detected by the NRT collector.

- a field "moduleData", to indicate few application data, as for instance the value identifying the subCellBfGroup.

- a field "swComponentList" containing the list of the L2 SW component  name recording probes.

The identifiers are  « L2PSUL », « L2PSDL », « L2PSTMR », « L2PSSGL » for L2PS,

and « L2LOUL », « L2LOSDU », « L2LOPDU » for L2LO (introduced by 5GI000077 futher implementation)

The number of names in this list is equal to the number of elements in the array object, excepted in case of error.

For the "swComponentData" object, an element contains:

- a field "swComponentId", set to the SW component name as described above

- a field "swComponentStatus", which is a status for this component, set to 0 if the data are correct, or another value corresponding to an errorId

- a field "markerVersion", set to 1 for the first version of structure of the marker, which may evolve later.

- a field "markerCount", providing the number of marker recorded in this part

- a field "markerList" containing all the marker values, in hexadecimal format, 16 hexadecimal characters per marker, in a Json array.

The list shall contain the marker in the chronological order, so the oldest marker first and the newest marker at the end.
