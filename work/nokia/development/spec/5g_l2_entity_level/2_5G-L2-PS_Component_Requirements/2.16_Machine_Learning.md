## 2.16.1 MLIP utilization by L2-PS

### 2.16.1.0-1 MLIP utilization by L2-PS MLIP Overview (ID: 13496053)

**2.16.1.0-1.0-1**  (ID: `13496055`)

MLIP utilization by L2-PS MLIP (Machine Learning Inference Platform) is a hardware accelerator for AI/ML tasks (see MLIP details in BBP specification using EE Magic Draw or [direct link](https://cc4twc.ext.net.nokia.com/collaborator/document/1b0fa8b3-64b5-4b9a-9d21-7e2dfc5625c9?viewId=c9705666-1f39-4563-8a4d-31283b5da069&viewType=model&sectionId=1398456e-21e8-4ec6-8f18-97a3906be71c) ). MLIP primary function is to perform inference, which is the process of using a pre-trained machine learning model to support real-time decisions. 5G-L2-PS utilizes MLIP for specific, performance-critical use cases.

The entire process of using MLIP for a specific use case can be broken down into two main phases:

  * activation of the ML use case (see [13496074](https://dn-prod.ext.net.nokia.com/rm/resources/BI_gA9gF0h5EfGPgoB1AmbVPQ)),
  * runtime inference communication (see [13496077](https://dn-prod.ext.net.nokia.com/rm/resources/BI_8FtkQEh5EfGPgoB1AmbVPQ)).

### 2.16.1.0-2 MLIP utilization by L2-PS MLIP API (ID: 13496071)

**2.16.1.0-2.0-1**  (ID: `13496072`)

MLIP utilization by L2-PS The BBP MCU HWAPI AIML provides a MlipSupport library, which is a library of APIs that enables 5G-L2-PS to function as a client, running inferences on the MLIP instance. As the client, 5G-L2-PS is responsible for correctly utilizing this library to enhance its scheduling decisions. This library serves as the bridge between the L2-PS software and the MLIP hardware. 5G-L2-PS calls to send inference requests and receive results, as specified in [13496077](https://dn-prod.ext.net.nokia.com/rm/resources/BI_8FtkQEh5EfGPgoB1AmbVPQ).

### 2.16.1.0-3 MLIP utilization by L2-PS ML Use Case Activation (ID: 13496074)

**2.16.1.0-3.0-1**  (ID: `13496075`)

MLIP utilization by L2-PS ML use case activation procedure consists of the following steps:

  * 5G-CP-RT initiates the activation process by sending a PsMl_ActivateUseCaseReq message to 5G-L2-PS (see [10981887](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2VuPx_u6Ee-AqvopbP1qhQ)). The activation request contains mlInferenceEngineAddress IE - a SICAD address which indicates 5G-L2-PS where to send its inference request(s).
  * Upon reception of a valid activation request, 5G-L2-PS stores the inference engine's SICAD address and all relevant parameters for the specific use case. 5G-L2-PS shall confirm successful activation by sending a PsMl_ActivateUseCaseResp message to 5G-CP-RT (see [10981906](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2VuPyvu6Ee-AqvopbP1qhQ)).



Once this flow is complete, 5G-L2-PS is ready to communicate with the MLIP and support runtime inference communication, as specified in [13496077](https://dn-prod.ext.net.nokia.com/rm/resources/BI_8FtkQEh5EfGPgoB1AmbVPQ).

### 2.16.1.0-4 MLIP utilization by L2-PS Runtime Inference Communication (ID: 13496077)

**2.16.1.0-4.0-1**  (ID: `13496078`)

MLIP utilization by L2-PS Once a use case is activated (see [13496074](https://dn-prod.ext.net.nokia.com/rm/resources/BI_gA9gF0h5EfGPgoB1AmbVPQ)), 5G-L2-PS is able to send inference requests to the MLIP during its regular processing cycle.

There are two approaches to 5G-L2-PS <> MLIP inference communication:

Method 1. Less timing critical. The communication follows a request-response sequence (like message based):

  1. Request from L2-PS: 5G-L2-PS sends an inference request message (specific per each use case) to the Inference Engine Scheduler using the SICAD address obtained during activation. This message contains the input data required by the DNN model for the specific use case.
  2. Inference on MLIP: The Inference Engine Scheduler forwards the request to the MLIP instance, which performs the inference using the pre-trained DNN model.
  3. Response to L2-PS: After completing the inference, the result is sent back to 5G-L2-PS via an inference response message (specific per each use case). 5G-L2-PS then decodes this message to extract the output data and proceeds with its internal logic.



Method 2. More timing critical. The L2-PS shall ensure the round-trip latency, measured from sending an input to receiving the corresponding output, is less than 10 microseconds, in order to complete the process within one slot. The communication between the L2-PS and MLIP is provided through direct calls of HwApi BBP function from BBP library.

### 2.16.2.3 DL MU-MIMO UE pairing Use Case

#### 2.16.2.3.0-1 DL MU-MIMO UE pairing Use Case Overview (ID: 13497388)

**2.16.2.3.0-1.0-1**  (ID: `13497389`)

DL MU-MIMO UE pairing Use Case ML based massive MU-MIMO DL UE pairing use case is defined in [11230030](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyri7PvSEe-AqvopbP1qhQ).

The use case is activated in 5G-L2-PS by reception of PsMl_ActivateUseCaseReq message with useCaseId IE set to MuMimo_Calculation (see [10981887](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2VuPx_u6Ee-AqvopbP1qhQ)).

When use case is activated, then 5G-L2-PS shall prepare input data for inference. For each eligible cell (for eligibility conditions, see [13355393](https://dn-prod.ext.net.nokia.com/rm/resources/BI_14I6RPA-EfC3ocOh_n7w7A)), 5G-L2-PS shall calculate two sets of inputs for the DNN model: NormBeamPF (normalized beam Proportional Fair values) and BeamSumRtxUeNum (sum of retransmission UEs per beam). For a model with 48 predefined DNN beams, NormBeamPF is representeded as 6x48 matrix and BeamSumRtxUeNum is 1x48 matrix. This results in a total of 336 input values (for inference inputs, see [13355414](https://dn-prod.ext.net.nokia.com/rm/resources/BI_aj_PxfBCEfC3ocOh_n7w7A)). The input data shall be delivered to MLIP for inference, as specified in <5G EL: Send request to MLIP>.

5G-L2-PS shall handle the reception of inference output data from MLIP, as specified in <5G L2: Receive response from MLIP>. The MLIP returns a set of "Q-values," one for each of the 48 beams. These Q-values shall be used by 5G-L2-PS to prioritize beams and UEs for Multi-User MIMO (MU-MIMO) pairing. L2-PS uses the Q-values to construct virtual UEs for MU-MIMO transmission, as specified in [13355508](https://dn-prod.ext.net.nokia.com/rm/resources/BI_1_G1MPBPEfC3ocOh_n7w7A).

The use case is deactivated in 5G-L2-PS by reception of PsMl_DeactivateUseCaseReq message with useCaseId IE set to MuMimo_Calculation (see [10981989](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2Vu21_u6Ee-AqvopbP1qhQ)).

#### 2.16.2.3.0-3 DL MU-MIMO UE pairing Use Case Send request to MLIP (ID: 13497417)

**2.16.2.3.0-3.0-1**  (ID: `13497418`)

DL MU-MIMO UE pairing Use Case 5G-L2-PS shall deliver inputs to MLIP instance. The inputs – as defined in [13355414](https://dn-prod.ext.net.nokia.com/rm/resources/BI_aj_PxfBCEfC3ocOh_n7w7A)\- are needed at MLIP instance to proceed with inference – as defined in [13355551](https://dn-prod.ext.net.nokia.com/rm/resources/BI_pm5QdfBREfC3ocOh_n7w7A).

To send inference requests to MLIP, 5G-L2-PS shall call the dedicated sendRequest (MlRtInferencePollRequestAPI) function from MLRT library (i.e. L2-PS delegates “sending” functionality to a local MLRT polling API).

sendRequest function:

  * processes an inference context, which is the structure holding input data for runtime inference,
  * ensures model_id exists and is valid,
  * processes a completion ID, where:
  * completion id exists and is not in used in other ongoing inference request,
  * completion_id assignation shall follow the rule to ensure unique number across multiple core applications:
    * completion_id = 0x000000ZY, where Z="core_id" and Y in range from 0 to 15,
    * completion_id < 512 due to Marlin MLIP hardware capability,
  * once those checks are successful, the inference request will be launched against HW accelerator.



Note: sendRequest function:

  * does not implement any transport logic itself or message-based communication,
  * does not use the received MLIP/SICAD address to route the request.

#### 2.16.2.3.0-4 DL MU-MIMO UE pairing Use Case Receive response from MLIP (ID: 13497420)

**2.16.2.3.0-4.0-1**  (ID: `13497421`)

DL MU-MIMO UE pairing Use Case 5G-L2-PS shall be able to handle responses from MLIP instance. 5G-L2-PS shall implement the receiveResponse and receive functions to receive the inference outputs (see [13355427](https://dn-prod.ext.net.nokia.com/rm/resources/BI_x8i1IPBDEfC3ocOh_n7w7A)) from the MLIP instance. The 5G-L2-PS, as Inference Client, shall poll the MLRT via the MlRtInferencePollResponseAPI function to retrieve an inference result. The receive function shall repeatedly call the MlRtInferencePollResponseAPI function. The polling shall terminate based on the first of the following conditions to be met:  
• a valid response appears, or,  
• timeout is exceeded, or,  
• number of retries is exceeded.

After the response reception, 5G-L2-PS makes the UE pairing decision and constructs the virtual UEs, as specified in [13355508](https://dn-prod.ext.net.nokia.com/rm/resources/BI_1_G1MPBPEfC3ocOh_n7w7A) and [13355540](https://dn-prod.ext.net.nokia.com/rm/resources/BI_JAbUk_BREfC3ocOh_n7w7A).   
If the response is not available, then 5G-L2-PS shall release the allocated completion_ID in the next slot.

#### 2.16.2.3.0-5 DL MU-MIMO UE pairing Use Case Allocate and release functions (ID: 13497422)

**2.16.2.3.0-5.0-1**  (ID: `13497423`)

DL MU-MIMO UE pairing Use Case The 5G-L2-PS component shall implement MLRT library memory management functions to ensure that memory allocation and deallocation use the same runtime-specific mechanisms. The implementation shall adhere to the following:

  * The allocate function shall obtain a memory buffer from the MLRT runtime. It shall return the pointer to the allocated buffer on success and a null pointer if the allocation fails. For each cell L2-PS should allocate separate memory buffer.
  * The release function shall return that buffer back to MLRT when the data is no longer needed.

#### 2.16.2.3.0-6 DL MU-MIMO UE pairing Use Case Timing and Fallback Mechanism (ID: 13497424)

**2.16.2.3.0-6.0-1**  (ID: `13497425`)

DL MU-MIMO UE pairing Use Case Communication with MLIP for the DL MU-MIMO UE pairing use case (related to method 2 specified in [13496077](https://dn-prod.ext.net.nokia.com/rm/resources/BI_8FtkQEh5EfGPgoB1AmbVPQ)) is time-sensitive. The inference result must be available within the same slot it was requested to be useful for the scheduler. The target round-trip latency for the inference - from L2-PS sending the request to receiving the response - is less than 10 microseconds.

A dedicated R&D parameter, rdTimeInferenceResp, defines the timeout duration for which 5G-L2-PS shall wait for a response from the MLIP. The default value shall be set to 10 µs, see [13448824](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FOj_JCi9EfGxldjOZQLezQ).

When a response from the MLIP is not received within the timeout specified by rdTimeInferenceResp, then 5G-L2-PS shall:

  * Discard the pending request for that slot.
  * Fallback to using the legacy MU-MIMO pairing algorithm.



When a response from the MLIP is received within the configured timeout, then L2-PS shall use the inference result for scheduling, as specified in [13497388](https://dn-prod.ext.net.nokia.com/rm/resources/BI_PQPaAkkrEfGPgoB1AmbVPQ).
