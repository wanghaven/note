Pre-Scheduler

The UL pre-scheduler maintains the “pre-scheduler candidate set” (5G_L2_378_replaced([11151479](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24ifmPvTEe-AqvopbP1qhQ))) of UEs with UL resource requests (such as VoIP or data transmission, or HARQ retransmission).

This set contains a list for each UL request type based on their priority class (defined in 5G_UP_ALG_10506_replaced([11194320](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv0cIfvSEe-AqvopbP1qhQ))).

[CB012734-E] If NRBTS/NRCELLGRP/actHighNumberOfUes = TRUE and rdActEnhBeamSelection = TRUE, the UL pre-scheduler maintains in addition a list per beam for the UL DRB traffic request class. The UE with this request class is added to the list associated with its primary beam. [End CB012734-E]

In each list, new UEs are added to the back of the list, such that UEs are ordered by the age in the list (5G_L2_378_replaced([11151479](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24ifmPvTEe-AqvopbP1qhQ))). The UEs in this set meet the criteria defined in 5G_UP_ALG_4807_replaced([11194840](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv3fe_vSEe-AqvopbP1qhQ)), with exceptions defined in 5G_UP_ALG_4805_replaced([11194818](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv3fd_vSEe-AqvopbP1qhQ)) (measurement gap), 5G_UP_ALG_4826_replaced([11195069](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv5UqfvSEe-AqvopbP1qhQ)) (reconfiguration transient phase), 5G_UP_ALG_6633_replaced([11195102](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv57sPvSEe-AqvopbP1qhQ)) (SCell release) and 5G_UP_ALG_4829_replaced([11195113](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv57svvSEe-AqvopbP1qhQ)) (A-CSI scheduling with large k2). UEs stay in the “pre-scheduler candidate set” until their resource request has been served (5G_L2_378_replaced([11151479](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24ifmPvTEe-AqvopbP1qhQ))), or the UE is no longer eligible for scheduling, e.g., due to one of the exceptions mentioned above.

[Before CB012734-E]

At each PUSCH scheduling occasion, the pre-scheduler creates the “Candidate Set 1” (CS1) (5G_UP_ALG_4825_replaced([11195013](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv4tnvvSEe-AqvopbP1qhQ))) based on the pre-scheduler candidate set.

[CB012734-E] At each PUSCH scheduling occasion,

  * If NRBTS/NRCELLGRP/actHighNumberOfUes = False or rdActEnhBeamSelection = False, the pre-scheduler creates the “Candidate Set 1” (CS1) (5G_UP_ALG_4825_replaced([11195013](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv4tnvvSEe-AqvopbP1qhQ))) based on the pre-scheduler candidate set.



-Else, the pre-scheduler performs the beam selection as defined in 5G_UP_ALG_27614_replaced([11194972](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv4tl_vSEe-AqvopbP1qhQ)) and determines the set of UEs the selected beam serves to create the “Candidate Set 1” (CS1).

-End.

[End CB012734-E]

For this set UEs are selected in the order of the pre-scheduler candidate set, i.e., in order of their priority class and their sub-priority.

The pre-scheduler selects either up to rdMaxNumUeTdUl UEs (5G_UP_ALG_4825_replaced([11195013](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv4tnvvSEe-AqvopbP1qhQ))) or all UEs in the pre-scheduler candidate set. The permitted maximum number of UEs may be reduced by the overload control (5G_L2_1833_replaced([11212618](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-mrWvvTEe-AqvopbP1qhQ)) and 5G_L2_1845_replaced([11212901](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-ogi_vTEe-AqvopbP1qhQ))).

Time Domain Scheduler

The Time Domain (TD) scheduler takes CS1 as the input for TD scheduling decisions.

[Before CB012734-E]

In case of analog beamforming, the TD scheduler selects all UEs corresponding to the beam which is associated with the highest priority UE (5G_UP_ALG_4865_replaced([11196363](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCelPvSEe-AqvopbP1qhQ))) for the “CS1 Selected” set. Here priority considers both the priority class (5G_UP_ALG_10322_replaced([11194333](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv0cI_vSEe-AqvopbP1qhQ))), as well as the sub-priority of the appropriate class (5G_UP_ALG_4772_replaced([11194367](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv0cKPvSEe-AqvopbP1qhQ))) in case multiple UEs have the same priority class, as described in 5G_L2_375_replaced([11151339](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24hRf_vTEe-AqvopbP1qhQ)).

[CB012734-E] In case of analog beamforming,

  * If NRBTS/NRCELLGRP/actHighNumberOfUes = False or rdActEnhBeamSelection = False, the TD scheduler selects all UEs corresponding to the beam which is associated with the highest priority UE (5G_UP_ALG_4865_replaced([11196363](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCelPvSEe-AqvopbP1qhQ))) for the “CS1 Selected” set. Here priority considers both the priority class (5G_UP_ALG_10322_replaced([11194333](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv0cI_vSEe-AqvopbP1qhQ))), as well as the sub-priority of the appropriate class (5G_UP_ALG_4772_replaced([11194367](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv0cKPvSEe-AqvopbP1qhQ))) in case multiple UEs have the same priority class, as described in 5G_L2_375_replaced([11151339](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24hRf_vTEe-AqvopbP1qhQ)).



-Else the selection of UEs corresponding to the beam associated with the highest priority UE is skipped as it's already performed in Pre-scheduler as described in 5G_UP_ALG_27614_replaced([11194972](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv4tl_vSEe-AqvopbP1qhQ)). As a consequence, “CS1 Selected” is identical to "Candidate set 1".

-End

[End CB012734-E]

In case of digital beamforming or if beamforming is not enabled, “CS1 Selected” is identical to CS1 (5G_UP_ALG_4867_replaced([11196407](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCenPvSEe-AqvopbP1qhQ))).

Based on “CS1 Selected”, the TD scheduler creates “Candidate Set 2” (CS2) (5G_UP_ALG_4867_replaced([11196407](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCenPvSEe-AqvopbP1qhQ))), which is a set of carrier-specific lists. For each UE in “CS1 Selected”, the TD scheduler assigns this UE to the appropriate carrier-specific list in CS2, such that each UE is only contained in one carrier-specific list. Some UEs may be excluded due to CRC NACK flood (5G_UP_ALG_4892_replaced([11197752](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwH-JvvSEe-AqvopbP1qhQ))) or PDCCH availability check (5G_UP_ALG_4895_replaced([11198055](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwIlMfvSEe-AqvopbP1qhQ))).

After CS2 has been defined, the TD scheduler builds the “to-be-scheduled UEs” set based on CS2 (5G_UP_ALG_4871_replaced([11196450](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCepPvSEe-AqvopbP1qhQ))). For each carrier, this set contains a list created by subsequently adding UEs in order of highest TD metric (5G_UP_ALG_4850_replaced([11195809](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv-0M_vSEe-AqvopbP1qhQ))) up to a maximum of rrmMaxNumUesFdmUl UEs (5G_UP_ALG_4960_replaced([11199244](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwMPk_vSEe-AqvopbP1qhQ))), while ensuring there are sufficient resources available (5G_UP_ALG_4871_replaced([11196450](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCepPvSEe-AqvopbP1qhQ))). In case of A-CSI, this process is instead defined by 5G_UP_ALG_4869_replaced([11196426](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCeoPvSEe-AqvopbP1qhQ)).

Frequency Domain Scheduler

Frequency Domain (FD) scheduler takes the “to-be-scheduled UEs” set as the input for allocating frequency resources, i.e., physical resource blocks (PRBs). For each carrier, in order of the priority (5G_UP_ALG_4968_replaced([11199418](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwNdt_vSEe-AqvopbP1qhQ))) the FD scheduler computes the required resources to serve the UEs request (5G_UP_ALG_4988_replaced([11199729](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwPS7PvSEe-AqvopbP1qhQ))). Here priority again consists of the priority class (5G_UP_ALG_10322_replaced([11194333](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv0cI_vSEe-AqvopbP1qhQ))) and the sub-priority of the appropriate class (5G_UP_ALG_4772_replaced([11194367](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kv0cKPvSEe-AqvopbP1qhQ))), as described in 5G_L2_375_replaced([11151339](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24hRf_vTEe-AqvopbP1qhQ)). Afterwards, the FD scheduler allocates the computed resources (5G_UP_ALG_5026_replaced([11201742](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwZD4vvSEe-AqvopbP1qhQ))).

The FD scheduler may skip UEs in the “to-be-scheduled UEs” set if the remaining time in the slot is not sufficient to serve all UEs (5G_L2_1859_replaced([11213118](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-purfvTEe-AqvopbP1qhQ))).

Implementation Note 1: In case of supercell mode enabled according to 5G_L2_7884_replaced([11153463](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24uFzvvTEe-AqvopbP1qhQ)), depending on the number of supercell subcells up to 6 FD Scheduler (EO) instances are running at the same time for parallelized scheduling of frequency domain resources among supercell subcells within each UL Scheduler (EO) instance.

PDCCH Scheduler

Based on the allocated resources from FD scheduler the PDCCH Scheduler creates the DCI (5G_UP_ALG_4928_replaced([11198763](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwKaafvSEe-AqvopbP1qhQ))) and sends it to the L1 in form of a “PDCCH Send Request”. Additionally, a “PUSCH Receive Request” is sent to L1 based on the FD scheduler allocation (5G_UP_ALG_5135_replaced([11204062](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwi05vvSEe-AqvopbP1qhQ))).

##### 2.3.2.1.1.0-1 Evaluation of UL Buffer Status Reports Check if a BSR is included in the UL MAC PDU (ID: 11151171)

**2.3.2.1.1.0-1.0-1**  (ID: `11151176`)

Evaluation of UL Buffer Status Reports ueBsrList in LoCtrl_UlMacPduReceiveInd is an array with maxNumOfLcgs elements. The element ueBsrList[x].bufferSizeIndex represents the Buffer Size Index for the Logical Channel Group with LCG ID = x:

  * If ueBsrList[x].bufferSizeIndex ≠ INVALID (=0xFFFF), then a Buffer Size Index is reported for LCG x.

  * Otherwise, no Buffer Size Index is reported for LCG x.




Note: The UE reports a BSR to the gNB only for the LCG of the RB configured in the gNB. In 5G18A release (NSA) only one DRB and no SRB can be configured on the gNB per UE. In each ueBsrList, L2-PS can check only the entry of the LCG which the RB is configured in.

[5GC000578-J] Note: SRB3 for NSA in gNB is also applied together with DRB, where the SRB3 are configured with different LCG then DRB. [End 5GC000578-J]

[5GC000578-J] Note: Different LCGs for SRB and DRB are introduced by, for example, SRB with LCG#0 and DRB with LCG#1. [End 5GC000578-J]

If ueBsrList[x].bufferSizeIndex ≠ INVALID for at least one LCG x, then a BSR has been reported in the received UL MAC PDU and L2-PS shall proceed from point 2 in 5G_UP_ALG_5131_replaced([11203806](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwg_wfvSEe-AqvopbP1qhQ)).

##### 2.3.2.1.1.0-2 Evaluation of UL Buffer Status Reports Check if this BSR is older than last received up-to-date BSR for the UE (ID: 11151181)

**2.3.2.1.1.0-2.0-3**  (ID: `11151199`)

Evaluation of UL Buffer Status Reports This is done by comparing the time when this UL MAC PDU was granted the first time (rrmInitUlTxTime[harqProcessIndex]) and the time when the last received up-to-date BSR was granted the first time (rrmLastUlBsrUpdateTime):

  * If rrmInitUlTxTime[harqProcessIndex] > rrmLastUlBsrUpdateTime, then this BSR is the most up-to-date and L2-PS shall process it further from point 3 in 5G_UP_ALG_5131_replaced([11203806](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwg_wfvSEe-AqvopbP1qhQ)).

  * Otherwise, this BSR is outdated and L2-PS shall not consider it further.




In the unlikely case the CG HARQ context has been reset for an immediate reuse (refer to 5G_L2_11023_replaced([11175832](https://dn-prod.ext.net.nokia.com/rm/resources/BI_26vfAvvTEe-AqvopbP1qhQ)) and 5G_L2_9153_replaced([11175877](https://dn-prod.ext.net.nokia.com/rm/resources/BI_26vfC_vTEe-AqvopbP1qhQ))), the above check on rrmInitUlTxTime value cannot be performed due to the absence of a valid HARQ context [PR802079] with the same SFN and slot number as in LoCtrl_UlMacPduReceiveInd [End PR802079], so the BSR is also considered as outdated.

Notes:

  * harqProcessIndex is the HARQ process index of the received UL MAC PDU and it is read from the homonym IE in LoCtrl_UlMacPduReceiveInd

  * A UE with carrier aggregation may send multiple UL TBs and hence multiple BSRs per TTI (triggered Regular or Periodic BSR on one carrier and possible Padding BSR on other carriers).

  * An outdated BSR was sent before the last up-to-date BSR (rrmInitUlTxTime[harqProcessIndex] < rrmLastUlBsrUpdateTime) but it is received only now due to retransmission delays.




CP2 reference: 5G_UP_ALG_5131_replaced([11203806](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwg_wfvSEe-AqvopbP1qhQ))

### 2.3.3.1 Estimation of available bandwidth

#### 2.3.3.1.0-1 Estimation of available bandwidth Determine number of PRBs required for PUCCH (numPrbToBeUsedForPucch) (ID: 11151739)

**2.3.3.1.0-1.0-3**  (ID: `11151757`)

Estimation of available bandwidth That requirement refers to 5G_UP_ALG_4876_replaced([11196884](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwGI9vvSEe-AqvopbP1qhQ)) explaining how to calculate numPrbToBeUsedForPucch.

numPrbToBeUsedForPucch is the number of cell-PRBs consumed from the overall PRB processing capacity of the L1 pool for PUCCH reception. Note, L1 processing capacity limitations are per OFDM symbol.

[PR788151]

5G_UP_ALG_4876_replaced([11196884](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwGI9vvSEe-AqvopbP1qhQ)) also describes totalNumPrbToBeUsedForPucch which is essentially sum of numPrbToBeUsedForPucch and the PUCCH PRBs (if any) to be used for carrying A/N of the common PUCCH (Msg4/SRB0...). L2-PS uses only numPrbToBeUsedForPucch within all the computational steps to derive availableUlPrb indicated in 5G_UP_ALG_4876_replaced([11196884](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwGI9vvSEe-AqvopbP1qhQ)).

Further, the PUCCH PRBs (if any) to be used for carrying A/N of the common PUCCH (Msg4/SRB0...) which are non overlapping with dedicated PUCCH PRBs are subtracted from the available UL PRBs (availableUlPrb as per 5G_UP_ALG_4876_replaced([11196884](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwGI9vvSEe-AqvopbP1qhQ))) during the FDM phase.

[End PR788151]

Following PUCCH F1 and F3 attributes are of interest: startPrb and numOfPrb as finally signaled in UlData_PucchReceiveReq, and concerning PUCCH F1 only, the initialCyclicShift and Orthogonal Cover Code. Note, the numOfPrb of UlData_PucchReceiveReq might be smaller than the numOfPrb configured via C-Plane through PsUser interface as UE adapts PUCCH F3 bandwidth if the UCI contains HARQ AN bits.

Multiple PUCCH F1 allocated on same PRB due to initialCyclicShift and Orthogonal Cover Code usage consume only one cell-PRB from L1 processing capacity together.

The intra-slot frequency hopping of PUCCH F1 and F3 to secondHopPrb has to be ignored for the purpose of calculating numPrbToBeUsedForPucch, i.e. even if a PUCCH with numOfPrb = 2 occupies four different PRBs in total, two PRB in first 7 symbols and another two PRB on second 7 symbols, it consumes only two cell-PRB from processing perspective.

So, to determine numPrbToBeUsedForPucch of a certain UL slot, L2-PS shall count the number of PRBs occupied by all PUCCHs for which DL scheduler will request reception from L1 - based on their startPrb and numOfPrb and considering the information given before. For that, L2-PS has to distinguish users with pre-allocated DPA resources in that UL slot (for carrier aggregation), called DPA user in the following, and users without pre-allocated DPA resources in that UL slot, called non-DPA users. DPA functionality is explained under 5G_L2_587_replaced([11158124](https://dn-prod.ext.net.nokia.com/rm/resources/BI_25JjmfvTEe-AqvopbP1qhQ)).

For non-DPA users, i.e. users scheduled only in their PCell, the UL Pcell scheduler knows exactly which PUCCHs will be requested by the DL Pcell scheduler for reception from L1 UL.

For DPA users, i.e. users that may be scheduled in their PCell and also in their Scell(s), the UL Pcell scheduler may not know in time which PUCCH will be requested by the DL Pcell scheduler for reception, especially if actK1OptForDpa = true. So, even if the DPA user has got a pre-allocated PUCCH F3 which is a resource of UE’s PUCCH resource set1, it may transmit its UCI on a different PUCCH resource nevertheless. Following cases might happen finally (after UL PCell scheduling was done) - informative:

A) DL scheduler will not request any PUCCH reception as UE was not scheduled in any DL slot of any carrier.

B) DL scheduler will request reception of the PUCCH F1 of SR configuration if UE was not scheduled and it has an SR occasion in same UL slot

C) DL scheduler will request reception of the PUCCH F1 of SR configuration and additionally of a PUCCH F1 of PUCCH resource set 0 if the UE was scheduled once or twice over all carriers and DL slots and has an SR occasion in same UL slot.

D) DL scheduler will request reception of a PUCCH F1 of PUCCH resource set 0 if UE was scheduled at most twice in first or third DL slot of any carrier (applicable to 2CC case only).

E) DL scheduler will request reception of the PUCCH F3 of CSI configuration if UE was not scheduled in any DL slot of any carrier but has periodic CSI (+SR) occasion in same UL slot.

F) DL scheduler will request reception of the pre-allocated PUCCH F3 (PUCCH resource set1 resource) if number of UCI bits (CSI report, if any + SR bit, if any + AN bits) is <= maxPayloadSize of PUCCH resource set1.

G) DL scheduler will request reception of the PUCCH F3 of PUCCH resource set2 (whereby using same PUCCH resource index as in case of the pre-allocated PUCCH F3 of set1) if number of UCI bits (CSI report, if any + SR bit, if any + AN bits) is > maxPayloadSize of PUCCH resource set1.

Due to this uncertainty and because any underestimation must be prevented, the case leading to highest cell-PRB consumption shall be assumed. Accordingly following algorithm shall be applied per DPA user:

If the UE has no periodic CSI occasion in the UL slot, then assume pre-allocated PUCCH F3 will be used and if UE has SR occasion in same slot, then consider PUCCH F1 of SR configuration reserved as well.

If the UE has a periodic CSI occasion in the UL slot, then assume the PUCCH F3 resource with largest bandwidth (configured number of PRB) will be used,

• either the PUCCH F3 of CSI configuration

• or the pre-allocated PUCCH F3

• or the PUCCH F3 of PUCCH resource set2 (if configured) accessed by same PUCCH resource index as the pre-allocated PUCCH F3.

Thereby, please note,

• it can be assumed that the PUCCH resource set2 resources are larger in bandwidth than the PUCCH resource set1 resources.

• it should better not be assumed that the PUCCH F3 of the CSI configuration is always configured larger in bandwidth than the set2 resource.

• any potential bandwidth reduction due to multiplexing with HARQ AN shall be ignored, that means numOfPrb configured by C-Plane for PUCCH F3 shall be used.

#### 2.3.5.2.0-1 Carrier-Based FD Scheduling Updating Carrier specific CS2 Lists (ID: 11152096)

**2.3.5.2.0-1.0-1**  (ID: `11152103`)

Carrier-Based FD Scheduling This is a detail description for the implementation of 5G_UP_ALG_10563_replaced([11196411](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwCenfvSEe-AqvopbP1qhQ)) with carrier aggregation.

5G-L2-PS shall perform carrier-based FD scheduling by means of the carrier specific CS2 lists and the priority list of carrieres received from TD scheduling, see 5G_L2_398_replaced([11151863](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24k72vvTEe-AqvopbP1qhQ)).

5G-L2-PS shall find the carrier with highest priority that is not yet handled in FD scheduling, and do the FD scheduling for the selected carrier. If more than 1 carrier have same priority, choose the one with the least carrier index.

After FD scheduling, 5G-L2-PS shall do the priority update for each CS2 list of the carriers that FD scheduling is not done.

1.If this UE is already scheduled in the previous FD scheduling for other carrier, 5G-L2-PS shall re-calculate the TD metric for this UE on this carrier. Go to the next UE in the CS2 list.

2.Go to next UE if this UE match none of above cases.

After check for all UEs in the list, re-sort the CS2 list based on the TD metric order. The TD metric for the highest priority UE is the new priority of the carrier.

5G-L2-PS shall continue to find a carrier with highest priority for FD scheduling until all carriers with CS2 list not empty are handled in FD scheduling.

#### 2.3.5.5.0-2 HARQ Processing HARQ Handling of PUSCH for Asymmetric DSS (ID: 12811523)

**2.3.5.5.0-2.0-1**  (ID: `12811524`)

HARQ Processing On an FDD carrier shared with LTE with actAsymmetricDss=TRUE, the number of PRB that can be allocated for the UEs per UL slot has some extra constraints compared to symmetric DSS due to the following reasons:

\- Available UL resources are composed of 2 non-contiguous areas which are NR dedicated area and Shared area because there is LTE PUCCH between the two areas.

\- The resource assigned in shared area to NR can change during run time upon receiving the EN DSS UL SUBCARRIER RESOURCE ALLOCATION REQUEST message from CRM. When the ‘Activation Time’ comes, new allocation will be applied and the number of available PRBs in a slot will change.

\- K2 is constant and equal to 2 since all DL slots are available due to NR dedicated area.

\- Adaptive retransmission feature is always activated with asymmetric DSS thanks to PDL rule resulting actUlAdaptReTxEnhDss=TRUE and actAdaptiveReTxResMcs=TRUE.

Refer to [11196526: Sufficient BWP FD resources are available](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwDFrfvSEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9).

**2.3.5.5.0-3.0-3**  (ID: `11152380`)

Scheduling of an Initial Transmission:

  * The transport block size is calculated as specified in 5G_UP_ALG_5084_replaced([11203344](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwejfPvSEe-AqvopbP1qhQ)).   * L2 shall allocate a UNIQUE absoluteHarqProcessIndex across all L1 resources within the considered L1 instance, so that L1 can pool over those indexes.

  * the MCS recommended by the link adaptation (rrmUlProposedMcs) and the one finally used (rrmUlUsedMcs), which can be different in case of MCS downgrade.

 On an FDD carrier shared with LTE and actLteNrDssFastAlloc=true, after receiving the EN DSS UL SUBCARRIER RESOURCE ALLOCATION REQUEST message from CRM and during the DSS UL allocation transition phase, UL scheduler shall determine the maximum number of PRBs can be used as below:

#### 2.3.5.5.0-4 HARQ Processing HARQ Handling of PUSCH Retransmissions (ID: 11152388)

**2.3.5.5.0-4.0-4**  (ID: `12811611`)

HARQ Processing Scheduling of a Re-transmission:

  * On an FDD carrier shared with LTE and actLteNrDssFastAlloc=true, after receiving the EN DSS UL SUBCARRIER RESOURCE ALLOCATION REQUEST message from CRM and during the DSS UL allocation transition phase, UL scheduler shall decide whether a retransmission can be scheduled or not as following:

\- Otherwise, the retransmission is scheduled as usual.

[End PR863094]

## 2.3.9 UL L1-L2 timing Requirement

### 2.3.9.0-6 UL L1-L2 timing Requirement Statistics of Late UL Messages Sent to L1 or UL response received from L1 at L2-PS: L2-PS must maintain late UL data receive request sent to L1 and total UL data receive request sent to L1 over a period of second. These statistics will be printed using VIP logs if there exists any late message during 1 second observation period. These counters are maintained for each message mentioned in 12710896: Time Stamping of L2-PS UL receive request messages to L1 . L2-PS must maintain late UL response messages received from L1 and total UL response messages received from L1 over a period of second. These statistics will be printed using VIP logs if there exists any late message during 1 second observation period. These counters are maintained for each message mentioned in 12710887: L1-L2 UL Receive Response message Timing . (ID: 12712334)
