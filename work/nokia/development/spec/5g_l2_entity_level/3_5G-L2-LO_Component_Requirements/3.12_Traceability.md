# 3.12 Traceability

## 3.12.1 L2 trace

### 3.12.1.0-2 L2 trace Genral Description (ID: 11219175)

**3.12.1.0-2.0-2**  (ID: `11219187`)

L2 trace
L2 trace functionality provides insight of L2 internal functionality and L2 interfaces. For 5G-L2-LO this means mainly MAC user plane protocol.

L2 trace shall support Streaming and Snapshot -modes.

Streaming mode shall stream trace information to external server by IP/UDP messages. Streaming mode shall be activated and deactivated by user request according to 5G\_L1L2\_FLOW\_769\_replaced([11205620](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BgmKwfvREe-AqvopbP1qhQ)).

Snapshot mode shall store trace information into internal file system. Snapshot mode shall be active continuously and it shall store information in a ring buffer manner. I.e data shall be saved into limited storage and when storage is full the old information shall be discarder to make room for new. Snapshot mode shall follow the functionality define in 5G\_L1L2\_FLOW\_762\_replaced([11205475](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BgljuPvREe-AqvopbP1qhQ)).

[Before CNI-115810]5G-L2-LO shall support L2 traces independently from Sherpa trace (5G\_L2\_2013\_replaced([11246731](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3CE2GPvTEe-AqvopbP1qhQ))).[CNI-115810][End CNI-115810]

L2 trace shall have dynamic content on item level. This means that a filtering parameter shall be set to define the trace items that are included (into Stream message or Snapshot file). The trace items shall have fixed size and structure.

### 3.12.1.3 Trace filtering

**3.12.1.3.0-1**  (ID: `11219598`)

Trace filtering
L2 trace shall support filtering on object level and on item level.

Filtering object is cell and it shall identified by list of parameter physCellId in trace start request L2LogDu\_StreamActivationtReq. 5G-L2-LO shall support up to eight cell filter list. Filter object list modification is not supported. In order to modify filter list the ongoing measurement must be stopped and new one started.

Trace items shall be activated / deactivated by R&D parameters that defines the filtering bitmap. See 5G\_L2\_2086\_replaced([11220240](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_W5R_vTEe-AqvopbP1qhQ)) and 5G\_L2\_2087\_replaced([11220246](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_W5SPvTEe-AqvopbP1qhQ)), separate parameters for Tx (items that are related to DL) and Rx (items that are related to UL) . Default filter state (ACT/DEACT)  is stated in the item table in 5G\_L2\_2058\_replaced([11219197](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_QLk_vTEe-AqvopbP1qhQ)).

Note that filtering R&D parameters are common to both stream and snapshot. Filtering parameters shall be read at the streaming start. Snapshot filtering shall be updated dynamically run time as it is collected all the time.

Note that modified trace item filter is active until next component restart.

Filtering bitmap mapping to trace items left for implementation. I.e. the exact bit to item mapping must be checked from source code.

### 3.12.1.4 Streaming mode

**3.12.1.4.0-1**  (ID: `11219610`)

Streaming mode
5G-L2-LO shall clear all stream messages at start.

5G-L2-LO shall add trace items into the stream message as long as there is free space. This includes items from different users, carriers and TTIs. Stream messages shall not exceed standard Ethernet frame size (1500 bytes). 5G-L2-LO shall transmit stream message immediately when it gets full.

At the streaming trace stop all stream messages shall be flushed out.

All sub-components shall send separate stream messages. If there are CPU several cores for same sub-component (e.g. L2-LO-DL-PDU) they also shall send independent stream messages.

Rationale:

Trace message sending is not tied to TTI boundary because L2-LO does not currently know the end of TTI.
