#### 3.6.1.2.0-3 Received RLC data volume Determine received amount of UL data per logical channel (ID: 11216716)

**3.6.1.2.0-3.0-2**  (ID: `11216727`)

Received RLC data volume For informing 5G-L2-PS, 5G-L2-LO shall sum up the amount of data received for a logical channel in the 5G-RLC PDUs of a 5G-MAC PDU (see 5G_L2_1997_replaced([11216768](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_AT-fvTEe-AqvopbP1qhQ))). For that, 5G-L2-LO shall consider 5G-RLC data PDUs as well as 5G-RLC control PDUs (STATUS PDU). Any AMD or UMD PDU header and AMD or UMD PDU segment header shall be excluded, i.e., 5G-L2-LO shall subtract the size of the AMD or UMD PDU (segment) header from the size of the 5G-RLC data PDU. Note, TMD PDUs do not contain any 5G-RLC PDU header.

The length of an AMD PDU header is three bytes.

The length of an AMD PDU segment header is five bytes.

Based on the latest agreement from 3GPP, for RLC PDU header size, SRB only requires 12 bits, therefore the below header size is updated for SRB:

  * For SRB, the length of an AMD PDU header is two bytes.
  * For SRB, the length of an AMD PDU segment header is four bytes.



UMD PDU header length has three options. The shortest one byte header is used for complete PDUs. The second option of two byte header is used for a first segment of segmented PDU. The longest option of four bytes header is used for other segments of segmented PDUs.

[CB006112-A]

When the rlcUmDrbSnLength is set to 6 bits in the drbAddReqLo structure which is contained in the LoUser_UserSetupReq and LoUser_BearerSetupReq message, the below header size is updated for RLC UM mode DRB.

 The length of an UMD PDU header with 6 bits SN and without SO is one byte

 The length of an UMD PDU header with 6 bits SN and with SO is three bytes

where:

rlcUmDrbSnLength is defined in 5G_L2_IF_2544_replaced([10983749](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2V7EFfu6Ee-AqvopbP1qhQ)).

[End CB006112-A]

Different PDU types

The SI field of a AMD/UMD PDU (5G-RLC data PDU) indicates whether the AMD/UMD PDU header contains Segmentation Offset or not (if SI indicates that PDU contains all SDU bytes or the first segment of SDU bytes, then SO is not included in the header).

The D/C field of a 5G-RLC AMD PDU indicates whether the PDU is a 5G-RLC AMD data PDU (D/C field set to 1) or 5G-RLC AMD control PDU (D/C field set to 0).

For 3GPP 5G NR, see 5G_UP_IF_3348_replaced([11529607](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tWMuN_v1Ee-AqvopbP1qhQ)) (38.322, 6.2.2 Formats) and subclauses.

## 3.6.2 MAC

### 3.6.2.1 5G-MAC PDU Disassembling

#### 3.6.2.1.0-2 5G-MAC PDU Disassembling MAC PDU demultiplexing (ID: 11216782)

**3.6.2.1.0-2.0-5**  (ID: `13463068`)

5G-MAC PDU Disassembling **MAC PDU demultiplexing**

Upon reception of uplink transport block from 5G-L1-UL by message _UlData_PuschReceiveRespLo, 5G-L2-LO shall_

  * copy the _harqProcessIndex & _absoluteHarqProcessIndex IEs received from _UlData_PuschReceiveRespLo message into the_ LoCtrl_UlMacPduReceiveInd message. L2-PS needs those IEs to associate the MAC CE with the HARQ process.****
  * demultiplex the 5G-MAC PDU. Note that one MAC-PDU maps directly to one Transport Block.



MAC PDU consist of consecutive MAC sub-PDUs. MAC sub-PDU can be one of following types:

  1. MAC data, where MAC sub-PDU consists of MAC subheader and MAC SDU.

  2. MAC control element, where MAC sub-PDU consists of MAC subheader and MAC CE.

  3. MAC padding, where MAC sub-PDU consists of MAC subheader and optional padding bytes.




MAC sub-PDUs are arranged in order shown above. I.e. first all data MAC sub-PDUs, etc.

5G-L2-LO shall detect the MAC sub-PDU type based on LCID, according to 5G_L2_2008_replaced([11215666](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-5mSvvTEe-AqvopbP1qhQ)). Also the same requirement defines the header structure.

From MAC data type sub-PDUs 5G-L2-LO shall extract the 5G-MAC SDUs and perform RLC layer processing according to 5G_L2_1982_replaced([11216621](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-_F3vvTEe-AqvopbP1qhQ)).

From MAC control element type sub-PDUs 5G-L2-LO shall extract the 5G-MAC CEs and provide to 5G-L2-PS as described below. The supported 5G-MAC CEs are defined in 5G_L2_9489_replaced([11215735](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-6NVfvTEe-AqvopbP1qhQ)).

5G-L2-LO shall handle unsupported MAC sub-PDUs according to 5G_L2_9486_replaced([11215824](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-6NX_vTEe-AqvopbP1qhQ)).

Any padding 5G-L2-LO shall ignore totally.

If 9 first bytes of transport block are zero, then 5G-L2-LO shall discard the whole transport block. _Rationale, this is for discarding zero content transport blocks that are caused by L1 false detection. LCH ID zero corresponds to 64bit CCCH data SDU and it is assumed that UE sends only one at a time._

If the transport block contains MAC sub-PDUs with _Reserved LCH IDs then 5G-L2-LO shall discard the whole transport block._ Rational, this means that TB is either corrupted or contains data with unknown size. In either case the demultiplexing cannot be performed in reliable manner.**

If the result of parsing transport block is any one of Mac Sdu has too many zero, Mac Sdu Length is zero, Mac Sdu length error , Mac Sdu has dirty data then 5G-L2-LO shall discard the whole transport block.

5G-L2-LO shall discard the whole transport block if rnti indicates this is a fake ue(rnti = 0xFFF1)

5G-L2-LO shall send a _LoCtrl_UlMacPduReceiveInd_ to L2-PS for every received transport block _(except if whole transport block was discarded) with the following content :_ **

  1. 5G-L2-LO shall determine the received amount of data per logical channel as described in 5G_L2_1990_replaced([11216716](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-_s7vvTEe-AqvopbP1qhQ)). 5G-L2-LO shall include the received data amount for all logical channels that fulfill both of the following conditions:

  2. The LCID field is > 0.

  3. The Length field is > 0.



If there is no such logical channel in the TB 5G-L2-LO shall set numOfBearers = 0 in the message.

2\. If the MAC PDU contained an UL Buffer Status Report (MAC CE), 5G-L2-LO shall handle it according to 5G_L2_9965_replaced([11216818](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_AUAPvTEe-AqvopbP1qhQ)).

  1. If the 5G-MAC PDU contains a C-RNTI MAC CE (RA Msg3) and the included C-RNTI belongs to an existing permanent 5GUE then 5G-L2-LO shall set the IE crnti of _LoCtrl_UlMacPduReceiveInd to the value included in the C-RNTI MAC CE. If the 5G-MAC PDU doesn't contain a C-RNTI MAC CE then the C-RNTI (IE rnti) carried in the_ UlData_PuschReceiveRespLo message shall be used in _LoCtrl_UlMacPduReceiveInd ._ **

#### 3.6.2.1.0-3 5G-MAC PDU Disassembling UL Buffer Status Report MAC CE handling (ID: 11216818)

  * If a short UL BSR is contained, 5G-L2-LO shall set the buffer size to 0 for all LCGs but the one indicated in the BSR. For the one indicated, a 5 bit BSR index is set.
  * If a long UL BSR is contained, 5G-L2-LO shall set 8 bit BSR for indicated LCG, see 5G_L2_9489_replaced([11215735](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-6NVfvTEe-AqvopbP1qhQ)), and set the buffer size to 0 for all other LCGs. [CAS-327226-C8Y3] If the long BSR contains no buffer size fields, for each LCG the buffer size shall be set to 0. 

If no BSR MAC CE is contained in the TB, then 5G-L2-LO shall set the 8 buffer sizes to an invalid value. [End CAS-327226-C8Y3]

  * [5GC000578-G] If a short truncated BSR is contained, 5G-L2-LO shall set the buffer size for the LCG indicated in the BSR. And for the LCG indicated, a 5 bit BSR index is set. For LCGs which are not indicated, the 5G-L2-LO shall set the 5 bit BSR index to an invalid value, meaning that no buffer status being reported to this LCGi, as described in 5G_UP_ALG_5131_replaced([11203806](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwg_wfvSEe-AqvopbP1qhQ)) handled at 5G-L2-PS.



Note: Short truncated BSR means more than one LCG has data but the UE can only report one LCG BSR, where short BSR means only one LCG has data and is reported.

  * [5GC000578-G] If a long truncated BSR is contained, the LCGs having data available or not is indicated via LCG indicator, as shown in 5G_L2_9489_replaced([11215735](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-6NVfvTEe-AqvopbP1qhQ)). The 5G-L2-LO shall set the buffer size to 0 for all LCGs indicated having no data available. And for LCGs having data available:



Note: Due to the padding space limitation, not all the LCGs having data available can be sent by the UE. As specified in TS38.321, section 6.1.3.1, the number of buffer status fields included is maximised, while not exceeding the number of padding bits. And as also noted, the number of the Buffer Size fields in the long Truncated BSR format can be zero. 

As shown in 5G_L2_1969_replaced([11215648](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-5mR_vTEe-AqvopbP1qhQ)), for long truncated BSR, the MAC subheader consists of four header fields R/F/LCID/L, where the header field L indicates the length of the corresponding variable-sized (MAC CE) long truncated BSR in bytes. As shown in 5G_L2_9489_replaced([11215735](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-6NVfvTEe-AqvopbP1qhQ)), for long truncated BSR, it has variable size and consists of a 8 bit (1 byte) field indicating the LCGs with data available and a buffer size field of 8 bits (1 byte) for LCGs being included/reported. Based on the above description, for LCGs having data available, the following should be conducted based on the size of long truncated BSR:

If (L is equal to 1 byte)

Then only LCG indicator being included in the long truncated BSR, and there is no buffer size fields included/indicated.

Note: As noted in TS38.321, the number of the Buffer Size fields in the Long BSR and Long Truncated BSR format can be zero.

Elseif (L is equal to 2 bytes)

Then there is 1 byte used for LCG indicator, and another 1 byte for one buffer size field. Therefore, 5G-L2-LO shall set the buffer size with 8 bit BSR index for the first LCG having data available in ascending order based on the LCGi.

Elseif (L is equal to 3 bytes)

Then there is 1 byte used for LCG indicator, and another 2 bytes for two buffer size fields. Therefore, 5G-L2-LO shall set the buffer size with 8 bit BSR index for the first two LCGs having data available in ascending order based on the LCGi.

......

......

......

Elseif (L is equal to 8 bytes)

Then there is 1 byte used for LCG indicator, and another 7 bytes for seven buffer size fields. Therefore, 5G-L2-LO shall set the buffer size with 8 bit BSR index for the first seven LCGs having data available in ascending order based on the LCGi.

Elseif (L is equal to 9 bytes)

Then there is an error, long truncated BSR would not include all 8 buffer size fields. If so, the long BSR should be used and reported by UE. Therefore, 5G-L2-LO shall set the 8 bit buffer sizes to an invalid value.

End

For LCGs which are not indicated/included or being truncated, the 5G-L2-LO shall set the 8 bit BSR index to an invalid value (as described in 5G_L2_IF_3473_replaced([10984729](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WCY1_u6Ee-AqvopbP1qhQ))), meaning that no buffer status being reported to this LCGi, as described in 5G_UP_ALG_5131_replaced([11203806](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwg_wfvSEe-AqvopbP1qhQ)) handled at 5G-L2-PS.
