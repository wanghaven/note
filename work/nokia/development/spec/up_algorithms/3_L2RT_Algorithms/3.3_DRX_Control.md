### 3.3.1.0-2 General Functions Support of long DRX cycle (ID: 11114199)

**3.3.1.0-2.0-1**  (ID: `11114202`)

General Functions
The MAC layer shall support DRX functionality with long DRX cycle if configured.

Note: the short DRX cycle if configured would not be supported in this feature.

Note: the long DRX cycle with long DRX cycle(>80ms) would not be supported in this feature.

Parameters for configuration of the DRX functionality includes: drx-onDurationTimer, drx-InactivityTimer, drx-HARQ-RTT-TimerDL, drx-HARQ-RTT-TimerUL, drx-LongCycle, drx-Retransmission-TimerDL, drx-Retransmission-TimerUL, drx-slotOffset etc.

Note: Only a single set of parameters is supported initially.

### 3.3.1.0-3 General Functions Support of DRX feature (ID: 11114206)

**3.3.1.0-3.0-1**  (ID: `11114209`)

General Functions
The DRX module in gNodeB provides function block to manage all UEs DRX active status while they are running in DRX mode(DRX is active). The DRX module update the UE DRX status should follow the procedure as defined in [3GPP-38.321], section 5.7. The scheduler would perform resource allocation for the UE or not based on the UE DRX status.

### 3.3.1.0-4 General Functions Exchange of DRX Configuration with control plane (ID: 11114213)

**3.3.1.0-4.0-1**  (ID: `11114216`)

General Functions
CP-RT will config the DRX Configuration for a UE at certain events like, e.g., Initial Context Setup, DRB modification.

C-Plane configures to U-Plane the DRX configuration for UE in PsUser\_UserSetupReq message. DRX function block in U-Plane will respond with the DRX Configuration Data for this UE.

In case of a UE release, DRX function block will release any UE specific DRX related data.

### 3.3.1.0-8 General Functions DRX interworking with SRS (ID: 11114252)

**3.3.1.0-8.0-1**  (ID: `11114254`)

General Functions
If L2-PS scheduled the aperiodic SRS(UL grant) for DRX UE while UE is not in DRX sleep status,  L2-PS shall send SRS receive request to L1-UL. Refer more to 5G\_UP\_ALG\_5057\_replaced([11201982](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwa5GvvSEe-AqvopbP1qhQ)).

* If the SRS measured beam age for the current refined beam is less than beamAgeThreshold and the beam gain from the latest SRS feedback for the current beam is greater than or equal to the beam gain when it was selected for the UE, 5G-L2-PS shall continue to use the same refined beam for the UE.

Note: Beam gain is IIR filtered power value of SRS measured RSSI value for the beam [See 5G\_UP\_ALG\_1999\_replaced([11112098](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kpS9avvSEe-AqvopbP1qhQ))].

* If the SRS measured beam age for the current refined beam is less than beamAgeThreshold and the beam gain from the latest SRS feedback for the current beam is not lesser by more than rdDrxRefinedBeamMargin [See 5G\_L2\_1263\_replaced([11184076](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27oP4_vTEe-AqvopbP1qhQ))] to the beam gain when it was selected for the UE, 5G-L2-PS shall continue to use the same refined beam for the UE.
* If another refined beam is detected (SRS measured) better than the current refined beam for the DRX UE during the active time, switch the UE to the new refined beam [See, 5G\_UP\_ALG\_2006\_replaced([11112105](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kpS9bPvSEe-AqvopbP1qhQ))].
* If the SRS measured beam age for the current refined beam is not less than beamAgeThreshold and another refined beam is detected (SRS measured) with beam gain greater than or equal to the current beam's gain when it was selected, 5G-L2-PS shall switch the UE to this newly detected refined beam.
* If the SRS measured beam age for the current refined beam is not less than beamAgeThreshold and another refined beam is detected (SRS measured) with beam gain not lesser by more than rdDrxRefinedBeamMargin [See 5G\_L2\_1263\_replaced([11184076](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27oP4_vTEe-AqvopbP1qhQ))]  to the current beam's gain when it was selected for the UE, 5G-L2-PS shall switch the UE to this newly detected refined beam. 5G-L2-PS shall also update this beam's selection gain to the same value as of current beam's selection gain.

In case beam gain for the current beam is lesser by more than rdDrxRefinedBeamMargin [See 5G\_L2\_1263\_replaced([11184076](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27oP4_vTEe-AqvopbP1qhQ))] to the beam gain when it was selected for the UE and/or SRS measured beam age for the current refined beam is more than beamAgeThreshold and no other suitable refined beam (SRS measured) beam is available, switch the DRX UE to the current best reported SSB beam. If no feedback for any SSB beam is available (beam age is greater than beamAgeThreshold) then switch the UE to the SSB beam corresponding to the current refined beam.

Note: beamAgeThreshold value is max of computed beam age threshold [See 5G\_UP\_ALG\_2254\_replaced([11114258](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kpjcE_vSEe-AqvopbP1qhQ))].

### 3.3.1.0-9 General Functions DRX interworking with beam management (ID: 11114258)

**3.3.1.0-9.0-1**  (ID: `11114262`)

General Functions
5G-L2-PS shall maintain the beam age information for an UE in DRX and keep it updated based on the latest beam report information. Beam age is the time (in ms) since the last beam report information is received for an UE.

5G-L2-PS shall compute the beam age threshold useing rdDrxBeamAgeThreshold [See 5G\_L2\_1255\_replaced([11184096](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27o24vvTEe-AqvopbP1qhQ))] for beam age comparision.

If the beam tracking is based on SSB (FR2 or FR1 without refined beam) :

If the number of SSB Beams <= 2, then the beam age threshold value shall be infinite

If the number of SSBs Beams > 2, then the beam age threshold value is

rdDrxBeamAgeThreshold \*64 / Number of SSB Beams.

If the beam tracking is based on refined beam (FR1) : The beam age threshold value is

rdDrxBeamAgeThreshold \*64 / Number of refined Beam.

Based on beam age and beam age threshold comparision, approprite DL and UL MCS selection for the UE is done on the current used beam to get the additional gain.

On top of the DRX parameters, UE active Time is maintained and corresponds to the capability for the UE to monitor the PDCCH.

Before the DRX configuration, the UE is in Active Time and monitors the PDCCH in each slot where search spaces are active.

After the DRX activation, the UE Active Time includes the time while

 drx-onDuration or drx-InactivityTimer is running (after receiving a new PDCCH after the DRX activation),

 drx-RetransmissionTimer[DL,UL] is running (pending HARQ process allowing retransmission),

 a Scheduling Request (SR) is sent on PUCCH and is pending.

 CBRA procedure is ongoing and not yet completed.

During the DRX configuration and after the delivery of the DRX configuration to the UE at the MAC layer, the gNB does not know when the DRX is activated in the UE. Nevertheless, the RRC processing delay defined by the 3GPP is the maximum time allowed to process the RRC message received and to be able to answer to the gNB with the new configuration. This delay is mainly 10 or 16ms according to the content of the RRC message sent to the UE.

Therefore, after the RRC processing delay without any grant sent during this time, the UE is in Active Time if there is a pending SR (sent on PUCCH) or if the drx-onDuration is running. During the RRC processing delay, if the gNB sends a DCI, it could be before or after the DRX activation and even during the time where the UE may discard the grant. If a positive SR is sent during the RRC processing delay, there is still an ambiguity regarding the DRX state (activated or not yet activated) even if the UE answers to the grant. After the RRC processing delay and without any RA procedure triggered by the UE, the new configuration is accepted by the UE and should be activated. Therefore, the gNB can considers that UE Active Time is managed and corresponds to the first occurrence of the two following conditions (Pending SR sent or drx-onDuration is running).

For the DRX deactivation, the gNB applies the DRX operation until the delivery of the RRC message by the MAC layer. Then the new setting is applied not later the RRC processing delay in the gNB if the MAC layer monitoring is activated (5G\_UP\_ALG\_17940\_replaced([11153286](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kslh8vvSEe-AqvopbP1qhQ))). Otherwise, it will be at the RRC reconfiguration complete indication form the C-Plane when the RRC layer monitoring is activated. The MAC layer monitoring reduces at the maximum the ambiguity related to the new configuration activation.

At the reception of the last part of the RLC-SDU of the RRC message with the poling bit set to 1, the UE has to add in its SRB1 buffer the RLC-Status PDU and to wait for an Uplink grant to send it until the first occurrence of Scheduling Request in order to request an uplink grant. The gNB needs to receive it before the end of the downlink RLC retransmission timer.

During the RRC processing delay, the UE has to:

 Validate the content of the RRC message. If the validation is not successful, the UE will trigger a RA procedure with a UL-CCCH (RRC reestablishment Request).

 Apply the new configuration. From the 3GPP (38.331), The UE must use the new configuration to send back the RRC message response to the gNB.

 Add the RRC reconfiguration complete in the SRB1 buffer for transmissions.

 Wait for the reception of UL grant or send on PUCCH a SR to get the expected UL grant.

HARQ RTT timer DL is one per DL HARQ process. It is defined as the minimum amount of NR-UNIT(s) before a DL assignment for HARQ retransmission is expected by the MAC entity. It will be started when the PDCCH indicates a DL transmission and the corresponding HARQ feeback on PUCCH or PUSCH is received as NACK.

DL DRX Retransmission timer shall be started if DL HARQ RTT timer is expired .

HARQ RTT timer UL is one per UL HARQ process. It defines the minimum amount of NR-UNIT(s) before a UL HARQ retransmission grant is expected by the MAC entity. It will be started when the UE sends a UL transmission and the UE is DRX Active at that time.

UL DRX Retransmission timer shall be started if UL HARQ RTT timer is expired and the data of the corresponding HARQ process was decoded (ACK/NACK).

If NRBTS/NRCELL.actUlSlotAgg set to ‘Enabled\_VoNr’ and  NRBTS/NRCELL.actDrxSleepOff is set to 'Disable' then gNB-DU shall check above conditions to possibly Start the UL HARQ RTT timer after the end of the first transmission only within a bundle of the PUSCH transmission. Note- Retransmission decision by gNB-DU shall be taken after the all transmission within a PUSCH bundle is received but the start boundary of retransmission window will be started after the expiry of UL HARQ RTT timer. For the PUSCH bundle,refer to 5G\_UP\_ALG\_19635\_replaced([11104383](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kn8SgPvSEe-AqvopbP1qhQ)). In the case, when NRCELL.actUlSlotAgg set to ‘Enabled\_VoNr’ and NRCELL.actDrxSleepOff is  set to 'Enable' then a special DRX setting will applied where drx-On duration timer will be same as drx-LongCycle length to avoid UE from going to sleep.

Purpose of HARQ RTT timer DL and HARQ RTT timer UL is to define the slot when the next possible retransmission can take place earliest.Similar to any dynamic grant for a UE as explained above, the HARQ RTT timer UL is started one per UL HARQ process for a UE for which a Configured Grant (CG) Type2 in Uplink has been activated for periodic UL PUSCH transmission. HARQ processes corresponding to CG are the configured number of HARQ processes for CG-Type2 configuration given by 5G\_UP\_ALG\_14592\_replaced([11172878](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuOgvfvSEe-AqvopbP1qhQ)). The HARQ RTT timer UL shall be started when the UE sends a UL transmission corresponding to an initial transmission of a CG-Type2 (configured grant) or when the UE sends a UL transmission corresponding to a retransmission of a CG-Type2 scheduled by a PDCCH addressed to CS-RNTI (i.e. dynamic grant that should be transmitted in DRX Active time).

Similarly, for the UE with an activated CG-Type2, the UL DRX Retransmission timer shall be started if UL HARQ RTT timer, explained above, has expired and the data of the corresponding HARQ process was decoded as ACK/NACK. The gNB may send, during this time, a UL DCI indicating a need for UL Retransmission by a PDCCH scrambled with CS-RNTI for the corresponding HARQ process.

if NRBTS/NRCELL.actDrxRetransTimerSupport is equal to False, Additionally, in order to serve potential RLC Ack transmission (as it could happen that the only traffic UE experiences is over CG), initial transmission in DL direction shall be allowed when UL DRX Retransmission timer corresponding to a CG process is running as follows:

- When the PDCCH for a CG HARQ restransmission is scheduled, DL transmission shall not be allowed.

- DL scheduling is not allowed in a slot N if N+k1 slot is configured with P-CSI and the only reason for which UE is DRX active is UL DRX Retransmission timer running

o Note: This requirement doesn’t introduces any change in handling of the initial DL transmission during the running time of DL DRX Retransmission timer.

In case of a negative feedback 'DTX' of a DL re-transmission is received, which means UE did not decode the PDCCH sucessfully, UE will not start the DRX RTT timer for the HARQ.  gNB shall stop scheduling the HARQ unless UE onDuration timer or inactivityTimer is running.

DRX retransmission timer is one per HARQ process. The purpose of DRX retransmission timer is to spawn a window within which the UE listens on the PDCCH for a retransmission. Refer more in [3GPP-38.321, section 5.7].

DRX Retransmission timer DL defines the maximum number of consecutive NR-UNIT(s) until a DL retransmission is received. DRX retransmission timer DL shall be started and stopped as per 5G\_UP\_ALG\_23617\_replaced([11114476](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kpkqP_vSEe-AqvopbP1qhQ)). If it is running, UE is DRX Active.

DRX Retransmission timer UL defines the maximum number of consecutive NR-UNIT(s) until a grant for UL retransmission is received. DRX Retransmission timer UL shall be started and stopped as per 5G\_UP\_ALG\_23615\_replaced([11114469](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kpkqPfvSEe-AqvopbP1qhQ)). If it is running, UE is DRX Active[PR846142] but, in case of PUSCH is CRC NOK, DL scheduler is not informed.[End PR846142]

**if NRBTS/NRCELL.actDrxRetransTimerSupport is equal to False, HARQ process retransmission would be scheduled while UE is not sleep. But to keep simply in implementation, while UE DRX onDuration timer and inactivity timer are both off, the HARQ process can retransmit only when its DRX retransmission timer is running,** and this is also applied to FR1 CA.\*\*\*\*

**if NRBTS/NRCELL.actDrxRetransTimerSupport is equal to True, then gNB-DU shall allow transmission when only DRX retransmission timer is running (i.e. DRX inactivity timer and DRX onDuration timer have already expired) as per following conditions:**

- DL HARQ retransmission timer is running, only DL retransmission is allowed.

- HARQ retransmission timer (started when PUSCH was successfully decoded) is running in UL, then

- new transmission in DL is allowed.

- new Transmission in UL is allowed.

- retransmission in UL is allowed (for some other HARQ process).

- HARQ retransmission timer (started when PUSCH was not successfully decoded) is running in UL, then

- retransmission in UL is allowed (When retransmission is still possible).

Note- If NRBTS/NRCELL.actUlSlotAgg set to ‘Enabled\_VoNr’ and  NRBTS/NRCELL.actDrxSleepOff is set to 'Disable' then retransmission decision by gNB-DU shall be taken after the all transmission within a PUSCH bundle is received. For the PUSCH bundle refer to 5G\_UP\_ALG\_19635\_replaced([11104383](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kn8SgPvSEe-AqvopbP1qhQ)).

SCell(s) DL scheduler maintains the DRX inactivity timer status for this SCell, and informs the DRX inactivity timer status to P(S)cell DL scheduler only when the status is changed, in order to reduce the message overhead. When SCell is released, SCell(s) DL scheduler shall inform the DRX inactivity timer status OFF to P(S)cell DL scheduler if its previous DRX inactivity timer status is ON.

P(S)cell DL scheduler works as a DRX master and maintains an overall DRX inactivity timer status based on the input from all SCell(s) DL scheduler and P(S)Cell own DRX inactivity timer status.

To synchronize overall DRX inactivity timer status, **the P(S)Cell DL scheduler shall broadcast the overall DRX inactivity timer status to all SCell(s) DL 5G-L2-PS and UL 5G-L2-PS whenever the overall DRX inactivity timer status changed from OFF to ON, or from ON to OFF.**

The inactivity timer status informed from SCell to PCell takes at least 1 slot (even more slot in inter-gNB CA case), and PCell broadcast overall DRX inactivity timer status to all SCell takes at least another 1 slot (even more slot in inter-gNB CA case).

If SCell has inactivity timer expired later than PCell, it may happen that UE enters DRX sleep status 2 slots earlier than gNB and any scheduling in these 2 slots are DTX. This is accepted as a restriction, but gNB shall avoid furher mismatch by two kinds of rollback:

a) SCell rollback: when SCell(s) DL 5G-L2-PS and UL 5G-L2-PS received overall DRX inactivity timer status OFF, DL 5G-L2-PS and UL 5G-L2-PS shall reset local and remote DRX inactivity timer status to OFF immediately.

b) P(S)Cell rollback: when P(S)Cell find in a past time all inactivity timer is OFF for P(S)Cell and all SCells, P(S)Cell DL scheduler shall broadcast the overall DRX inactivity timer status OFF to all SCell(s) DL 5G-L2-PS and UL 5G-L2-PS.

Note: In case of more than one SCell, the P(S)Cell shall collect the inputs from all SCell(s), calculates the inactivity timer status considering P(S)Cell and all SCells and broadcasts the information to all SCell(s).

Note: it's suggest to configure drxInactivityTimer equal to or larger than 3ms when FR1 CA is activated in order to handle the delay of DRX inactivity status transfer from SCell to PCell, otherwise more DTX may happen because PCell DRX inactivity timer status may be togged when SCell DRX inactivity timer status is arrived so that P(S)Cell rollback does not work as expect.

**[Before CNI-137616-B] rd parameter rdEnable6137Fr1CaDrxSolution is set to true by default.** [End CNI-137616-B]\*\*\*\*
