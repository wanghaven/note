**3.28.1.0-1.0-1.0-4**  (ID: `11172763`)

CG resource pool management
At the cell setup procedure **[CB013388-C]or the cell reconfiguration procedure** [End CB013388-C], the 5G-L2-PS checks that the CG resources Blocks allows to handle the minimum PUSCH transmission in each uplink BWP area when the CG resources are within the same area. The constraints are detailed hereafter.\*\*\*\*

If the check is failed, the 5G-L2-PS rejects the cell setup procedure **[CB013388-C]or cell reconfiguration procedure**[End CB013388-C] by sending a negative acknowledgment to the C-Plane with additional information (see 5G\_UP\_ALG\_14786\_replaced([11172769](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuN5rfvSEe-AqvopbP1qhQ))).\*\*\*\*

The minimal number of consecutive Resource Blocks (RB) for a PUSCH transmission in the BWP is defined by rrmMinSupportedPuschPrb (5G\_UP\_ALG\_31776\_replaced([11199457](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kwNdvvvSEe-AqvopbP1qhQ))).

The rdMinNumPrbsPusch check in all uplink BWPs, must be done for a DCI Format 0\_0 configured with a common search space. Therefore, the location of the minimum RB for PUSCH available must be within the RB start of the uplink BWP to check and the number of RB of the initial uplink BWP minus the rrmMinSupportedPuschPrb, **but only in case that CG pool (see 5G\_UP\_ALG\_14663\_replaced([11172788](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuOgsPvSEe-AqvopbP1qhQ))) overlaps** at least partially this PRB range.

In TDD mode, in addition to the check above, in case of initial uplink bandwidth part, if CG pool (see 5G\_UP\_ALG\_14663\_replaced([11172788](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuOgsPvSEe-AqvopbP1qhQ))) overlaps at least partially the PRB range between lower PRB and higher PRB minus the rdMinNumPrbsPusch for MSG3 allocation (see in 5G\_UP\_ALG\_3419\_replaced([11152630](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kshQifvSEe-AqvopbP1qhQ))), rrmMinSupportedPuschPrb PRBs must be additionally searched between lower PRB and higher PRB (see in 5G\_UP\_ALG\_3419\_replaced([11152630](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kshQifvSEe-AqvopbP1qhQ))) minus the rrmMinSupportedPuschPrb.

With the time domain allocation and the smallest MCS, the rrmMinSupportedPuschPrb must guarantee the presence of at least one byte of data on top of the MAC header and the RLC-AM header for SRB1.

In order to find, at least, one group of consecutive PUSCH RBs in each uplink BWP, the 5G-L2-PS invalidates the RBs overlapping each uplink BWP and corresponding to :

 The blanked RBs.

 The dedicated PUCCH (largest size, if dynamic dedicated PUCCH is activated).

 The common PUCCH for SA or SA + NSA cell.

 The PRACH.

When the PUSCH multiplexing in PRACH slot is allowed, the minimal number of consecutive Resource Blocks (RB) for a PUSCH transmission in each BWP area must not overlap the PRACH. Otherwise, the 5G-L2-PS does not take into account the PRACH to validate the minimal number of consecutive Resource Blocks (RB) for a PUSCH transmission in each uplink BWP area.

 The Configured Grant RBs.

### 3.28.1.0-2 CG resource pool management CG pool initialization (ID: 11172788)

**3.28.1.0-2.0-2**  (ID: `11172797`)

CG resource pool management
At the cell setup procedure **[CB013388-C]or the cell reconfigure procedure**[End CB013388-C] the UL CG Type2 feature is activated (NRCELL.actUlCgType2 set to true), the 5G-L2-PS creates a two dimensions map of CG resources in time and frequency domains for the cell.\*\*\*\*

In frequency domain, the NIDD parameters NRCELL.ulCgPrbStartOffset and NRCELL.ulCgMaxNumPrbs, define respectively the starting Common Resource Block (CRB) and the number of RB allocated to the CG pool. The size of the resource in the CG pool is a multiple of 4 RBs.

In time domain, the map is in unit of slots and ranges to the number of slots over 8 radio frames (8 x 10 x 2µ slots) (i.e. maximum value of NRBTS/NRULCG\_PROFILE.periodicity in slots) . The DL slots or special slots are set as invalid.

It is not expected that the PRACH/PUCCH (common and dedicated) regions are overlapping the CG resource blocks (check/validation performed in 5G\_CP\_ARC\_ALG\_8363\_replaced([11280771](https://dn-prod.ext.net.nokia.com/rm/resources/BI_TpYUHvvcEe-AqvopbP1qhQ))).

**[CB013388-C]If CG pool re-initialization is triggered by cell reconfiguration (after validation in [11172745](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuN5qPvSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)) that with change of   
1. NRCELL.actUlCgType2 changed from false to true, or  
2. (NRCELL.ulConfiguredGrantType2.ulCgMaxNumPrbs or NRCELL.ulConfiguredGrantType2.ulCgPrbStartOffset) changed when NRCELL.actUlCgType2 is true,**

then gNB-DU shall re-initialize the CG pool with new received values after all UEs are CG deactivated.   
Note: In case additional reconfiguration(s) received during transient phase of UE CG deactivation, the last value takes effect after all UEs are CG deactivated.**[End CB013388-C]**

The available Uplink RBs of the CG pool are set to available.

### 3.28.1.0-3 CG resource pool management CG Link Adaptation (MCS/RB determination) (ID: 11172802)

**3.28.1.0-3.0-5**  (ID: `11172847`)

CG resource pool management
11172847:
CG resource pool management
To determine the MCS and number of RBs for CG, the 5G-L2-PS runs the following Link Adaptation (LA) hereafter.
It uses the corrected average SINR  (computed with OLLA and ILLA) as requested via  5G\_UP\_ALG\_15178\_replaced([11131181](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krBbt_vSEe-AqvopbP1qhQ)), and the expected rank (see 5G\_UP\_ALG\_3226\_replaced([11139957](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krq79_vSEe-AqvopbP1qhQ))) (i.e. a change in any of corrected average SINR or UL rank shall trigger the evaluation of the CG activation). The legacy PMI selection (5G\_UP\_ALG\_3231\_replaced([11139991](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krq7_fvSEe-AqvopbP1qhQ))) is applied for CG.
To determine the Transport Block Size (TBS) with MCS and the number of layers, the 5G-L2-PS uses:
 A number of symbols corrsponding to the smallest length of PUSCH allowed for regular scheduling (not for the RAR constraints) reported in the PUSCH TDRA list to the C-Plane during the cell setup procedure **[CB013388-C]or the cell reconfigure procedure** [End CB013388-C].***\*
 The CDM group without data is selected whatever the MCS selected.
 The number of RBs (nRbCgUl), initialized to the minimal allocation allowed (4RBs) and increased until the maximum number of RB on the CG pool by setp of 4 RB according the capability to find a MCS allowing an enough large TBS (see hereafter).
If the number of RBs does not allow to find the expected TBS after the MCS scanning, the 5g-L2-PS increases the number of RBs (nRbCgUl) by adding 4 RBs to attempt again to find a MCS to handle the expected TBS until the Link Adaptation procedure is completed sucessfully. If the CG allocation is not possible when the nRbCgUl reached the max of RBs allowed for the expected CG profile (cgProfileRbMax corrsponding to the NRBTS.ulCgProfile-maxPrbNum), the CG UE is not activated or the CG deactivation is triggered in case of CG modification failure (5G\_UP\_ALG\_15651\_replaced([11172853](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuOgufvSEe-AqvopbP1qhQ))).
When a combination of MCS, symbols, RB allows to handle the expected ulCgProfile.grantedPacketSize, the 5G-L2-PS rescales the averaged SINR (rrmCorrAvgSinr) according to the number of RBs (see 5G\_UP\_ALG\_3090\_replaced([11131515](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krDQ5fvSEe-AqvopbP1qhQ))), removes/subtracts the static SINR margin for CG (cgSinrOffset* \*is not exceeding 3dB and is computed according to the target MCS computed and the largest MCS allowed for the MCS table used by the UE) and the CG BLER loop conrol (cgOlla). Additionally an SINR offset for HR and CG (i.e., rrmSinrOffset defined in 5G\_UP\_ALG\_15846\_replaced([11130560](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krANk_vSEe-AqvopbP1qhQ))) is subtracted also if UE is in CG-HR(Configured Grant-High Reliability) mode.**
The 5G-L2-PS determines whether SINR (5G\_UP\_ALG\_3100\_replaced([11132185](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krFGGvvSEe-AqvopbP1qhQ))) associated to the target MCS (allowing the expected TBS) is compatible with this rescaled and corrected SINR, verifies that the effective coding rate is not higher than the one allowed (rdMaxPuschEffectiveCodeRate) and finds the highest MCS allowed **and compatible with the coding rate rule. If UE is in CG-BR mode SINR is determined by the BR related Look-up table (e.g., 10% BLER) and also depended on whether 64qam/256qam table or 64qamLowSE table is used. If UE is in CG-HR mode SINR is determined by the HR related Look-up table (e.g., 1% BLER) and also depended on whether 64qam table or 64qamLowSE table is used.**
The additional SINR offset (cgSinrOffset **is not exceeding 3dB and is computed according to the target MCS computed and the largest MCS allowed for the MCS table used by the UE) and SINR offset for HR and CG (i.e., rrmSinrOffset defined in 5G\_UP\_ALG\_15846\_replaced([11130560](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krANk_vSEe-AqvopbP1qhQ))) are used to reduce the need to update the UE with a new set of LA parameters (number of RBs and MCS).**
A BLER loop control for CG is initialized (cgOlla in the figure hereafter) to 0 at the CG activation procedure and updated according to the CG HARQ feedbacks indicating an acknowledgement or a negative one over a monitoring period (5G\_UP\_ALG\_15651\_replaced([11172853](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuOgufvSEe-AqvopbP1qhQ))).
In case of transmission of CG-PUSCH, if UE is configured with LowSE MCS table or 64QAM MCS table and the UE works in CG-HR (Configured Grant-High Reliability) mode, the cgMCS shall be restricted in similar manner as defined in 5G\_UP\_ALG\_17281\_replaced([11131699](https://dn-prod.ext.net.nokia.com/rm/resources/BI_krD38_vSEe-AqvopbP1qhQ)) for rrmUlProposedMcs if the high reliability LA mode is configured.

### 3.28.1.0-4 CG resource pool management CG HARQ feedback monitoring and actions (ID: 11172853)

**3.28.1.0-4.0-3**  (ID: `12725446`)

CG resource pool management
12725446:
CG resource pool management
The 5G-L2-PS monitors the CG HARQ feedbacks over a CG monitoring period (cgMonitoringPeriod) defined by a set of CG HARQ feedbacks (nCgAckNackInd) which are indicating successful or negative acknowledgement for initial CG transmission.
This monitoring  period is restarted at each time the CG modification is triggered or when the end of the CG monitoring period is reached.
The CG modification could be triggered when one of the following conditions is met:
 Over a CG monitoring period (cgMonitoringPeriod), All CG HARQ feedbacks (nCgHarqAckNackInd) of initial CG transmission are acknowledged (nAckInd) and the Link Adaptation output (cgMcsSel, nRbCgUlSel) is different from the one configured for the CG.
 The number of negative acknowledgement (nNackInd) received for initial CG HARQ transmission is greater or equal to a threshold (cgNackThreshold) and the Link Adaptation output (cgMcsSel, nRbCgUlSel) is different from the one configured for the CG.
**[Before PR837473-25R3] ; When the CG LA output is not able to provide a valid MCS and number of RBs, the CG deactivation is triggered.**[End PR837473-25R3]***\**\*[PR837473-25R3] ;  When the CG LA output, triggered by a negative acknowledgment of a CG HARQ feedback, is not able to provide a valid MCS and number of RBs, the CG deactivation is triggered. Otherwise, when the CG LA output, triggered by consecutive successful acknowledgments of a CG HARQ feedbacks, is not able to provide a valid combination of MCS and RBs due to SINR degradation after the dynamic grants, the U-Plane maintains the current MCS and number of RBS as the periodic CG transmissions are successfully decoded by the gNB.**[End PR837473-25R3]***\*
The CG LA procedure for CG modification is the same as the one triggered for CG activation (see 5G\_UP\_ALG\_14664\_replaced([11172802](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuOgs_vSEe-AqvopbP1qhQ))) with the current value of the BLER loop control (cgOlla). It corresponds to "CG Link adaptation output" block in the diagram below.

### 3.28.2 CG configuration/deconfiguration procedure

#### 3.28.2.0-1 CG configuration/deconfiguration procedure Configuration parameters which are calculated by gNB DU (ID: 11172878)

**3.28.2.0-1.0-2**  (ID: `12429144`)

CG configuration/deconfiguration procedure
As part of CG configuration to the UE (see 5G\_UP\_5893\_replaced([11348488](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ubWBFPvkEe-AqvopbP1qhQ))), gNB-DU shall derive PuschConfiguration -> configuredGrantConfig -> configuredGrantTimer and nrofHARQ-Processes to be provided in RRC setup or RRC reconfiguration to the UE:

*-configuredGrantTimer represents, de facto, the time, expressed in number of periodicities, since the last transmission or retransmission of the same Harq process after which UE will implicitly assume that the PUSCH was correctly decoded, if no other explicit retransmission was received during this period.*

* consequently, gNB makes sure that its value is computed such that the round trip delay corresponding to a CG transmission is supported by the network such that that PUSCH status is available in the packet scheduler early enough such that a retransmission could be scheduled before timer expiration.

*- configuredGrantTimer =roundup(harqRTTDelay\_slots/periodicity)*

where:

* periodicity as configured (see 5G\_UP\_5893\_replaced([11348488](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ubWBFPvkEe-AqvopbP1qhQ)))
* harqRTTDelay\_slots represent the maximum Harq round trip delay in slots for the current cell technology (FDD/TDD). It shall be computed with respect (sum of) for:

- UL scheduler advance time to the starting boundary of the same DL slot number on the air interface

- Maximum k2 configured for PUSCH (excluding the k2 for msg3 scheduled from RAR and k2aCsi computed for A-CSI scheduling in TDD)

- Maximum time for CRC reception for PUSCH from 5G-L1-UL relative to the starting boundary of the same DL slot on the air interface.

- Additionally, a fixed value of 3 slots is considered for TDD, when buffered UL is enabled (NRBTS.actTddPrePooling set to true [Before CNI-136563] OR when R&D parameter rdBufferedUlMode (see 5G\_L2\_1232\_replaced([11188091](https://dn-prod.ext.net.nokia.com/rm/resources/BI_28GJ4fvTEe-AqvopbP1qhQ))) is set to “2: Test”)[End CNI-136563].

*-nrofHARQ-Processes represents the number of Harq processes dedicated to CG and it must be at least* configuredGrantTimer (otherwise same Harq process is never implicitly acknowledged by the UE so UE never initiates a new transmission for the process). In addition it must be set such that the possibility of running into Harq stall (considering the remaining processes allocated to dynamic grant) is to be minimized. It shall be consequently computed as:\*\*

*- nrofHARQ-Processes = 1+*configuredGrantTimer\*\*

## 3.28.4 CG state machine

### 3.28.4.0-1 CG state machine CG state machine principles (ID: 11172972)

**3.28.4.0-1.0-1**  (ID: `11172976`)

CG state machine
- A state machine instance is initialized per UE whenever a CG configuration is received.

- For a UE,  CG state machine is not meaningful in case the CG is deconfigured.

- When UE enters a CG associated state, it could be assigned to a request class that is used to trigger UE's addition to the UL Pre-Scheduler. Packet scheduler shall further decide the effective priority class that shall be considered by the TD scheduler.

- The following states: CG-Activated, CG-WaitForDeactivationOrder, CG-WaitForDeactivationConfirmScheduling, CG-WaitForDeactivationConfirm [CB008124-CA], CG-WaitForModification, CG-WaitForModificationConfirm [End CB008124-CA] shall be called 'firing' states in a sense that, from UE perspective, UE is CG activated state and CG PUSCH shall be fired by the UE in a CG occasion.

- Per state, multiple events are considered. An event could have a specific component (only for this state) and a common component (common for all the CG firing states).

- For a given event, when triggered, the specific event component, if defined, shall be triggered/executed first then the common event component, if defined, shall be triggered/executed after.

o Example:

 TD-SCH event, when triggered:

 specific TD-SCH event shall be triggered/executed first, from the state machine definition

 common TD-SCH event shall be triggered/executed after, from the multi state common definition

- The documented events per state are introduced with the scope of illustrating the logic in the specifications scope and they have a logical role only. Implementation is free to select a different approach if it fits better to the implementation scope.
