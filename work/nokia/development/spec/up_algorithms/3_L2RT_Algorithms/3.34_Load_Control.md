Overload control detection
Enhanced Corrective overload control detection

---

**3.34.2.0-3.0-2.0-1**  (ID: `13249247`)

Overload control detection
The gNB-DU shall adjust to the scheduler processing capacity variation in every DL/UL scheduling slot by performing the following actions:

* For every DL/UL scheduler, at the start of slot scheduling, the gNB-DU shall calculate the processing capacity correction based on the actual and/or predicted slot-specific variations of the scheduler. The calculation shall consider the following inputs:
* Any late start for the scheduling.
* Any adaptation required due to predicted fluctuation in non-real-time processing requirements.
* For deployments with multi-schedulers per core, any under- or over-utilization of the provided CPU processing budget by schedulers that ran earlier in the slot.
* The inherent slot-specific variations in the processing requirements of the scheduler, namely the patternRatio mechanism (see EFS section [12644897](https://dn-prod.ext.net.nokia.com/rm/resources/BI_JA6m8kKxEfCH-cG2YrVgkA?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9) for more details).
* For deployments with multi-schedulers per core, the gNB-DU shall distribute the slot-specific processing capacity variations among all schedulers in proportion to their allocated budget share.
* The gNB-DU shall use the computed processing capacity correction (w.r.t. allocated budget provided by adaptive scheduler budget and load balancer) to adapt the maximum candidate set 1 (CS1) size proposed by the preventive overload control for the scheduler, if needed.

**3.34.2.0-6.0-2.0-1**  (ID: `11216081`)

RT processing load severity metric per DL/UL scheduler

---

**3.34.2.0-6.0-2.0-1.0-1**  (ID: `11216088`)

Overload control detection
gNB DU shall support the calculation of a Real Time (RT) processing load severity metric to detect high RT processing load per DL or UL scheduler. This severity metric shall make use of the reduction in the maximum CS1 list size as one of the inputs, which is the outcome of the corrective overload control function (5G\_UP\_ALG\_17159\_replaced([11215942](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxWGLPvSEe-AqvopbP1qhQ))). The input i.e maximum CS1 list size, shall undergo an arithmetic mean computed over a considerably long evaluation period and then fed to an IIR filter to smoothen the final output as shown below. More the reduction in the average CS1 list maximum size, more the overload.

The reduction in the maximum CS1 list size is relative to the fixed value of the maximum CS1 list size supported (rdMaxNumUeTdDL/rdMaxNumUeTdUL: the maximum number of UEs for TD DL / UL scheduler. The value and range depends on processing capacity defined in EFS), hence the severity metric can show the degradation much more too soon (for e.g when the cell is lowly loaded). Therefore DL or UL scheduler shall compare the average CS1 list maximum size with the average number of active UEs with pending data (DL/UL) or pending SR/Proactive UL grants (UL) in the pre-scheduler candidate list. The latter shall also be averaged over the same evaluation period using same method used for the maximum CS1 list size as mentioned above. The difference between the two shall be another input to the severity metric calculation along with the average CS1 list maximum size as given below.

l2PsRtCellLevelSeverityMetric = 100 - 100 \* (MIN (rdMaxNumUeTdDl (or rdMaxNumUeTdUl) , avgCs1ListSizeMax + MAX (0, avgCs1ListSizeMax - avgPreCsListSize))) /  rdMaxNumUeTdDl (or rdMaxNumUeTdUl)

where:

rdMaxNumUeTdDl is defined in 5G\_L2\_3697\_replaced([11182022](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27YYNvvTEe-AqvopbP1qhQ)) and is used when the severity metric is computed for DL

rdMaxNumUeTdUl is defined in 5G\_L2\_3702\_replaced([11182963](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27fs-fvTEe-AqvopbP1qhQ)) and is used when the severity metric is computed for UL

avgCs1ListSizeMax is calculated as shown below.

avgCs1ListSizeMax(t) = avgCs1ListSizeMax(t-1) \* (1 - rdSevMetricMeasFilterCoeff) + rdSevMetricMeasFilterCoeff \* cs1ListSizeMaxEachEvalPeriod(t)

where:

cs1ListSizeMaxEachEvalPeriod(t) gives the arithmetic mean of the input (i.e the actual maximum CS1 List size per slot in DL and UL with valid K1 and K2 values, respectively) computed over the most recent evaluation period 't'.

avgCs1ListSizeMax(t) gives the overall 'smoothened' output of the cs1ListSizeMaxEachEvalPeriod over consecutive evaluation periods. Its value at t=0 shall be initialized with cs1ListSizeMaxEachEvalPeriod(0)

avgPreCsListSize is the average of the number of active UEs with pending data in the pre-scheduler candidate list calculated as shown below.

where:

avgPreCsListSize(t) = avgPreCsListSize (t-1) \* (1 - rdSevMetricMeasFilterCoeff) + rdSevMetricMeasFilterCoeff \* preCsListSizeEachEvalPeriod(t)

where:

preCsListSizeEachEvalPeriod(t) gives the arithmetic mean of the input (i.e the actual maximum prescheduler candidate list size per slot in DL and UL with valid K1 and K2 values, respectively ~~non-zero users with pending data (DL/UL) or pending SR/Proactive UL grants (UL)~~. For prescheduler candidate list see 5G\_UP\_ALG\_4209\_replaced([11174627](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kudxRfvSEe-AqvopbP1qhQ)) ) computed over the most recent evaluation period 't'.

avgPreCsListSize(t) gives the overall 'smoothened' output of the preCsListSizeEachEvalPeriod over consecutive evaluation periods. Its value at t=0 shall be initialized with preCsListSizeEachEvalPeriod(0).

For rdSevMetricMeasFilterCoeff see 5G\_UP\_ALG\_32164\_replaced([11217252](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxda7fvSEe-AqvopbP1qhQ)).

The rationale for the evaluation period to be reasonably longer (recommended to be 160ms) is to counter the outliers in the form of periodic spikes or lows in the actual CS1 size per slot, due to factors outside of overload control like measurement gaps, DRXs etc.

Thus, to summarise the severity of the RT processing load shall be calculated every evaluation period and shall be based on following conditions as inputs:

- the average CS1 list maximum size

- average number of active UEs with pending data in the pre-scheduler candidate list.

gNB DU shall support the calculation of a Non-Real Time (RT) processing load severity metric to detect high NRT processing load at L2-pool level. This severity metric shall use the number of pending signalling procedures at L2-pool level. gNB DU shall update this metric once every radio frame.

The metric is given below as:

l2PoolNrtSeverityMetric = max { 0, (numOfSignallingPsUserProcedures - rdNbProcedureContextsLow) / (rdNbProcedureContextsMaxLimit - rdNbProcedureContextsLow) } \* 100%

where for the following;

- rdNbProcedureContextsLow shall define the number of signalling procedures below which scheduler shall  continue processing these procedures. See 5G\_UP\_ALG\_32160\_replaced([11217214](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxda5_vSEe-AqvopbP1qhQ)).

- rdNbProcedureContextsMaxLimit shall define the number of signalling procedures above which the scheduler shall suspend processing of these procedures. See 5G\_UP\_ALG\_32158\_replaced([11217201](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxda5fvSEe-AqvopbP1qhQ)).

- numOfSignallingPsUserProcedures shall define the number of pending signalling procedures in the L2-pool.

[PR833406] The pool level severity metric can be different across different cells in the same L2-pool [End PR833406]

**3.34.2.0-6.0-2.0-3**  (ID: `11216106`)

Overload control detection
soft-RT/NRT processing load severity metric per core

**3.34.2.0-6.0-2.0-3.0-1**  (ID: `11216114`)

Overload control detection
gNB DU shall support the calculation of a soft-Real Time (soft-RT)/Non-Real Time (RT) processing load severity metric to detect high NRT processing load per scheduler processing core. A scheduler processing core hosts RT and NRT processing jobs of one or more cells.

This severity metric shall account for the additional soft-RT/NRT processing budget on top of nominal NRT processing budget for a slot with DL or UL scheduling or no scheduling. More the NRT processing share, more the NRT overload. The slot processing budget is calculated by the adaptive scheduler budget calculation (5G\_UP\_ALG\_24621\_replaced([11216305](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxYiZfvSEe-AqvopbP1qhQ))) which gives the processing budget for the RT and soft-RT/NRT per slot. The output of the above shall be averaged using the same averaging method for the maximum Cs1 list size (as defined in 5G\_UP\_ALG\_31301\_replaced([11216081](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxXUQfvSEe-AqvopbP1qhQ)) i.e an arithmetic mean computed over a considerably long evaluation period (160ms) and then fed to an IIR filter to smoothen the final output) and taken as input to this severity metric.

The metric is given below as:

**[Before CB013709-A-UPD001]**

If NRBTS.actAdaptiveSchedulerBudgetOlc is set to TRUE then:

l2PsNrtCoreLevelSeverityMetric = max{ 0, (rdMaxAdaptiveSlotProcessingRatio\_perCent/100 – avgSlotProcessingRatio) / (1 – rdMaxAdaptiveSlotProcessingRatio\_perCent /100) } \* 100%

* rdMaxAdaptiveSlotProcessingRatio\_perCent define the maximal scheduler processing duration. See 5G\_UP\_ALG\_32162\_replaced([11217224](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxda6fvSEe-AqvopbP1qhQ)).

otherwise

l2PsNrtCoreLevelSeverityMetric = max{ 0, (rdOptimalSlotProcessingRatio\_perCent/100 – avgSlotProcessingRatio) / (1 – rdOptimalSlotProcessingRatio\_perCent/100) } \* 100%

* rdOptimalSlotProcessingRatio\_perCent define the maximal scheduler processing duration. See 5G\_UP\_ALG\_32500\_replaced([11217241](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxda6_vSEe-AqvopbP1qhQ)).

**[CB013709-A-UPD001]**

If NRBTS.actAdaptiveSchedulerBudgetOlc is set to TRUE then:

l2PsNrtCoreLevelSeverityMetric = max{ 0, (rrmRefSlotProcessingRatio\_perCent– avgSlotProcessingRatio\*100) / (rrmRefSlotProcessingRatio\_perCent - rdMinAdaptiveSlotProcessingRatio\_perCent) } \* 100%

where,

rdMinAdaptiveSlotProcessingRatio\_perCent is defined in[11190862: rdMinAdaptiveSlotProcessingRatio\_perCent](https://dn-prod.ext.net.nokia.com/rm/resources/BI_28Zr6PvTEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9)

For FR1 FDD on ABIL

* rrmRefSlotProcessingRatio\_perCent shall define the reference slot processing ratio and shall be calculated as per 5G\_L2\_1833\_replaced([11212618](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-mrWvvTEe-AqvopbP1qhQ)).

otherwise

-  rrmRefSlotProcessingRatio\_perCent = rdMaxAdaptiveSlotProcessingRatio\_perCent (See 5G\_UP\_ALG\_32162\_replaced([11217224](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxda6fvSEe-AqvopbP1qhQ))).

otherwise

l2PsNrtCoreLevelSeverityMetric = max{ 0, (rrmRefSlotProcessingRatio\_perCent – avgSlotProcessingRatio\*100) / (rrmRefSlotProcessingRatio\_perCent - rdMinAdaptiveSlotProcessingRatio\_perCent) } \* 100%

* rrmRefSlotProcessingRatio\_perCent shall define the reference slot processing ratio and shall be calculated as per 5G\_L2\_1833\_replaced([11212618](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2-mrWvvTEe-AqvopbP1qhQ)).

**[End CB013709-A-UPD001]**

avgSlotProcessingRatio is defined as follows:

avgSlotProcessingRatio(t) = avgSlotProcessingRatio(t-1) \* (1 - rdSevMetricMeasFilterCoeff) + rdSevMetricMeasFilterCoeff \* slotProcessingRatioEachEvalPeriod(t)

where:

* slotProcessingRatioEachEvalPeriod(t) gives the average of the input (i.e the slot processing budget per slot), computed over the recent most evaluation period 't'.
* avgSlotProcessingRatio(t) gives the overall 'smoothened' output of the slotProcessingRatioEachEvalPeriod over consecutive evaluation periods.

For rdSevMetricMeasFilterCoeff see 5G\_UP\_ALG\_32164\_replaced([11217252](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxda7fvSEe-AqvopbP1qhQ)).

### 3.34.3.0-8 Overload control actions Countermeasures mapped to Severity metrics (ID: 11216293)

**3.34.3.0-8.0-1**  (ID: `11216299`)

If **[Before CB013709-E] the rdActExtendedUpOverloadHandling (5G\_UP\_ALG\_31311\_replaced([11216763](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxaXn_vSEe-AqvopbP1qhQ))) is set to TRUE** [CB013709-E] NRBTS.tmpActFeat1(Bit 0) *is set to 0* *[End CB013709-E],***\***\*

From the overall system perspective, these countermeasures can be classified as QoS impacting and Performance impacting.

- QoS impacted countermeasures **shall represent preferred set of countermeasures. These countermeasures shall not impact prioritized KPIs (e.g Throughput, PRB utilization, scheduled UEs/TTI).**

- Performance impacted countermeasures shall represent set of countermeasures that are allowed to impact the prioritized KPIs temporarily to bring down CPU processing.

The countermeasures shall follow the below listed generic principles:

- The countermeasures shall be ordered according to corresponding minimum severity metric **threshold.**

- Several countermeasures shall be bundled, when necessary if they belong to same severity metric threshold.

-     [PR839866] A countermeasure corresponding to a specific functionality shall be applied when either one of the following conditions is satisfied:

* The feature activation flag corresponding to the functionality is enabled. For example, DL MU-MIMO related countermeasures shall only be applied when DL MU-MIMO is enabled in the cell. [PR846325] The functionality can be enabled at NRBTS level as well. [End PR846325]
* The functionality is still available in the cell. For example, in case of NBR, even if the activation flag is disabled online, there can be NBR bearers still existing in the cell. In that case, NBR related countermeasures shall be applied.

[End PR839866]

Overload mitigation actions based on power-saving features involve simplifyin

- The activation threshold of a certain countermeasure defines the related severity metric value at which the countermeasure shall be activated.

- The deactivation threshold of a certain countermeasure defines the related severity metric value at which the countermeasure shall be deactivated.

- The activation and deactivation threshold values shall include the relative hysteresis value to avoid the impulsive reaction of the countermeasures that may occur to the overall system, when the severity metric value increases or decreases

- When activation threshold for multiple countermeasures is reached at the same time, gNB-DU shall activate one countermeasure (or one bundle of countermeasures) at a time starting with the lowest threshold.

- When deactivation threshold for multiple countermeasures is reached at the same time, gNB-DU shall deactivate one countermeasure (or one bundle of countermeasures) at a time starting with the highest threshold.

- When the severity metric value lies between the activation and deactivation thresholds then the status of the countermeasure(s) changes only when severity metric exceeds/goes below the thresholds, i.e, if the countermeasure is active, it remains activated until the severity metric reaches the deactivation threshold, otherwise if the countermeasure is inactive, it remains inactive unless the severity metric exceeds the activation threshold.

### 3.34.3.0-10 Overload control actions Activation of DL Pipelined Scheduling (ID: 13423825)

**3.34.3.0-10.0-1.0-1**  (ID: `13423845`)

Overload control actions
When DL pipelined scheduling is enabled (see [13423826: Activation Criteria](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Imxw4RiSEfGFMKAOrP5qqw?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Frm%2Fcm%2Fstream%2F_UBmNVp5EEe2A-5U7tzedRA)), the gNB-DU shall enable a pipelining solution for DL scheduling. This solution shall utilize the CPU core assigned for the UL scheduler as a secondary DL scheduling core in slots where UL scheduling is not required. The gNB-DU shall operate based on the availability of the secondary core as follows:

1. DL Parallel Scheduling Slots: In slots where the secondary CPU core is available for DL scheduling, the gNB-DU shall operate in a parallel scheduling mode as follows:
2. The primary DL CPU core shall perform initial DL pre-scheduling and part of  Time Domain (TD) scheduling.
3. The primary DL CPU core shall trigger the secondary CPU core to perform Frequency Domain Multiplexing (FDM), Spatial Domain (SD), and Frequency Domain (FD) scheduling.
4. After the secondary core completes its tasks, the primary CPU core shall perform all post-DL scheduling tasks for the slot.
5. Across DL parallel scheduling slots, there is a possibility of out-of-order scheduling i.e. The DL pre-scheduling of a slot n+1 might be performed before the  post-scheduling of the previous slot n. In that case,  the gNB-DU shall not consider any UEs from the CS2 list of the previous slot for scheduling in the next slot to avoid concurrent memory handling issues.
6. DL Non-Parallel Scheduling Slots: In slots where the secondary CPU core is not available for DL scheduling (i.e., it is performing UL scheduling), the gNB-DU shall perform all DL scheduling tasks on the single primary DL core.

Notes:

* This requirement only provides a basic view of the multi-core DL scheduling process and its primary motivation is to clarify why a UE might not be scheduled in consecutive slots as it affects system KPIs/SFS functionaliies. Please refer to EFS for the exact details on the DL pipelining solution.
* To improve scheduling efficiency, the process for handling UEs excluded for scheduling in the next slot during out-of-order execution shall be further evaluated in CP3 to determine if the considered but unscheduled UEs (i.e. UEs that are part of CS2 list but not scheduled in the slot) can be reported to the primary DL core more quickly. By accelerating their post-processing, they could be made eligible for consideration in the subsequent slot, thereby preventing unnecessary scheduling delays caused by the pipelined execution model.

## 3.34.4 Adaptive Scheduler Budget

### 3.34.4.0-1 Adaptive Scheduler Budget Overview (ID: 11216313)

**3.34.4.0-1.0-2**  (ID: `12573205`)

Adaptive Scheduler Budget
If NRBTS.actAdaptiveSchedulerBudgetOlc is set to true, gNB-DU shall dynamically adjust the overall processing time available to all schedulers (UL and DL) running on the same core such that there is enough capacity remaining to process the asynchronous messages of each scheduler. [CB014680-D]  In TDD FR1, the slots in which maximum number of schedulers performs corresponding DL/UL scheduling shall be excluded from the overall processing time adjustments. Note, the determination of number of schedulers in a slot shall be done based on slots in which corresponding UL/DL scheduling is performed and not based on the on air slot types in the slot pattern. For ex. in FR1 TDD 1 cell per core deployments, the slots in which both DL & UL schedulers of the cell perform scheduling in the core shall be excluded. [End CB014680-D]

* gNB-DU shall determine the slotUtilization for each core as described in 5G\_UP\_ALG\_24680\_replaced([11216326](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxYibvvSEe-AqvopbP1qhQ)). It shall be calculated in each slot.
* The slotUtilization shall be averaged over time with exponential filtering to determine the averageSlotUtilization.
* The averageSlotUtilization shall be used in each slot to determine the available processing time budget to be allocated to the schedulers.

In case multiple schedulers are executed on the same core, the overall available processing time budget determined above shall be further distributed between the schedulers as described in 5G\_UP\_ALG\_17003\_replaced([11216341](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxYicPvSEe-AqvopbP1qhQ)).

The available processing time budget allocated to the schedulers shall always be greater than a minimum value such that all processing time is not allocated to message processing. This is to allow scheduling DL/UL messages so that there won't be sleeping cells.

**3.34.5.0-2.0-3.0-1**  (ID: `12572284`)

Load Balancing among Schedulers
The gNB-DU shall assess the load condition of each DL/UL scheduler 'i' by computing schedRatio(i) metric. This metric is defined as the ratio between:

* [Before PR854004] The sliding window average of number of UEs scheduled per slot [PR854004] The sum of scheduled Ues within the sliding window for the corresponding scheduler  [End PR854004], and
* The sliding window average of number of UEs present in the pre-scheduler candidate set for the corresponding scheduler.

The formula is given by:

[Before PR854004]

schedRatio(i) = max{1/N, avgScheduledUes(i)}/max{1/N, avgNumUesInPreSchedCandSet(i)}

[PR854004]

schedRatio(i) = max{1, sumScheduledUes(i)}/max{1/N, avgNumUesInPreSchedCandSet(i)}

[End PR854004]

where:

* 'N' is the number of slots present in sliding window and it shall be defined in EFS.
* avgScheduledUesPerSlot(i): The linear average of number of UEs scheduled per TTI over a sliding window of size 'N' for scheduler 'i'. DL and UL schedulers considers only those slots where respective scheduling opportunities exist.
* [PR854004] sumScheduledUes(i): The sum of scheduled number of UEs over a sliding window of size 'N' for scheduler 'i'.[End PR854004]
* avgNumUesInPreSchedCandSet(i): The linear average of number of UEs in pre-scheduler candidate set over the same sliding window 'N' for scheduler 'i'. DL and UL schedulers considers only those slots where respective scheduling opportunities exist.

[PR854004] Rationale for schedRatio computation: The inverse of the schedRatio represents the required time period (in multiples of sliding window duration)  to schedule all the pending UEs in the pre-scheduler candidate set. For. ex. If avgNumUesInPreSchedCandSet=100 and the sumScheduledUes = 20 over the sliding window, it would mean that 5 times sliding window duration will be required to schedule all the existing UEs in pre-scheduler candidate set in the scheduler. [End PR854004]

The scheduler load is considered inversely proportional to schedRatio(i). That is, a lower schedRatio(i) indicates higher load, and vice versa.

Requested Processing Time per scheduler

**3.34.5.0-3.0-2.0-6.0-1**  (ID: `11216565`)

Load Balancing among Schedulers
L2PS shall evaluate for each scheduler the required processing time to perform allocations without time constraint depending on its load situation.

In Low Load condition, the required processing time is the time to schedule a single UE. A scheduler is considered in low load whenever it has less than one UE in the prescheduler list in average.

In Medium Load condition, the required processing time is the actual execution time with the time margin to schedule one additional user. A scheduler is considered in medium load whenever it does not apply any restriction to the prescheduler set or the scheduled users.

In High Load condition, the required processing time is the estimation of the required processing time to schedule the minimum between all users in the presecheduler set and the maximum scheduled users per slot. A scheduler is considered in high load situation whenever the target scheduled users cannot be reached due to preventive overload control.

Note: load conditions may be adjusted to the actual processing time function. It is assumed a higher processing cost to schedule the first UE and then a linear increase. However, the cost of scheduling users may be linear with a few users and grow exponentionally due to blocking rate in prescheduler and CS1 list. It is left to implementation to determine the minimum processing time for the scheduler and the determination of the actual required processing time.
