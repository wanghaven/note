## 4.23.0-4 Common for Bearer Type Change and Mobility Procedures Concurrent Handover Procedures (ID: 13396588)

**4.23.0-4.0-1**  (ID: `13396590`)

Common for Bearer Type Change and Mobility Procedures The concurrent handovers under consideration here are the cases where there is an ongoing Inter-gNB change of PDCP anchor (Inter-gNB Handover or Bearer Type change procedures) which involves DL PDCP Data forwarding and then while the forwarding is ongoing (end marker timer is running), there is a second handover (Inter or Intra-gNB) initiated.

gNB-CU-UP shall perform the following steps when Bearer Suspend / Release procedure is initiated for a second handover while the DL data forwarding is ongoing for a previous handover:

  * Suspension and Termination of Ongoing Data forwarding
  * IF EndMarker has not been received OR End Marker Timer is running:
    * Terminate the end marker timer and mark this as the end of data forwarding.
    * Release the ingress Forwarding tunnel so that no more packets can be received over the GTP-U forwarding tunnel. This is applicable both to DRB-level forwarding and PDU Session data forwarding.
    * NOTE: GTP-U Error Indication shall be raised for any new packets received on this terminated forwarding tunnel. The Source gNB-CU-UP shall ignore the error indication.
  * Handling of the second handover:
  * If DL data transmission over F1U has already started, the last successfully delivered PDCP SN shall be taken into account so that the forwarding shall start from the next PDCP SDU.
  * The handling of the second handover shall be as per the existing requirements:
    * For SA Inter-gNB handover, refer to [11227387: SA Inter gNB Mobility](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kycSWvvSEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9)
    * For SA Intra-gNB handover, refer to [11227132: SA Intra gNB Mobility](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kybEMfvSEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9)
    * For NR DC SN addition / PSCell Addition, refer to [11226029: NR DC PSCell Addition for SN Terminated Bearer](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyR6SfvSEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9) , [11226402: Intra gNB NR DC Split Bearer type setup](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyU9l_vSEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9)
    * For NSA Intra-gNB handover, refer to [11226818: NSA Intra gNB Mobility](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyYn8PvSEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9)
    * For NSA Inter-gNB handover, refer to [11227076: NSA Inter gNB Mobility](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyadIvvSEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9)



NOTE: Concurrent NSA Intra-gNB / Inter-gNB handover is possible only post a EN DC SgNB Addition.

## 4.23.0-5 Common for Bearer Type Change and Mobility Procedures StandAlone (SA) (ID: 11225827)

**4.23.0-5.0-1**  (ID: `11225833`)

Common for Bearer Type Change and Mobility Procedures

### DL Data Forwarding from Source gNB-CU-UP

---

**4.23.0-5.0-1.0-4**  (ID: `13106153`)

Common for Bearer Type Change and Mobility Procedures gNB-CU-UP shall handle the DL data forwarding at Source gNB as follows:

  * In case when Xn-U / Ng-U /X2-U DL DRB data forwarding tunnel is provided for the given DRB,
  * Data forwarding shall be performed as described in 5G_UP_ALG_24409_replaced([11225861](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyQFGvvSEe-AqvopbP1qhQ)).
  * For the data received from the Ng-U DL data tunnel, SDAP entity shall classify the received SDAP SDUs as per the QFI and send the SDAP PDU to the PDCP enity. PDCP entity shall forward the data on the respective Xn-U / Ng-U /X2-U DL DRB data forwarding tunnel.

  * In case when Xn-U / Ng-U /X2-U **DL DRB data forwarding tunnel is not provided for the given DRB AND DL PDU session data forwarding tunnel is provided.**

  * Data forwarding shall be performed as described in 5G_UP_ALG_24411_replaced([11225875](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyQsIfvSEe-AqvopbP1qhQ)).
  * For the data received from the Ng-U DL data tunnel, SDAP entity shall classify the received SDAP SDUs as per the QFI and send the SDAP PDU to the PDCP enity. PDCP entity shall forward the data on the respective DL PDU session data forwarding tunnel.

  * In case when Xn-U / Ng-U **/X2-U DL DRB data forwarding tunnel is not provided for the given DRB AND DL PDU session data forwarding tunnel is not provided**

  * No DL data forwarding is done



[CB010559-I-UPD002]

RRE handling during Inter-gNB HO

  * Upon receiving the new DL GTP-U tunnel information during ongoing data forwarding for Inter-gNB HO, the gNB-CU-UP shall stop sending packets to the old DL GTP-U forwarding tunnel and start using the new DL GTP-U forwarding tunnel for forwarding all the subsequent packets. This includes any data still held in its PDCP buffers and any new data that arrives from the UPF including the End Marker packet.
  * Data forwarded earlier to the previous DL GTP-U forwarding tunnel shall not be forwarded again and is considered lost. SN assigned PDCP PDUs lost due to this reason during DL DRB data forwarding may invoke reordering of PDUs that are arriving out of sequence at RRE target gNB. Reordering operation shall not happen for DL PDU session data forwarding since SDUs are forwarded in this case.
  * In case an End Marker was received at source gNB and already forwarded to previous DL GTP-U forwarding tunnel, source gNB shall create an End Marker GTP-U packet and send it to the new DL GTP-U forwarding tunnel.



[End CB010559-I-UPD002]

UP Interruption time:

Source gNB shall include timestamp 'x' in the "Source gNB Timestamp" GTP-U extension header (Nokia proprietary) of the forwarded packet(s) during the data forwarding for Inter-gNB handover. Refer 5G_UP_IF_14443_replaced([11547447](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tY9HRvv1Ee-AqvopbP1qhQ)) for details on "Source gNB Timestamp" GTP-U extension header.

Timestamp 'x' shall be added to first few (Ex: 3) forwarded data packets to increase reliability of delivering timestamp information. Same Timestamp information 'x' shall be embedded in multiple forwarded data packets.

'x' is timestamp at source gNB when data flow is suspended during Inter-gNB handover or Inter-RAT handover to LTE.

**Note:**

  * gNB-CU-CP can disable Data forwarding for certain DRB(s). In such case gNB-CU-CP indicates explicitly if DL data forwarding needs to be done or not for the given DRB.
  * In case when Data forwarding is enabled for the DRB and when Xn-U / Ng-U /X2-U DL DRB data forwarding tunnel is not allocated, and if the data forwarding is enabled for this DRB, gNB-CU-CP should indicate that the DRB specific buffered data needs to be forwarded over DL PDU session data forwarding tunnel.
  * The DL PDU session data forwarding tunnel can be setup on Xn-U or Ng-U interfaces
  * Irrespective of the data forwarding tunnel provided by core network, in case IRAT direct data forwarding is to be done on X2-U/NG-U/Xn-U, then PDCP SN and QFI shall not be added in GTP-U extension header.
  * In case of IRAT direct data forwarding tunnel, only DL DRB data forwarding is used. **X2-U interface can be configured only in case of IRAT direct data forwarding.**

**4.23.0-5.0-2.0-1**  (ID: `11225867`)

  * PDCP SDU with SN assigned:

2) For DRB using RLC AM mode

For DRB using RLC UM mode

  * PDCP SDU as GTP-U payload. PDCP SN must not be included in the GTP-U extension header

  * PDCP SDU without SN assigned:




1) PDCP SDU as GTP-U payload

**4.23.0-5.0-3.0-1**  (ID: `11225878`)

**4.23.0-5.0-4**  (ID: `11225888`)

Common for Bearer Type Change and Mobility Procedures DL GTP-U End Marker from Source gNB-CU-UP

---

**4.23.0-5.0-4.0-1**  (ID: `11225894`)

Common for Bearer Type Change and Mobility Procedures gNB-CU-UP at Source gNB shall perform the DL GTP-U End Marker related actions as follows:

1) Send DL GTP-U End Marker as the last GTP-U packet on each of the Xn-U / Ng-U DL DRB data forwarding tunnel and the DL PDU session data forwarding tunnel, after all buffered PDCP PDUs/SDUs have been forwarded to target gNB.

2) Send any subsequently received DL GTP-U End Marker from UPF, over Xn-U /Ng-U DL data forwarding tunnels towards target gNB.

3) Discard any subsequent user data packet or GTP-U signalling message (except DL GTP-U End Marker message) received over the NG-U tunnel.

4) In case when the DL data forwarding is requested for the default DRB, the DL GTP-U End Marker shall be sent for all the QFIs that were handled by the default DRB (refer Note 1).

Note:

1\. In case when default DRB is configured, the QFI(s) of all the DL PDCP SDUs shall be recorded and are used for sending the DL GTP-U End Markers (refer Rationale).

2\. In case where the target entity allocates all the requested DRB forwarding tunnels, data forwarding shall be done on the Xn-U /Ng-U DL DRB data forwarding tunnel only. DL PDU session data forwarding tunnel if allocated, will be injected with DL GTP-U End Marker without any data forwarded on it.

3\. In case where DRB specific forwarding tunnel is unavailable and only DL PDU session data forwarding tunnel has been allocated, all the buffered DRB specific data and any new data from NG-U shall be forwarded over the DL PDU session data forwarding tunnel.

4\. For the case when all the DRB tunnels are allocated - the end marker is injected on PDU session tunnel (when there is no data), only after all the DRB data is sent out of Source gNB. To avoid any discarding of forwarded data at target.

Rationale:

  * Current E1 interface does not support the signalling of the QFI(s) accepted for data forwarding. This issue is raised to 3GPP RAN3. The current workaround will be replaced once the final solution is agreed in 3GPP.

  * Only the unacknowledged PDCP PDUs shall be transmitted, if the PDCP Status report is received.

• Data forwarding is complete on the Xn-U / Ng-U DL PDU session data forwarding tunnel.

  * Incase when Xn-U / Ng-U DL data forwarding is disabled for the DRB



DL data transmission shall be allowed when:

  * DL F1-U TEID is received

**4.23.0-5.0-10**  (ID: `11225983`)

Common for Bearer Type Change and Mobility Procedures Handling of DL GTP-U End Marker at Target gNB-CU-UP

---

**4.23.0-5.0-10.0-1**  (ID: `11225990`)

Common for Bearer Type Change and Mobility Procedures gNB-CU-UP shall do the following:

If DL GTP-U End marker message is received over the Xn-U / Ng-U DL DRB data forwarding GTP-U tunnel:

  * Stop End marker timer if it is running.

  * Discard any new GTP-U packet received over Xn-U / Ng-U DL DRB data forwarding tunnel.

  * Discard any subsequently received End marker message.




If DL GTP-U End marker message is received over the DL PDU session data forwarding GTP-U tunnel:

  * Stop End marker timer if it is running.

  * Discard any new GTP-U packet received over DL PDU session data forwarding GTP-U tunnel.

  * Discard any subsequently received End marker message.

  * Inform all the DRBs of the PDU session that data forwarding on the PDU session tunnel is complete.




If End marker timer has expired (no End marker received):

\- Discard any new GTP-U packet received over Xn-U DL DRB data forwarding GTP-U tunnel.

  * If the end marker timer was started for the DL PDU session tunnel, indicate to all the DRBs of the PDU session that data forwarding on the PDU session tunnel is complete.
