# 4.18 RLC uplink reassembly window handling

## 4.18.0-2 RLC uplink reassembly window handling Handling of RLC AM uplink reassembly window due limited buffering capability (ID: 11225548)

**4.18.0-2.0-2**  (ID: `11225562`)

RLC uplink reassembly window handling On reception of AMD PDUs or byte segments of AMD PDUs which are within the RLC re-assembly window, 5G-L2-HI shall,

If the byte segments of the AMD PDU is received and the corresponding SN was not received before and total number of RLC PDU waiting for re-assembly > [Before CB008963-B] rdUlRlcReassemblyContextCount

[CB008963-B] min(20% of AM_Window_Size, rdUlRlcReassemblyContextCount) [End CB008963-B] (for DRBs and SRBs) discard the received byte segments.

Rationale:

In worst case, the number of RLC PDU waiting for re-assembly is based on the AM_Window_Size. Since the value is quite large when compared to the buffering capability, we need to limit the number of RLC PDU waiting for re-assembly.

[Before CB008963-B] The value of rdUlRlcReassemblyContextCount shall be defined based on buffering capability.

[CB008963-B] Since 12-bit RLC SN was introduced, earlier rdUlRlcReassemblyContextCount value is very high. Hence, a different MAX value needs to be defined based on buffering capability. So, minimum value is selected between %age of window size and rdUlRlcReassemblyContextCount. [End CB008963-B]

Note : Discarding of RLC PDU segment inside the receive window due to buffer limitation will cause that in the RLC STATUS PDU, these discarded PDU are not marked as NACKED. This causes that the UE RLC TX window does not move forward causing UE RLC Tx window to stall.

## 4.18.0-3 RLC uplink reassembly window handling RLC Status reporting (ID: 11225570)

**4.18.0-3.0-1**  (ID: `11225574`)

RLC uplink reassembly window handling The maximum size for status PDU is fixed to 1864 bits. (Refer to Rationale)

In case when the RLC STATUS PDU cannot fit the NACK SN list for the RLC SN(s) due to maximum size restriction.

1\. Build the RLC STATUS PDU with the NACK SN List that can fit into the RLC STATUS PDU, leaving space for 1 segment, so that the last segment will NACK the remaining part of the SDU which could not fit in the status PDU.

2\. Set the ACK_SN = Highest NACK_SN + 1. NACK_SN is derived from the RLC STATUS PDU built in step 1

Rationale:

5G-L2-HI is not aware of the actual transport block size determined by packet scheduler, so a fixed maximum limit is defined. Status PDU max size is defined as number of bits on MCS 0 per 100 PRBs on one component carrier. This is considered the minimum allocation from a scheduler.

## 4.18.0-4 RLC uplink reassembly window handling DRB Release due to missing UL RLC retransmission (ID: 11225581)

**4.18.0-4.0-1**  (ID: `11225588`)

RLC uplink reassembly window handling 5G-L2-HI shall maintiain the counter for couting number of RLC STATUS PDU's that contain NACKed segment(s) for RLC SN = RX_Next.

When the above counter exceeds rdulMaxRetxThreshold,

  * 5G-L2-HI shall request CPlane to release the DRB.



The counter increments are started only after RX_Next_Highest > RX_Next.

The counter is reset every time new RLC SDU/SDU byte segment with RLC SN = Rx_Next is received.

rdulMaxRetxThreshold is a R&D parameter defined in 5G_UP_ALG_13383_replaced([11225596](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyOP5_vSEe-AqvopbP1qhQ)).

Rationale

1) UE should retransmit the byte segment(s) which are NACKED upon reception of RLC STATUS PDU. However if the UE after multiple attempts does not retransmit the missing byte segement(s), it could possibly indicate a mismatch between UE and gNB.
