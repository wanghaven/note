# 4.9 PDCP Traffic Shaping for Downlink data

## 4.9.0-2 PDCP Traffic Shaping for Downlink data Overview (ID: 11224527)

**4.9.0-2.0-2**  (ID: `11224542`)

PDCP Traffic Shaping for Downlink data [Before CB008490-DT]

On the E2E X2-U path between SgNB and MeNB, there are several bottlenecks which can lead to drops of PDCP PDUs. This feature introduces the traffic shaping at 5G-L2-HI (CU) for downlink traffic over X2-U interface to limit packet drops.

The traffic shaping is applied for Downlink data over X2-U in the following cases:

  * The EN-DC traffic towards the eNB over the X2-U tunnel during EN-DC call,
  * The buffered DL data forwarded over the X2-U interface during mobility.



[CB008490-DT]

On the E2E backhaul path between two gNBs or between gNB and MeNB, there maybe bottlenecks which can lead to drops of PDCP PDUs. This feature introduces the traffic shaping at 5G-L2-HI (CU) for downlink traffic over X2-U / Xn-U interface to limit packet drops.

The traffic shaping is applied for Downlink data over X2-U and Xn-U in the following cases:

  * The downlink traffic from PDCP towards the RLC over the X2-U or Xn-U tunnel,
  * The buffered DL data forwarded over the X2-U or Xn-U interface during mobility, NR DC SN Addition and bearer takeback procedures.



NOTE: For NR DC split bearers, L2 traffic shaping is applied only in the case of Inter-gNB NR DC split bearers for Xn-U link only.

[End CB008490-DT]

## 4.9.0-4 PDCP Traffic Shaping for Downlink data Bearer level Traffic Shaping of Downlink Data over X2-U or Xn-U (ID: 11224568)

**4.9.0-4.0-1**  (ID: `11224574`)

PDCP Traffic Shaping for Downlink data Computation of Maximum Burst Size Limit

---

**4.9.0-4.0-1.0-3**  (ID: `11224594`)

PDCP Traffic Shaping for Downlink data **[Before CB008490-DT]**

The **maxBurstLimitPerBearerPerMsX2 is applicable for DL data over X2-U for each NSA split bearer calculated as follows:**

Initialization at the start of each DDDS Period:

currentTransmissionOccasion = 0

Calculation at the start of each DL transmission (millisecond):

  * percentageOfBurstAllowed = [0.3, 0.3, 0.15, 0.15, 0.1, ...]
  * maxBurstLimitPerBearerPerMsX2 (in bytes) = percentageOfBurstAllowed [currentTransmissionOccasion] * (maxTransferDelayXx * 2) * eTH (in Bytes/ms)
  * Increment currentTransmissionOccasion



Note: percentageOfBurstAllowed after 4th occasion is considered to be 0.1 for all successive transmission occasions, there is no limit to the number of transmission occasions. The rationale for this is to cater for traffic bursts that arrive towards the end of the DDDS period, the flow control delay limits are applied prior to traffic shaping, thus ensuring we are always within the delay limits.

Rationale 1:

  * The number of Downlink transmissions per ms within a DDDS period can be limited by the number of timer events occurring precisely at 1ms. When the number of timer events is not equal to DDDS period, there is a loss of the amount of data that can be sent over X2-U. To overcome this, the bearer level bursts are shaped in such a manner that the initial transmissions are bigger bursts than the later ones, this allows for maximum possible downlink transmission over X2-U within a limited number of transmission occasions (1ms).

  * If maxBurstLimitPerBearerPerMsX2 (in bytes) calculated is less than the size of a single PDCP PDU, the size of one PDCP PDU shall be considered as the burst limit. Refer **Rationale 2.**




Rationale 2**:**

 If the maxBurstLimitPerBearerPerMsX2 is smaller than the size of a single PDCP PDU, it is possible that no DL data can be sent over the link. To avoid link starvation, **one PDCP PDU shall be allowed if the calculated burst size limit is smaller than PDU size.**

where:

estTputXx (in bytes/ms) - Estimated Throughput for X2-U as calculated in 5G_UP_ALG_5553_replaced([11219630](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxrdWfvSEe-AqvopbP1qhQ)). The estTputXx if calculated in different units (e.g: bytes/s), suitable adaptations shall be made to the above calculation.

rdMaximumBurstLimitX2 (in ms) - R&D parameter defined in 5G_UP_ALG_17801_replaced([11224734](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJXYfvSEe-AqvopbP1qhQ))

**[CB008490-DT]**

PDCP Traffic Shaping for Downlink data [Before CB008490-DT]

If the L2 Traffic Shaping feature is activated with NRBTS/actL2TrafficShapingX2 set to True:

For each NSA bearer, in given 1 ms time period (refer Rationale)

  * The total size of PDCP PDUs sent over X2-U link shall be maintained
  * The total number of bytes sent over X2-U link shall not exceed the maxBurstLimitPerBearerPerMsX2 (refer to 5G_UP_ALG_17799_replaced([11224574](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyIJSPvSEe-AqvopbP1qhQ))).
  * Once the maximum limit is reached, PDCP PDUs over X2-U can be sent in the next millisecond.



Rationale:

Although the average throughput rate over X2-U is well within the transport capacity, an instantaneous burst of data towards X2-U can result in buffer overrun and packet loss in the intermediate switches / common transport. Hence it is necessary to have the shaping also done at a smaller interval of 1ms so that the instantaneous bursts of a bearer is within the limits.

  * The total number of bytes sent over X2-U or Xn-U link shall not exceed the maxBurstLimitPerBearerPerMsXx (refer to 5G_UP_ALG_17799_replaced([11224574](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyIJSPvSEe-AqvopbP1qhQ))).

## 4.9.0-8 PDCP Traffic Shaping for Downlink data Bearer Burst Control (ID: 11224701)

**4.9.0-8.0-1**  (ID: `11224706`)

PDCP Traffic Shaping for Downlink data Per Bearer Burst Control

---

**4.9.0-8.0-1.0-2**  (ID: `11224720`)

PDCP Traffic Shaping for Downlink data In case when number of packet sent over the given interface for the bearer during the current ms cycle >= packetPerMs, 5G-L2-HI shall assume that the given interface is not available till the end of the current ms cycle.

In case when number of packet sent over the given interface for the bearer during the last ms cycle >= packetPerMs AND current time - time instant of the last packet sent on the give interface < 100 microsecond, 5G-L2-HI shall assume that the given interface is not available

packetPerMs = rdF1PacketLimt, when the given interface is F1-U

packetPerMs = rdXxPacketLimt, when the given interface is X2-U

rdF1PacketLimt and rdXxPacketLimt are R&D parameter defined in 5G_UP_ALG_10956_replaced([11224760](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJXZfvSEe-AqvopbP1qhQ)) and 5G_UP_ALG_13753_replaced([11224776](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJXZ_vSEe-AqvopbP1qhQ))

Note

  1. The granularity of the burst control is per ms. Hence the skipping of sending data is kept within the same ms.



Eg :- Flow control start counting at t0. At t0 + 0.25 ms, the packet limit is met. After the limit is met, the interface is marked as not available for next 1-0.25 = 0.75 ms.

  1. Since burst control is done per ms, it is possible that gNB can generate one burst at end of last ms cycle and second burst at start of the current ms cycle. Since the ms cycle are not aligned across different SW component, i could result in 2 time the burst.



To avoid this, 100 microsecond gap is added between 2 burst.

[CB008717-E]

  1. For NR DC Split bearers, the packetPerMs limit shall be individually applicable to each of the two F1-U links (MCG and SCG). The rdF1PacketLimt shall be considered for both the F1-U links.



[End CB008717-E]

Rationale

1) Packet drop within gNB occur when L2-HI send large burst toward the eNB. To avoid the packet drop, burst control needs to be performed to minimise the amount of burstiness.
