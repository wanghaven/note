# 4.10 PDCP TCP MSS Clamping

## 4.10.0-2 PDCP TCP MSS Clamping Overview (ID: 11224799)

**4.10.0-2.0-1**  (ID: `11224806`)

PDCP TCP MSS Clamping
The network performance is affected when there is excessive IP packet fragmentation in the backhaul, i.e, when the IP payload size (MTU) exceeds the maximum possible size configured in the network. The IP payload size includes the protocol overhead with TNL encapsulation (GTP-U/UDP/IP, optionally IPsec), thus increasing the chances for fragmentation.

During the TCP session establishment, UE negotiates a maximum TCP payload (segment) size with their peers for each TCP session, by setting the Maximum Segment Size (MSS) value in the TCP SYN or TCP SYN ACK packets. The TCP MSS clamping feature allows the gNB to override the UE set value with its own value ( by accounting for the protocol overheads). The clamping is achieved by performing a Deep Packet Inspection (DPI) of each PDCP SDU (both DL and UL) to check for TCP SYN or SYN + ACK packets and overriding the MSS value if neccessary. DPI of PDCP SDUs are perfomed at L2-HI-CU inside gNB.

## 4.10.0-5 PDCP TCP MSS Clamping Check for IP Options or Extension Headers (ID: 11224845)

**4.10.0-5.0-1**  (ID: `11224851`)

PDCP TCP MSS Clamping
For IPv4:

* Get the IP version with the Version field and the IP header length from the Internet Header Length (IHL) field.
* IF version is 4 AND IHL value is greater than 5 (i.e 5 words or 20 bytes), the IPv4 Options are received
* Reference:Internet Protocol, RFC 791, <https://datatracker.ietf.org/doc/html/rfc791>

For IPv6:

* Get the IP version with the Version field and the Next Header field
* IF version is 6 AND Next Header field is not set to TCP (decimal value 6), the IPv6 extensions are received
* Reference:Internet Protocol Version 6, RFC 2460, <https://datatracker.ietf.org/doc/html/rfc2460>

## 4.10.0-6 PDCP TCP MSS Clamping TCP Checksum Calculation (ID: 11224858)

**4.10.0-6.0-1**  (ID: `11224863`)

PDCP TCP MSS Clamping
The TCP checksum is calculated as the 16 bit one's complement of one's complement sum of all the 16bit words in the TCP header and data.

NOTE: For further details on TCP checksum implementation, the following can be referred:

* Computing the Internet Checksum <https://datatracker.ietf.org/doc/html/rfc1071>
* Computation of Internet Checksum via Incremental Update <https://datatracker.ietf.org/doc/html/rfc1624>

## 4.10.0-7 PDCP TCP MSS Clamping MSS Clamping Procedure (ID: 11224870)

**4.10.0-7.0-1**  (ID: `11224885`)

PDCP TCP MSS Clamping
11224885:
PDCP TCP MSS Clamping
The following actions are performed on each DL and UL PDCP SDU:
 
\* Feature Enable Check:
\* 5G-L2-Hi CU shall check for the TCP MSS clamping feature enable as defined in 5G\_UP\_ALG\_13093\_replaced([11224815](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJXbfvSEe-AqvopbP1qhQ)).
\* If the feature is not enabled OR if the CPU overload actions have been initiated (refer 5G\_UP\_ALG\_13514\_replaced([11229095](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kymDVvvSEe-AqvopbP1qhQ))), TCP MSS value shall not be updated and the following steps are skipped.
 
\* Check for IP Options and Extensions:
\* From the IP header, check whether the IP Options for IPv4 or IP Extension Headers for IPv6 are received, as defined in 5G\_UP\_ALG\_13803\_replaced([11224845](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyJ-cvvSEe-AqvopbP1qhQ))
\* If IP Options for IPv4 or Extension headers for IPv6 are present, TCP MSS value shall not be updated, and the rest of the following steps are skipped.
 
\* Check for TCP SYN packet:
\* From the IP header, check the Protocol field for IPv4 or the Next Headerfield for IPv6 to determine the transport layer protocol. For Transmission Control Protocol (TCP), the value shall be 6 in decimal (Reference: ).
\* If the protocol is not TCP, the MSS clamping shall not be updated, and the rest of the steps are skipped.
 
\* If the Protocol is TCP, check the Control bit for the SYN flag (for both SYN and SYN+ACK TCP packets). If the SYN flag is set, the MSS value shall be checked for update
\* If the SYN flag is not set, TCP MSS value shall not be updated, and the rest of the following steps are skipped.

**4.11.0-2.0-4.0-1**  (ID: `11224967`)

PDCP HFN/SN Desynchronisation
This requirement applies to all types of traffic, RLC AM/UM mode, non-ciphered PDUs.

For all incoming PDCP PDU and if ciphering is disabled, PDCP SN is checked for missing PDUs.

If difference between previously received PDCP PDU SN and current received PDCP PDU SN is greater than "half of SN space", then SN desynchronisation is detected and action described in 5G\_UP\_ALG\_25219\_replaced([11224985](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyKlh_vSEe-AqvopbP1qhQ)) shall be  executed. PM counter HFN\_SN\_DESYNC\_DETECT\_DRB is incremented.

Note: “Half of SN space” is computed using configured pdcp-SN-SizeUL. For example, if the configured value is 18 bit, and if the difference of current received PDCP PDU SN and previously received PDCP PDU SN is greater than or equal to 2 (18-1) = 2 17 = 131072, then the currently received PDCP data PDU shall be dropped.

### 4.11.0-3 PDCP HFN/SN Desynchronisation Packet Validation algorithm (ID: 11224972)

**4.11.0-3.0-1**  (ID: `11224978`)

PDCP HFN/SN Desynchronisation
Packet Validation works using IP header. IP header from PDCP PDU shall be checked to validate the packet after deciphering.

Note:

1. IPv6 packet contains Version field filled with 6 and IPv4 header contains Version field filled with 4. IPv6 does not contain header length field.
2. IPv4 contains IP header length field. This is 4-bit field and valid value starts from 5 .. 15. Value 5 represent 5*4 = 20 bytes header length. IP header could contain options field which adds additional bytes and hence IP header length could increase upto 60 bytes (=15*4).

Note (from LBT4576): The validation of a single IP packet is intended to be very fast - not impacting the system performance. It relies only on checking the IP version in the header and header length of IPv4.

A single packet check will statistically not detect ~1/8 of invalid packets (version field is 4bit long and has only 2 valid values- 4 and 6). But with the 2nd mis-deciphered packet the chance of not detecting invalid packets is only ~1/64, with the 3rd packet it is ~1/512, and so on. Hence the HFN detection will be anyway very quick.

When the IP Packet Version is filled with 6 (IPv6 packet), then Packet Validation shall return successful.

When the IP Packet Version is filled with 4 (IPv4 packet) and IPv4 header length field contains value greater than or equal to 5 (which means IPv4 header length is greater than or equal to 5\*4 = 20 bytes), then Packet Validation shall return successful.

In all other cases, this procedure shall return failure.
