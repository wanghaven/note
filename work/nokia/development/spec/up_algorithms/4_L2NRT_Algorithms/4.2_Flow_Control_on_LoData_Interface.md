estimateLoBuffer = max(0 , estimateLoBuffer  – (t –t0) \* eTh)

Note: If the rdRlcDisableDecay = false, then decay part is enabled. Decay shall be done for all above cases namely:

2. CA is configured (and l2LoAvailableForDataSending is set to True or is set to False)

endif

Rationale: Time decay is not representative of scheduling delays always. So, by using the time decay, L2-LO buffer status is not estimated properly by L2-HI-DU because of which L2-HI-DU sends more data and fills up L2-LO buffer fast. As data packets wait longer in the L2-LO queue, the queue delay eventually exceeds the configured threshold (maxRlcQueuingDelay) + dddsPeriod.   
At this point, the RLC layer sends a Data Buffer Status report of DBS = 0 to the L2-HI-CU. Receiving the DBS = 0 signal, the L2-HI-CU stops refilling the buffer for the L2-HI-DU, causing the L2-HI-DU's buffer to become empty. The problem is then compounded by a DDDS reporting lag. A periodic DDDS report is not sent during the next period after the Emergency DDDS to update the L2-HI-DU's status. During this skipped periodic DDDS, the L2-HI-DU's buffer remains empty, and this 'zero' status is what gets reported to the L2-PS scheduler. Ultimately, because the L2-PS scheduler sees a buffer status of zero from the L2-HI-DU, it concludes there is no data to send and therefore does not activate a Secondary Cell (SCELL), missing a critical opportunity to increase data rates and recover performance.

rdRlcDisableDecay is R&D parameter defined in [13460643: rdRlcDisableDecay](https://dn-prod.ext.net.nokia.com/rm/resources/BI_XQWbZTGWEfGxldjOZQLezQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)

[End PR877989]

## 4.2.0-9 Flow Control on LoData Interface Proactive resending of RLC PDUs due to high RLC PDU delay with CA (ID: 11221262)

**4.2.0-9.0-1**  (ID: `11221268`)

Flow Control on LoData Interface
 Identify and update the lowest SN (lowestTxPdcpSN) among the reported “Highest transmitted NR PDCP Sequence Number” across the 5G-L2-LO instances.

 Identify the Highest SN (highestTxPdcpSN) among the reported “Highest transmitted NR PDCP Sequence Number” across the 5G-L2-LO instances as below,

- If all the Pdcp SN(s) between the current highestTxPdcpSN(Note 1) and lowestTxPdcpSN are reported as transmitted, or if the “RLC PDU delay due to CA” exceeds tReassemblyIBCA then reset the “RLC PDU delay due to CA” .

 Update the highestTxPdcpSN to the highest SN among the reported “Highest transmitted NR PDCP Sequence Number” across the 5G-L2-LO instances

 Mark the time stamp upon identifying the highestTxPdcpSN to account the “RLC PDU delay due to CA” between the highestTxPdcpSN and lowestTxPdcpSN across the 5G-L2-LO instances.

- Until all the Pdcp SN(s) between the current highestTxPdcpSN(Note 1) and lowestTxPdcpSN are reported as transmitted,

 Retain the current highestTxPdcpSN(Note 1) and continue to count the “RLC PDU delay due to CA”.

 Start monitoring the delay (“RLC PDU delay due to CA”) between highestTxPdcpSN and lowestTxPdcpSN when the highestTxPdcpSN is updated.

 If the “RLC PDU delay due to CA” exceeds the "rlcPduDelayThCa" (Note 2), the RLC PDUs between highestTxPdcpSN and lowestTxPdcpSN that are not indicated as transmitted shall be re-sent ahead of the unsent RLC PDUs(RLC PDUs that are to be sent for the first time) on the available 5G-L2-LO instance(s) as per the flow control decisions in 5G\_UP\_ALG\_5757\_replaced([11221478](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx11ZvvSEe-AqvopbP1qhQ)).

The chosen 5G-L2-LO instance shall be used for RLC PDU re-sending as defined in 5G\_UP\_ALG\_16407\_replaced([11221290](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx0nS_vSEe-AqvopbP1qhQ)).

 In case when all the RLC PDUs which were identified for re-sending were not re-sent due to unavailability of 5G-L2-LO instance(s), the re-sending of these unsent RLC PDUs shall be re-tried when a 5G-L2-LO instance for re-sending RLC PDUs becomes available for data sending, until the “RLC PDU delay due to CA” does not exceed tReassemblyIBCA - rdMaxRlcPduDelayMarginCa.

Rationale:

While updating the highestTxPdcpSN and lowestTxPdcpSN, the idea is to stick to the highestTxPdcpSN until all the PDCP SN(s) which are less than highestTxPdcpSN are transmitted. This will help in identifying the PDU delay which can cause the RLC re-assembly timer expiry.

Also, when there are multiple 5G-L2-LO instances configured, the lowestTxPdcpSN shall be updated to the lowest "highest transmitted PDCP SN" among all the 5G-L2-LO instances.

Note:

1. The highestTxPdcpSN value that is already in use which was updated based on previous DDDS reports.

2. rlcPduDelayThCa is a NIDD parameter received from C-plane.

3. tReassemblyIBCA is a NIDD parameter received from C-plane.

4. rdMaxRlcPduDelayMarginCa is a RAD parameter defined in 5G\_UP\_ALG\_23528\_replaced([11223024](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx-_UfvSEe-AqvopbP1qhQ)).

## 4.2.0-11 Flow Control on LoData Interface DL Data sending for the delayed RLC PDUs when CA is configured (ID: 11221290)

**4.2.0-11.0-1**  (ID: `11221298`)

Flow Control on LoData Interface
The delayed RLC PDUs as identified in 5G\_UP\_ALG\_16397\_replaced([11221262](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx0nR_vSEe-AqvopbP1qhQ)) shall be resent ahead of the new/unsent RLC PDUs over the available 5G-L2-LO instance(s) such that,

* If the RLC PDU that is being resent was originally sent over the same 5G-L2-LO instance that is chosen for resending, skip the re-sending of such RLC PDU(s). (Rationale)
* If the RLC PDU is reported as received in the status report, then skip the resending of such RLC PDU(s).
* Mark the RLC PDU that is getting resent to 5G-L2-LO instance as a retransmission.
* Update the current PDU delay and estimated 5G-L2-LO buffer for the corresponding 5G-L2-LO instance as defined in 5G\_UP\_ALG\_5788\_replaced([11221810](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx3qmfvSEe-AqvopbP1qhQ)).
* Upon re-sending the delayed RLC,
* For the first status report that is received requesting for retransmission of the same RLC PDU, skip the retransmission of such RLC PDUs, and increment the retransmission count. ## 4.2.0-32 Flow Control on LoData Interface Select the 5G-L2-LO instance for data sending based on PDU sending rate (ID: 11221694)

**4.2.0-32.0-1**  (ID: `11221698`)

Flow Control on LoData Interface
Compute the time taken for a PDU to be delivered on the primary and the available secondary 5G-L2-LO instances

estPerPduDelivTimePrimary = avgPduSizeInRlcBuffer/eTh of the primary 5G-L2-LO

For all the available secondary 5G-L2-LO instance(s):

estPerPduDelivTimeSecondary = avgPduSizeInRlcBuffer/eTh of the corresponding secondary 5G-L2-LO instance

Compute the number of PDUs that can be sent over corresponding 5G-L2-LO instances (primary or the available secondary 5G-L2-LO instances) as defined in 5G\_UP\_ALG\_14397\_replaced([11221707](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx3DifvSEe-AqvopbP1qhQ))

 In case when the “noOfPdus” that could be sent on a 5G-L2-LO instance for the allowed max queuing delay is 0, mark the 5G-L2-LO instances as Not available.

 In case when the “noOfPdus” that could be sent for all the available 5G-L2-LO instances for the allowed max queuing delay is 0,

 Mark the 5G-L2-LO instance(s) with highest eTh among the available 5G-L2-LO instances as available, and the remaining 5G-L2-LO instances as unavailable.

 In case when among the available 5G-L2-LO instances,

If the highest eTh is same for primary and the secondary 5G-L2-LO instance(s), the primary 5G-L2-LO instance shall be marked as available, and the secondary 5G-L2-LO instance(s) is marked as unavailable for data sending.

 Choose the 5G-L2-LO instance whose “noOfPdus” metric is highest among the available 5G-L2-LO instances –

i.e: maxNoOfPdus = MAX (primaryNoOfPdus, secondaryNoOfPdus1, ……. , secondaryNoOfPdusn)

 For each of the available 5G-L2-LO instance (Primary and/or Secondary 5G-L2-LO instance(s)),

 If the estimated “per PDU” delivery time exceeds the maximum “per PDU” queuing delay threshold i.e estPerPduDelivTime > rdPerPduMaxQueuingDelayTh,

o If the PDU ratio [maxNoOfPdus/(“noOfPdus” of the corresponding 5G-L2-LO instance)] is greater than the rdPduRatioThreshold, Mark the corresponding 5G-L2-LO instance as Not available, else the 5G-L2-LO instances are marked as available.

 If the estimated “per PDU” delivery time is less than the maximum “per PDU” queuing delay threshold (estPerPduDelivTime < rdPerPduMaxQueuingDelayTh), mark the corresponding 5G-L2-LO instance as available

Note:

1. rdPerPduMaxQueuingDelayTh  is an RAD parameter defined in 5G\_UP\_ALG\_5896\_replaced([11222947](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx-YRPvSEe-AqvopbP1qhQ))

2. rdPduRatioThreshold is an RAD parameter definedin 5G\_UP\_ALG\_5898\_replaced([11222961](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx-YRvvSEe-AqvopbP1qhQ))

3. avgPduSizeInRlcBuffer -(Average PDU size in RLC buffer) shall be estimated as below

o Total amount of “Bytes in the RLC buffer”/ “Number of PDUs in the RLC buffer”.

4. In case when the buffer is empty, the estimated delay shall be 0, hence both primary and secondary 5G-L2-LO instances are marked as available.

5.   eTh – the current estimated throughput as defined in 5G\_UP\_ALG\_5736\_replaced([11221148](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx0ANvvSEe-AqvopbP1qhQ))

Implementation Note:

To avoid any performance issues with floating point operations, the ratio computation results can be stored as scaled integer values.

[A more detailed note could be added during the CP3 phase].

Rationale:

The objective is to avoid the t-reassembly timer expiry at the UE, due to data sending on more than one 5G-L2-LO instance.

The rdPerPduMaxQueuingDelayTh is assumed to be a fraction of the t-reassembly timer so that:

If the per PDU queuing delay on a 5G-L2-LO instance exceeds the rdPerPduMaxQueuingDelayTh, it has to be checked if the difference in the number of PDUs that can be sent on Primary and secondary 5G-L2-LO instances is within the limit (rdPduRatioThreshold) to ensure that the SNs of the PDUs delivered at the UE are not too far apart which could result in t-reassembly expiry.

## 4.2.0-47 Flow Control on LoData Interface Activation and deactivation of TCP Boost Proactive Uplink Grants (PUG) (ID: 11221931)

**4.2.0-47.0-1**  (ID: `11221937`)
