## 4.8.0-3 L4S - Low Latency, Low Loss, Scalable throughput L4S Queue Classifiction and Marking (ID: 11224242)

**4.8.0-3.0-1**  (ID: `11224247`)

L4S - Low Latency, Low Loss, Scalable throughput The 5G-L2-HI shall classify the packets based on ECN bits of IP inner header.

When ECN bits modulo 2 == 1 i.e. ECN bits = ECT(1) or CE

packet goes to the so-called L-queue

else i.e. ECN bits = not-ECT or ECT(0)

packet goes to the C-queue

Where the ECN bits stand for:

+-----+------+

| ECN FIELD |

+-----+------+

ECT CE

0 0 Not-ECT

0 1 ECT(1)

1 0 ECT(0)

1 1 CE

The classification is only used for double-queue variants.

[CB010936-A-UPD001]

"Marking" means setting the ECN bits to 11 (i.e. Congestion Experienced, CE), which requires a recalculated IP Header checksum.

[End CB010936-A-UPD001]

**4.8.0-8.0-1.0-1**  (ID: `11224319`)

L4S - Low Latency, Low Loss, Scalable throughput 11224319: L4S - Low Latency, Low Loss, Scalable throughput Selecting the NRDRB_ L4S.l4sDlVariant = 1 activates the DL L4S "Variant 1" which is intended for best-effort bearers, where the non-L4S queue delay may increase to several hundred ms. First, the queue classifier splits the traffic and sends C-packets to a regular queue managed by PI2 (like CB008441), and L-packets to a shallow queue where the packets are marked in case of congestion. Both queues are merged again using the WRR dispatcher with priority set to L-traffic (e.g. 90/10) to ensure L4S priority.

A virtual queue is applied to L4S traffic to limit the L4S rate to a target of X1% of the bearer capacity (estimated by the DDR). If there is no L4S, the classic traffic can use all the capacity regardless of WRR and X1, X2 settings. If there is no Classic, L4S would use up to X1% of the bearer capacity. Note that the closer X1 is to 100%, the greater the risk of latency spikes as capacity fluctuates. If there is classic traffic, X2 defines the bandwidth sharing factor between the two queues. If X2 is 1, then both L4S and Classic share the bandwidth equally.

This would work as long as the Marker/Discard/PI target of both queues are greater than the target DU delay (maxRlcQueueingDelay * 2). Otherwise, the CU will try to limit the rate to reduce flow control delays, which will have no effect.

The X2 cannot be so high that L4S obtains a throughput share higher than its WRR weight ratio. For example, if X2 is such that L4S receives 95% BW and WRR weight is 90%, then 5% of L4S traffic will be blocked by the CU creating a queue.

The NRBTS.l4sFeedbackValidityPeriod parameter defines the period after which DDR will no longer be considered and 5G-L2-HI (CU) will stop the Virtual Queue based marking.

It will be up to the implementation whether marking/discarding will be done first and then the WRR, or vice versa.

L4S - Low Latency, Low Loss, Scalable throughput 5G-L2-Hi shall determine the rates RC, RL and R for the C-traffic, L-traffic, and combined traffic respectively based on the number of bytes that the CU sends to the DU in the period between one DDDS and the next.

[CB010936-A-UPD002]

5G-L2-Hi shall determine the smoothed rate SRC from the rate RC through an Exponentially Weighted Moving Average (EWMA) filter using the following formula:

SRC(t) = alpha * RC(t) + (1 - alpha) * SRC(t-1)  


Where,

RC(t) is current traffic rate

SRC(t) is the current average rate

SRC(t-1) is the previous average. Initial value when previous average is not computed for SRC(t-1) shall use NIDD parameter: minThFlowCtrl.

alpha is calculated as: alpha = min(1, T_update / 25ms) / 16.

where T_Update is the time difference in milliseconds between consecutive DDDS.

[End CB010936-A-UPD002]

L4S - Low Latency, Low Loss, Scalable throughput [Before CB010936-A-UPD003]

This function gives the probability based on the marking factor. To do this, it counts the number of marked packets (m) and the total number of packets in a period of T_ Update = 25ms. It then calculates the probability as p = m/t. It cannot be used directly to drop Classic packets as this would be too aggressive, so it is smoothed in the next block of "Prob. smoother".

[CB010936-A-UPD003]

This function gives the probability based on the marking decision of the real or virtual Step markers (which only check if that packet is exceeding either of the thresholds, not considering if the packet is classic or L4S). To do this, it counts the number of "mark decisions" (m) and the total number of packets in a period of T_ Update = 25ms. Note these counters need to count all packets and mark decisions, not taking into account its ECN/L4S capability. It then calculates the probability as p = m/t. The mark decision for an L4S capable packet is applied by setting the ECN field to CE (0x03). For a classic packet this marking decision is only used to contribute to the drop probability calculation (next to the L4S packet mark decisions). Both this mark decision and p (the average marking probability in the 25ms interval) cannot be used directly to drop Classic packets as this would be too aggressive. The calculated p first needs to be smoothed in the next block of "Prob. smoother" and that output needs to be applied as a drop probability.

Note that the counters are pegged only by the packets that are actually marked and send to the UE.

If p=m/t (of last 25 ms) causes a packet to be discarded, then the discarded packet shall be counted for the next t and for the next m if the packet exceeded the marking threshold.

[End CB010936-A-UPD003]

This block is only valid if rdL4sDropClassic = 1. If it is 0, all classic (non-L4S) traffic of Variant 3 is dropped unconditionally, and if it is 2, the classic traffic is never dropped.

L4S - Low Latency, Low Loss, Scalable throughput The "Prob. smoother" is an Exponentially Weighted Moving Average (EWMA) filter of the marking probability, with a gain of 1/16 (default when T_ Update = 25ms). When T_Update is less than 25ms, gain = min(1, T_update / 25ms)/16. The probability is then divided by K = NRBTS.l4sDualPi2CouplingRatio (default 2) and squared. If this amount consistently exceeds a given threshold, gNB marks a certain fraction of the packets.

The parameter NRDRB_ L4S.l4sUlSystematicOffset (which is included in calculation of ƒ see later for details) is added in case of "manual" tuning when it is found that the marking/discarding is too aggressive, which may lead to underutilization of the UL. The NRBTS.l4sFeedbackValidityPeriod parameter defines the period after which UL congestion information will no longer be considered and 5G-L2-HI (CU) will stop marking and dropping in UL.

The „Smooth Marking Probability“ block is used to adjust the marking probability to Prague Congestion Control for low throughput bearers, to avoid a larger queue bias when the marking probability needs to be bigger (when the rate needs to be low).

---

**4.8.0-11.0-3.0-1**  (ID: `11224496`)

L4S - Low Latency, Low Loss, Scalable throughput Since it cannot be ruled out that UE pushes Classic traffic to the L4S-enabled bearer, excess Classic traffic in such a bearer will have to be discarded so that the sender of UE Classic traffic backs off. TCP is very sensitive to discard, so a drop of a fraction of a percent won't hurt performance, but will still allow the 5G-L2-HI (CU) to control Classic rate. Excessive Classic traffic is detrimental to bearer latency because both types of traffic are maintained by the UE as a single queue per bearer, however it can be disabled with _NRDRB_ L4S.l4sUlLatencyShield = false, as in the "?" decision block in the diagram._

The "Smooth Dropping Probability" is an Exponentially Weighted Moving Average (EWMA) filter of the marking probability, with a gain of 1/16 (default when T_ Update = 25ms).

Due to PR815672 it was decided to suspend the "Classic Drop" action when last UCI == 0, so that psm calculations are performed as usual and only packets are not discarded.
