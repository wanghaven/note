# 4.5 PDCP Uplink reordering window handling

## 4.5.0-2 PDCP Uplink reordering window handling Handling of UL PDCP SDUs in reordering window due limited buffering capability (ID: 11223864)

**4.5.0-2.0-2**  (ID: `11223879`)

PDCP Uplink reordering window handling If the number of PDCP SDU in PDCP reordering queue >= [Before CB008963-B] rdUlPdcpReorderingElementCount

[CB008963-B] min(20% of Window_Size, rdUlPdcpReorderingElementCount) [End CB008963-B],

5G-L2-HI shall discard new PDCP SDU and perform the below action.

Deliver to upper layers in ascending order of the associated COUNT value ,

  * Next PDCP SDU starting from RX_DELIV.
  * all stored PDCP SDU(s) with consecutively associated COUNT value(s) starting from SDU delivered previously.



Update RX_DELIV to the COUNT value of the last PDCP SDU delivered to upper layers;

Update RX_DELIV to the COUNT value of the last PDCP SDU delivered to upper layers + 1;

If there is at least one stored PDCP SDU:

  * restart t-Reordering;
  * starting from RX_DELIV search the next stored PDCP SDU and note it’s COUNT value by x. update RX_REORD to x + 1



5G-L2-HI shall write a log note about discarding PDCP SDUs. Log note shall identify the impacted bearer.

Rationale:

[CB008963-B] In worst case, the number of PDCP SDU waiting for reordering is based on the Window_Size. Since the value is quite large when compared to the buffering capability, we need to limit the number of PDCP PDU waiting for reordering.

Since 12-bit PDCP SN is introduced, earlier value of rdUlPdcpReorderingElementCount is very high. Hence, a different MAX value needs to be defined based on buffering capability. So, minimum value is selected between %age of window size and rdUlPdcpReorderingElementCount. [End CB008963-B]
