# 4.34 CPU Load Handling

## 4.34.0-2 CPU Load Handling CPU load calculation in CU (ID: 11228962)

**4.34.0-2.0-2**  (ID: `12525250`)

CPU Load Handling 5G-L2-HI in gNB-CU shall calculate CPU Load in gNB-CU as shown below.

5G-L2-HI shall calculate the _L2HiLoad as below,_

_AverageL2HICPULoad_ new _= (rdAvgL2HICPULoadFilteringConst * AverageL2HICPULoad_ old _+__(100 -rdAvgL2HICPULoadFilteringConst) * current measured L2HI CPU load)/100._***** *

where, _current measured L2HI CPU load =_

(sum of instantaneous CPU load for all cores which are used by 5G-L2-HI)/number of L2HI core(s)

rdAvgL2HICPULoadFilteringConst is R&D parameter defined in 5G_UP_ALG_5927

The _current measured L2HI CPU load shall be calculated every 100ms._

Update _L2HiLoad =_ AverageL2HICPULoad _new._ **

[Before CB013725-J]

5G-L2-HI shall calculate the _TrswLoad_ as below,**

**_AverageTrswCPULoad_ new _= (rdAvgTrswCPULoadFilteringConst * AverageTrswCPULoad_ old *+ (100 - rdAvgTrswCPULoadFilteringConst) * current measured Trsw CPU load)/100.*******

where the _current measured Trsw CPU load =_

(sum of instantaneous CPU load for all cores which are used by TRS SW)/number of TRSW core(s))

rdAvgTrswCPULoadFilteringConst is R&D parameter defined in 5G_UP_ALG_5929

The _current measured Trsw CPU load shall be calculated every 100ms by calling the RCP API._

Update _TrswLoad =_ AverageTrswCPULoad _new_ **

[End CB013725-J]

UPUE Load is defined as indicated below

[Before CB013725-J]

_UPUELoad= Max (_ L2HiLoad, _TrswLoad)._ **

The _UPUELoad is updated periodically every time_ L2HiLoad or _TrswLoad is updated._ **

_NOTE:_ L2HiLoad and _TrswLoad shall be set to zero during the load measurement configuration._ **

[CB013725-J]

_UPUELoad=_ L2HiLoad**

The UPUELoad is updated periodically every time L2HiLoad is updated.

NOTE: L2HiLoad shall be set to zero during the load measurement configuration.

[End CB013725-J]

## 4.34.0-3 CPU Load Handling CPU load calculation in DU (ID: 11228977)

**4.34.0-3.0-1**  (ID: `11228984`)

CPU Load Handling 5G-L2-HI in gNB-DU shall calculate CPU Load in gNB-DU as shown below.

5G-L2-HI shall calculate the L2HiLoad as below,

AverageL2HICPULoadnew = (rdAvgL2HIDUCPULoadFilteringConst * AverageL2HICPULoadold + (100 – rdAvgL2HIDUCPULoadFilteringConst ) * current measured L2HI CPU load)/100.

where, current measured L2HI CPU load =

(sum of instantaneous CPU load for all cores which are used by 5G-L2-HI)/number of L2HI core(s)

The current measured L2HI CPU load shall be calculated every 100ms by calling BBP API: AaPro Cpu load data.

Update L2HiLoad = AverageL2HICPULoadnew.

UPUE VM Load is defined as indicated below

UPUELoad= L2HiLoad.

The UPUELoad is updated periodically every time L2HiLoad is updated.

NOTE: L2HiLoad shall be set to zero during the load measurement configuration.

For rdAvgL2HIDUCPULoadFilteringConst refer to 5G_UP_ALG_5933.

## 4.34.0-7 CPU Load Handling CPU Overload Handling (ID: 11229056)

**4.34.0-7.0-1**  (ID: `11229063`)

CPU Load Handling General

---

**4.34.0-7.0-1.0-1**  (ID: `11229069`)

CPU Load Handling This section details the overall CPU Overload handling in 5G-L2-Hi.

The CPU Overload handling is implemented in multiple steps with each step successively reducing the processing load. The actions are staged in the increasing order of their impact to the core functionality of L2-HI, the least impacting first and the most impactful at the end. Each action is undertaken independently and sufficient time gap allowed for the load reduction.

The actions are undertaken in the following order:

  1. Disable the non-critical functions

  2. Flow Control Actions

  3. PDCP SDU Discards

Rationale

a) DRB Selection

The following DRB selection is proposed

  * Option 1
  * DRB have higher X2-U TP shall be preferred over X2-U
  * DRB have higher F1-U TP shall be preferred over F1-U

  * Option 2

  * DRB with lower number of PDCP SDU shall be sent over X2-U.

  * Option 3

  * Random selection.



Option 3 was chosen as other option lead to SW complexity and CPU overhead. Details of Random selection is defined is implementation detail.

b) DL data split disabling

When the DL data split is disabled, we shall sending data only over one interface. Since no data is sent on the other interface, DDDS polling shall be done with longer DDDS period(every rdMaxDDDSSkipPeriod) leading the reduced number of DDDS message. This shall result in reduction in CPU load.
