# 4.19 Handling of RLC Max Retransmission

## 4.19.0-2 Handling of RLC Max Retransmission (ID: 13387827)

**4.19.0-2.0-2**  (ID: `13365822`)

Handling of RLC Max Retransmission
For RLC PDUs that have already reached maximum number of retransmissions, 5G-L2-HI shall consider that corresponding PDCP PDU was unsuccessfully delivered.

* If threshold value of maximum number of retransmissions is set to infinity, 5G-L2-HI shall keep such PDUs in the radio bearer data buffer and continue retransmissions according to normal rules.
* If the threshold value of maximum number of retransmissions is NOT set to infinity, 5G-L2-HI shall inform the failure to the control plane. The corresponding RLC PDU(s) shall be retained in the radio bearer data buffer but no further [CNI-152894-A] RLC transmissions and [End CNI-152894-A] retransmissions shall be initiated. 5G-L2-HI shall ensure that only first occurrence (per bearer) of reaching max RLC retransmissions triggers the failure message to the control plane. [CNI-152894-A] 5G-L2-HI shall discard RLC SDUs stored in the RLC buffer after reaching Max RLC retransmission limit. [End CNI-152894-A]

Note:

1. A special value outside of the 3GPP range is introduced for infinite re-transmissions. This is only for testing purposes, and this value will not be exposed in the NIDD. This value is only available on the HiUserDu interface.
2. 3GPP specification does not clearly state what should be done with the RLC PDUs, which have been retransmitted Maximum number of retransmissions other than indicating the error to C-plane (RRC). Hence, the RLC PDU is kept in the buffer until the C-plane initiates the action, but RLC PDUs shall not be further [CNI-152894-A] transmitted or [End CNI-152894-A] re-transmitted. [Before CNI-152894-A] Also the data transmission for which the bearer is not stopped as the number of UEs supported in 5G17 is very low. In future releases we can stop the transmission on the affected bearer to save the air interface resources. [End CNI-152894-A]
3. [CNI-152894-A-UPD001] A gNB internal flag, stopRlcTxAfterMaxRetx (located in NRBTS.actInternalImprovements1, Bit 9), shall be implemented to control the RLC entity's behavior after the Max RLC retransmission threshold has been reached.
4. When stopRlcTxAfterMaxRetx is set to 1 (Default): Upon reaching the maximum number of RLC retransmissions, the RLC entity shall be prevented from performing further transmissions of the RLC PDUs. RLC SDUs remaining in the buffer shall be discarded.
5. When stopRlcTxAfterMaxRetx is set to 0: Upon reaching the maximum number of RLC retransmissions, the RLC entity shall be allowed to continue transmissions of the RLC PDUs in the buffer, and the RLC SDUs shall be retained in the buffer. [End CNI-152894-A-UPD001]
