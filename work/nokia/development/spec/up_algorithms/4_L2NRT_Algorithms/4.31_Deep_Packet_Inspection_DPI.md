**4.31.0-2.0-1.0-2**  (ID: `11228729`)

Deep Packet Inspection (DPI) Selection of DRB for monitoring

---

**4.31.0-2.0-1.0-2.0-1**  (ID: `11228735`)

Deep Packet Inspection (DPI) gNB-CU-UP shall select the UL/DL DRB for DPI monitoring as soon as the DRB setup is completed by applying the following **selection criteria:**

1\. If the actDpiGoodput **[CB013979-A] or/and actDpiRtt**[End CB013979-A] is enabled for the 5QI type of the DRB profile****

2\. Current active count of DRBs for DPI are less than configured maximum DRB (dpiMaxDrb)

3\. Current active count of TCP flows are less than configured maximum TCP flows (dpiMaxTcpFlows)

4\. Processing units are not in Overload condition

DRBs matching all the above criteria are selected for DPI monitoring after DRB setup since gNB-CU is not aware of the type of traffic (viz. TCP/UDP) expected to flow on this DRB until the start of the data flow. Hence, dpiTcpWaitTime shall be started to monitor TCP traffic.

Once the DL DRB is selected for DPI monitoring, gNB-CU-UP shall notify the associated UL DRB about the selection so that associated UL DRB can create DRB DPI context and monitor TCP packets only for those DRBs. So, when DL DRB is setup and selected for monitoring, the same is notified to associated UL DRB of the same UE and vice versa.

Once the DRB is selected for monitoring, DPI monitoring is started based on criteria described in 5G_UP_ALG_32518.

Target gNB-CU-UP shall select UL/DL DRB for DPI monitoring after handover (like Inter-gNB HO, Inter-CU HO, Inter-RAT HO to gNB) as part of the DRB setup.

**4.31.0-2.0-1.0-3**  (ID: `11228742`)

Deep Packet Inspection (DPI) Start DPI monitoring based on entry criteria

---

**4.31.0-2.0-1.0-3.0-1**  (ID: `11228748`)

Deep Packet Inspection (DPI) gNB-CU-UP shall start DPI monitoring for the selected DRB only when the following criteria is met:

  1. Current active count of TCP flows monitored across all the DRB(s) are less than or equal to configured TCP flows (dpiMaxTcpFlows) [PR848258] or maximum range of configured TCP flows (maximum range of dpiMaxTcpFlows in NIDD) as detailed in Note 1 [End PR848258].



**Note 1: If current TCP flow count reaches configured maximum TCP flows (dpiMaxTcpFlows) and in case there are more new TCP flows connecting for the monitored DRBs, then DPI procedure shall support monitoring for new TCP flows connected until the pre-calculated maximum memory limit. This is to get the maximized DPI results from the monitored DRB.**

In this case, no more DRBs shall be selected for DPI monitoring.

Note 2: Monitoring higher number of bearers causes higher CPU load impact and hence it is limited using dpiMaxDrbs NIDD parameter. It is found that monitoring more TCP flows does not cause high CPU load impact compared to high amount of bearer monitoring and hence monitoring more number of TCP flows upto maximum range of dpiMaxTcpFlows was recommended.

  1. First TCP packet is received on the DRB within a timer period of dpiTcpWaitTime.



**Rationale: Some DRBs may send UDP/other traffic initially or continuously and hence this timer is used to determine if there is any TCP traffic within this time limit, then only DPI monitoring is started. This timer shall also be used to determine if TCP flow is continuous (timer is restarted on receiving every TCP packet) and if there are big gaps in TCP traffic and timer expires, then that DRB shall be removed from the DPI monitoring list.**

gNB-CU-UP shall detect new TCP flows, monitors TCP flows within a TCP connection handled by the monitored DRB and stores contextual information related with DRB and TCP flows.

For TCP Goodput measurement, after the creation of DRB and TCP flow context as specified in 5G_UP_ALG_32626, gNB-CU-UP shall perform DPI monitoring as described in 5G_UP_ALG_32599 and compute goodput as specified in 5G_UP_ALG_32602.

[05492012] 

Note: Occationally few packets of a given TCP flow can be mapped to a different QFI/5QI which lead to change in mapping of QoS to DRB. This can impact accuracy of the DPI measurements.

[End 05492012]

**4.31.0-2.0-1.0-4**  (ID: `11228756`)

Deep Packet Inspection (DPI) Stop DPI monitoring based on exit criteria

---

**4.31.0-2.0-1.0-4.0-1**  (ID: `11228764`)

Deep Packet Inspection (DPI) gNB-CU-UP shall stop DPI monitoring when any one of the exit criteria listed below is reached:

DPI monitoring and measurement shall be stopped for the monitored TCP flow on a selected DRB

  1. When TCP RST or TCP FIN is received on a TCP flow



DPI monitoring and measurement shall be stopped for selected DRBs (and for all TCP flows) on following conditions:

  1. Number of TCP packets monitored greater than dpiMaxDlFirstPackets or dpiMaxUlFirstPackets for the DRB across all TCP flows.



**Rationale: Number of TCP packets for the whole DRB across all TCP flows shall be compared with the threshold. If it is checked on TCP flow basis, there are chances that packets flowing in a TCP flow may not reach Max packets threshold due to less traffic and hence monitoring cannot be stopped for long time.**

  1. DRB release due to any condition

  2. When TCP packets are not received until the expiry of dpiTcpWaitTime timer. After timer expiry, DRB shall be removed from DPI monitoring.

  3. On Inter-gNB handover, at target gNB DPI measurement feature is not activated on that DRB profile.

  4. In case dpiFirstMilliseonds is set to negative value and the dpiFirstMilliSeconds timer expires.




[Before 05426021]

  1. In case DPI monitoring is stopped for all TCP flows of the DRB.



[End 05426021]

[PR848258]

  1. While trying to add new TCP flow for a DRB to monitoring list and when TCP flow count for all DRBs exceeds the pre-calculated maximum memory limit (for usage about pre-calculated maximum memory limit refer Note: 1 in: [11228742: Start DPI monitoring based on entry criteria](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kykOIvvSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)), DRB monitoring shall be stopped for current DRB (which is attempting to add a new TCP flow to monitoring list) and PM counters shall be reported. This is done to give space for additional TCP flows being added for existing DPI monitored DRBs. Maximum memory limit shall be MAX range configuration value of dpiMaxTcpFlows in NIDD.



[End PR848258] 

**Note: When gNB-CU-UP stops DPI monitoring in one direction, it shall stop DPI monitoring of the bearer related to that UE in the opposite direction.**

DPI monitoring and measurement shall be stopped for all monitored DRBs:

  1. Overload condition is reached and handled as described in 5G_UP_ALG_32583.



When the feature is deactivated online, gNB-CU-UP shall continue DPI monitoring for already monitored DRBs and update PM counters, but no new DRBs shall be added to monitoring list in this case.

Once the DPI monitoring is stopped for the DRB, gNB-CU-UP shall update the PM counters related with the TCP measurements and the DRB shall be removed from the DPI monitoring list.

**4.31.0-2.0-1.0-5**  (ID: `11228769`)

Deep Packet Inspection (DPI) Creation of DRB and TCP flow context

---

**4.31.0-2.0-1.0-5.0-1**  (ID: `11228776`)

Deep Packet Inspection (DPI) gNB-CU-UP shall maintain DRB DPI context in the TCP packet flow direction which stores information about multiple TCP flow context and received TCP packet count for comparing with dpiMaxDlFirstPackets or dpiMaxUlFirstPackets threshold. DRB DPI context also accumulates Volume and Active time reported from each TCP flow when TCP flow is closed.

gNB-CU-UP shall store TCP flow related contextual information in TCP flow context for each DRB. gNB-CU-UP shall detect new TCP flow when first TCP packet is received in a direction (which means a TCP segment is successfully transmitted in the reverse direction). For Example, first TCP packet is received in UL direction based on first TCP segment successfully transmitted to UE in DL direction and first TCP packet received in DL direction based on first TCP segment successfully sent to TCP server in UL direction.

Target gNB-CU-UP shall detect the first TCP packet for each TCP flow in case of incoming handover (when gNB-CU-UP changes like Inter-gNB HO, Inter-CU HO, Inter-RAT HO to gNB, etc) in which case, TCP connection is already established through the source gNB (which will already have some existing TCP flow contexts at source gNB) and hence target gNB-CU-UP shall start DPI monitoring for existing and newly established TCP flows as described in 5G_UP_ALG_32518. Forwarded packets from source gNB shall also be monitored for incoming TCP packets.

Note: In Intra-gNB HO, Intra-DU HO, Inter-DU HO, Intra-Frequency or Intra-cell HO, gNB-CU-UP is not changed and hence DPI monitoring shall be continued with the existing TCP flow contexts of the DRB.

Each TCP flow context requires some (but not limited to) of the following information for book-keeping:

TCP flow related information {Src IP, Dest IP, Src Port, Dest Port}, Previous TCP ACK SN (for computing goodput volume or measuring short volume burst), Previous TCP ACK received time (for computing total time and inactive time), Goodput measurement direction, TCP flow volume, TCP flow total time, TCP flow inactive time, etc.

**4.31.0-2.0-1.0-6**  (ID: `11228783`)

Deep Packet Inspection (DPI) CPU Overload control

---

**4.31.0-2.0-1.0-6.0-1**  (ID: `11228789`)

Deep Packet Inspection (DPI) On CPU overload, gNB-CU-UP shall reuse the existing overload control mechanism specified in 5G_UP_ALG_13357.

On reaching rdCpuNonCriticalLoadThreshold CPU load threshold, DPI monitoring shall be done only for current monitored DRBs as described in 5G_UP_ALG_13514. New DRBs setup shall not be added to the monitoring list as given in 5G_UP_ALG_32517.

On reaching rdCpuCriticalLoadThreshold CPU load threshold, DPI monitoring shall remove/flush the list of currently monitored DRBs after reporting the PM counters as in 5G_UP_ALG_13515. DPI monitoring is stopped and no new DRBs will be selected or monitored

On reaching rdCpuNormalLoadThreshold CPU load threshold, DPI monitoring shall be started on those DRBs that are newly setup as in 5G_UP_ALG_13366.

**4.31.0-2.0-2.0-1**  (ID: `11228803`)

Deep Packet Inspection (DPI) Goodput measurements

---

**4.31.0-2.0-2.0-1.0-1**  (ID: `11228809`)

Deep Packet Inspection (DPI) DL goodput measurement shall be performed in UL direction just before TCP packets are sent to core after all PDCP procedures are completed in UL.

Downlink DPI monitoring: core -> CU -> UE -> CU [TCP ACK monitoring before sending] -> core

For DPI monitoring, each UL data packet (after all PDCP procedures are completed), gNB-CU-UP shall check if it is a TCP packet. If the packet is a TCP packet, then TCP header shall be checked for presence of TCP ACK. If TCP ACK is present, then computation of DL Goodput shall be handled based on 5G_UP_ALG_32602.

Similarly, UL goodput measurement shall be performed in DL direction as soon as TCP packets are received at gNB-CU-UP from core before any PDCP procedures are initiated in DL.

Uplink DPI monitoring: UE -> CU -> core -> [TCP ACK monitoring when received] CU -> UE

For DPI monitoring, each DL data packet (when received at gNB-CU-UP), gNB-CU-UP shall check if it is a TCP packet. If the packet is a TCP packet, then TCP header shall be checked for presence of TCP ACK. If TCP ACK is present, then computation of UL Goodput shall be handled based on 5G_UP_ALG_32602.

gNB-CU-UP shall restart the dpiTcpWaitTime timer on receiving each TCP packet to continue monitoring. When TCP packets are not received for more than dpiTcpWaitTime and the timer expires, then DPI monitoring is stopped.

**4.31.0-2.0-2.0-2**  (ID: `11228815`)

Deep Packet Inspection (DPI) Goodput calculations

---

**4.31.0-2.0-2.0-2.0-1**  (ID: `11228820`)

Deep Packet Inspection (DPI) gNB-CU-UP computes Volume considering difference of consecutive TCP ACK SNs for each TCP flow ignoring short volume bursts. Short volume burst shall be computed as small data transfers less than or equal to dpiMinTrafficVolThreshold in a period of dpiInactivityDetectionTime or greater. gNB-CU-UP shall compute Total volume considering the data transfer happening on all TCP flows. For example, if v1 is volume of transfer in first TCP flow, v2 is volume of transfer in second TCP flow and so on, then

Total volume for the DRB for active time = v1 + v2 + ...

gNB-CU-UP shall compute active time at the DRB level storing it under DRB DPI context. It is done as follows:

Total time of transfer for DRB = min(time when last TCP ACK was received for the DRB, time when DPI monitoring is stopped) - time when first TCP packet was detected for the DRB

Note: min(time when last TCP ACK was received for the DRB, time when DPI monitoring is stopped) is needed to ensure that the amount dpiTcpWaitTime is not included in the calculation if DPI monitoring is stopped due to dpiTcpWaitTime expiry

Inactive time for DRB = Cumulative time difference between consecutive TCP ACKs across all TCP flows at the DRB level which is greater than or equal to dpiInactivityDetectionTime and small data transfers less than or equal to dpiMinTrafficVolThreshold [PR879555] and no TCP DUPACKs are detected within the inactivity period. 

Inactive time computation is suspended as specified in [13464780: DUPACKs Handling](https://dn-prod.ext.net.nokia.com/rm/resources/BI_GKR2ADQtEfGxldjOZQLezQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9) when DUPACKs are detected. **[End PR879555]**

Active Time for the DRB = Total time of transfer for DRB - Inactive time for DRB

gNB-CU-UP shall compute:

Goodput = Total volume for the DRB for active time / Active time for the DRB

and shall update the Goodput computed to relevant PM counter histogram bin after DPI monitoring is stopped based on exit criteria described in 5G_UP_ALG_32519.

**4.31.0-2.0-2.0-3**  (ID: `13464780`)

Deep Packet Inspection (DPI) **DUPACKs Handling**

---

**4.31.0-2.0-2.0-3.0-1**  (ID: `13464781`)

Deep Packet Inspection (DPI) During periods of TCP packet loss, a TCP receiver generates a series of Duplicate Acknowledgment (DUPACK) packets to signal the need for retransmission. From a simple traffic volume perspective, this recovery period can be indistinguishable from genuine user inactivity, as no new data is being acknowledged. By specifically detecting the DUPACK pattern, the gNB-CU-UP can distinguish between genuine user inactivity and a network-level recovery event. This prevents the gNB from incorrectly triggering inactivity time procedures, which leads to more accurate goodput measurements.

The gNB-CU-UP shall implement a mechanism to detect TCP Duplicate Acknowledgment (DUPACK) patterns for each TCP flow on a DRB where Deep Packet Inspection (DPI) is active. A TCP ACK packet shall be identified as a DUPACK if it contains an acknowledgment number equal to the highest previously seen cumulative TCP acknowledgment number for that flow. The gNB-CU-UP shall track the consecutive DUPACKs and the timestamp for each received ACK. Upon detection of the first DUPACK for a flow, the gNB-CU-UP shall immediately suspend the inactivity time calculation for that DRB. The time period during which DUPACKs are being received shall not be added to the cumulative "Inactive time for DRB" even if the conditions defined by dpiInactivityDetectionTime and dpiMinTrafficVolThreshold are met. DUPACK detection shall be performed for TCP segments with TCP ACKs with or without TCP payload.

DUPACK Detection pattern:

First ACK: The gNB-CU-UP receives a TCP ACK for sequence number "x". This is TCP ACK acknowledging the highest cumulative acknowledgment number.  
Second ACK: First DUPACK: The gNB-CU-UP receives the same TCP ACK for sequence number "x" again. Receiving First DUPACK indicates the start of the DUPACK pattern and gNB-CU-UP shall suspend the inactivity time calculation.  
Third ACK: Second DUPACK: The gNB-CU-UP receives the another TCP ACK for sequence number "x" again.  
Fourth ACK: Third DUPACK: The gNB-CU-UP receives the another TCP ACK for sequence number "x" again.

The DUPACKs can be received continuously and the inactivity time computations shall be suspended until next sequence TCP ACK is received. Normal inactivity time computations shall resume only after a new TCP ACK is received that advances the cumulative acknowledgment number, indicating that the packet loss has been resolved.

For consecutive TCP ACKs carrying the same Acknowledgment Sequence Number, the gNB-CU-UP shall classify them as part of a DUPACK sequence, thereby distinguishing them from Keep-Alive ACKs, if the time interval between each consecutive ACKs does not exceed 5000 ms.

**4.31.0-2.0-3.0-2**  (ID: `11228846`)

Deep Packet Inspection (DPI) RTT and jitter calculations

---

**4.31.0-2.0-3.0-2.0-1**  (ID: `11228853`)

Deep Packet Inspection (DPI) A TCP segment is considered acknowledged when the TCP "Acknowledgment Number" (received in the other direction) is [Before 05426021]greater than or equal to its TCP "Next Sequence Number".[05426021]greater than its TCP Sequence Number. Alternatively, the TCP header has the TCP SACK option and the TCP Sequence Number is greater than or equal to the SACK Left Edge and less than the SACK Right Edge. Consider that up to four SACK blocks can be reported within a single TCP packet.[End 05426021]

The RTT will be derived as the timestamp difference between the original arrival of the packet and its acknowledgement. Furthermore, jitter will be the absolute value of the difference between the current RTT and the previous one.

Note that TCP segments of length 0 (and those without SYN or FIN flags) do not change the TCP Sequence Number and therefore cannot be specifically acknowledged. So the RTT for these packets cannot be calculated.

After performing these calculations, the corresponding counters shall be pegged:

 RTT sum

 RTT sample count

 jitter sum

 jitter sample count

 number of measurements exceeding the threshold set by dpiRttMaxDlThreshold/dpiRttMaxUlThreshold
