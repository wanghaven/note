## 2.1.1 L1 Variants

### 2.1.1.0-1 L1 Variants General (ID: 11094074)

**2.1.1.0-1.0-1**  (ID: `11094078`)

L1 Variants In the whole chapter, 'L1 Variant' attribute shall be filled for all the FUN requirements which are linked to 5G-L1-DL/5G-L1-UL as described in the following:

  * **L1 Variant is set to ‘generic’**
  * When the requirement is generic and it does not depend on L1 HW capability or restriction; typically:
  * Support of 3GPP functionalities/configurations
  * Specification of precoding matrices
  * Etc
  * The requirement could be common or specific for a given technology
  * The requirement could be supported by one or multiple L1 HWs in a given release and it’s linked to the 1st subfeature introducing the functionality
  * If the requirement is ported to another L1 HW, the porting subfeature shall be linked only to the configuration requirement which is L1 HW specific
  * it’s recommended to add references to the configuration requirements in the respective generic one
  * L1 Variant shall not be set to another value (or L1 HW specific)
  * A configuration requirement indicating the L1 HWs supporting the generic requirement should be added (see next bullet)

  * **L1 Variant is set to one or more L1 HWs**

  * When the requirement depends on L1 HW capability/restriction; typically:
  * UL receiver algorithms
  * Configuration requirements which give the list of L1 HWs supporting generic requirements
  * The requirement shall add reference to the generic requirement
  * The requirement could be common or specific for a given technology



L1 Variant shall not include ‘generic’

Generally, requirements with L1 Variant equals to ‘generic’ and the respective configuration requirements (providing the list of L1 HWs) are located in the same chapter

### 2.1.1.0-2 L1 Variants Updating requirements with "L1 Variant" set to Generic (ID: 12776202)

**2.1.1.0-2.0-1**  (ID: `12776677`)

L1 Variants When updating an existing functional requirement with "L1 Variant" set to true (i.e. adding new version), two cases are possible:

  * Case1: If the new update applies only for a subset of L1 Variants (i.e. not applicable for all L1 Variants supporting the legacy FUN requirement): 
  * A tag/link to the subfeature introducing the new update shall be added in L1 HW configuration requirement
  * Case 2: If the new update applies for all L1 Variants supporting the legacy FUN requirement
  * Not to add tag/link to the subfeature introducing the new update in L1 HW configuration requirement

## 2.1.2 Linking to NEs

### 2.1.2.0-1 Linking to NEs General (ID: 11094106)

**2.1.2.0-1.0-1**  (ID: `11094109`)

Linking to NEs In the whole chapter, the requirements shall be linked to Layer 1 NEs, see [11369597](https://dn-prod.ext.net.nokia.com/rm/resources/BI_udq8YvvkEe-AqvopbP1qhQ).

Following NEs shall be used:

  * 5G-L1-UL (BB) and/or 5G-L1-DL (BB) if the functional requirement is related to physical layer or FrontHaul procedure/functionality/algorithm/etc in the BaseBand (BB)
  * 5G-L1-UL (Radio) and/or 5G-L1-DL(Radio) if the functional requirement is related to physical layer or FrontHaul procedure/functionality/algorithm/etc in the Radio
  * 5G-L1-UL (FHE-CPRI GW) and/or 5G-L1-DL (FHE-CPRI GW) if the functional requirement is related to physical layer procedure/functionality/etc in the FHE

## 2.1.4 SFS requirements wrt eCPRI FH configuration

### 2.1.4.0-1 SFS requirements wrt eCPRI FH configuration When a requirement is valid for all the supported eCPRI FH configurations (ID: 11094128)

**2.1.4.0-1.0-1**  (ID: `11094132`)

SFS requirements wrt eCPRI FH configuration When a FUN requirement is valid for all the supported eCPRI FH configurations (with “L1 Variant” attribute set to generic or to subset of L1 HWs):

Case1: if the requirement applies to the DU for all the FH configurations (i.e. CPRI/eCPRI/etc):

  * The use of the word “eCPRI”, “CPRI”, etc shall be avoided to keep a generic requirement
  * Only a link to 5G-L1-UL (BB) or 5G-L1-DL (BB) shall be added



**Examples:[11101692](https://dn-prod.ext.net.nokia.com/rm/resources/BI_knauFPvSEe-AqvopbP1qhQ), [11103657](https://dn-prod.ext.net.nokia.com/rm/resources/BI_knyhjPvSEe-AqvopbP1qhQ)**

Case2: If the requirement is valid only for eCPRI FH configurations (i.e. eCPRI 7-2a/eCPRI 7-2e/eCPRI 7-2b ULPI/ etc)

A) If the full requirement applies to the DU for all the FH configurations

  * Use simply the word “eCPRI” in the text to highlight the FH limitation when necessary (eg. [11109974](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ko_bYPvSEe-AqvopbP1qhQ))
  * Add a link to 5G-L1-UL (BB) or 5G-L1-DL (BB)
  * The differentiation between eCPRI 7-2a, 7-2e, 7-2b ULPI is not mandatory

  * Use the words “For eCPRI 7-2e and/or eCPRI 7-2b ULPI, DU shall….” to refer to L1 (BB) for eCPRI 7-2e and/or eCPRI 7-2b ULPI
  * Use the words “For eCPRI 7-2e and/or eCPRI 7-2b ULPI, Radio shall….” to refer to L1 (Radio) for eCPRI 7-2e and/or eCPRI 7-2b ULPI
  * Add a link to 5G-L1-UL (BB)/5G-L1-UL (Radio) or 5G-L1-DL (BB)/5G-L1-DL (Radio)

### 2.1.4.0-3 SFS requirements wrt eCPRI FH configuration L1 HW configuration requirements (ID: 11094143)

**2.1.4.0-3.0-1**  (ID: `11094147`)

SFS requirements wrt eCPRI FH configuration **Starting from August 2023, only the terminology in[11094120](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmZaYvvSEe-AqvopbP1qhQ) is allowed. The text hereafter was updated accordingly.**

For the tables in the L1 HW configuration requirements (see [11094074](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmYzVfvSEe-AqvopbP1qhQ)):

  * For the L1 Variants which do not support the split between DU and Radio for the physical layer functionalities, keep only the L1 Variant name as per [11094082](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmYzV_vSEe-AqvopbP1qhQ)
  * For the L1 Variants which support the split between DU and Radio for the physical layer functionalities, see [11094082](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmYzV_vSEe-AqvopbP1qhQ):
  * Use “L1 Variant name” to refer to the requirement for L1(BB) in a dedicated column/row
  * Before the support of eCPRI 7-2b ULPI FH: Use “L1 Variant name (Radio FH)” to refer to the requirement for L1 (Radio) in a dedicated column/row
  * After the support of eCPRI 7-2b ULPI FH: Use “L1 Variant name (Radio)” to refer to the requirement for L1 (Radio) in a dedicated column/row and precise the FH after the table when needed; otherwise the FH type could be only added in the respective Generic requirement
  * Add a link to the NEs that shall support the requirement, see [11094106](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmYzZPvSEe-AqvopbP1qhQ)



For example:

  * “Thor” in the text refers to Thor in BB
  * Before the support of eCPRI 7-2b ULPI FH: “Thor (Radio 7-2e)” refers to Thor in the Radio with 7-2e
  * After the support of eCPRI 7-2b ULPI FH: “Thor (Radio)” refers to Thor in the Radio with 7-2e and/or 7-2b ULPI.
  * “Thor (FHE)” refers to Thor in the FHE with CPRI-GW
