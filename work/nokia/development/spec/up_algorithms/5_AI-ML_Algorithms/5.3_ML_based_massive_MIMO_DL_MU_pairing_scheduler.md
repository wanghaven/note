# 5.3 ML based massive MIMO DL MU pairing scheduler

## 5.3.0-1 ML based massive MIMO DL MU pairing scheduler ML based DL MU pairing algorithm only considers UEs performing WB BF as eligible (ID: 13355399)

**5.3.0-1.0-1**  (ID: `13371883`)

ML based massive MIMO DL MU pairing scheduler ML based DL MU pairing algorithm only considers UEs performing Wideband (WB) oDFT Beamforming (BF) as eligible.

Sub-band BF is supported by [12137660](https://dn-prod.ext.net.nokia.com/rm/resources/BI_0nff0gAVEfCD3seUwaI0tg?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9) . When sub-band BF is enabled (NRBTS.NRCELL.actDlSrsSubbandBm=true), UEs performing sub-band BF are excluded from the MU-eligible UEs list for both Downlink MU MIMO enhancement, refer [11142415](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kr4XVfvSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9) and ML based DL MU pairing.  
These sub-band BF UEs are treated separately under sub-band BF feature and are not part of the standard MU-eligibility processing by Downlink MU MIMO enhancement or ML based DL MU pairing .

## 5.3.0-2 ML based massive MIMO DL MU pairing scheduler ML based DL MU pairing algorithm inherits and use the same interworking solution defined in Downlink MU MIMO enhancement feature (ID: 13448799)

**5.3.0-2.0-1**  (ID: `13448801`)

ML based massive MIMO DL MU pairing scheduler ML based DL MU pairing inherits and use the same interworking solution defined in [11142415](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kr4XVfvSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9).

For example, for the interworking with Massive MIMO muting (FR1) for energy efficiency, half TXs path switch off by mute mode beamset when cell is in reducedEnergySaving. No UE will be paired when cell is in reducedEnergySaving. Half TXs path switch on by unmute mode beamset when cell is in notEnergySaving and UE will be paired when cell is in notEnergySaving in case ues are satisfied for paring.

Inherit exhaustive scheduling function interworking solution with Downlink MU MIMO enhancement, after virtual UE construction, WRR replaced with exhaustive scheduling.

**5.3.0-3.0-1**  (ID: `13375162`)

Only MU pairing (including construction of virtual UEs and determination of number of PRBs for each UE) are impacted. The other part in DL scheduler are same as legacy.

The first 3 steps (Creating CS2 list in TD scheduler; Building list of MU eligible UEs; Calculating maximum DL MU layers) are same as legacy features. Next we focus on the changes by ML based MU pairing function.

  1. Identify MU-AI eligible cell. MU-AI eligible cell is the cell which has enough MU eligible UEs. The definition of MU eligible UEs is same as legacy. The reason of identify MU-AI eligible cell is to exclude those cells which have only few UEs can be MU paired. By excluding those cells the ML model on MLIP can use more resources to serve the cells which have many UEs to be MU paired. For details refer to [13355393: Identify MU-AI eligible cells](https://dn-prod.ext.net.nokia.com/rm/resources/BI_14I6RPA-EfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9).
  2. Create the required inputs for inference. The inputs to inference models are: NormBeamPF and BeamSumRtxUeNum. 48 DNN beams for 64TRX RRU/32TRX RRU are predefined. A DNN beam is a beam that represents the groups of nearby oDFT beams. Every oDFT beam is mapped to the ‘nearest’ DNN beam. For each UE, it belongs to a DNN beam according to its best oDFT beam. The DNN beam PF (NormBeamPF) is calculated according to the MU PF values of UEs belonging to the DNN beam. BeamSumRtxUeNum is the sum of number of retransmission UEs in DNN beam. For details refer to [13355414: Construct inference inputs to MLIP for inference](https://dn-prod.ext.net.nokia.com/rm/resources/BI_aj_PxfBCEfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9).
  3. During waiting for the results from inference model, L2-PS shall construct virtual UE by using legacy CB008305 method. And calculate SumMuPF for the first virtual UE by legacy method. It will be compared with MU PF for the first virtual UE by AI ML method. For details refer to [13355427: Safety Net Check](https://dn-prod.ext.net.nokia.com/rm/resources/BI_x8i1IPBDEfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9).
  4. The results from inference model should be feedback on time. If the time exceed the predefined threshold (R&D parameter rdTimeInferenceResp, refer to [13355405: R&D Parameters](https://dn-prod.ext.net.nokia.com/rm/resources/BI_4KaCYfA_EfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9), then L2-PS will no longer wait for the feedback and perform the MU pairing by legacy method. If the feedback is on time, then gNB will construct the virtual UEs according to output (Q-value) from inference.
  5. Construct virtual UEs according to output from inference. For details refer to [13355540: Virtual UE construction](https://dn-prod.ext.net.nokia.com/rm/resources/BI_JAbUk_BREfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9).
  6. If there are still remaining PRB resources available after step 5, then continue to construct virtual UEs for MU eligible UEs by using legacy method.
  7. After constructing virtual UEs, the remaining handling are same as legacy features.

## 5.3.0-5 ML based massive MIMO DL MU pairing scheduler DNN beams definition and mapping (ID: 13355408)

**5.3.0-5.0-1**  (ID: `13355409`)

ML based massive MIMO DL MU pairing scheduler There are 48 DNN beams’ s DOA without weight are predefined as [DNN beam](https://nokia.sharepoint.com/:f:/r/sites/ASCOP/AS%20Documents/FS1TAs-CFAMs/CB/CB015500%20-%20CB015999/CFAM/CB015912%20Moving%20the%20whole%20CB009016%20content%20to%20trunk/CP1%26CP2/DNN%20beam?csf=1&web=1&e=U7dUyE)

A DNN beam is a beam that represents the groups of nearby oDFT beams. Every oDFT beam is mapped to the ‘nearest’ DNN beam. For each UE, it belongs to a DNN beam according to its best oDFT beam.

The scheduler shall map user beam (oDFT beams) to DNN beam in the following way:

  * Compute distance between DNN beam’s DOA and oDFT beam’s DOA, d = sqrt( (θh1-θh2)^2+ multiplier*(θv1-θv2)^2 ), where θh1, θv1 are horizontal and vertical directions of oDFT beam derived from Cplane based on [11296085: LCS AoA of oversampled DFT beams generation](https://dn-prod.ext.net.nokia.com/rm/resources/BI_TqFesPvcEe-AqvopbP1qhQ?vvc.configuration=https://dn-prod.ext.net.nokia.com/rm/cm/stream/_EvoPVu3rEe-rL47x_rCCmw), θh2, θv2 are the predefined horizontal and vertical directions of DNN beam.
  * For 64Trx multiplier = 2.4, and for 32Trx multiplier = 2.76
  * Each oDFT beam would map to a DNN beam with which has the shortest distance.



For the POC, oDFT beam and DNN beam mapping table can be calculated one time during cell start up and stored for use.

For the rank 4 UEs, rank downgrade is still supported, rank 1/2 beam selected for mapping this UE to DNN beam. 

Following the mapping of oDFT beams to DNN beams, each DNN beam will have its list of UEs sorted in descending order based on the UE’s scheduling weight.

**5.3.0-6.0-1**  (ID: `13355415`)

ML based massive MIMO DL MU pairing scheduler 13355415: ML based massive MIMO DL MU pairing scheduler There are two matrices need to be constructed by L2PS as the inputs for MLIP to run inference. One is NormBeamPF, the other is BeamSumRtxUEnum as show below.

NormBeamPF = (Pmatrix – mean(Pmatrix))/stDev(Pmatrix)

Where,

  * mean(Pmatrix) is the average value of the Pmatrix.
  * stDev(Pmatrix) is the standard deviation of the Pmatrix



Pmatrix is a alphaNum x DNNbeamNum matrix. Number of elements of matrix P is therefore equal to the number of different alpha values times the number of beams. For POC case, we have 6 alpha values and 48 predefined DNN beams, therefore there will be 6*48=288 elements in the P matrix.

Element pij of Pmatrix is equal to pij=Beam_PF(alpha= αi, beam=j).

Beam_PF(alpha= αi, beam=j) is computed for all UEs in every beam and every alpha as follows:

Beam-PF is a weighted sum method that is calculated as follows:

Phase 1: Initialization [Done once per beam]

  1.      1. Sort the UEs according to decreasing Priority Class. Within each priority class, the users are sorted according to decreasing scheduler metric.
  2. For each UE, determine the number of requested RBGs(RAT0) or PRBs(RAT1).



Phase 2: Determine/Calculate NumRBs to High-Priority UEs [Done once per beam]

1\. Until either all RBGs (RAT0) or PRBs (RAT1) are taken or all High-priority UEs have been assigned, sequentially assign the high-priority UEs their requested RBGs (RAT0) or PRBs (RAT1).

Phase 3: Determine/Calculate NumRBs to Non-High-Priority Ues [Done once per beam]

  1.      1. Determine the number of RBGs(RAT0) or PRBs(RAT1) untaken by the high Priority UEs.
  2. Split and assign the untaken RBs approximately equally among the non-high-priority UEs.



Phase 4: Calculate BeamPF using UE PFs and UE NumRBs. [varies with alpha and L]

  1.      1. Calculate the BeamPF as a weighted sum of UE PF(alpha, L). For initial UEs, each UE’s weight is its determined NumRB divided by total RBs. For HARQ retransmission UEs, each UE’s weight is 1 because when calculating PF factor for HARQ retransmission UE, the used PRB is same as the initial transmission allocated to UE.



Each UE’s PF(α, L) is defined as PF(α, L) =Rcurr/Ravg , which is similar to [11175151: Definition of per UE and per carrier scheduling metric for DL](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kug0nPvSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9), the difference is that MU-SINR is used instead of SU-SINR to determine current DL data rate.

  * Where the Rcurr is the current DL data rate per TTI in bytes for the UE determined by its MU-SINR(α, L) for initial transmission UE; for HARQ retransmission UE, Rcurr is the same DL data rate as initial transmission because gNB will use the same TBS for HARQ retransmission.
  * The user reported CQI can be viewed as SU-SINR. 



[image-1][image_desc]The image contains a mathematical equation. The equation is: MU-SINR(α, L) = $\frac{P/L}{P\alpha/L+I+N} = \frac{1}{\alpha + \frac{I+N}{P} \cdot L} = \frac{1}{\alpha + \frac{L}{SU-SINR}}$[/image_desc][image_path]https://storage.googleapis.com/dng_files/images/img-01151052.png[/image_path][/image-1] For P-matrix input, _L=4,_ α _i_ ∈ {0, 0.3, 0.6,0.9,1.2,1.5}** * Ravg is the average DL data rate per TTI in bytes for the UE (see [11175317: Calculation of the average 5G UE DL Data Rate](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuiCuvvSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)).

BeamSumRtxUEnum is a 1xDNNbeamNum vector. For POC case, we have 48 predefined DNN beams, therefore there will be 1 x 48=48 elements in the BeamSumRtxUEnum. Each vector is sum of the number of retransmission UEs for each DNN beam. BeamSumRtxUENum[DNNBeam_i]= sum(RtxUEaccociatedBeam_i)

Pmatrix is a 6 x 48 matrix, followed by BeamSumRtxUEnum with 1 x 48, totally 7 x 48 matrix with 336 elements. The inputs to inference is get by reading the final matrix by row.

## 5.3.0-8 ML based massive MIMO DL MU pairing scheduler Safety Net Check (ID: 13355427)

**5.3.0-8.0-2**  (ID: `13492550`)

ML based massive MIMO DL MU pairing scheduler The output from inference model is Q-value, which is a 1xDNNbeamNum vector, For POC case, we have 48 predefined DNN beams, therefore there will be 1 x 48=48 elements in the Q-value.

Q-value is used to prioritize the beams and UEs to be paired. The output from inference model should be feedback on time. If the time exceed the rdTimeInferenceResp (rdTimeInferenceResp is R&D parameter refer to [13355405: R&D Parameters](https://dn-prod.ext.net.nokia.com/rm/resources/BI_4KaCYfA_EfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)) , then L2-PS will no longer wait for the feedback and perform the MU paring by legacy method. If the feedback is on time, then gNB will construct the virtual UEs according to output (Q-value) from inference (refer to requirements [13355540: Virtual UE construction](https://dn-prod.ext.net.nokia.com/rm/resources/BI_JAbUk_BREfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)).

During the time of waiting for feedback from inference, L2-PS construct the first virtual UE by using the legacy DL MU MIMO paring method, and calculate SumMuPF of the first virtual UE for safety net.

[Before CB015912-F]

Safety net can guard against performance degradation: If SumMuPF(Legacy selected Beams) > SumMuPF(DNN selected beams) then continue to construct the virtual UEs by legacy method and MU paring is by legacy algorithm. Otherwise, construct the virtual UEs by ML method and the MU paring is by ML algorithm.

[CB015912-F]  
gNB-DU shall use rdLegacySumMuPfMultipler for SumMuPF comparison.

If (SumMuPF(Legacy selected Beams) multiplied by rdLegacySumMuPfMultipler) is greater than SumMuPF(DNN selected beams),

gNB-DU shall perform MU pairing using the legacy algorithm (ref. [11144070: Virtual UE construction](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kr7ao_vSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)).

else,

gNB-DU shall perform MU pairing using the ML-based algorithm by constructing virtual UEs according to the Q-value (ref. [13355540: Virtual UE construction](https://dn-prod.ext.net.nokia.com/rm/resources/BI_JAbUk_BREfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)).

[End CB015912-F]

SumMuPF(Legacy selected Beams) is defined as the summary of BeamPF for the first virtual UEs construct by legacy methods and mapped to associated DNN beam with details as follows:

  1.      1.         1. Construct first virtual UEs as DL MU MIMO enhancement feature.
     2. Mapping each UE within first virtual UEs to DNN beam according to [13355408: DNN beams definition and mapping](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Yhjv4_BAEfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)
     3. Calculate BeamPF for selected DNN beams according to [13355408: DNN beams definition and mapping](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Yhjv4_BAEfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)
     4. Calculate the summary of BeamPF for selected DNN beams



SumMuPF(DNN selected beams) is defined as the summary of BeamPF for DNN selected beams by MU algorithm according to [13355508: DNN beams selection for MU pairing](https://dn-prod.ext.net.nokia.com/rm/resources/BI_1_G1MPBPEfC3ocOh_n7w7A?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9).

R&D parameter rdSaftyNetCheck is used to enable/disable safety function. As SumMuPF(Legacy selected Beams) and SumMuPF(DNN selected beams) need to be print in TTI trace, those two values always need to be calculated. Parameter rdSaftyNetCheck only to control the comparation of two values and the following selection between legacy paired methods and ML based MU pairing.
