# 3.5 LoData

## 3.5.0-1 LoData Document Owner: Andrzej Cholewinski (Nokia) (ID: 10985780)

## 3.5.1 General

**3.5.1.0-3**  (ID: `10985801`)

General
5G-L2-LO receives RLC PDUs from 5G-L2-HI on its fronthaul PEP on ASIK.

On ASIK, each L2-RT core has 2 PEP with 1 EM queue attached to each PEP.

In order to get data coming from L2-HI through TRSW, 5G-L2-LO SDU core shall get RLC PDU from Fronthaul PEP  using queue\_id created with API em\_queue\_create pointing to Fronthaul PEP (Second\_device\_id).

The corresponding EQID is loDataDlRlcPduReceiverAddress.

For L2-RT, connectivity is reversed at HWAPI side. L2-RT shall still use same API em\_queue\_create as the one used in 5GC000555 but pointing now to backplane RRC to communicate with L2-HI (DU).

#### 3.5.2.1.0-4 RLC PDU Send Request Exceptions (ID: 12809651)

**3.5.2.1.0-4.0-2**  (ID: `12806311`)

RLC PDU Send Request
**Expected Exceptions:**

See MSC [11185509](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Be794fvREe-AqvopbP1qhQ): Bearer Delete in gNB-DU without User Delete.

5G-L2-HI may send [Before CB015504-C]LoData\_DlRlcPduSendReq[CB015504-C]LoData\_DlSrbRlcPduSendReq or LoData\_DlDrbRlcPduSendReq[End CB015504-C] to 5G-L2-LO for LCID that does not exist if bearer deletion in 5G-L2-LO happens before bearer deletion in 5G-L2-HI.

Generic exception handling rules can be found in the chapter [11374912](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueKrmfvkEe-AqvopbP1qhQ).

5G-L2-LO shall discard the DL PDU if:

* LoDataUeID IE is not known (user does not exist) or;
* logicalChannelIdentity IE is not known (bearer does not exist) or;
* the request message is not valid. [CP3] 5G-L2-LO shall check the value range of parameters: numOfPdus, LoDataUeID, logicalChannelIdentity,txType and sizeOfRlcData. If any one is out of range, message is invalid.

Note: 5G-L2-LO do not check sfn, slot and slotOffset, as they are used for updating RLC delay counters only in 5G-L2-LO(5G18A 5GC425-Y).

* Refer to [Before CB015504-C]5G\_L2\_IF\_933\_replaced([10985833](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WK7tPu6Ee-AqvopbP1qhQ))[CB015504-C] [12805843](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tFeucJ-EEfCkxIKMlo4L6w) and [12805904](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BgIJgZ-MEfCkxIKMlo4L6w)[End CB015504-C].

**Unexpected Exceptions:**

In case of an unexpected exception, 5G-L2-LO shall discard the message.

 sfn

 slot

 btuNr

 slotDurationUs

 slotsInFrame,

but only log an error. The UL PDU shall be handled.

Rationale: these parameters are used only for updating counters.

Unexpected Exceptions:

In case of an unexpected exception, 5G-L2-HI shall discard the message.

#### 3.5.2.9.0-2 DL UM RLC SN Index Indication Description (ID: 12348145)

**3.5.2.9.0-2.0-1**  (ID: `12407322`)

DL UM RLC SN Index Indication
This message valid both intra-gNB and inter-gNB CA(specific Local Pcell and Scell part) scenario.

There is the purposes to introduce this message: Configure the L2Lo SN Index for 5G-L2-LO instance(static configuration).

During new scell addition:

For Scell, the L2Lo SN Index from 5G-L2-HI shall be received before the new data split from 5G-L2-HI and after related bearer been setup cause by "ScellAddition";

For Pcell, the L2Lo SN Index only be configured during the first Scell additional procedure with fixed value(l2LoSnIndex = "L2Lo-0" and rlcSnSubRangeFlag= "ENABLED").

During Scell removal：

For Scell, there is no indication message from 5G-L2-HI；

For Pcell, the indication of the L2Lo SN Index and SN Full Range only happened on the last Scell removal procedure with fixed value(l2LoSnIndex = "L2Lo-0" and rlcSnSubRangeFlag= "DISABLED").
