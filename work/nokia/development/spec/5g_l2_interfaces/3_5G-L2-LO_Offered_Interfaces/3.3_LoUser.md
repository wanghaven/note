#### 3.3.2.1.0-4 User Setup Message validation (ID: 10983621)

**3.3.2.1.0-4.0-1**  (ID: `13254475`)

User Setup
Upon reception of the LoUser\_UserSetupReq message, 5G-L2-LO shall check the validity of the request message.

Upon reception of the message LoUser\_UserSetupReq, 5G-L2-LO DL receiver shall check the value range for ueAddCause, numSrbs, srbAddReqLo, numDrbs, drbAddReqLo, if any of them is out of range, the message is invalid.

5G-L2-LO DL receiver shall also check the value of nrCellIdentity, if not match, the message is invalid.

5G-L2-LO shall reply with LoUser\_UserSetupResp message and set the status IE = Ok if the request message is valid and the user was setup successfully.

For exception handling, see 5G\_L2\_IF\_804\_replaced([10983627](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2V6dAPu6Ee-AqvopbP1qhQ)).

#### 3.3.2.1.0-5 User Setup Exceptions (ID: 10983627)

**3.3.2.1.0-5.0-3**  (ID: `10983643`)

User Setup
Expected Exceptions:

If at least one bearer can't be set up, 5G-L2-LO shall not set up any of the given bearers and send a LoUser\_UserSetupResp with "status" = NOK, "cause" field is not used.

Generic exception handling rules can be found in the chapter 5G\_UP\_1795\_replaced([11374912](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueKrmfvkEe-AqvopbP1qhQ)).

Exception handling rules specifically for object setup can be found in the requirements 5G\_UP\_1805\_replaced([11375008](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueLSpvvkEe-AqvopbP1qhQ)) and 5G\_UP\_1806\_replaced([11375035](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueLSqfvkEe-AqvopbP1qhQ)).

5G-L2-LO shall reply with LoUser\_UserSetupResp message, set the status IE = NotOk, and set an appropriate cause value if:

* ueIdDu IE is not valid or it already exists or;
* crnti IE is not valid or it already exists or;
* allocation of TEID (Tunnel Endpoint Indentifier) failed for all requested bearers;
* the request message is not valid or;
* all the requested bearers and resources were not allocated successfully or;
* If the number of unique loDataFlowControlClientAddress(existing and new DRB(s)) per 5G-L2-LO is >= max number of loDataFlowControlClientAddress.

max number of loDataFlowControlClientAddress is defined in 5G\_L2\_IF\_947\_replaced([10986024](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WMJ1_u6Ee-AqvopbP1qhQ))

[CB008911-B]

• nrPCellIdentifier is invalid or is not present under certain conditions - see IE description in 5G\_L2\_IF\_2620\_replaced([10983649](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2V6dBPu6Ee-AqvopbP1qhQ)) for details.

[End CB008911-B]

For available cause values, see 5G\_UP\_1846\_replaced([11375790](https://dn-prod.ext.net.nokia.com/rm/resources/BI_uePkGPvkEe-AqvopbP1qhQ)).

In case of failure, if 5G-L2-LO is not able to do a proper internal rollback for all sub-systems, and the state of the 5G-L2-LO is inconsistent, 5G-L2-LO shall indicate this in the response message, set status IE = "NotOk" and set the needForRollback IE = TRUE. This is to indicate that clean-up is expected from 5G-CP-RT, and that 5G-CP-RT should not presume that no resources were allocated by the 5G-L2-LO.

when 5G-L2-LO UL respond to 5G-L2-LO DL with needForRollback IE = TRUE, 5G-L2-LO DL shall respond to CP-RT with status IE = "NotOk" and set the needForRollback IE = TRUE.

when 5G-L2-LO UL respond to 5G-L2-LO DL with cause IE = "ECause5G\_ObjectAlreadyExists", 5G-L2-LO DL shall respond to CP-RT with status IE = "NotOk" and cause IE = "ECause5G\_ObjectAlreadyExists".

Refer to [5G\_L3\_4810\_replaced([11019158](https://dn-prod.ext.net.nokia.com/rm/resources/BI_A-_Scf_KEe-D3seUwaI0tg))] about Error handling in 5G-CP-RT.

Unexpected Exceptions:

In case of an unexpected exception, 5G-L2-LO shall not send response message.

Error is logged for both expected and unexpected exception.
