#### 4.6.2.1.0-2 Bearer Setup Description (ID: 10989880)

**4.6.2.1.0-2.0-3**  (ID: `10989898`)

Bearer Setup
Upon reception of the request,

5G-L2-HI shall create RLC entities and configure them with parameter values given in the message.

5G-L2-HI shall initialize the internal parameters as defined in 5G\_UP\_ALG\_5798.

If tcpPugBoostProfileParams is received, TCP Proactive Uplink Grants (PUG) is considered as enabled  and current TCP PUG state for the bearer is deactivated.

If cookie information is present, perform actions required for PCMD as defined in 5G\_L2\_11511\_replaced([11231388](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AbQQ_vTEe-AqvopbP1qhQ)).

For sending the periodic PCMD trace report messages to PCMD Trace Framework (Drain Address) refer 5G\_L2\_9508\_replaced([11231780](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AeTkfvTEe-AqvopbP1qhQ)).

[PR724241]

If actOptDataFlowRateEst is set to TRUE, then 5G-L2-HI shall calculate eTh as defined in 5G\_UP\_ALG\_5736.

[End PR724241]

#### 4.6.2.1.0-3 Bearer Setup Message validation (ID: 10989905)

**4.6.2.1.0-3.0-1**  (ID: `13434101`)

Bearer Setup
Upon reception of the HiUserDu\_BearerSetupReq message, 5G-L2-HI shall check the validity of the request message.

5G-L2-HI shall reply with HiUserDu\_BearerSetupResp message and set the status IE = Ok if the request message is valid and the bearer was setup successfully.

For exception handling, see 5G\_L2\_IF\_1185\_replaced([10989911](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WoOv_u6Ee-AqvopbP1qhQ)).

#### 4.6.2.1.0-4 Bearer Setup Exceptions (ID: 10989911)

**4.6.2.1.0-4.0-3**  (ID: `10989929`)

Bearer Setup
**Expected:**

Generic exception handling rules can be found in the chapter [11374912](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueKrmfvkEe-AqvopbP1qhQ).

Exception handling rules specifically for object setup can be found in the requirements [11375008](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueLSpvvkEe-AqvopbP1qhQ) and [11375035](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueLSqfvkEe-AqvopbP1qhQ).

5G-L2-HI shall reply with HiUserDu\_BearerSetupResp message, set the status IE = NotOk, and set cause value as per below:

**NoUserPlaneResourcesAvailable(1):**

* the resources were not allocated successfully.

**InvalidParameter (2):**

* bearerGroup IE is not valid - CU DRB is configured in different bearer group than specified in DU bearer setup for collocated RLC;
* bearerGroup IE out of range or;
* Pair of loDataDlRlcPduReceiverAddress IE and loDataUeID IE for all data paths is not unique in HiUserDu\_BearerSetupReq (see Note) or;
* DL CA (SecondaryL2LoParams.size() > 0):
* is configured with rlcMode = AM AND invalid l2LoDataDlRlcPduReceiverAddress;
* snFieldLenght = 12bit;
* **[CB009103-B] IEs of the [SCellBwpParams](https://dn-prod.ext.net.nokia.com/rm/resources/TX_ZSUyAPu6Ee-AqvopbP1qhQ) structure out of range.**
* IEs of the [CellDetails](https://dn-prod.ext.net.nokia.com/rm/resources/TX_TU5xUvu6Ee-AqvopbP1qhQ) structure out of range. **[End CB009103-B]**
* HiUserDu\_BearerSetupReq cannot contain Remote SCell configuration - Remote data paths are not valid yet, xgTeid Allocation request is required first.
* bearerIndex IE out of range or;
* logicalChannelId IE out of range or;
* snFieldLength IE set to a value other than 12 bits for SRB;
* actProactResendRlcPdu structure present AND rlcMode = UM.

**ObjectAlreadyExists (3):**

* bearerIndex IE with same index exists already;

**[CB009103-A]**

**TeidAllocationFailure (7):**

* Xg TEID allocation fails**.**

**[End CB009103-A]**

For available cause values, see [11375790](https://dn-prod.ext.net.nokia.com/rm/resources/BI_uePkGPvkEe-AqvopbP1qhQ).

In case of failure, if 5G-L2-HI is not able to do a proper internal rollback for all sub-systems, and the state of the 5G-L2-HI is inconsistent, 5G-L2-HI shall indicate this in the response message, set status IE = "NotOk" and set the needForRollback IE = TRUE. This is to indicate that clean-up is expected from 5G-CP-RT, and that 5G-CP-RT should not presume that no resources were allocated by the 5G-L2-HI.

**Unexpected:**

In case of an unexpected exception, 5G-L2-HI shall not send response message.

Error is logged for both expected and unexpected exception.

Note:

For the duration of CB014067 (3CC UL CA) development in -A subfeature this has been temporary removed for SRBs and DRBs (using RLC UM) as loDataUeID from different L2-LO instances is not unique and loDataDlRlcPduReceiverAddress is set to 0 for UL only bearers. In -D subfeature proper validation will be added and this exclusion removed.

#### 4.6.2.2.0-2 Bearer Modify Description: (ID: 10990245)

**4.6.2.2.0-2.0-3**  (ID: `10990265`)

Bearer Modify
5G-L2-HI in gNB-DU shall perform the following upon reception of HiUserDu\_BearerModifyReq message from C-Plane:

- Stop periodic reporting of F1-U DDDS.

- In case when rlcReestablishRequired = TRUE, perform RLC re-establishment as described in 5G\_L2\_2179\_replaced([11220902](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_bKvPvTEe-AqvopbP1qhQ)).

- In case when rlcReestablishRequired is TRUE and cause = [Before CB008490-BIF] MobilitySuspend[CB008490-BIF] Suspend[End CB008490-BIF] for UM bearer suspend DL transmission.

- In case when cause = [Before CB008490-BIF] MobilityResume [CB008490-BIF] Resume[End CB008490-BIF] for UM bearer resume DL transmission.

- In case when rlcReestablishRequired = TRUE and SecondaryL2LoParams.size() == 0, release all the existing SCells and perform RLC re-establishment.

- In case when f1UParams is present (refer to 5G\_L2\_IF\_1199\_replaced([10990309](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2Wqq-Pu6Ee-AqvopbP1qhQ)) for IE details), reset the flow control parameters, timers, flags associated with the F1 interface, and update the F1-U Tunnel Parameters as received in the message.

- Update the RLC bearer context with parameters received in the message.

- In case when old F1-U TEID is already present and allocateF1UTEID = TRUE is received, the old DL F1-U TEID shall be released (Refer Note: 1)

- In case when allocateF1UTEID is TRUE, Allocate DL F1-U TEID (refer 5G\_L2\_IF\_970\_replaced([10986733](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WRCX_u6Ee-AqvopbP1qhQ)) for TEID allocation) and send the DL F1-U TEID in response message.

- DL GTP-U PDUs that are received on the old DL F1-U tunnel shall be discarded.

- Upon reception of DL GTP-U PDUs on the new DL F1-U tunnel, the old DL F1-U TEID shall be released

- In case when SecondaryL2LoParams.size() > 0, data are sending over multiple 5G-L2-LO as shown in 5G\_L2\_2596\_replaced([11227585](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AC1x_vTEe-AqvopbP1qhQ)).

- Override existing secondary 5G-L2-LO instance(s) configuration with one received in the message. Each difference shall result in SCell activation/deactivation/change.

- In case when modifiableRlcAmParams are present, update RLC AM related parameters. Modification is done only when inter band UL CA is activated/deactivated for the UE.

- In case when suspend set to TRUE is received for a given secondary 5G-L2-LO instance(s) suspend traffic and do not split data to this SCell(s). Used only in case of Inter-gNB CA.

- In case of Inter-gNB CA when SCell is removed (decreased SecondaryL2LoParams.size() is received) release allocated Xg-U TEID(s).

-     perform actions required for PCMD as defined in 5G\_L2\_11511\_replaced([11231388](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AbQQ_vTEe-AqvopbP1qhQ)).

Notes:

1.   When there are consecutive allocateF1UTEID =TRUE requests without any data transmitted from L2-HI CU, DU shall always maintain the latest 2 DL F1-U TEID and delete the previous DL F1-U TEIDs.

For example,

- When current DL F1-U TEID is 1 and allocateF1UTEID =TRUE is received, then both TEID1 and TEID2 is maintained until new data is received.

- Now, without any data received on TEID2, if another allocateF1UTEID =TRUE is received. TEID2 and TEID3 are maintained at DU deleting TEID1.

#### 4.6.2.2.0-4 Bearer Modify Exceptions (ID: 10990276)

**4.6.2.2.0-4.0-5**  (ID: `13321691`)

Bearer Modify
**Expected:**

Generic exception handling rules can be found in the chapter [11374912](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueKrmfvkEe-AqvopbP1qhQ).

Exception handling rules specifically for object modification can be found in the requirements [11375119](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueL5tvvkEe-AqvopbP1qhQ) and [11375147](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueL5v_vkEe-AqvopbP1qhQ).

In case actProactResendRlcPdu parameters are present, 5G-L2-HI shall omit them and proceed with message validation.

5G-L2-HI shall reply with HiUserDu\_BearerModifyResp message, set the status IE = NotOk, and set cause value as per below:

**InvalidParameter (2):**

* logicalChannelId IE is not known or;
* Pair of loDataDlRlcPduReceiverAddress IE and loDataUeID IE for all data paths is not unique in HiUserDu\_BearerModifyReq (see Note 3) or;
* DL CA (SecondaryL2LoParams.size() > 0):
* is configured for GBR, DC-GBR DRB;
* is configured for SRB;
* rlcReestablishRequired = TRUE;
* IEs of the [SCellBwpParams](https://dn-prod.ext.net.nokia.com/rm/resources/TX_ZSUyAPu6Ee-AqvopbP1qhQ) structure out of range.
* Addition of more than 1 remote SCell while there is already DL CA [PR862782] with remote SCell(s) [End PR862782] configured (see Note 1);
* xgTeidAllocationRequest:
* received in consecutive BearerModify message in secondaryL2LoParams under index of already existing SCell;
* is followed by another xgTeidAllocationRequest under the same SCell index.
* timerReassembly is received for UM bearer or out of range;
* dscp IE is out of range**;**
* suspend IE = TRUE:
* after receiving HiUserDu\_BearerSetupReq where no SCells were allocated;
* on SCell release just after successful Xg-U TEID allocation in 5G-L2-HI (DU);
* suspend IE = FALSE:
* after receiving suspend IE = TRUE in previous HiUserDu\_BearerModifyReq for this SCell;

**ObjectAlreadyExist (3):**

* Inter-gNb CA: xgTeidAllocationRequest received for existing data path

**ObjectDoesNotExist (4):**

* bearerIndex IE is not known or;
* bearerGroup IE is not known or;
* Inter-gNb CA: remote SCell configuration not preceded by xgTeidAllocationRequest;

**TeidAllocationFailure (7):**

* the resources (xgTeid) were not modified successfully;
* Xg TEID allocation fails for addition of new SCell while there is already CA configured for bearer **(see Note 2)**

For other available cause values, see [11375790](https://dn-prod.ext.net.nokia.com/rm/resources/BI_uePkGPvkEe-AqvopbP1qhQ).

In case of failure, if 5G-L2-HI is not able to do a proper internal rollback for all sub-systems, and the state of the 5G-L2-HI is inconsistent, 5G-L2-HI shall indicate this in the response message, set status IE = "NotOk" and set the needForRollback IE = TRUE. This is to indicate that clean-up is expected from 5G-CP-RT, and that 5G-CP-RT should not presume that no resources were modified by the 5G-L2-HI.

**Unexpected:**

In case of an unexpected exception, 5G-L2-HI shall not send response message.

Error is logged for both expected and unexpected exception.

**Note:**

1.  In case there is already DL CA configured for the bearer (SecondaryL2LoParams.size() > 0) [PR862782] with remote SCell(s)[End PR862782], and step-wise SCell addition was enabled by operator, then only one remote SCell per BearerModify message can be added. [PR862782] This validation excludes the DRB addition scenario under mixed intra/inter SCells configuration, where intra SCell(s) is/are already added during bearer setup procedure. [End PR862782]

1. Existing configuration of SCells shall be kept unchanged.

3. For the duration of CB014067 (3CC UL CA) development in -A subfeature this has been temporary removed for SRBs and DRBs (using RLC UM) as loDataUeID from different L2-LO instances is not unique and loDataDlRlcPduReceiverAddress is set to 0 for UL only bearers. In -D subfeature proper validation will be added and this exclusion removed.

In case of an unexpected exception, 5G-L2-HI shall not send response message.

Error is logged for both expected and unexpected exception.

Note: Even if the message interface supports both "Ok" and "NotOk" status IE value, only "Ok" is sent because in case of an exception, no response is sent.

This is because the message is practically an indication that RLF is detected and C-Plane does not need status in the response message. Hence, this is optimized so that no response is sent in case of an expected or unexpected exception.
