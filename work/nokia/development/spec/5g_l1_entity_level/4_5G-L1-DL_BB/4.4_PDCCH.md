In TDD, L1 should be able to store up to 6 messages (without the Ethernet, BIP and SW event headers) per sub-cell. This is because in worst case scenario:

1) T2max(refer to chapter 4.4) is greater than the duration of 2 slots and smaller than the duration of 3 slots.

2) Up to two DlData\_PdcchSendReq messages could be received per slot.

3) The PDCCH is transmitted on the 1th or 2th symbol of the NR-slot which means that when DlData\_PdcchSendReq messages for slot t+3 arrives the PDCCH symbols of slot t will be already transmitted.

In FDD, L1 should be able to store up to 4 messages (without the Ethernet, BIP and SW event headers) per sub-cell. This is because in worst case scenario:

1) T2max is greater than the duration of 1 slot and smaller than the duration of 2 slots.

2) Up to two DlData\_PdcchSendReq messages could be received per slot.

3) The PDCCH is transmitted on the 1th or 2th symbol of the NR-slot which means that when DlData\_PdcchSendReq messages for slot t+2 arrives the PDCCH symbols of slot t will be already transmitted.

#### 4.4.2.2.8 DCI processing

##### 4.4.2.2.8.1 Generate dciIndex

**4.4.2.2.8.1.0-2**  (ID: `12694533`)

Generate dciIndex
With eCPRI 7-2e, as DCI configuration info and encoded payload need to be sent to Radio from DU side in separate messages, one parameter dciIndex is defined to pair each DCI configuration info(sent in message DlDataFH\_PdcchSendReq) and encoded DCI payload(sent in message DlDataFH\_PdcchEncodedPayloadSendReq). This dciIndex is generated by 5G-L1-DL(BB) based on parameter startDciIndex from message DlData\_PdcchSendReqL1sw and should follow rule below:

1) startDciIndex denotes dciIndex of first DCI info in each DlData\_PdcchSendReqL1sw message.

2) dciIndex value for other DCI info should be assigned incrementally

For example:

startDciIndex of one DlData\_PdcchSendReqL1sw message is XX

this DlData\_PdcchSendReqL1sw message has 5 DCIs

Then dciIndex of there 5 DCIs should be: XX, XX+1,XX+2,XX+3,XX+4

Note:

1) If there is only one DlData\_PdcchSendReqL1sw for DL DCIs for each subcell in one slot and startDciIndex value in this message should be 0 when message is for DL DCI.

2) If there is only one DlData\_PdcchSendReqL1sw for UL DCIs for each subcell in one slot and startDciIndex value in this message should be 128 when message is for UL DCI.

3) When there are more than one DlData\_PdcchSendReqL1sw messages for DL DCI L1 should handle like this for each message:

for each DlData\_PdcchSendReqL1sw message, get value of  startDciIndex and set  DciIndex  of first DCI to be same value of startDciIndex, then startDciIndex+1 for secondary DCI, then startDciIndex+2 for third DCI,  ...

The same procedure applies to UL DCI.

[CB012115-SR-GA, CB012115-SR-HA] 5G-L1-DL(BB) and 5G-L1-DL(Radio) shall also support requirements above for TDD FR1 eCPRI-7-2e on Odin. [End CB012115-SR-GA, CB012115-SR-HA]

For a PDCCH resource set (CORESET + first symbol), a combination is defined with the following attributes: {first symbol, number of symbols, start Common Resource Block (CRB, see 38.211 §4.4.4.3) & number of RBs}.

These attributes are derived from the *DlData\_PdcchSendReq message (see 5G\_L1\_IF\_74\_replaced([11512566](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRqbGvvxEe-AqvopbP1qhQ))) as follow:*

* The first symbol corresponds to the parameter *startSymbolNumber, and the number of symbols corresponds to the parameter* numOfSymbols, refer to 5G\_L1\_4348.\*\*

* last CRB = 6*(64 - last MSB bit in* coresetFreqDomain bitmap) - 1 + *coresetFreqDomainRbShift*\*
* number of RBs = 6 *number of '1' in* coresetFreqDomain bitmap\*

**4.4.4.1.2.1.0-2**  (ID: `13383886`)

General description of CORESET configuration
The information needed to configure the coreset are: *startSymbolNumber, numOfSymbols, aggregationLevel,* startCce, ***cceRegMappingType, regBundleSize, interleaverRows, coresetFreqDomain, coresetFreqDomainRbShift, precoderGranularity, and* nShiftModNumOfRegBundles**, which comes the message *DlData\_PdcchSendReq (See 5G\_L1\_IF\_74\_replaced([11512566](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRqbGvvxEe-AqvopbP1qhQ))), also refer to 38.211 section 7.3.2 for detailed information.****\****\*

The *startSymbolNumber field together with the* numOfSymbols field in the *DlData\_PdcchSendReq message* indicates which symbol(s) are allocated to send the DCI content provided in the same message. See 5G\_L1\_4348 for supported values.\*\*\*\*

A PDCCH consists of one or more CCEs indicated by the *aggregationLevel field in the* DlData\_PdcchSendReq message, the following values are supported: see 5G\_L1\_132 for supported values.\*\*

The start CCE is indicated by the *startCce field in the* DlData\_PdcchSendReq message. Whether the CCE to REG mapping is interleaved or not is indicated through the *cceRegMappingType parameter in the* DlData\_PdcchSendReq message*. The REG bundle size is indicated by the* regBundleSize field in the *DlData\_PdcchSendReq message.***\***\*

When the interleave mode is used, a field *interleaverRows is used to indicate the interleaver size. The field* nShiftModNumOfRegBundles is also introduced for the calculation of the interleaver, this field represents (nshift)mod (NCORESETREG / L) , the upper limit range of this field is (max. number of PDCCH symbols (3) \* max. number of PDCCH CCE (45) \* CCE size (6)) / (min. regBundleSize (2)) - 1 = 404.  Note that, both fields are ignored in non-interleaved mode.\*\*

Also see 5G\_L1\_2296\_replaced([11440690](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e_c53PvvEe-AqvopbP1qhQ)) for CCE to REG mapping, start CCE and interleaver details.

FDM with PDSCH
FDM between PDCCH and PDSCH in the same symbol shall be supported in TDD FR1.

Following cases of FDM shall be supported:

* Use case 1:
* PDCCH and PDSCH FDMed in same symbol (symbol 1). PDSCH is scheduled on the PRBs that do not overlap with CORESET. It’s used in the slots with single or multiple UEs scheduled in DL. (see 5G\_L1\_15128\_replaced([11440906](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e_eH8_vvEe-AqvopbP1qhQ)) for details).
* [PR855098] 5G-L1-DL (BB) shall support PDCCH and PDSCH FDM (frequency domain multiplexing) in both symbol 1 and symbol 2 without overlapping with CORESET. It is used in slots with single or multiple UEs scheduled in DL. [End PR855098]
* Use case 2:
* PDCCH and PDSCH FDMed in same symbol (symbol 0; symbol 1; both symbol 0 and symbol 1). L1 shall support PDSCH scheduled on the PRBs that do not overlap with CORESET, or overlap with CORESET but do not overlap with DCI. It’s used in the slots single UE scheduled in DL **with or without dummy PDCCH packets in the first symbol.**
* PDSCH shall not be scheduled on the symbol which has dummy PDCCH packets. **(see 5G\_L1\_15129\_replaced([11440955](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e_eH9vvvEe-AqvopbP1qhQ)) for details).**
* Use case 3:
* PDCCH and PDSCH FDMed in same symbol (symbol 0; symbol 1; both symbol 0 and symbol 1). L1 shall support PDSCH scheduled on the PRBs that overlap with DCI, rate matching around full or partial DCI is needed. It’s used in the slots single UE scheduled in DL **with or without dummy PDCCH packets in the first symbol. PDSCH shall not be scheduled on the symbol which has dummy PDCCH packets.** (see 5G\_L1\_15132\_replaced([11440994](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e_eH-vvvEe-AqvopbP1qhQ)) and 5G\_L1\_15112\_replaced([11438635](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e_VlGvvvEe-AqvopbP1qhQ)) for details).\*\*\*\*

* Use case 5:

### 4.4.5.2 PDCCH encoding

#### 4.4.5.2.1 General encoding processing

**4.4.5.2.1.0-2**  (ID: `13383934`)

General encoding processing
For PDCCH, Error detection is provided on DCI transmissions through a 24 bits Cyclic Redundancy Check (CRC) and the CRC parity bits are scrambled with the corresponding rnti in DlData\_PdcchSendReq after attachment. The information bits after CRC attachment are encoded via Polar coding, then Rate matching is done based on the actually allocated physical resource. The exact mathematical details of each function can be found in chapter 7.3 of 3GPP 38. 212 therefore they are not copied here.

When the parameter *conformanceTestMode is true then:*

N bits PN sequence which is generated by L2-PS should be used as data input of polar encoder. This PN sequence is sent to L1 as payload of DlData\_PdcchSendReq message. L1 uses this PN sequence as the input of encoder.

Loner
For conformance test, when the parameter conformanceTestMode is true then N bits at the output of “Polar Encoder” should be zeros. L1-FPGA implementation can be done as follow:

o Stops the execution of the processing blocks “24-bits CRC attachment”, “XOR rnti bits with the last 16 CRC bits”, “Polar encoder” and generates directly the N bits of zeros.

o Keeps the normal execution of the processing blocks “24-bits CRC attachment”, “XOR rnti bits with the last 16 CRC bits”, “Polar encoder”. The N bits at the output of the “Polar Encoder” will be zeros because the bits of the parameter dciPayload are zeros when conformanceTestMode is true.

Note: the original solution in CB005881(PN23 sequence received from L2-PS is used as input of encoder, L2-PS send PN23 sequence to L1 using the DCI payload of DlData\_PdcchSendReq message. L1 processing is the same with normal mode. The processing of force encoder output into all “0” data is not used from CB005881.) has been postpone to CB007200.
