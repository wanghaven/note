EMA and EG buffers are used to order such BIP transmissions and in case of congestion of BIP/eFramer egress circuit buffer loss can happen. So, the solution is pure SW-centric: configuration of BIP is changed to obey "EG-BIP CH Id to that of stream Id" rule. As a result, BIP/eFramer will correctly react to back-pressure situation (in case of EG-FIFO buffer overflow) and EG buffers will be managed correctly (without leaking buffer pointers). When configuring EG BIP0 in Loner/eLoner, BIP_EG_STREAM.CH_ID should have the same value as number of used stream.  Example for stream 0 CH_ID should be 0, for stream 4 CH_ID should be 4. This way of configuring CH_ID enables EMA EG_FIFO overflow checking and prevents dropping EG task/losing EG buffers in case of short EG_FIFO full. The following information are transferred to L1:

- l1SpMaxNumStreamPrb: Maximum number of allocated stream PRB per subpool

- l1SpMaxNumLayerPrb: Maximum number of allocated layer PRB per subpool

• The SFN in which L1 shall apply the new reconfigurations; this SFN is the second SFN of the next pooling interval (see the below figure).

Exceptional scenarios:

• If for any reason, L2 is not received UlPool\_BbResourceReconfResp message in the current pooling interval window, BBRM shall calculate the following PRB capacities and L2PS shall apply these values:

- currentStreamPrbCapacityPerSp = min{currentStreamPrbCapacityPerSp, proposedStreamPrbCapacityPerSp}

- currentLayerPrbCapacityPerSp = min{currentLayerPrbCapacityPerSp , proposedLayerPrbCapacityPerSp}

In other words, BBRM calculates current capacities as the minimum of capacities between current and proposed capacities for stream and layer PRBs per subpools. Note that these values are not sent to L1.

L2PS uses the new L1 intra capacities in the second SFN of the next pooling interval (refer to 5G\_UP\_ALG\_6788\_replaced([11212354](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxAu-fvSEe-AqvopbP1qhQ))).
