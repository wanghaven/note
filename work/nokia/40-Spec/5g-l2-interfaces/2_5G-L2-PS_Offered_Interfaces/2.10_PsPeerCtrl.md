## 2.10.1 General

### 2.10.1.0-1 General Description (ID: 10979146)

**2.10.1.0-1.0-4**  (ID: `12815800`)

General
**Description: Control of 5G-L2-PS peer for inter-L2PS instance or intra-L2PS inter-NrCellGroup carrier aggregation.**

Each PsPeerCtrl message listed as an IE of structure type in 5G\_L2\_IF\_2688\_replaced([10980851](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2Vls6vu6Ee-AqvopbP1qhQ)) (i.e. used for IgNB CA):

– must not exceed the limit of maximum size, as specified in 5G\_L2\_7644\_replaced([11146728](https://dn-prod.ext.net.nokia.com/rm/resources/BI_24HBzvvTEe-AqvopbP1qhQ)),

– shall be extendible for backwards compatibility through the extension points (see 5G\_CP\_OTH\_9057\_replaced([11597451](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Y25PG_wOEe-AqvopbP1qhQ))),

[CNI-140264]

–   shall contain Fixed-version Versioned Structure Arrays(FVVSA) , which allow for graceful evolution of interfaces while maintaining backward compatibility. Existing array of structures to be replaced by FVVSA.

-FVVSA version shall be incremented with newly added element into versioned structure.

-The sequence nor the type of existing fields inside the versioned structures cannot be changed

-Previously defined constants inside a versioned structure cannot be changed or removed

-The PsPeerCtrl interface version shall be incremented with the introduction of FVVSA.

-Messages for which message size exceed the maximum size after migration to FVVSA, shall not be converted to FVVSA.

[End CNI-140264]

– shall contain extension point in the message’s main structure,

– if contains a substructure of ‘union’ type, the union shall be defined as an extendible union,

– shall contain extension points also in their substructures, starting from top substructures downwards, unless the message with two fields/IEs of ‘unit32’ type inside such extension point exceeds the aforementioned maximum size limit.

**Used by: 5G-L2-PS**

**Addressing: [5G\_L1L2\_FLOW\_58\_replaced([11176868](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BeQodvvREe-AqvopbP1qhQ))]**

### 2.10.1.0-2 General Message Prioirty over PsPeerCtrl Interface (ID: 10979159)

**2.10.1.0-2.0-1**  (ID: `10979163`)

General
Over the PsPeerCtrl interface, procedures(messages) defined by 5G\_L1L2\_FLOW\_107\_replaced([11178727](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BeZLYPvREe-AqvopbP1qhQ))/5G\_L2\_IF\_644\_replaced([10979213](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2VZfp_u6Ee-AqvopbP1qhQ)) have High/Low priority.

Message priority is based on whether timing is critical for Pcell and Scell DL/UL scheduling co-ordination.

Pcell and Scell exchange PsPeerCtrl address during NrRel setup proceudre(Including Non-timing critical address and timing critical address if needs.)

For timing critical messages/procedures, PsPeerCtrl message should be sent to timing criritial address (related event queue id with high priotity);

For Non timing critical messages/procedures, PsPeerCtrl message should be sent to  Non-timing critical address (related event queue id with lower priotity);

By default,  Non-timing critical address is applied(ie: pCellDlPsPeerCtrlAdress/pCellUlPsPeerCtrlAdress).

For message which is timing critical, should be listed in 5G\_L2\_IF\_2522\_replaced([10979279](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2VaGtfu6Ee-AqvopbP1qhQ)).

### 2.10.1.0-3 General Interface versioning (ID: 10979168)

**2.10.1.0-3.0-4**  (ID: `12815801`)

General
The PsPeerCtrl interface version shall be:

* Stored in the PsPeerCtrl (MuLTI) interface definition file as a constant of the same type, as *xpProtocolVersion (see 5G\_L2\_IF\_2514\_replaced([10966351](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2UNz6vu6Ee-AqvopbP1qhQ)))**[CB008140-JE], and specified individually for each (sub)feature, as the latest interface version in a release under 5G\_L2\_IF\_4962\_replaced([10979181](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2VZfoPu6Ee-AqvopbP1qhQ))**[End CB008140-JE].*\*\*\*\*
* Incremented manually by a developer for every commit if it adds or changes any IE in any message listed in NrIgnbCaUpPayload definition (5G\_L2\_IF\_2688\_replaced([10980851](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2Vls6vu6Ee-AqvopbP1qhQ))), except if:

– the added IE is within an extension point,

– the changed IE is within an extension point and 5G-L2-PS shall support dual mode of operation i.e. legacy mode for backwards compatibility, and new mode (see 5G\_CP\_OTH\_9057\_replaced([11597451](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Y25PG_wOEe-AqvopbP1qhQ)) and the note below),

[CNI-140264]

–   the added IE is within a versioned structure in FVVSA.

[End CNI-140264]

**Note 1: If a (sub)feature adds or changes an IE within an extension point, and the (sub)feature is not activated on a (prime- or ext-) gNB with 5G-L2-PS instance running the new software release, or 5G-L2-PS supports dual mode of operation (legacy/present and new/future), a backwards compatibility is expected for interworking with a 'legacy' 5G-L2-PS instance on a second gNB (respectively ext- or prime-) over the inter-gNB PsPeerCtrl interface, despite the interface is changed (upgraded) on one end only.**

**Note 2: As this requirement is not SCT testable (because of a human factor i.e. manual incrementation of the interface version by a developer), the implementation of this requirement can be verified only by some other means, for example with a CI script which runs during the code integration.**

### 2.10.2.11 PCell UL Scheduling Ind (P-&gt;S)

#### 2.10.2.11.0-1 PCell UL Scheduling Ind (P-&gt;S) Overview (ID: 10980101)

**2.10.2.11.0-1.0-3**  (ID: `10980115`)

PCell UL Scheduling Ind (P->S)
Messages:

PsPeerCtrl\_PcellUlSchedulingInd

Direction:

[Before CB006449-AA]

For all cases apart from UL CA 2CC F+F and PCell on ABIL:

PCell DL -> SCell DL (pCellUlSchedulingIndType set to DlHarqBitsAndUlGrant case)

For UL CA 2CC F+F and PCell on ABIL:

PCell DL -> SCell DL (pCellUlSchedulingIndType set to DlHarqBitsOnly case)

PCell UL -> SCell DL (pCellUlSchedulingIndType set to UlGrantOnly case)

[CB006449-AA]

For UL CA 2CC (F+F and PCell on ABIL) or (T+T):

PCell DL -> SCell DL (pCellUlSchedulingIndType set to DlHarqBitsOnly case)

PCell UL -> SCell DL (pCellUlSchedulingIndType set to UlGrantOnly case)

For all other cases of UL CA:

PCell DL -> SCell DL (pCellUlSchedulingIndType set to DlHarqBitsAndUlGrant case)

[End CB006449-AA]

Rationale: Due to timing limitations UL CA (F+F with PCell on ABIL)[CB006449-AA] or (T+T) [End CB006449-AA] cannot work as all other legacy cases do. In this particular situation PCellUlSchedulingInd is received by SCell after it sends PuschReceiveReq message to L1. The problem is solved by sending partially filled (indicated by enum values of DlHarqBitsOnly or UlGrantOnly)  PCellUlSchedulingInd messages, just after information regarding DL Harq bits or UL Grant is available.

Purpose:

This procedure is used by P(S)Cell to notify Pcell DL/UL scheduler result to Scell on overlaping UL slots. Scell use it to decide if there is needs for UCI(AN bits or P-CSI) over Scell PUSCH:

* if UE's PCell UL scheduling happens, inform its SCell not to multiplex AN bits or P-CSI over Scell PUSCH
* if UE's PCell UL scheduling does not happen but DL scheduling happen, inform its SCell to multiplex AN bits over Scell PUSCH

Precondition:

Both cells have been setup in User Plane and NrRel is setup between them.

UL CA UE has successfully activated SCell for one Ue at least.

Addressing:

The request is sent to the PsPeerCtrl address of the SCell DL scheduler (PsCell\_NrRelSCellSetupReq.sCellDlPsPeerCtrlAdress, see 5G\_L2\_IF\_2331\_replaced([10972685](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2UwmcPu6Ee-AqvopbP1qhQ))).

If rdIncreaseUlSchedulingIndPriority BIT 0 is set to 1(see 5G\_L2\_15260\_replaced([11181742](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27V7_fvTEe-AqvopbP1qhQ))), then send the msg to the queue with highest priority (PsPeerCtrl\_DlNrRelPCellUpdateResp.sCellDlTimeCriticalAddress, see 5G\_L2\_IF\_2523\_replaced([10979313](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2VaGu_u6Ee-AqvopbP1qhQ))).

#### 2.10.2.23.0-3 Inter-gNB Carrier Aggregation Backward compatibility handling of FVVSA (ID: 12746145)

**2.10.2.23.0-3.0-2**  (ID: `12874188`)

Inter-gNB Carrier Aggregation
The following terms are used in determining the FVVSA version.

Selected denotes the version of the FVVSA structure generated by the sending entity.

Latest denotes the latest version of a FVVSA structure supported by a given entity.

Sender procedure   
When building a message to be sent, Latest version of the versioned structure arrays is used.

Full receiver procedure  
As receiving entity, shall support all versions of versioned structure arrays from version 0 to the Latest.  
Receiver shall check the Selected version of the received FVVSA. The following procedure shall be performed for each of these elements:  
  
if Selected >= Latest  
    Read the FVVSA using Latest version  
    If Selected>Latest, IEs which are not supported by the latest version will be ignored  
else  
    switch (selected)  
    case 0:  
        Read the FVVSA using version 0 format  
        break  
    case 1:  
        Read the FVVSA using version 1 format  
        break  
  
    ...  
    end switch  
  
    All IEs introduced between versions selected+1 and latest shall be set to default or invalid values.  
end if

#### 2.10.2.31.0-3 Scell Request Rar Transmission Description (ID: 13427004)

**2.10.2.31.0-3.0-1**  (ID: `13409061`)

Scell Request Rar Transmission
1. L2-PS from SCell requests PCell L2-PS to send Random Access Response message to UE. Values in request are evaluated as for regular RAR message (see [11152611: Assembling MAC PDU for RA Response (RA-Msg2)](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kshQhvvSEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9))
2. L2-PS on PCell sends RAR to UE so that it will send MSG3 in requested slot. Values RAPID, TAC, FDRA, RNTI are copied directly from indication message. TDRA is taken from tdraToK2Mapping so that UE shedule MSG3 in slot described by msg3Sfn and msg3Slot.
3. L2-PS on SCell waits for MSG3 transmission.
