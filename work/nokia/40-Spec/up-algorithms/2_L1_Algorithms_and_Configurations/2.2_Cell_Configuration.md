* \*Number of antenna for SRS SU MIMO stands for the number of spatial streams at the input of the SRS channel estimation [11106189](https://dn-prod.ext.net.nokia.com/rm/resources/BI_koUtEPvSEe-AqvopbP1qhQ)

(\*\*)   For PUCCH, for eCPRI 7-2e the maximum number of streams at the input of the DU and the Radio with C16’2(Radio) depends on DU subcellType: it is 8 with C8’2(DU) subcellType.

* Number of MIMO Layers refers to the input to the DU from the Radio for PUSCH (or equivalently the output from the Radio to the DU) with eCPRI 7-2e, see [11380349](https://dn-prod.ext.net.nokia.com/rm/resources/BI_uemJZ_vkEe-AqvopbP1qhQ),[11380307](https://dn-prod.ext.net.nokia.com/rm/resources/BI_uemJYfvkEe-AqvopbP1qhQ).
* For PRACH, N refers to NRBTS.NRCELL.nbrOfSsbPerRachOccasion.
* For SRS BF, the number of streams is equal to the number of TRX at the input of:
* the DU for eCPRI 7-2a ([11106894](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kocBz_vSEe-AqvopbP1qhQ))  for C2(?), C4, C8.
* the Radio for eCPRI 7-2e  ([11107174](https://dn-prod.ext.net.nokia.com/rm/resources/BI_koco2vvSEe-AqvopbP1qhQ)) for C16'2,  C16'4 and C16'8.

\*The number of antennas in DL refers to number of spatial streams at the output of:

* the DU for eCPRI 7-2a and 7-2e, see [11377292](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ueVqsfvkEe-AqvopbP1qhQ) and [11380378](https://dn-prod.ext.net.nokia.com/rm/resources/BI_uemJavvkEe-AqvopbP1qhQ) respectively
* the Radio for 7-2e, see [11380378](https://dn-prod.ext.net.nokia.com/rm/resources/BI_uemJavvkEe-AqvopbP1qhQ)

\*The number of antennas for SRS (Precise Positioning) represents the number of raw SRS streams received from Radio and input to DU for SRS measurement ([11107759](https://dn-prod.ext.net.nokia.com/rm/resources/BI_koiIb_vSEe-AqvopbP1qhQ)) in C4S and C8S cell type.

Note:  
For TDD FR1 eCPRI:  
- C4, C8 UL subcell for L1 Thor, C4 and C8 UL subcell for GPU L1 HW is only supported for long PUCCH, not for short PUCCH.   
- C4 and C8 UL subcell for GPU is only supported for NRBTS.NRCELL.nbrOfSsbPerRachOccasion = 1

### 2.2.5.2 Baseband PRB Pooling

#### 2.2.5.2.0-1 Baseband PRB Pooling Baseband PRB Pooling concept (ID: 11094484)

**2.2.5.2.0-1.0-5**  (ID: `11094502`)

Baseband PRB Pooling
The PRB pooling is supported for FDD FR1 in DL/UL directions **and TDD in DL/UL direction.**

When PRB pooling enable, the PRBs pooling resources within L1 pool would be split among L1 subpools according to the cell slots size, see L1 SW configuration in [11585088](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BPv4QPwGEe-AqvopbP1qhQ?oslc_config.context=https%3A%2F%2Fdn-prod.ext.net.nokia.com%2Fgc%2Fconfiguration%2F9).

Within L1 subpool, a static part of PRBs are reserved for each subcell, the remaining PRBs resources can be shared among the subcells.

* **minPRBs:**
* A minimum PRBs is reserved for each L1 subcell see 5G\_CP\_ARC\_ALG\_4269\_replaced([11277368](https://dn-prod.ext.net.nokia.com/rm/resources/BI_TpMt7fvcEe-AqvopbP1qhQ)) for DL/UL FDD and 5G\_CP\_ARC\_ALG\_6521\_replaced([11277447](https://dn-prod.ext.net.nokia.com/rm/resources/BI_TpNU8vvcEe-AqvopbP1qhQ)) for UL TDD.

For DL, those PRBs are reserved to ensure that the primary subcell of the cells has enough PRB for common signals and for calls startup.

For UL, those PRBs are reserved for PRACH, PUCCH, if any PRB remaining they can be allocated for PUSCH and SRS.

* A minimum PRBs is reserved for each cell, see 5G\_CP\_ARC\_ALG\_6975\_replaced([11277487](https://dn-prod.ext.net.nokia.com/rm/resources/BI_TpNU-PvcEe-AqvopbP1qhQ)) for DL TDD. This minimum number of PRBs ensures that the primary subcell of the cells has enough PRB for common signals and for calls startup.

PRB reservation is done for cell level but resolution is at subcell level.

Those PRBs are reserved for common channels (SSB), reference signal traffic (CSI-BM) as well as for a minimum of PDSCH traffic for every subcell. If any PRBs remain, they can be allocated for PDSCH.

The current max number of PRBs for CORESET 0 for

TDD FR1 is 48 (5G\_UP\_ALG\_134\_replaced([11095535](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmmOuvvSEe-AqvopbP1qhQ))),

FDD FR1 is 48 (5G\_UP\_ALG\_128\_replaced([11095468](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmlnrPvSEe-AqvopbP1qhQ)) and 5G\_UP\_ALG\_7284\_replaced([11095507](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmmOtPvSEe-AqvopbP1qhQ)))

then DL numMinPRBAlloc is 48.

Rationale : Number of PRBs required for SSB : 20 and, the smallest number of PRBs for CSI-RS for BM : 24, summing up to 44 PRBs.

Thus, setting numMinPRBAlloc to 48 PRBs is sufficient as it can accommodate the SSB & CSI-RS for BM and still have some PRBs remaining for PDSCH transmission.

* **Bandwidth - minPRBs: Those PRBs are for PUSCH or for PDSCH resource allocation and can be processed per symbol by using the pooling resources within the L1 subpool.**

For TDD FR1 DL/UL PRBs can be pooled on both L1 subpool and L1 pool level.

For FDD FR1 DL/UL, when NRBTS.actSlowPrbPoolingFddFr1Enh is set to TRUE, DL/UL PRBs can be pooled on both L1 subpool and L1 pool level.

For TDD FR1 DL/UL and FDD FR1 DL/UL:

L1 shall support the PRBs pooling resources reconfiguration for each L1 subpool by giving the maximum number of Stream-PRBs and Layer-PRBs. In another word, the PRBs pooling is slow processed between L1 subpools relying on the reconfiguration.

$WDFTV\_q = [1 \quad e^{j\frac{2\pi q}{M}} \quad e^{j\frac{2\pi q \cdot 2}{M}} \quad ... \quad e^{j\frac{2\pi q \cdot (M-1)}{M}}]$[/image\_desc][image\_path]https://storage.googleapis.com/dng\_files/images/HTML\_m22dng\_5G\_UP\_Algorithms\_file\_175.002.png[/image\_path][/image-1],

M: number of TRX rows;  q: vertical DFT beam index; *q* ∈ *{0, 1, … , M-1};*\*\*

3. Construct 2-D DFT beam with formula:

[image-2][image\_desc]WDFT\_m = Kron(WDFT V\_q, WDFTH\_p)[/image\_desc][image\_path]https://storage.googleapis.com/dng\_files/images/HTML\_m22dng\_5G\_UP\_Algorithms\_file\_175.003.png[/image\_path][/image-2],

m: 2D DFT beam index, *m= q*N+p, m *∈* {0, 1, …, N-2, N-1, ..., M*N-1};*\*\*

**Output**

* 2-D DFT beam.

For example, for 64 TRX(4x8x2), The TRXs are 4 rows and 8 columns and 2 polarizations. There are 8 DFT beams in horizontal direction and each DFT beam weight vector WDFTHp length is 8.  And there are 4 DFT beams in vertical direction and each DFT beam weight vector WDFTVq is 4. Finally, there are 32 2-D DFT beams for each polarization and the beam weight vector length is also 32.

Rational: 2-D DFT beams are used for calculating EIRP coefficient with short term BWVs, see 5G\_UP\_ALG\_21899\_replaced([11107528](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kogTPfvSEe-AqvopbP1qhQ)), that is the reason why DFT beams shall be generated and stored in DU(L1-UL)

Note: The DFT beams pattern is calculated at Cplane at cell setup stage, so the order of DFT beams must be the same as Cplane, refer to 5G\_CP\_ARC\_ALG\_2060\_replaced([11278098](https://dn-prod.ext.net.nokia.com/rm/resources/BI_TpPxMvvcEe-AqvopbP1qhQ))
