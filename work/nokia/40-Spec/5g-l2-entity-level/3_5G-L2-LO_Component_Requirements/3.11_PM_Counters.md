## 3.11.0-5 PM Counters Nullable and non-nullable PM counters (ID: 11218499)

**3.11.0-5.0-1**  (ID: `11218505`)

PM Counters
5G-L2-LO shall support nullable and non-nullable PM counters.

When the measured value in the given reporting period is 0:

* If a PM counter is declared as non-nullable, 5G-L2-LO shall update PM Agent with 0 by the end of the reporting period.
* If a PM counter is declared as nullable, 5G-L2-LO shall refrain from updating PM Agent by the end of the reporting period.

Each PM counter is non-nullable, unless specified otherwise.

## 3.11.0-7 PM Counters PM Counter Search in NIDD (ID: 12747294)

**3.11.0-7.0-1**  (ID: `12747306`)

PM Counters
The following instructions explain a simple method to search for PM counters in NIDD which is recommended to unexperienced NIDD users.

* Open NIDD in <https://nidd-prod1.ext.net.nokia.com/NIDD/>.
* Select PM -> Counters Search.
* Set Domain = SingleOAM.
* Set Product = MRBTS PM/FM.
* Set the PI ID or the NetAct Name to the value of the wanted counter (E.g.: PI ID = M55303C13001 or NetAct Name = PUCCH\_FORMAT1\_DTX).
* Set the Category of the wanted counter:

Category = Customer, if the counter has only the character 'C' in its PI ID (E.g.: PI ID = M55303C13001).

Category = R&D, if the counter has the characters 'CRD' in its PI ID (E.g.: PI ID = M55308CRD52001).

Note: If the Category is set to the wrong value, then the counter will not be found.

* Click the 'Search' button.
* NIDD displays a table of all available versions of the counter.
* Select the version which corresponds to the release or package of the wanted counter.

See <https://confluence.ext.net.nokia.com/display/EE/Counter+Search> for the PM counter search function.

Search results can be exported to an Excel sheet.

### 3.11.1.3 NR DL VoNR Mute Measurement PM Counters

#### 3.11.1.3.0-1 NR DL VoNR Mute Measurement PM Counters VoNR mute metrics calculation for PM counters (ID: 11218864)

**3.11.1.3.0-1.0-3**  (ID: `13372772`)

NR DL VoNR Mute Measurement PM Counters
[CB015803-A]

For the purpose of PM counters update, 5G-L2-LO shall calculate metrics for 5QI-1 DRBs according to the following procedure.

Trigger:

When the vonrPmCount structure is present in the cell context, then upon reception of LoCtrl\_Fr1HarqStatusUpdateReq (FR1, see 5G\_L2\_IF\_6399\_replaced([10985489](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WIfcfu6Ee-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9))) or LoCtrl\_Fr2HarqStatusUpdateReq (FR2, see 5G\_L2\_IF\_6673\_replaced([10985572](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WJGgfu6Ee-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9))), 5G-L2-LO shall perform the steps below.

Procedure:

1. 5G-L2-LO shall filter the harqBufferIndexList and process only those elements where the isVoNRRelated parameter is TRUE.
2. For each of these elements, 5G-L2-LO shall identify the corresponding Transport Block (TB) (see 5G\_L2\_1976\_replaced([11217544](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_ElavvTEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9))) and the 5QI-1 DRB context.
3. If the TB contains at least one 5QI-1 MAC sub-PDU constructed from the RLC buffer of the 5QI-1 DRB (5G\_L2\_IF\_2544\_replaced([10983749](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2V7EFfu6Ee-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)), 5G\_L2\_1960\_replaced([11216443](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2--exPvTEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9))), then 5G-L2-LO shall apply the following logic:

Mute period is not active for the DRB

If harqStatus is NACK or DTX and isFinalHarqReTx is TRUE:

* 5G-L2-LO shall start adding the number of unsuccessful(NACK or DTX) 5QI-1 MAC sub-PDUs from the TBs to Lost VoNR Packet Number for the DRB
* If start time (I) of the Cumulative Voice Break Time further called mute period is not set for the DRB, then 5G-L2-LO shall store the SFN and SLOT from the message and the current HFN as the start time (I).

Mute period is active for the DRB

If DRB has active mute period, the harqStatus of the last TB with isFinalHarqReTx set to TRUE is NACK. 5G-L2-LO shall:

* Listen for another (initial or reTx HARQ) TB with harqStatus equals ACK. Mark the SFN and SLOT from the msg and current HFN as the end time(t).

Else If DRB has active mute period, the harqStatus of the last TB with isFinalHarqReTx set to TRUE is DTX . 5G-L2-LO shall:

* Listen for two consecutive ACK in initial or reTx HARQ of other TBs. For the first of them mark SFN and SLOT from the msg and current HFN as the end time(t).

DRB with an active mute period is removed

If DRB has active mute period and the DRB is removed, refer to VoNR mute metrics management ([11218884](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_NITPvTEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9))

Mute Period Finalization:

When the end time (t) is set, 5G-L2-LO shall:

* Calculate mute period as end time(t) - start time(I)
* Calculate Voice packet interval, verify if the session is active or silent according to 5G\_PM\_1990\_replaced([12457907)](https://dn-prod.ext.net.nokia.com/rm/resources/BI_r-AmSgpBEfCT35inaBGo7g)
* If the session is active:
* Update the 'Number of VoNR active sessions for downlink air interface' PM Counter (5G\_L2\_20551\_replaced([11218930](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_NvU_vTEe-AqvopbP1qhQ?oslc_config.context=https://dn-prod.ext.net.nokia.com/gc/configuration/9)))
* If the mute period duration is greater than the VoNR mute threshold parameter, update the voice break status to 'valid mute'.

  + Update the Distribution of DL Air interface mute
  + Reset the mute period and Lost VoNR packet number

Note: Multiple feedback statuses for a single 5QI-1 DRB may be present within a single LoCtrl\_Fr1/Fr2HarqStatusUpdateReq. 5G-L2-LO shall process these statuses in the order they are received.

Note: In each case HFN shall be retrieved using CCS API. If current SFN is lower than SFN received in the msg, HFN shall be decremented by 1. When calculating the mute period or decrementing the HFN, 5G-L2-LO shall take into consideration HFN range.

[End CB015803-A]

#### 3.11.3.1.0-1 Overload monitoring counters Number of packets received at L2-LO from L2-HI (ID: 11219131)

**3.11.3.1.0-1.0-1**  (ID: `11219136`)

Overload monitoring counters
Measurement: NR L2 overload report in DU for RLC and MAC

NetAct name: L2\_PDU\_DL\_RECEIVED

PI ID: M55381CRD50001

Note: PM counter definitions can be found in NIDD as explained in 5G\_L2\_13022\_replaced([11192143](https://dn-prod.ext.net.nokia.com/rm/resources/BI_28iOyPvTEe-AqvopbP1qhQ)).

Update time: If rdEnableOverloadCounters flag is set to TRUE (Ref.: 5G\_L2\_22221\_replaced([11220634](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_Z8k_vTEe-AqvopbP1qhQ))), The counter is updated when an L2-LO instance receives an RLC PDU from L2-HI.

Increment value: Increment by one each for each incoming RLC PDU packet

---

#### 3.11.3.1.0-2 Overload monitoring counters Number of packets from L2-HI that L2-LO discarded due to lack of space in buffer (ID: 11219144)

**3.11.3.1.0-2.0-1**  (ID: `11219152`)

Overload monitoring counters
Measurement: NR L2 overload report in DU for RLC and MAC

NetAct name: L2\_PDU\_DL\_DISC\_BUFFER\_FULL

PI ID: M55381CRD50002

Note: PM counter definitions can be found in NIDD as explained in 5G\_L2\_13022\_replaced([11192143](https://dn-prod.ext.net.nokia.com/rm/resources/BI_28iOyPvTEe-AqvopbP1qhQ)).

Update time: If rdEnableOverloadCounters flag is set to TRUE (Ref.: 5G\_L2\_22221\_replaced([11220634](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_Z8k_vTEe-AqvopbP1qhQ))), the counter shall be updated when an L2-LO instance discards an RLC PDU from L2-HI due to lack of space in its buffer.

Increment value: Increment by one each for each discarded RLC PDU packet.
