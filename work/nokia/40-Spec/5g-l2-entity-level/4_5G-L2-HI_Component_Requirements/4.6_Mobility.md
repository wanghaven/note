### 4.6.1.0-3 Common Handling of HiUserDu_BearerSuspendReq (ID: 11225158)

**4.6.1.0-3.0-1**  (ID: `11225163`)

Common
RLC entity shall perform the following upon reception of HiUserDu\_BearerSuspendReq message from C-plane during Inter-gNB NSA:

- Discard RLC SDUs received over F1-U interface (\*)

- Suspend transmission of downlink RLC PDUs to 5G-L2-LO.

- Suspend transmission of uplink RLC SDUs over F1-U interface.

- Discard uplink RLC PDUs received from 5G-L2-LO (\*)

- All the timers active at RLC should be stopped (for example:t-PollRetransmit, t-reassembly etc., )

- [Before 5GC000704-A] Generate and send F1-U DDDS message with "Final frame indication" IE set to '1'(refer 5G\_UP\_ALG\_5813\_replaced([11222269](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx6G4PvSEe-AqvopbP1qhQ)) for DDDS message generation). [End 5GC000704-A]

[End 5GC000704-A][End 5GC000704-C]

- [Before 5GC000704-V][5GC000704-C] For DRB : Generate and send F1-U DDDS message (refer 5G\_UP\_ALG\_5813\_replaced([11222269](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kx6G4PvSEe-AqvopbP1qhQ)) for DDDS message generation). [End 5GC000704-C][End 5GC000704-V]

### 4.6.1.0-4 Common Handling of DL PDCP STATUS REPORT (ID: 11225170)

**4.6.1.0-4.0-1**  (ID: `11225175`)

Common
Handling of pdcpStatusReportWaitTimer

---

**4.6.1.0-4.0-1.0-4**  (ID: `11225206`)

Common
In case pdcpStatusReportWaitTimer > 0 is received in HiUserCu\_BearerModifyReq (with cause: **Suspend) or HiUserCu\_BearerSetupReq (refer Rationale) message:**

For all bearer types 5G-L2-HI shall start pdcpStatusReportWaitTimer in below scenarios:

 Intra-gNB mobility:

o For PDCP Data Recovery:

o Upon reception of HiUserCu\_BearerModifyReq with cause: Suspend;

o For PDCP re-establishment:

o Upon reception of HiUserCu\_BearerModifyReq with cause: Resume OR first UL GTP-U PDU on the target/new F1-U tunnel;

 Inter-gNB mobility:

o Upon reception of first UL GTP-U PDU on the target F1-U tunnel.

* **[CB008490-D] Inter-gNB NRDC PSCell addition:**

o Upon reception of first UL GTP-U PDU or SCG F1-U DDDS from SN gNB-DU. **[End CB008490-D]**

* **[CB008490-F] Inter-gNB NRDC PSCell release with bearer takeback:**

o Upon reception of first UL GTP-U PDU or MCG F1-U DDDS from MN gNB-DU. **[End CB008490-F]**

In case **[CB008490-F] of** Intra-gNB **[End CB008490-F] PSCell Release with takeback (delete NR-DC split bearer) when**\*\*\*\*

* pdcpStatusReportWaitTimer > 0 is received in HiUserCu\_BearerModifyReq with cause=SuspendSecondaryLink (5G\_L2\_11894\_replaced([11228018](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AFSCfvTEe-AqvopbP1qhQ))) AND
* DL PDCP Status report is NOT received,

then at the reception of HiUserCu\_BearerModifyReq with cause=ReleaseSecondaryLink (5G\_L2\_11892\_replaced([11227924](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AEq-_vTEe-AqvopbP1qhQ))), 5G-L2-HI (CU):

* starts pdcpStatusReportWaitTimer

5G-L2-HI shall stop pdcpStatusReportWaitTimer upon reception of DL PDCP STATUS REPORT.

**Rationale:**

Since the HiUserCu\_BearerModifyReq is sent by 5G-CP-NRT after UE RACH procedure, the pdcpStatusReportWaitTimer value (which indicates CNI activation) is not known at 5G-L2-HI when the first UL GTP-U PDU is received. For Inter-gNb mobility scenarios **[CB008490-D] or inter-gNB NRDC PSCell addition**[End CB008490-D][CB008490-F] /release with bearer takeback **[End CB008490-F] (PDCP re-establishment case) HiUserCu\_BearerSetupReq is used to carry pdcpStatusReportWaitTimer value.**\*\*\*\*

---

**4.6.1.0-4.0-2.0-5**  (ID: `11225246`)

Common
5G-L2-HI shall on receiving DL PDCP STATUS report:

 Stop pdcpStatusReportWaitTimer if running,

 For [Before CNI-122587] bearerType = ENDC [CNI-122587] typeOfBearer = NSA [End CNI-122587],

 For case when last HiUserCu\_BearerModifyReq was received with cause = BearerSetup (i.e. SgNB addition or Inter gNB mobility) AND DL data transfer is allowed (i.e. DL data transfer is ongoing),

 Release PDCP PDU(s) that are reported as received in the DL PDCP STATUS report AND not yet transmitted by PDCP entity.

 In case HiUserCu\_BearerModifyReq is not yet received with cause = BearerSetup (i.e. DL data transfer has not yet started),

 Store the DL PDCP status report and process after DL data transfer is allowed.

 For case when last HiUserCu\_BearerModifyReq was received with cause = Suspend OR when the DL data transfer is resumed over new F1-U tunnel as defined in 5G\_L2\_2560\_replaced([11226382](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_8IHfvTEe-AqvopbP1qhQ))

 Release the PDCP PDU(s) transmitted over old F1-U tunnel which are indicated as acked in the PDCP STATUS report. PDCP PDU(s) are not released if they are retransmitted in 5G\_UP\_ALG\_6088\_replaced([11227013](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyZ2FfvSEe-AqvopbP1qhQ)).

 For [Before CNI-122587] bearerType [CNI-122587] typeOfBearer [End CNI-122587] = SA,

* In case of Inter-gNB mobility [CB008490-D] or Inter-gNB NRDC PSCell addition [End CB008490-D][CB008490-F] /release with bearer takeback [End CB008490-F]  (pdcpCounts are present - refer Note)
* In case when DL Data forwarding is complete and DL data transfer is allowed as per 5G\_L2\_2285\_replaced([11223213](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_pNIvvTEe-AqvopbP1qhQ)),
* The PDCP PDUs that are indicated as received in the DL status report and have not been transmitted by the PDCP entity shall be released.
* In case when DL data forwarding is ongoing and DL data transfer is NOT allowed as per 5G\_L2\_2285\_replaced([11223213](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_pNIvvTEe-AqvopbP1qhQ)),
* Store the DL PDCP Status report and process it after data transfer is allowed.
* In case of intra gNB mobility (pdcpCounts not present - refer Note)
* Prioritize sending the un-acked PDCP PDUs, if not already sent during 5G\_L2\_2535\_replaced([11226125](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_658vvTEe-AqvopbP1qhQ)).
* In case of [CB008490-D] Intra-gNB NRDC [End CB008490-D] PSCell Release with takeback (delete NR-DC split bearer) AND pdcpStatusReportWaitTimer > 0 is received in HiUserCu\_BearerModifyReq with cause=SuspendSecondaryLink (5G\_L2\_11894\_replaced([11228018](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AFSCfvTEe-AqvopbP1qhQ))) AND DL Status report is received 5G-L2-HI (CU):
* stops the pdcpStatusReportWaitTimer if it is started and not yet expired
* releases all the acked PDCP PDUs indicated in the status report
* starts the DL transmission of the marked PDCP PDUs over MCG F1-U according to the DL order (5G\_L2\_11916\_replaced([11228032](https://dn-prod.ext.net.nokia.com/rm/resources/BI_3AFSC_vTEe-AqvopbP1qhQ)))

Note:

1. ### 4.6.1.0-6 Common Handling of first UL PDCP PDU during DL data suspend (ID: 11225316)

**4.6.1.0-6.0-4**  (ID: `12800344`)

Common
Upon reception of first UL PDCP PDU during DL data suspend [CB015319-A] for RLC AM bearer [End CB015319-A], 5G-L2-HI (CU) shall:

1. [CB015319-A] Send UL PDCP Status Report (refer [11531456](https://dn-prod.ext.net.nokia.com/rm/resources/BI_tWisdfv1Ee-AqvopbP1qhQ) for details)

1. In case pdcpStatusReportWaitTimer > 0 received in HiUserCu\_BearerModifyReq with cause=Suspend and HiUserCu\_BearerModifyReq with cause=Resume was not received AND:
2. UL PDU is not PDCP Status Report:
3. Buffer incoming DL PDCP SDU(s)
4. UL PDU is PDCP Status Report
5. Resume DL data transmission
6. In case pdcpStatusReportWaitTimer = 0 received in HiUserCu\_BearerModifyReq with cause=Suspend and HiUserCu\_BearerModifyReq with cause=Resume was not received:
7. Resume DL data transmission [End CB015319-A]

[Before CB015319-A]

1. Buffer incoming DL PDCP SDU(s) in case pdcpStatusReportWaitTimer > 0 received in HiUserCu\_BearerModifyReq with cause=Suspend and UL PDU is not PDCP Status Report.
2. Resume DL data transmission in case pdcpStatusReportWaitTimer = 0 received in HiUserCu\_BearerModifyReq with cause=Suspend and HiUserCu\_BearerModifyReq with cause=Resume was not received [End CB015319-A]

### 4.6.1.0-7 Common Detect old and new F1-U tunnel for the UL data during mobility (ID: 11225346)

**4.6.1.0-7.0-1**  (ID: `11225352`)

Common
In case of below listed scenarios, there shall exist source and target F1-U tunnels (Until the Source F1-U tunnel is deleted upon resuming the DL data on the target F1-U tunnel) that shall be identified by:

Scenarios are

 Mobility (Intra gNB Intra DU or Inter DU).

 [5GC000575-A] Security Key Change.

1. Tunnel Endpoint IDs (UL F1-U TEID, DL F1-U TEID).

- Source UL F1-U TEID:

The UL F1-U TEID that is currently allocated in the bearer context at 5G-L2-HI (PDCP).

- Target UL F1-U TEID:

Allocated by PDCP entity of 5G-L2-HI upon receiving HiUserCu\_BearerModifyReq with:

 “allocateF1UTEID” = TRUE.

2.

- Source gNB-DU IP address:

The remote IP address for F1-U interface that is currently allocated in the bearer context at 5G-L2-HI (PDCP).

- Target gNB-DU IP address:

The remote IP address for F1-U received at PDCP entity of 5G-L2-HI in HiUserCu\_BearerModifyReq with:

 “Number of F1-U parameter” = 1

 “bearerSuspend” =TRUE

While both the source and target F1-U tunnels exist:

For detection of UL GTP-U PDU on the target F1-U tunnel, 5G-L2-HI shall check if

 UL F1-U TEID of the received GTP-U PDU = Target UL F1-U TEID.

For detection of UL GTP-U PDU on the source F1-U tunnel, 5G-L2-HI shall check if:

 UL F1-U TEID of the received GTP-U PDU = Source UL F1-U TEID.

Upon resuming DL data on the target F1-U tunnel, the source tunnel and the associated resource shall be deleted (refer 5G\_L2\_2535\_replaced([11226125](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_658vvTEe-AqvopbP1qhQ))), and the target tunnel shall become the current source tunnel.

### 4.6.1.0-8 Common Maintain status of PDCP Key change (ID: 11225359)

**4.6.1.0-8.0-1**  (ID: `11225365`)

Common
5G-L2-HI shall maintain variable pdcpKeyChangeOngoing to track the status of PDCP security key change procedure.

On Bearer Setup, pdcpKeyChangeOngoing  shall be set to FALSE.

5G-L2-HI on receving HiUserCu\_BearerModifyReq with [Before CNI-91051] pdcpKeyChange = TRUE [CNI-91051]securityParams is filled [End CNI-91051] shall set pdcpKeyChangeOngoing = TRUE.

Once PDCP re-establishment is completed, pdcpKeyChangeOngoing shall be set to FALSE.

### 4.6.1.0-9 Common Allocate F1-U UL TEID when Bearer Modify request is received with allocateF1UTEID == TRUE (ID: 11225372)

**4.6.1.0-9.0-1**  (ID: `11225379`)

Common
When allocateF1UTEID is set to TRUE in the bearer modify request, the 5G-L2-HI shall allocate a new F1-U UL TEID and send it in the bearer modify response message. 5G-L2-HI shall release the temporary F1-U target tunnel ID if it is already present and store the newly created TEID as the new target tunnel ID.

[Rationale]This is required to handle the cases where the current handover is cancelled by control plane after the F1-U tunnel ID has been allocated. Control plane can request a new tunnel ID if a new handover gets triggered for the user at a later point of time.

5G-L2-HI shall also release the target tunnel ID if its still present during bearer delete. This is required if the current handover is aborted after a F1-U TEID is allocated and bearer is deleted due to any reason.

**4.6.1.0-12.0-1.0-1**  (ID: `11225577`)

Common
5G-L2-Hi shall perform the following steps order to indicate first GTP-U packet over new target TEID tunnel during resume procedure in mobility scenario and to ensure DDDS over target tunnel will be validated properly:

 Receive first UL GTP-U packet (DDDS) over new target tunnel

 Reset flow control/PDCP

 Resume traffic

 Validate and process DDDS

Rationale: When PDCP is still suspended, RLC may send first DDDS over new target tunnel. Such DDDS won’t provide any Highest Transmitted/Delivered SNs values, because L2-HI did not send anything over new tunnel yet. In suspend state, Flow Control is not reset yet and holds state according to the source tunnel. This means DDDS over target tunnel will be validated in respect to Flow Control state of the source tunnel and in the result validation will not validate it.

**4.6.2.2.1.0-4.0-1**  (ID: `11225837`)

 After the PDCP ulCount and dlCount  is sent in HiUserCu\_BearerModifyResp message, there could still be some UL PDCP SDUs that shall be received at the PDCP entity of 5G-L2-HI in source gNB due to the order of data suspension at gNB-DU and gNB-CU.  After the PDCP ulCount and dlCount  is sent in HiUserCu\_BearerModifyResp message, there could still be some UL PDCP SDUs that shall be received at the PDCP entity of 5G-L2-HI in source gNB due to the order of data suspension at gNB-DU and gNB-CU. To avoid discarding these UL PDCP SDUs, the UL data shall continue to be processed and sent to higher layers until the receipt of "DDDS with FFI = 1" OR  "DL End Marker from NG-U".

 Since the PDCP ulCount is updated to the target gNB before the suspension of UL data at the source gNB, it could result in target gNB requesting for the already delivered UL PDCP SDUs from the UE, which shall be discarded if re-transmitted by the UE.

[End CB005876-G]

#### 4.6.2.3.0-2 Intra gNB Resume the DL data on the target F1-U tunnel (ID: 11226125)

**4.6.2.3.0-2.0-4**  (ID: `12800454`)

Intra gNB
To resume the DL data on the target F1-U tunnel 5G-L2-HI needs to (refer to [11225316](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_2BdfvTEe-AqvopbP1qhQ)):

* receive
* First UL GTP-U PDU on the new F1-U tunnel, excluding the F1-U DDDS containing "Cause Report" = 1 (see [11225346](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_2BfPvTEe-AqvopbP1qhQ)) OR
* HiUserCu\_BearerModifyReq (with cause = Resume)
* [CB015319-A] For DRB mapped to RLC AM:
* transmit the UL PDCP Status report as the first Downlink PDCP PDU [End CB015319-A]
* receive PDCP Status Report in case when pdcpStatusReportWaitTimer > 0 in  HiUserCu\_BearerModifyReq with cause = Suspend was received

At the reception of HiUserCu\_BearerModifyReq with cause = Resume, 5G-L2-HI:

* Initialises the F1-U flow control parameters ([11218906](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxnL5vvSEe-AqvopbP1qhQ)).
* Resumes ROHC compression operation for DL PDCP SDUs.
* If flowControlActive is True performs F1-U DDDS polling over new/target F1-U tunnel as described in 5G\_UP\_ALG\_5381\_replaced([11218534](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxkvrvvSEe-AqvopbP1qhQ)).
* Sets f1DataSendSuspended = false and resume data transmission over F1-U for DRB as defined below:

The DL data transmission priority order is defined below:

* If ROHC feedback was generated and stored when PDCP was suspended, transmit the PDCP control PDU with interspersed ROHC feedback.
* Un-acked PDCP PDU(s).
* If pdcpStatusReportWaitTimer > 0 received in HiUserCu\_BearerModifyReq and DL PDCP Status is received:
  + Start sending only un-acked PDCP PDU(s) starting from SN in the FMC of DL PDCP Status Report
  + Release the PDCP PDU(s) indicated as successfully received via the FMC and Bitmap, else:
* If dataUnblock/dataDuplication is set to True,
  + Start sending the un-acked PDCP PDU(s) starting from SN = "last delivered PDCP SN" which were previously sent over F1-U, in Dl User Data, else
* Start sending the un-acked PDCP PDU(s) starting from SN = "last submitted PDCP SN" which were previously sent over F1-U.
* Buffered PDCP SDU(s).

The PDCP PDU(s) and PDCU SDU(s) are sent based on the flow control decision, and normal flow control operation resumes.

* Release the old UL F1-U TEID.

Note:

* After the DL data is resumed on the target F1-U tunnel, any subsequent F1-U DDDS on the old UL tunnel is ignored and for classical GTP-U Error Indication is not raised but for Cloud GTP-U Error Indication is raised.
* The old UL F1-U TEID is released only after the either UL data or F1-U DDDS is received on the new tunnel.

References:

* For detection of new F1-U tunnel, refer [11225346](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_2BfPvTEe-AqvopbP1qhQ)
* For detection of first UL GTP-U PDU on the new F1-U tunnel, refer [11225346](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_2BfPvTEe-AqvopbP1qhQ)
* For the handling of DDDS, refer [11217803](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxgePPvSEe-AqvopbP1qhQ)
* For detection of UL GTP-U PDU on the old F1-U tunnel, refer [11225346](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_2BfPvTEe-AqvopbP1qhQ)

Rationale:

Re-transmitting PDCP PDU(s) will have the below implication

1) Reduced possibility of PDCP re-ordering timeout at UE

* PDCP re-ordering is triggered when PDCP detects missing PDCP SN. Hence transmitting the un-acked PDU over new PDCP PDU reduces the possibility of PDCP re-ordering at UE.

2) Duplication and discarding

* Since PDCP PDU’s were already sent, it is possible that UE had already received them. In this case it shall discard the re-transmitted PDUs.

### 4.6.3.3 Inter gNB

#### 4.6.3.3.1 Source

##### 4.6.3.3.1.0-1 Source Initiate the data forwarding over X2-U DL data forwarding tunnel (ID: 11226653)

**4.6.3.3.1.0-1.0-3**  (ID: `11226677`)

Source
5G-L2-HI shall perform the below,

1. Update the PDCP ulCount and dlCount IEs in the HiUserCu\_BearerModifyResp message:

- Update dlCount IE to current value of PDCP state variable TX\_NEXT.

- Update ulCount IE to the current value in PDCP state variable RX\_DELIV and send the HiUserCu\_BearerModifyResp message.

- Continue processing UL PDCP PDU(s) and send UL PDCP SDU(s) to higher layer.

1. IF F1-U DDDS and/or X2-U DDDS is received with FFI=1 based on the configured DL data split mode OR DL GTP-U End Marker is received over S1-U interface:

- Suspend the UL data transfer.

- Discard any stored UL PDCP SDUs and discard any newly received PDCP SDUs.

END

1. IF forwardingTunnelParameters structure is filled:

IF s1EndMarkerCount != 0 (i.e. End marker is already received, Refer 5G\_L2\_2231\_replaced([11222429](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_kUr_vTEe-AqvopbP1qhQ))):

- Discard the DL PDCP SDUs received over S1-U interface (\*).

ELSE (i.e. s1EndMarkerCount  = 0 which indicates End marker is not yet received):

- Buffer the DL PDCP SDUs received over S1-U interface.

- DL PDCP SDUs received over S1-U interface after the GTP-U End marker message shall be discarded.

- Buffered DL PDCP SDUs/PDUs and GTP-U End marker message shall be forwarded as defined in 5G\_L2\_6994\_replaced([11223077](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2_n_DfvTEe-AqvopbP1qhQ)).

END

ELSE

- No data forwarding to be executed at the source gNB;

- Discard the DL PDCP SDUs received over S1-U interface. This means discarding of newly received data, already buffered data is not discarded.

- Set s1EndMarkerCount = 0 (i.e. discard already received End marker message).

END

1. If [Before CB008490-DTIF] L2TrafficShapingX2FwdParams [CB008490-DTIF] L2TrafficShapingXxFwdParams (5G\_L2\_IF\_6905\_replaced([10988127](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WcBcvu6Ee-AqvopbP1qhQ))) [End CB008490-DTIF] structure is filled:
2. calculate maxFwdLimitPerBearerPerMs as per 5G\_UP\_ALG\_17881\_replaced([11224641](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kyIwU_vSEe-AqvopbP1qhQ)).

Rationale:

1. After the PDCP ulCount and dlCount is sent in HiUserCu\_BearerModifyResp message, there could still be some UL PDCP PDUs that shall be received at the PDCP entity of 5G-L2-HI in source gNB due to the order of data suspension at gNB-DU and gNB-CU. UL PDCP PDUs may also be received over X2-U interface. To avoid discarding these UL PDCP SDUs, the UL data suspension shall be handled as mentioned in points “b” and “c” above.

2. Since the PDCP ulCount is updated to the target gNB before the suspension of UL data at the source gNB, it could result in target gNB requesting for the already delivered UL PDCP SDUs from the UE, which shall be discarded if re-transmitted by the UE.

Note:

1) This object refers to parameters related to “X2-U” interfaces and changes without prefix in parameter name, because L2-HI component implements functionalities of X2-U and Xn-U the same way. However Xn functionality will be implemented in next feature.
