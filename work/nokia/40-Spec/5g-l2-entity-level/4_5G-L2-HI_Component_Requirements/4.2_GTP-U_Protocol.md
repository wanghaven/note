## 4.2.1 GTP-U TEID

### 4.2.1.0-1 GTP-U TEID TEID Format (ID: 11221929)

**4.2.1.0-1.0-1**  (ID: `11221941`)

GTP-U TEID
TEID format (32 bits) for the TEID allocated by 5G-L2-HI is defined below:

[image-0][image\_desc]The image is completely black, with no discernible features or content.[/image\_desc][image\_path]https://storage.googleapis.com/dng\_files/images/HTML\_m24dng\_5G\_L2\_Entity\_Level\_file\_12716.001.png[/image\_path][/image-0]

TunnelType:

 Identifier of tunnel where data comes from. There are the following tunnel types:

- (0) forwarding tunnel (X2-U, Xn-U)

- (1) F1-U tunnel

- (2) Xx tunnel (X2-U, Xn-U, Xg-U)

- (3) tunnel for data from core (S1-U or NG-U)

L2-Hi Mapping Index:

 Mapping index is a 14-bit identifier which is used to identify the bearer or PDU session. A single bearer can have more than one mapping index. It is internal to 5G-L2-HI and is not used by TRSW.

L2 Path Id:

* TRSW uses L2 path id to retrieve "Local Queue id" from stored TEID mapping table to send the GTP-U message to L2NRT or L2RT instance.
* L2 path id should be unique within, hence Board Slot id and  instance id are added along with table index to generate a unique path id.
* L2 path id format is internal to 5G-L2-HI and is not known by TRSW.

Discriminator:

* This bit is used to identify if the TEID belongs to 5G-L2-HI (set to '0') or 5G-L2-LO (set to '1').

Instance Id:

* It represents the L2-HI or L2-LO instance id (see 5G\_UP\_2447\_replaced([11389245](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ufN0cfvkEe-AqvopbP1qhQ))).

Non-zero bit:

* This bit is used to identify if the TEID belongs to 5G (set to '1') or LTE (set to '0').

BoardSlot Id:

* It represents the board location(slot No) in AirScale is (0~7)
* It is not relevant in L2-HI CU in Cloud deployment. L2-HI shall set it to '0' and is ignored by OAM/TRS.

Table Index:

* Table index is a 5-bit unique identifier which is used to identify the 32-bit 5G-L2-HI or 5G-L2-LO local queue id within the TEID mapping table (refer 5G\_L2\_IF\_971\_replaced([10986749](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WRpYvu6Ee-AqvopbP1qhQ))).

### 4.2.1.0-4 GTP-U TEID TEID (Tunnel Endpoint Identifier) validation for received GTP-U PDU (ID: 11222013)

**4.2.1.0-4.0-2**  (ID: `11222024`)

GTP-U TEID
**[Before CNI-109641-A]**

5G-L2-HI shall obtain queue id from the TEID of GTP-U PDU.

If queue id is valid (5G-L2-HI has the information of the valid queue range for each interface),

 Bearer Context and IP address validity

- Fetch Bearer/PDU session mapped to the TEID.  Mapping is maintained as defined in Refer to 5G\_L2\_IF\_970\_replaced([10986733](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2WRCX_u6Ee-AqvopbP1qhQ)).

- If Bearer/PDU session is found, fetch the local IP address of the interface to which the TEID was allocated.

- If the local IP address = destination IP address of the GTP-U PDU, consider TEID as valid.

If above conditions are not met, consider TEID as invalid.

**[CNI-109641-A]**

When L2-HI receives new GTP-U PDU, then shall:

- extract TEID

- search Bearer context  using that TEID

o If not found, then L2Hi shall consider TEID as invalid and send GTP Error Indication (5G\_L1L2\_FLOW\_267\_replaced([11194005](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Bfn6YPvREe-AqvopbP1qhQ)), 5G\_L1L2\_FLOW\_1888\_replaced([11194024](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Bfn6YvvREe-AqvopbP1qhQ)))

- check if destination IP address of the PDU is same as **[CB009266-SR-A] one of the local IP addresses(either IPv4 or IPv6) stored in the bearer context for the forwarding tunnel when configured with 2(IPv4 and IPv6) IP addresses or** [End CB009266-SR-A] local IP address stored in bearer context **[CB009266-SR-A]for other cases**[End CB009266-SR-A]***\****\*

o If they don’t match, then L2Hi shall consider TEID as invalid and send GTP Error Indication

**[End CNI-109641-A]**

**[CB009266-SR-A]**

Note :

1. During SgNB addition/modification,if localIp version for forwarding tunnel is configured as DualStack, then CP-NRT wound send both ipV4 and ipV6 addresses to 5G-L2-HI. 5G-L2-HI shall store both of these addresses as local IP addresses. MeNB would send the data packets only to one IP address(either ipV4 or ipV6) depending on the IP version configured at MeNB. Hence while validating the incoming packet on the forwarding tunnel configured with DualStack version, 5G-L2-HI should check for both ipV4 and ipV6 local addresses to compare against the detintation IP address mentioned in the GTP-U PDU.
2. If the remote(MeNB) IP address verion for the forwarding tunnel is known, then 5G-CP-NRT would send only one IP address(either ipV4 or ipV6) to 5G-L2-HI.

**[End CB009266-SR-A]**
