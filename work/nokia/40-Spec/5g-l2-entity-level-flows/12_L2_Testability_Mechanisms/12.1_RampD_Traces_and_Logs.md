#### 12.1.1.1.0-2 Storage of 5G-L2-PS History Buffer L2DataCollector startup on ABIO and ABIP and ABIR (ID: 11204401)

**12.1.1.1.0-2.0-3**  (ID: `13406010`)

Storage of 5G-L2-PS History Buffer
At startup, one ring buffer is allocated on the 5G-L2 instance (called "DPDK ring buffer" on ABIO and "event machine unscheduled queue" on ABIP [CB014419-SR-EB] /ABIR [End CB014419-SR-EB]) containing pointers to a memory pool. **on ABIO L2DataCollector creates the ring buffer while on ABIP [CB014419-SR-EB] /ABIR [End CB014419-SR-EB] L2-PS creates it because L2DataCollector is not an Event Machine EO.**

Memory pools are used by 5G-L2 application to log the History Buffers.

The ring buffer **is a dedicated shared memory area between Linux and 5G-L2 application.**

#### 12.1.1.1.0-5 Storage of 5G-L2-PS History Buffer L2-PS ring buffer (ID: 11204444)

**12.1.1.1.0-5.0-3**  (ID: `13406027`)

Storage of 5G-L2-PS History Buffer
TTI storage mechanism optimization shall enable larger time frame capability.

Redesign L2-PS ring buffer, with the minimum code impact:

The requirement is to use one poolType of 1.4K element size, and write more than one event in.

The number of element in the pool will fit exactly the number of bulk enqueue in the ring buffer. So all pools elements are used when the ring buffer is full.

For ABIP [CB014419-SR-EB] /ABIR [End CB014419-SR-EB], Event Machine unscheduled queues and Event Machine event pools are used. The number of elements in the pool and the size of the ring buffer are set to 65536 to align with previous hardwares.

On ABIP [CB014419-SR-EB] /ABIR [End CB014419-SR-EB], each element contain a Software Event Header, unused by the History Buffer framework, but requirement for the Event State Verification (ESV) platform tool.

The current ring buffer framework initialization not change, one ring buffer for all core and one element by core.

The uplink, downlink and (slot,SFN) sort will be done by the data collector.

The size of one element is fixed arbitrary to the size of an ethernet frame payload (to anticipated next evolutions).

As we have elements of different size, the size of the pool element will not fit exactly to the sum of elements sizes, so a padding will be used.

And the data collector will be impacted to manage this multiplicity of event in one element and to manage the padding.

see more detail  in chapter 9 and 10  in confluence page: <https://confluence.ext.net.nokia.com/pages/viewpage.action?spaceKey=5GRD&title=Refactoring+of+the+buffer+management+for+snapshot+mode+implementation>

### 12.1.1.2 Snapshot of 5G-L2-PS History Buffer

#### 12.1.1.2.0-2 Snapshot of 5G-L2-PS History Buffer L2DataCollector actions at snapshot request (ID: 11204490)

**12.1.1.2.0-2.0-2**  (ID: `11204508`)

Snapshot of 5G-L2-PS History Buffer
When receiving this snapshot request, AaTrbl client on BB2 calls the callback functions previously registered by L2DataCollector (prepare, store, delete).

The L2DataCollector dequeues this [Before CB007487-A] DPDK ring buffer [CB007487-A] ring buffer, defined in 5G\_L1L2\_FLOW\_684\_replaced([11204444](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Bgh5U_vREe-AqvopbP1qhQ)), [End CB007487-A] and stores element “Entry” in a file until the buffer is empty.

Then the L2DataCollector resets memory for the dequeued elements.

See 5G\_L1L2\_FLOW\_763\_replaced([11205484](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BgljufvREe-AqvopbP1qhQ)) for related MSCs.

The snapshot file isn’t deleted after transfer, L2DataCollector keeps a depth of 10 snapshot files per 5G-L2-PS instances. These files can be retrieved later on, using rcp commands.

Note: The snapshot file will be retrieved using AaFile service, from CC&S, up to the OAM VNFC (where Master NodeID runs) without any action from L2DataCollector.

Note: As backup and in integration phase end user can use as in LTE the 5G-L2-PS design tool referenced as “fetch Tool”, who allow to collect directly TTI-trace snapshot from an external PC.

See APIID146\_5GTTITraceFileFetchInterface in the confluence page: <https://confluence.ext.net.nokia.com/pages/viewpage.action?spaceKey=5GSWAT&title=TTI+Trace+Design+and+Interface+Documentation#TTITraceDesignandInterfaceDocumentation-APIID121TtiTraceInterfaceRtNrt>

Snapshots Documentation
BTS shall maintain a list (trbl\_log\_list.csv file) that includes information of SC/SE own log files which are registered for the technical log collection to be part of the BTS snapshot.

Generally, SC/SE printing to the log file of Snapshot is responsible for the entry in a trbl\_log\_list.csv (not the SC/SE acting as a proxy for printing). Application of using AaSyslog the approach is that BBP is responsible for providing snapshot documentation for 'default domain' (e.g. runtime\_DEFAULT.log). If a specific buffer file exists (e.g. runtime\_SCName.log), then snapshot documentation is provided by the SC (PR476745).

Rationale: content of trbl\_log\_list.csv file must be defined on EFS level so that each impacted SC/SE follows the same approach.

Note: After PR587653, New SFS level req OM\_GEN\_SFS\_SYSM.8280 has been created (as a correction to PR587653/PR587640) to replace 5G\_RAN\_OP\_620 from 5G21B onwards.

Detailed content definition: trbl\_log\_list.csv shall be semicolon-delimited, UTF8-encoded plain-text CSV file with following columns (inside columns comma used as separator) in order:

file container; filename ; deployment; min size in kiB ; max size in kiB; mandatory flag; deployable unit; coverage; decoder name;  comment/ description file name; instructions for optional file generation; filetype; scrambler; blacklist

Each content line (including "column title" line also) shall have end-of-line (EOL) marker to signifying the end of a line. Unix format EOL (0a) shall be used. PR449954: empty lines are not allowed.

SCs/SEs can use the ‘trbl\_log\_list\_verificator’ tool for the format verification of the trbl\_log\_list.csv file. The instructions and location for the tool can be found on: <https://bts.inside.nsn.com/twiki/bin/view/Main/TatSIngleReferenceApi43>

**12.1.9.1.0-2.0-2.0-1**  (ID: `11206001`)

Snapshots Documentation
Rules:

“file container” (optional). Used only when more than one log file is combined in one container, otherwise the field has to stay empty. Container is usually zipped, format should be detected via extension. If file container is set then it shall be named as registered by AaTrblService or file name visible in RP1 messaging (e.g. in communication with TRSW and RF modules). Exception is OAM as there is no service for file name verification. In this case the plain name of the file stored to disk is to be used.

(PR476745): Following notes shall be taken account when documenting “file container”:

* Application of using AaSyslog for a specific buffer file, file container shall be mentioned when more than one log file is combined in ne container.
* Application of using Aatrbl, file container can be left empty as SC don’t know whether the log file is included to a container.

“filename“ ( mandatory). Filename shall be the same as used in registering the file into snapshot. If SC cannot give any regular expression, then *.* may be used (PR446168).

(PR476745): Following notes shall be taken account when documenting “filename”:

* Fixed part shall be documented as text string (e.g. runtime, memdumps, etc.).
* Dynamic part consists of variable part of filename (e.g. node\_id, date, timestap, etc.) and should be documented by using fixed keywords specified in “trbl\_log\_list entry generator” tool.
* Extension part shall be documented as text string (e.g. .txt, .bin, etc.).
* Technical log master (OAM) adds BTS prefix automatically to the beginning of filename when uploading a file, so it is not needed to be documented as a part of filename here.
* If single log file is compressed by SC itself, filename shall be created, registered to techlog/snapshot and documented to end to an extension identifying used compression method, (For example: runtime.zip or startup.log.xz).

In case compression is performed by technical log master (OAM) during log file uploading, filename shall be created, registered to techlog/snapshot and documented as name of the file before compression i.e. without any additional compression extension identifying compression method used by technical log master (OAM).

Files generated by external requestor (WebEM, NMS) can be handled with their own rules.

“deployment” (optional). If left blank, the file is present in any deployment. If set, the file is only present in the given deployment variant. If several entries are used, they are separated with a comma. Used to distinguish Cloud and Classical deployment with following values: 'Classical BTS', 'Cloud BTS CUCP', 'Cloud BTS CUUP',  'Cloud BTS DU' (PR476745).

“min size in kB” and “max size in kB” (mandatory) . The values are describing the minimum and maximum size of the uncompressed file. Refer the max sizes for the common files from the RPREQ\_3486 “Platform log size budget”. For compressed files **C** prefix is used. If container is defined the format shall be: /. Both size values of single element and all elements shall be either numeric or given as a place holder defined in size\_type.dat, e.g.: l\_\_SIZE\_TYPE\_B\_TAG\_CCS\_SERVICE\_AACONFIG\_RAD\_SHELL\_CHANGE\_LOG\_SIZE\_\_.

The single/all size specification can be prefixed with compression ID **C** or not, like single element size specification, e.g.: \_\_C\_100/3000 is valid ***C*\_100 is valid,** \_C\_\_100, C\_\_3000 is NOT valid.

“mandatory flag” (mandatory).  Specifies whether file is required to be present always or optionally in techlog/snapshot for given “coverage” level. Mandatory file is always created and is present in techlog/snapshot with given "coverage” level. Optional file is created conditionally and is present in techlog/snapshot with given "coverage” level.

Configuration and deployment based log files are optional:

-     Logs which depend on R&D flags are optional

-     Core- and memory dumps are optional

-     Logs which depend on the BTS configuration (not SW build) are optional

o    This includes additional logs for deployable units e.g. in case of Baseband plug-in unit, at least logs of minimum configuration is mandatory.

Valid values for “mandatory flag”:

-     “mandatory flag” = 1 meaning  “Mandatory”

-     “mandatory flag” = 0 meaning  “Optional”

“deployable unit” (mandatory). Variable names defined in “trbl\_log\_list entry generator” tool shall be used. Real deployable units shall be specified in subfield:

-     For files (SCs) whose deployable unit can be determined by fixed node ID for example MCU SCs, node ID shall be used in subfield.

-     For SCs whose deployable unit is determined by some other way than fixed node IDs, constant names shall be used in subfield. 5GC001590-AA: SCs deployed on gNB-CU in Cloud can use SC name, if the same value is already used in Classical gNB.

If more than one deployable unit or real deployable unit applies, entries are separated by a comma.

Supported variable names, constant names and instructions how mapping between constant names and real units (node IDs, hexadecimal values not allowed) can be retrieved, shall be specified in “trbl\_log\_list entry generator” tool.

“coverage” (mandatory). Indicating the coverage level in which the file is registered into techlog/snapshot (CC&S AaTrbl service). Coverage levels specified in EAaTrblCoverage are used (M=Minor, N= Normal, F=Full).

Note for testing: EAaTrblCoverage levels are mapped to coverage levels used in BTS EM user interface for technical log request are mapped as follow:

-     Most important data (M & N)

-     All data (F)

“decoder name” (optional).  The name of the executable/script used for decoding the file with all necessary options . Recommended in case of binary files with a proprietary format. If blank, no specific decoder is needed (e.g. for ASCII files). It has to be executable format, e.g. it must contain a valid command line using the file name as an optional argument. If decoder searches for technical log file name automatically, filename is not necessary part of decoder name string.

“comment / description file name” (mandatory). The name of the log description file which explains the usage of the log file. The description file shall be provided and maintained by each SC/SE. Details of description file see 5G\_RAN\_OP\_621.

“instructions for optional file generation” (mandatory for optional files). For optional files it is mandatory describe information and instructions how to provoke or trigger file generation i.e. conditions or events which need to be meet for file generation. If “instructions for optional file generation” cannot be selected from list then SCM should provide such instruction in “comments“ field. Default value is N/A and multiple choices is allowed.

“filetype” (mandatory). Indicates the type of log file.  Allowed values are ASCII and non-ASCII.

“scrambler” (optional). When the log file is not an ASCII log file, and it can be scrambled using a tool (may be a general tool, or a tool developed by SC team), indicates the name and usage of the tool.  If the log file is ASCII, or when there is no scrambler available, field is left empty.

“blacklist” (mandatory). When the log file should be excluded from the output of scrambling, and should not be provided to teams that are un-authorized to view sensitive data (e.g. offshore teams), the field is set to Y. Otherwise, the field I set  to N. This field can be set to Y if the file is non-ASCII, has sensitive information, and no scrambler tool is available to scramble the non-ASCII file.
