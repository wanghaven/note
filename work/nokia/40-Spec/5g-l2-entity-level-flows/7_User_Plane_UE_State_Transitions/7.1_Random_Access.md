**7.1.1.3.0-2.0-5.0-2**  (ID: `11186600`)

Description
Upon receipt of the RA-MSG3, L2-PS shall:

 start the timer sendRachMsg4WithOutHIMessage (see 5G\_UP\_ALG\_17990\_replaced([11153010](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ksjswvvSEe-AqvopbP1qhQ))).  
Note: This timer is started only for an SA call and is used for RA-MSG4 sending supervision.  
It is set based on NRCELL.raContentionResolutionTmr (see 5G\_UP\_ALG\_17990\_replaced([11153010](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ksjswvvSEe-AqvopbP1qhQ))).   
If it expires, only a MAC-CE CRI (Contention Resolution Identity) shall be sent in RA-MSG4; it is the maximum time to wait for a RRCSetup from the gNB-CU to allow sending the MAC CE CRI along with the RRCSetup in the RA-MSG4.   
For more information, please refer to 5G\_L1L2\_FLOW\_180\_replaced([11186665](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BfBdfPvREe-AqvopbP1qhQ)) dedicated to the description of the MSG4/SRB0 transmission.

 Stop the temporary UE context supervision timer rdL2PsTempUeTimer (5G\_L2\_3401\_replaced([11187219](https://dn-prod.ext.net.nokia.com/rm/resources/BI_27_cNvvTEe-AqvopbP1qhQ))) as a RA-MSG3 is received.   
Note: This timer is started upon receipt of the RA-MSG1 (5G\_UP\_ALG\_18931\_replaced([11152830](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ksieo_vSEe-AqvopbP1qhQ))).   
If it expires (no valid RA-MSG3 received. ‘Ghost preambles’ might be detected (i.e., noise detected as valid preamble signal), the Temporary C-RNTI and associated UE context are released. For more information see 5G\_UP\_ALG\_18931\_replaced([11152830](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ksieo_vSEe-AqvopbP1qhQ)).

 Start rrmL2PsTempUeTimerExt\_mSec (see 5G\_UP\_ALG\_18931\_replaced([11152830](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ksieo_vSEe-AqvopbP1qhQ)))  
Note: The L2-PS timer rrmL2PsTempUeTimerExt\_mSec is used to supervise the RRC Connection establishment procedure (the timer value is set according to the maximum permitted duration between the RRCSetupRequest (RA-MSG3) to the RRCSetupComplete (RA-MSG5))  
If this timer expires, the C-RNTI and associated UE context is deleted.  
Rationale: if the TS 38.331 T300 UE timer expires, the UE aborts the RRC Connection establishment procedure and reports the Error to its RRC layer.

Overview

The rrmL2PsTempUETimerExt\_mSec timer corresponds to the 3GPP T300 UE timer (5G\_UP\_ALG\_18931\_replaced([11152830](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ksieo_vSEe-AqvopbP1qhQ))) and is used to supervise the RRC connection establishment, i.e. from receipt of the RA-MSG3 to the RA-MSG5. The MAC CE CRI (Contention Resolution Identity) shall be sent before expiry of the contention resolution timer. Therefore, the L2-PS shall schedule RA-MSG4 according to the appropriate DL class priority (5G\_UP\_ALG\_9891\_replaced([11174018](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuYRsfvSEe-AqvopbP1qhQ))) and the UE aging of the contention resolution timer, and the RA-MSG5 according to the appropriate DL class priority (5G\_UP\_ALG\_9891\_replaced([11174018](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kuYRsfvSEe-AqvopbP1qhQ))) and the UE aging of the rrmL2PsTempUETimerExt\_mSec (corresponds to the 3GPP T300 UE timer).

The sendRachMsg4WithOutHIMessage timer (5G\_UP\_ALG\_17990\_replaced([11153010](https://dn-prod.ext.net.nokia.com/rm/resources/BI_ksjswvvSEe-AqvopbP1qhQ))) is the maximum time to wait for the RRC message from the gNB-CU.

The scenario for sending RA-MSG4 depends on whether the sendRachMsg4WithOutHIMessage timer expires before or not the UserSetup procedure is initiated. If the sendRachMsg4WithOutHIMessage timer expires, the MAC CE CRI of RA-MSG4 is sent alone and the RRC message is sent later separately once the gNB-DU gets it from the gNB-CU (F1AP message relayed by CP-RT using the message LoUser\_CcchDataSendReq (5G\_L2\_IF\_827\_replaced([10984111](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2V9gXPu6Ee-AqvopbP1qhQ))). Sending the RRC message later than the MAC CE CRI is tolerated because the timer rrmL2PsTempUeTimerExt\_mSec (used to supervise the RRC Connection establishment) is much longer than the Contention Resolution timer and both timers are started at the same time, i.e. upon receipt of the RA-MSG3 (see 5G\_L1L2\_FLOW\_178\_replaced([11186265](https://dn-prod.ext.net.nokia.com/rm/resources/BI_Be_oRvvREe-AqvopbP1qhQ))).

The temporary UE contexts are changed to permanent contexts once LoUser\_UserSetupReq (5G\_L2\_IF\_2620\_replaced([10983649](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2V6dBPu6Ee-AqvopbP1qhQ))) and PsUser\_UserSetupReq (5G\_L2\_IF\_390\_replaced([10974651](https://dn-prod.ext.net.nokia.com/rm/resources/BI_2U9ayfu6Ee-AqvopbP1qhQ))) are received from CP-RT (see 5G\_L1L2\_FLOW\_123\_replaced([11180017](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BefSBvvREe-AqvopbP1qhQ))).

Note: In the subsequent descriptions:

“UserSetup done” means that the procedure described in 5G\_L1L2\_FLOW\_123\_replaced([11180017](https://dn-prod.ext.net.nokia.com/rm/resources/BI_BefSBvvREe-AqvopbP1qhQ)) is completed successfully.

Overview

When the gNB-CU cannot accept a RRCSetupRequest due to a lack of resources (e.g. maximum number of UEs capacity already reached), the gNB-CU sends a RRCReject to gNB-DU (L2-PS) via the message LoUser\_CcchDataInd with the IE isRrcReject set to True. This RRCReject can be piggybacked along with the MAC CE CRI or not and the UE context may be a temporary or a permanent one at the time to send the MAC CE CRI. Therefore, there are four possible scenarios to send this RRCReject. There is no RA-MSG5, in RRCReject case.

### 7.1.3.3 Exceptions

#### 7.1.3.3.0-1 Exceptions [Exc.2] RA preamble (ID: 11187598)

**7.1.3.3.0-1.0-1**  (ID: `11187605`)

Exceptions
In case several UEs have transmitted a RA preamble in the same PRACH occasion, only the TA matching with the highest RX power is considered for future PUSCH decoding.

The subsequent Contention resolution procedure will select a unique UE.

#### 7.1.3.3.0-2 Exceptions [Exc.10 &amp; 16] PUSCH (RA-MSG3) with interfering UEs (ID: 11187613)

**7.1.3.3.0-2.0-1**  (ID: `11187621`)

Exceptions
In case several UEs have transmitted a RA preamble in (2), UEs with a lower RX power and a possible different TA interferes with the strongest UE.

If interference is too high, the decoding of the PUSCH of the strongest UE after maximum number of retransmissions fails, as described in Alternative (18).

The Contention resolution timer expires and current RA attempt procedure fails as described in (34b).

The UE will make a new attempt and the randomization of time between two RA attempts will avoid a new conflict with the interfering UEs.

**7.1.3.3.0-3.0-1**  (ID: `11187635`)

Exceptions
In the very unlikely case PUSCH scheduling takes too much time, the Contention timer expires without Contention resolution as described in (34b) and the current RA attempt procedure fails as described in (34b).

The UE will make a new RA attempt (if maximum number of attempts has not been reached).

#### 7.1.4.2.0-13 Description (13) and (20) UlData_PuschReceiveRespHarqU (first UL HARQId 0) (ID: 11187860)

**7.1.4.2.0-13.0-1**  (ID: `11187867`)

Description
Direction: 5G-L1-UL -> 5G-L2-PS

Message definition: 5G\_L1\_IF\_209\_replaced([11513897](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRzlD_vxEe-AqvopbP1qhQ))

The 5G-L1-UL sends to the 5G-L2-PS the PUSCH CRC of the initial transmission or retransmission of the first UL HARQ Id 0 (pseudo Ra Msg3):

- If CRC is OK, the CFRA access is successful at U-Plane side, refer to (22).

- If CRC is NOK and the maxHARQ-Msg3Tx has not been reached (13), an HARQ retransmission is scheduled.

- If CRC is NOK, no CRC OK has been received for another HarqId (19) and the maxHARQ-Msg3Tx has been reached (20), it is likely that the UE has not correctly decoded the PDSCH containing the RA Msg2 and is therefore going to retransmit the RA Preamble, so the procedure loops to step (3).  
The RA state is set to msg3NackedCfra to indicate that HarqId0 is free to be reused for a new pseudo Msg3 unlike [Exc.3] (note that HarqId0 is reserved for this purpose in current sequence and cannot be used for another reason).

### 7.1.4.3 Exception

#### 7.1.4.3.0-1 Exception [Exc.3] Second RA preamble detection (ID: 11187926)

**7.1.4.3.0-1.0-1**  (ID: `11187932`)

Exception
A UE that has not been able to successfully decode the RA Msg2 is going to repeat the RA Preamble (RA Msg1). If (20) has not been reached yet to wait for this RA preamble again in (3), the new RA preamble is received between (4) to (20) whereas HarqId0 is already busy:

* DL 5G-L2-PS sends a new RA Msg2 to the UE as for (4) to (9) sequence.
* UL 5G-L2-PS ignores the UL grant notification from the DL and do not schedule decoding of the corresponding new pseudo RA Msg3 (HarqId0 already busy).  
  The UE is likely to retransmit this pseudo RA Msg3 thanks to HarqId0 retransmission request (16) of previous sequence. If this HarqId0 (20) or any HarqId (19) is CRC OK, CFRA access is successful in Uplane, refer to (22).

---

#### 7.1.4.3.0-2 Exception [Exc.5] RA-MSG2 cannot be scheduled in time (ID: 11187940)

**7.1.4.3.0-2.0-1**  (ID: `11187948`)

Exception
In the unlikely case the 5G-L2-PS cannot schedule the DCI 1\_0 of the RA-MSG2 within the ra-ResponseWindow windows, the UE is going to repeat the RA-MSG1. Unlike [Exc.3], the procedure restarts from (3) as if RA-MSG1 had not been received.

---

#### 7.1.4.3.0-3 Exception [Exc.19] and [Exc.20] Extremely bad UL radio link condition (ID: 11187954)

**7.1.4.3.0-3.0-1**  (ID: `11187961`)

Exception
In case no PUSCH at all can be decoded after RA Msg2, the CFRA access procedure fails on U-Plane point of view.

* In SA, a new UE access is delayed because the UE cannot be released in absence if DL traffic (Flow Control has not been started yet).
* In NSA, the CFRA access is successful in C-Plane that receives the RRC completion indication over the SRB of the Primary NodeB. C-Plane requires a specific defense for the U-Plane failure.
