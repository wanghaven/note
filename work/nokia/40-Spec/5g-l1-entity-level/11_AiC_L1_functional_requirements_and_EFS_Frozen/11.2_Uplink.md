## 11.2.2 Cell Configuration

### 11.2.2.1 General

#### 11.2.2.1.0-10 General 5G-L1-HI shall send copy of UlData_PuschReceiveReq to Queue with ID from UlCell_SetupReq-&gt;harqFeedbackQueueID, when this ID is not equal to 0. (ID: 11472865)

### 11.2.2.2 Subcell types

#### 11.2.2.2.0-5 Subcell types 5G-L1-HI shall support the same subcell type (UlCell_SetupReq-&gt;ulSubcellType) for all UL subcells setup within single 5G-L1-HI instance (mixing of UL subcell types is not allowed). (ID: 11472889)

### 11.2.2.3 Bandwidth

#### 11.2.2.3.0-9 Bandwidth 5G-L1-HI shall support 100 MHz UL subcell in TDD FR1 (UlCell_SetupReq-&gt;ulBandwidth = 273 with UlCell_SetupReq-&gt;scs = khz_30). (ID: 11472934)

### 11.2.2.4 Beamforming

#### 11.2.2.4.0-1 Beamforming 5G-L1-HI shall support UL FD (Frequency Dependent) beamforming when UlCell_SetupReq-&gt; ulEcpriFdBeamforming = {true}. (ID: 11472941)

### 11.2.3.5 PRACH common requirements

**11.2.3.5.0-21**  (ID: `11473199`)

PRACH common requirements
5G-L1-HI shall support and handle broken antenna detection for 4 RX configuration:

 In case of 1 Rx antenna broken, the number of valid RX antennas = 3.

 In case of 2 Rx antenna broken, the number of valid RX antennas = 2.

 In case of 3 Rx antenna broken, the number of valid RX antennas = 1.

according to algorithm described in section 2.5.7 “Broken Antenna Detection and Antenna-Specific Scaling” in 5GMax-5G New Radio Algorithm Specification (5G\_RAN\_UP\_30198).

#### 11.2.4.2.4 Information type

**11.2.4.2.4.0-7**  (ID: `11473743`)

Information type
5G-L1-HI shall report decoded UlData\_PucchReceiveRespHarqD  -> subcells -> pucchResources -> uciBits using HARQ-AN bits as specified in 38.213 section 9.2.5.1 (v15.10.0)

Padding bits are located on the least significant bits of the last octet.

If number of UCI bits is lower than 8, padding bits are also set on least significant bits of first octet (see 5G\_L1\_IF\_14741\_replaced([11516557](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FSEq1vvxEe-AqvopbP1qhQ))).

**11.2.4.3.4.0-10**  (ID: `11474070`)

Information type
When SR bit is multiplexed with the CSI report for PUCCH F2, SR bit is decoded regardless of:

UlData\_PucchReceiveRespPs -> srBitDetecion value.

In this case SR bit is included in

UlData\_PucchReceiveRespPs -> numOfBitsOfUciInformation

and shall be passed transparently as a bit string to L2 in

UlData\_PucchReceiveRespPs -> uciBits.

#### 11.2.6.2.6 Delay management for SRS based beamforming

**11.2.6.2.6.0-3**  (ID: `11475843`)

Delay management for SRS based beamforming
5G\_L1\_HI shall receive API\_BBB\_SET\_SRS\_PATTERN\_CONFIG\_REQ\_MSG-> eaxcReceiverDelayConfigs-> receiverDelay = []. For each SRS eAxC stream, it indicates the delay between RU starts sending the 1st SRS eAxC stream and current SRS eAxC stream. Receive

delay is SRS eAxC stream specific. For the very 1st SRS eAxC stream of a given SRS transmission period of this RU, receive delay value is 0. OAM shall calculate

receive delay values for all SRS eAxC stream based on the allowed number of Srs streams per symbol, which depends on frameStructureType, nrCellType and

eCPRI rate. OAM shall send receive delay to L1 for all configured NDM SRS eAxC streams.

#### 11.2.6.3.5 TPMI selection

**11.2.6.3.5.0-10**  (ID: `11476228`)

TPMI selection
5G-L1-HI shall use  UlData\_SrsReceiveReq->subcells[]->srsSuMimoStruct->srsReceiveReqUes[]->fullPowerPuschPowerScalingRatio for calculation of UlData\_SrsReceiveRespPs->subcells[]->srsReceiveRespPsUes->ulPmiRank1 selection and UlData\_SrsReceiveRespPs->subcells[]->srsReceiveRespPsUes->ulPmiRank1Sinr when:

* UlData\_SrsReceiveReq->subcells[]->srsSuMimoStruct->srsReceiveReqUes[]->puschTransCoherence = nonCoherent

UlData\_SrsReceiveReq->subcells[]-->srsSuMimoStruct-> srsReceiveReqUes[]->fullPowerPuschPowerScalingRatio is a bitmap with the size of 8 bits, only the LSBs 6 bits of the u8 are valid. (see 5G\_L1\_IF\_14078\_replaced([11514394](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FR3PZ_vxEe-AqvopbP1qhQ)))

Each 2 bits indicate power scaling factor (see 3GPP 38.213 section 7.1) for TMPI snr. fullPowerPuschPowerScalingRatio is in format

0b 00 xx yy zz, where:

* xx is scaler for snr TPMI0,
* yy is scaller for snr for TPMI1,
* zz is scaler for snr for TPMI2, which is sum of snr for TPMI0 and TPMI1

xx, yy and zz can have values:

* 00 -> coresponding TPMI is not used
* 01 -> power scaling factor 1
* 10 -> power scaling factor 2
