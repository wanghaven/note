##### 3.1.1.1.1.1 Link configuration

**3.1.1.1.1.1.0-2**  (ID: `11423300`)

Link configuration
L1SW shall handle API\_L1SW\_CPRI\_CONFIGURE\_AxC\_INFO\_REQ\_MSG  message.

Message granularity: One at initial setup follow by one or more as needed during run time reconfiguration.

L1SW shall:

\*Terminate the message,

\*Extract parameters,

\*Perform parameters coherency checks,

\*Send to OAM API\_L1SW\_CPRI\_AxC\_INFO\_RESP\_MSG message.

In absence of error, L1SW shall proceed with subcell configuration coherently with [Dl, Ul]Cell\_SetupReq message processing.

In case of error, L1SW shall ignore the message.

##### 3.1.1.1.1.2 Link activation

**3.1.1.1.1.2.0-1**  (ID: `11423312`)

Link activation
Links are activated as a result of autonomous start up or OAM interface request.

Serdes setting and calibration need to be triggered with CPRI link activation message. This way OAM can ensure that serdes calibration is done after the system clocks are stable.

L1SW shall follow the procedure described in CPRI core user’s guide, in chapter 5. Following remarks to the procedure described in CPRI core user’s guide apply:

• Only supported line rate is 9.8 Gbps.

• Fast C&M channel activation when P pointer is set to 40-63, according to Link configuration chapter description.

• EDF (Ethernet Data Forwarding) setup is not supported.

• L1SW shall support Master mode only.

• All timing related settings shall be calculated according to the procedure described in BB L1 synchronization requirements and EFS (chapter 4).

• CPRI protocol version should be set to 2.

• L1SW shall authorize activation of CPRI links up to the maximum capability supported for both FDD and TDD (see 5G\_L1\_2013\_replaced([11423534](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-kwHPvvEe-AqvopbP1qhQ))).

**3.1.1.1.1.4.1.0-1**  (ID: `11423335`)

P pointer reporting
L1SW shall report P pointer to OAM any time link state reach > state D.

Subscription:

OAM shall subscribe to P pointer reporting service by sending API\_L1SW\_CPRI\_SUBSCRIBE\_REQ\_MSG message to L1SW.

L1SW shall respond to OAM with API\_L1SW\_CPRI\_SUBSCRIBE\_RESP\_MSG message. See 5G\_L1\_IF\_341\_replaced([11515538](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FR8u_fvxEe-AqvopbP1qhQ)) for details on subscription.

Reporting:

L1SW shall report P pointer value to OAM through API\_L1SW\_PORT\_ETHERNET\_POINTER\_IND\_MSG message. See 5G\_IF\_15132.

##### 3.1.1.1.1.13 Snapshot T14 reporting

**3.1.1.1.1.13.0-5**  (ID: `11423887`)

Snapshot T14 reporting
L1SW handling of T14 in case of link state transition sequence F/G=>B=>F/G without synchronization loss:

Such state transition sequence may happen at any time during operation.

While in normal operation (Link in F or G state):

[PR730168]

if L1SW detect F/G to B transition without synchronization loss, then it shall:

not perform T14 measurement

report to OAM last T14 value before F/G to B transition.

Else (F/G to B transition with synchronization loss) L1SW shall:

wait for CRX\_STATE = HFNSYNC

loop to step 12 from snapshot T14 measurement procedure description in 5G\_L1\_14770\_replaced([11424400](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-qPovvvEe-AqvopbP1qhQ)).

Endif.

UL subcell L2-L1 BIP messages processing trigger rules:

Subcell UL channel processing is triggered every slot based on incoming BIP real time messages from L2-PS (UlData\_xxxxxReceiveReq).

Before starting real time processing as indicated in UlData\_xxxxxReceiveReq messages, L1SW check that conditions related to subcell, link state, Leka\_Rx stuck recover, sdi alarm and finally cpri delay configuration are met.

Initial setup:

During initial setup, after CPRI link have been enabled (primary and secondary), timing parameters are exchanged between OAM and L1FH SW FH delay is measured on each link and delay compensation is operated per timing group. Then comes cell setup which map AxCs (pair of) to CPRI links. Real time channels processing then starts in both DL and UL based on L2-PS scheduling.

CB007184 conditions cell setup to availability of all HW resources expected to be used for considered cell. In case of subcell split mode (C4 subcell split across 2 CPRI links connecting to 2 separate RUs), this means both primary and secondary CPRI links have been enabled (state F/G respectively). As a consequence delay measurements (T14) must have been successfully performed on any of those links.

Run time operation:

In case of HW resource problem on fronthaul (broken link or RU issue), alarms (LOS/LOF/RAI/SDI) are raised by L1FH SW for considered CPRI link and sent to OAM.

If either of LOS, LOF or RAI alarm is detected on CPRI link receiver, link state drops to state B. If second link used in C4 subcell is up (state F/G), UL IQ data transport is ensured and C4 subcell is in degraded mode. Subcell is ready and Leka\_Rx can process BIP messages relative to AxC streams carried by remaining link.

In case of SDI detection on either link, C4 subcell is considered as not ready and it is expected that faulty RU recovers from this alarm before restarting UL processing on Leka\_Rx.

Leka\_Rx stuck: In some situations, Leka-Rx processing blocks may be stuck(1). Recovery actions are then performed to restore service with minimum interruption time. Recovery from Leka\_stuck is going through a full Leka-Rx reset and CPRI subsystem reset. If one CPRI link across the 2 links of C4 subcell is down after CPRI subsystem restart, remaining link will still ensure synchronization and IQ data transport, so degraded cell mode can be supported and UL processing on remaining AxC streams can be performed.

(1): For Leka-Rx stuck and recovery description, check Loner uplink datapath recovery (requirement 5G\_L1\_5593\_replaced([11424216](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-pBi_vvEe-AqvopbP1qhQ))).

Based on above described use case (intial set up or run time operation) and depending on CPRI alarms and Leka\_Rx status, UL subcell is considered as ready or not ready for channel processing from L1SW prospective.

Subcell not ready: If scenario definition is not met, subcell is considered not ready and UL channel processing request messages received on L2-L1 interface are blocked (no processing for considered UL subcell / AxC streams).

A CPRI link is considered as up and running (cpri link ok) if it have reached >= state D.

##### 3.1.2.2.1.1 T14 computation

**3.1.2.2.1.1.0-1**  (ID: `11424201`)

T14 computation
*cpriLoopbackDelay = CRX\_MF\_MEAS capture – cpri\_t14\_calibration\_offset*

With,

 CRX\_MF\_MEAS capture: BCN-N2 time stamp of basic frame start on CPRI Rx port (based on K28.5 sequence detection). Provided by L1-FPGA CPRI core register named CRX\_MF\_MEAS.

 cpri\_t14\_calibration\_offset: L1-FPGA internal processing latency comprising the CPRI serdes and CPRI core timing offset to be applied in T14 calculation.

cpri\_t14\_calibration\_offset = Serdes Tx delay + Serdes Rx delay + CPRI core internal delay

o Serdes delay(1) is the arithmetic sum of a static delay and a variable delay, which variable delay is brought by two FIFOs present in the data-path for each direction. See CPRI subsystem design specification [12698057](https://dn-prod.ext.net.nokia.com/rm/resources/BI_xksSIFyQEfCPFofqMEsORA).

o Serdes Tx static latency(2) = 169 CPRI clock cycles (1)

o Serdes Tx variable latency (ns) = [1024]-1\*[tx\_fifo + tx\_pcs\_fifo] (FIFO filling level registers, per CPRI link)

o Serdes Rx static latency = 275.5 CPRI clock cycles (1)

o Serdes Rx variable latency (ns) = [1024]-1\*[rx\_fifo + rx\_pcs\_fifo] (FIFO filling level registers, per CPRI link).

o CPRI core internal delay is the arithmetic sum of the following contributors (unit: UTU) (see [12698043](https://dn-prod.ext.net.nokia.com/rm/resources/BI_JCywwFyQEfCPFofqMEsORA) for details):

o CTX\_START = 2 \* CK\_BCN\_UTU + 7,5 +/-2.5 (t1)

With CK\_BCN\_UTU = 5, CTX\_START = 17.5 +/-2.5 (t2)

o CTX = 70 (t3)

o CRX\_MF\_START = (2.5 +/- 0.5) \* CK\_BCN\_UTU = 12.5 +/-2.5 (t4)

o CRX = 10 (t5)

o CRX\_MF\_MEAS = 3 \* CK\_BCN\_UTU = 15 (t6)

 Note: This delay is a constant and must not to be mixed up with the internal register CRX\_MF\_MEAS used in the formula for providing the basic frame start BCN-N2 time stamp value.

o CRX\_CDC: Rx buffer FIFO latency (variable delay)

CRX\_CDC = 27.5 + CRX\_CKFIFO\_LVL\*5 +/-2.5 (t7)

 With CRX\_CKFIFO\_LVL: CPRI Rx clock crossing FIFO level (L1-FPGA register). Valid range = 4 to 253.

o In final, CPRI core internal delay = t1 + t2 + t3 + t4 + t5 + t6 + t7.

Notes:

(1): Serdes delay is to be converted from ns to UTU before summing with CPRI core delay.

(2): Static latencies in Tx and in Rx are provided by Intel for Stratix 10L and H Tile transceiver PHY.

### 3.1.2.3 uplink datapath recovery in case of CPRI link instability

**3.1.2.3.0-1**  (ID: `11424216`)

uplink datapath recovery in case of CPRI link instability
The purpose of this functionality is to provide a mechanism controlled by L1 SW to perform a controlled recovery in a situation where some anomaly/ interferences has e.g. caused LEKA-RX to get stuck or CPRI link problems. The goal is to achieve known good state with minimal impact to base station normal operation.

Triggering conditions:

 CPRI abnormality: No optical signal power on receiver, CPRI link out-of-sync, SFP module issues.

 LEKA-RX L1\_Low buffer overflow: Different possible origins amongst which clock crossing FIFO instability during start up (see [12698043](https://dn-prod.ext.net.nokia.com/rm/resources/BI_JCywwFyQEfCPFofqMEsORA) for details).

Both trigger conditions should be present to engage recovery action.

 Since 5G20B: Recovery should be initiated in case where all LEKA-RX processing cells are deleted. As far CPRI subsystem is concerned this  situation happen when all link go down.

Recovery action: L1SW performs LEKA-RX reset when CPRI link comes up to state F/G and antenna carriers have been configured (e.g. n\_ul received from OAM). Recovery procedure is performed also when last uplink cell has been deleted from Loner.

Detailled procedure is provided in [12698064](https://dn-prod.ext.net.nokia.com/rm/resources/BI_DOR_AFyREfCPFofqMEsORA)

BFCM messages are used only in TDD mode. In FDD mode processing of *DlData\_PatternConfigReq messages is not needed and this message is not received by L1. In gNB case* DlData\_PatternConfigReq messages are received.\*\*

Beamforming can be enabled or disabled thanks to  *actbeamforming parameter in the DlCell\_SetupReq message (DOORS requirement 5G\_IF\_16).*

At reception of DlCell\_SetupReq, L1\_SW shall check the value of *actbeamforming parameter. If set to FALSE, DlData\_PatternConfigReq message shall be ignored by L1SW. For debug purposes, it shall be possible to log unexpected reception of DlData\_PatternConfigReq message while* actbeamforming  is set to FALSE.\*\*

CPRI link setup and timing setting procedure need to be finished prior to cell setup procedure may be completed. Once timing parameters setting is finished and cell setup is done, L1SW starts BFCM message handling. L1SW is required to receive *DlData\_PatternConfigReq message from L2 PS and fill in L1-FPGA registers, so the BFCM messages can be transmitted over CPRI VSB control channel to the RF module.*

*DlData\_PatternConfigReq message from L2 is transmitted over 10GE interface.*

One ABIL shall be able to handle in the same BFCM message as many*DlData\_PatternConfigReq messages as the number of subcells C2 or C4 to be set up, up to the maximum number defined in section 9.1.1 for the flexible cell mapping support.*

To support this capability, *DlData\_PatternConfigReq message contains the following parameters:*

 subcellId parameter,

 patternIdPolList (max size = 14 control blocks x 8 beams),

 numOfXpolBeams (1,2,4),

 bfcmOffset.

**DlData\_PatternConfigReq message conversion for beamforming interface:**

The conversion algorithm linked to the subcell multiplicity is described in 5G\_IF\_59. This algorithm produces a single BFCM message containing in each control message block the BFCM control vectors of each cell, with:

 in the case of two C4 subcells, 4 vectors for subcell#0 starting at bfcmOffset = 0 followed by 4 vectors for subcell#1 starting at bfcmOffset = 4,

 in the case of four C2 subcells, 2 vectors for subcell#0 starting at bfcmOffset = 0, 2 vectors for subcell#1 starting at bfcmOffset = 2, subcell#2 starting at bfcmOffset = 4, 2 vectors for subcell#3 starting at bfcmOffset = 6.

The resulting BFCM message is then written by L1-FPGA on each active CPRI core BFCM control register. This step uses existing L1-FPGA mechanisms (no impact on L1-FPGA).

**Cell block / unblock management in dual cell mode:**

In dual cell mode, L1SW shall check for each slot, the reception of both DlData\_PatternConfigReq messages.

 Cell block occurence: In case L1SW stops receiving the beam forming pattern configuration message from one cell/subcell while in dual cell mode i.e. one of both DlData\_PatternConfigReq ingress buffer is not written with new data for next slot (value for slot N = value for slot N-1), then L1SW will activate a single cell mode flag1. As long as this flag is active, BFCM processing block will process the remaining DlDataPatternConfigReq as in single cell mode. This means that for each control block, BFCM control vectors field assigned to the remaining subcell will be placed as in dual cell mode and BFCM control vectors field assigned to missing cell will be filled with invalid vectors.

 Cell unblock occurence: Once DlData\_PatternConfigReq message from both cells/subcells are received by corresponding messages ingress buffers, L1SW will:

o Deactivate the single cell mode flag,

o Process BFCM control vectors from both subcells as done before the cell block occurrence (see conversion algorithm linked to subcell multiplicity described in 5G\_IF\_59).

1: The single cell mode flag shall be cleared at cell setup.

For information, each RF module terminates the message and applies the corresponding cell BFCM vectors based on mask configured by OAM. Message flow based on two subcells use case is shown in the figure below.

### 3.1.2.11 CPRI L1 reset handling

**3.1.2.11.0-1**  (ID: `11424438`)

CPRI L1 reset handling
This requirement depicts the existing situation as to CPRI L1 inband reset handling from L1 prospective. Although this description is targeting ABIL platform, same behavior is expected on Loki/Thor HW platforms.

CPRI interface standardization:

The CPRI L1 inband reset objective is to give an RU HW reset capability from SM side in case other existing RU SW reset capabilities are inoperative due to some RU SW/HW issue. This reset is part of CPRI L1 inband protocol, and is transported through Z.130.0 control byte of CPRI frame.

Z.130.0 byte definition:

\*b0: Reset; 0: No reset; 1: Reset. DL: reset request; UL: reset acknowledge. Valid reset request and acknowledgment are subject to confirmation criteria (>10 hyperframes in DL, 5 hyperframes in UL).

\*b1: RAI; b2: SDI; b3; LOS; b4: LOF. 0: alarm cleared. 1 alarm set.

\*b5 to b7: reserved bits.

For more details, see [12698175](https://dn-prod.ext.net.nokia.com/rm/resources/BI_LyDkoFyWEfCPFofqMEsORA)

Current implementation:

CPRI L1 reset request is not managed (whatever baseband platform).

For L1 this means:

 Z.130.0:b0 : **= 0, default value in CPRI Tx L1 Inband Protocol register of CPRICORE001 IP (ABIL)**

 No management of L1 Inband reset request bit Z.130.0:b0 by L1SW.

Default value (Z.130.0:b0 : = 0) is transmitted in every hyperframe towards RU.

Alternative for RU reset:

In case of loss of M-plane communication between OAM and RU SW, OAM trigger a laser shut down on the concerned CPRI port (by sending appropriate HW-API message to L1 platform).

Subsequent loss of signal is then detected on RU optical receiver, triggering an RU reset.
