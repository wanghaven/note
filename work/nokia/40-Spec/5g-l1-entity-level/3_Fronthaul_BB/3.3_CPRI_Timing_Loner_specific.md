### 3.3.1.1 Loner hardware clock

**3.3.1.1.0-1**  (ID: `11415253`)

Loner hardware clock

On ABIL BB card there is located RP1 burst receiver on Loner FPGA. ABIL Xeon-D processors are also requiring accurate BCN counter as L2 RT is deployed on Xeon. On both Xeon-D BBP SW uses BCN latching mechanism to have accurate SW BCN counter. For that Loner L1-SW needs to provide interface for reading latched BCN counter value.

BCN latching is performed by using dedicated GPIO lines to Loner FPGA’s BCN sync in pins from both Xeon-Ds. When Xeon sets GPIO output on, Loner FPGA will latch current BCN counter value and stores it value to Xeon’s dedicated register. L1-SW provides message interface on UDP protocol to read stored register value.

## 3.3.2 Link Synchronization

### 3.3.2.1 Keep uninterruption in PHY during CPRI link break

#### 3.3.2.1.0-1 Keep uninterruption in PHY during CPRI link break After cell activation, 5G-L1-DL and 5G-L1-UL shall keep the subcells processing data without interruption when: The CPRI link mapping to the subcells is break. The CPRI link mapping to the subcells detected FIFO overflow/underflow. (ID: 11415017)

### 3.3.2.2 Link sync state report

**3.3.2.2.0-1**  (ID: `11415038`)

Link sync state report
L1 shall report frame sync state to OAM when CPRI port entering HFNSYNC state.

[PR660778]

On ABIL, reporting frame sync state when port leaving HFNSYNC is not supported by L1, will be supported in the future.

[End PR660778]

L1 reports CPRI port sync state via parameter frameSyncState in API\_L1SW\_PORT\_FRAME\_SYNC\_IND\_MSG

Refer to 5G\_L1\_9468\_replaced([11424372](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-qPoPvvEe-AqvopbP1qhQ)) for port frame sync state reporting procedure.

### 3.3.2.3 TX K28.5 adjustment

**3.3.2.3.0-1**  (ID: `11415053`)

TX K28.5 adjustment
L1 shall support to adjust CPRI TX K28.5 output with configuration of txCorrection.

txCorrection is configured by OAM to L1 via API\_L1SW\_CPRI\_CONFIGURE\_LINKS\_REQ\_MSG.

Note: txCorrection shall be set to same value to all ports of one ABIL.

CPRI TX Offset report

## 3.3.3 Timing Management

### 3.3.3.1 air interface timing configuration

**3.3.3.1.0-1**  (ID: `11415028`)

air interface timing configuration
For each CPRI link, L1 shall get frameStartSFN and frameStartSub10ms from API\_L1SW\_CPRI\_DELAY\_CONFIG\_REQ\_MSG, as extended downlink air interface timing offset.

Definition of frameStartSFN:

SFN-level component of the frameStart, in order to shift the air interface timing.

range: [-32769, 32768]

unit: N/A

Definition of frameStartSub10ms:

UTU-level component of the frameStart, in order to shift the air interface timing. It’s always set to positive.

range: [0, 12287999]

unit:  UTU

### 3.3.3.6 Offset_A calculation

**3.3.3.6.0-1**  (ID: `11415393`)

Offset\_A calculation
Offset\_A calculation

The LEKA TX block of L1-FPGA shall add CT marker to the first radio frame sample of IQ data generated by this block with 10 ms periodicity.

The CT marked sample must arrive at CPRI block within a certain timing window before the start of the 1st radio frame sample transmission on CPRI (refer to figure in 5G\_L1\_5609\_replaced([11415403](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-OKxPvvEe-AqvopbP1qhQ))). This window is 256 samples long, equivalent to 4 CPRI basic frames equal to 1440 UTU. Middle of basic frame was selected as optimum point of insertion and two basic frames advance is added to make sure that the CPRI core is configured in time, thus (64/2 [middle of basic frame] =) 32 + 128 [2 BFs for CPRI core configuration] = 160 BCN = 800 UTU is subtracted for Offset\_A calculation.

Then, Offset\_A calculation is as below (in UTU, should convert to loner BCN):

Offset\_A = [ (NDL -3) mod 38400 ] \*320 + 160

### 3.3.3.9 Nul adjustment in L1

**3.3.3.9.0-1**  (ID: `11415414`)

Nul adjustment in L1
NUL adjustment in L1

According to the timing inaccuracy described in 5G\_L1\_10032\_replaced([11415414](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-OKxvvvEe-AqvopbP1qhQ)), in case the uplink data arrives much earlier with worst timing error(here the worst case is 490ns earlier),  L1 uplink physical channels(PRACH, PUSCH, PUCCH) cannot work well due to algorithm.

[PR650007]

As first introduced by PR482744 in history, moving NUL left is needed to increase negative timing error tolerance as a tentative solution (meanwhile with setting SW\_OFFSET).

After getting NUL, L1SW shall do the correction as below:

Nul\_corrected = (Nul - UL\_OFFSET) mod 38400

where "mod" is defined as:

x mod y = x – y \* floor ( x / y )  if y != 0

x mod y = x                             if y == 0

Note:

 UL\_OFFSET is set from L1 internal configurable R&D parameter:   
 UL\_OFFSET\_FDD\_FR1 (default value = 1)

 UL\_OFFSET could be set to 0 or 1, positive value means shift left (earlier in timeline),

 NUL adjustment in L1 is a tentative solution and only applicable for CPRI-FDD-FR1 on Loner platform.

[End of PR650007]

### 3.3.3.10 Configuration of txCorrection

**3.3.3.10.0-1**  (ID: `11415420`)

Configuration of txCorrection
Configuration of txCorrection

L1 needs to:

 Update CPRI TX k28.5 position by compensating txCorrection to the CTX\_BCN\_N2 (N2 register of CPRI TX start timer)

CTX\_BCN\_N2 = ceil ( ( CPRIOffset\_configure - CPRI\_INTERNAL\_TX\_DELAY\_UTU - txCorrection ) mod 12288000 \* 4 / 5 )

 Loner CPRI subsystem will ignore 2 LSB of CTX\_BCN\_N2 to make internal TX K28.5 start at (next) BCN 5 UTU boundary.  
CTX\_BCN\_N2\_effective = (CTX\_BCN\_N2 + 0x3) & ~0x3

 Based on aligned start N2, L1 re-calculates CPRIOffset and reports to OAM.  
cpriOffset\_report = ( CTX\_BCN\_N2\_effective \* 5 / 4 + CPRI\_INTERNAL\_TX\_DELAY\_UTU + txCorrection ) mod 12288000

In above formula:

CPRIOffset\_configure = 0   (hardcode in both OAM and L1)

CPRI\_INTERNAL\_TX\_DELAY\_UTU = 111 (hardcode in L1)

mod operation:

x mod y = x - y \* floor(x/y)     if y != 0

x mod y = x                          if y = 0

Note: positive txCorrection will decrease real configured cpriOffset .

### 3.3.3.11 Configuration of rxCorrection

**3.3.3.11.0-1**  (ID: `11415425`)

Configuration of rxCorrection
Configuration of rxCorrection

L1 needs to:

 adjust txCorrection and rxCorrection to the T14

t14\_report = ingressFrameOffset\_report – cpriOffset\_report

= ( crx\_mf\_meas \* 5/4 - CPRI\_INTERNAL\_RX\_DELAY\_UTU - fifo\_lvl \* 5 + rxCorrection – cpriOffset\_report ) mod 12288000

In above formula:

crx\_mf\_meas, fifo\_lvl are read from Loner registers

CPRI\_INTERNAL\_RX\_DELAY\_UTU = 170                  (hardcode in L1)

cpriOffset\_report is calculated as in 5G\_L1\_15285\_replaced([11415420](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-OKx_vvEe-AqvopbP1qhQ))

mod operation:

x mod y = x - y \* floor(x/y)     if y != 0

x mod y = x                          if y = 0

Note: Compensating txCorrection and rxCorrection to T14 will help OAM to calculate more accurate T12/T34 which help to determine more accurate air interface timing . L1 does not need to adjust any internal timing with rxCorrection, rxCorrection shall be small enough, e.g., |rxCorrection| < 320 UTU.

### 3.3.3.13 BFCM timing

**3.3.3.13.0-1**  (ID: `11415482`)

BFCM timing
BFCM is an interface between L1 and RU, which controls the TDD switching points with respect to air Interface timing as well as time-domain beamforming patterns on a per-symbol basis. BFCM is not relevant for FDD mode. For more details about BFCM, please check reference 14.

BFCM transmission in L1-FPGA is triggered by CTI block at time Tfast\_vsb\_trigger  , which expected to be received on RU ahead of 1st I/Q data of every slot, to covers FPGA serialization time, miniHUB/DFE processing time and a safety margin for various antenna module implementations in RUs.

Tfast\_vsb\_trigger  = Offset\_A – Offset\_vsb

The BFCM advance time Offset\_vsb is set from L1 configuration file, currently it is set to (unit loner BCN = 5 UTU):

TDD FR1: 21299   => ~87us  (updated by PR476196)

TDD FR2: 4915   => ~20us

Tfast\_vsb\_trigger  is moved to previous frame if negative.

The BFCM may be transmitted across several CPRI VSB subchannels, L1 shall take it into account to ensure the BFCM transmitting time advance.

More information about BFCM can be found in reference 17 of 5G\_L1\_5579\_replaced([11415194](https://dn-prod.ext.net.nokia.com/rm/resources/BI_e-MVmfvvEe-AqvopbP1qhQ)).
