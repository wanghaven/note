#### 4.1.1.1.4 Downlink data address distribution

##### 4.1.1.1.4.1 Description

**4.1.1.1.4.1.0-5**  (ID: `12713427`)

Description
**Purpose:**

As 5G-L1-DL I want to get 5G-L2-PS interface address upon reception of downlink data address request from 5G-L2-PS and respond with a set of addresses for services provided by 5G-L1-DL.

**References:**

o Downlink Addressing Procedure: ([11512316](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRol4_vxEe-AqvopbP1qhQ))

o *DlData\_AddressReq message: ([11512343](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRol6fvxEe-AqvopbP1qhQ))*

o *DlData\_AddressResp message: ([11512347](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRol6vvxEe-AqvopbP1qhQ))*

**Actors:**

5G-L1-DL, 5G-L2-PS

**Preconditions:**

5G-L1-DL started and a subcell created in 5G-L1-DL.

**Description:**

1) 5G-L1-DL receives *DlData\_AddressReq per subcell from 5G-L2-PS.*

2) Store 5G-L2-PS *DlData interface addresses.*

3) Response message *DlData\_AddressResp is sent to 5G-L2-PS with a set of addresses per subcell for services provided by 5G-L1-DL.*

**Exceptions:**

In case of unexpected exception. *DlData\_AddressResp is sent with IE* status = NOK and IE *cause set according to the exception cause ([11512347](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRol6vvxEe-AqvopbP1qhQ)).*\*\*

**Acceptance criteria:**

Downlink 5G-L2-PS interface address are set successfully and *DlData\_AddressResp is sent with IE* status = OK and IE *casue = NoError and containing proper addresses for services provided by 5G-L1-DL.*\*\*

1) 5G-L1-DL receives *DlCell\_DeleteReq from 5G-CP-RT.*

2) Received request is validated.

3) Subcell data is cleared from shared memory.

4) *DlCell\_DeleteResp is sent back to 5G-CP-RT.*

**Exceptions:**

In case of a rejected request:

* Error is logged.

If response sending fails deleted subcell is kept as inactive and an error is logged.

**Acceptance criteria:**

DL subcell is deleted and *DlCell\_DeleteResp is sent back to 5G-CP-RT with IE* status = OK and IE *cause = NoError.*\*\*

#### 4.1.2.2.2 Configurations for split mode

Configurations for split mode
L1-SW shall support parameters *numPhaseCompensationLutIndex and* dlScPerCarrierPart parameters delivered in *DlCell\_SetupReq. Parameter* numDlScPerCarrierPart defines in how many frequency parts the total bandwith shall be processed. Parameter *dlScPerCarrierPart defines how many subcarriers are processed per frequency part. Values in* dlScPerCarrierPart are listed from lowest to highest frequency part.**\***\*

*DlCell\_SetupReq provides* phaseCompensationLutIndex array for each carrier parts. Number of arrays are defined by *numPhaseCompensationLutIndex. First LutIndex arrays will be mapped to carrier part with lowest frequency and Second LutIndex arrays to carrier part with second lowest frequency etc.*\*\*

L1-SW shall configure and process subcell in wideband NR legacy mode, when *numDlScPerCarrierPart is in use and value is >1.*

### 4.1.2.5 DL FDD supercell

#### 4.1.2.5.1 FDD supercell concept

**4.1.2.5.1.0-1**  (ID: `11426474`)

FDD supercell concept
In L1 DL, a FDD supercell consists of 2~6 L1 DL subcells. DL transmission for FDD supercell shall be based on broadcasting the same IQ signal (including all physical channels and physical signals) to all subcells of the supercell (5G\_UP\_13098).

* 1 primary subcell

In primary DL subcell, 5G-L1-DL shall be responsible for receiving DlData\_\*SendReq messages and processing all DL physical channels and physical signals, as a regular FDD DL subcell.

* 1~5 DL fronthaul duplication subcells.

Fronthaul duplication subcell is different from regular FDD DL subcell. Fronthaul duplication subcell is not visible to 5G-L2. No DLData\_*Req messages (DlData\_AddressReq, DlData\_SlotTypeReq and DlData\_*SendReq) sent to fronthaul duplication subcell.

Fronthaul duplication subcell shall duplicate all the DL transmissions of the primary L1 DL subcell. The duplication function shall be implemented on L1 fronthaul level.

Each fronthaul duplication subcell shall always have the same number and order of DL streams as the primary subcell.

| FR1-FDD-7-2a-eCPRI | - | - | - |
| FR1-FDD-OBSAI | - | [CB009543-A] Starting from 26R1 onwards: Supported [End CB009543-A] | [CB009543-A] Starting from 26R1 onwards: Supported [End CB009543-A] |

#### 4.1.2.5.2 FDD supercell setup and deletion

**4.1.2.5.2.0-1**  (ID: `11426490`)

FDD supercell setup and deletion in L1 shall follow the same procedures of normal FDD subcell setup and deletion ([11512114](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRmwtvvxEe-AqvopbP1qhQ)).

FDD supercell setup

When superCellConfig is presented in DlCell\_SetupReq message (presence flag of superCellConfig is '1') in FDD, 5G-L1-DL shall setup the FDD super cell by single DlCell\_SetupReq message with additional parameters in *superCellConfig ([11512175](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRnXyvvxEe-AqvopbP1qhQ))*.\*\*

* 5G-L1-DL shall use the parameters in DlCell\_SetupReq message, except the structure *superCellConfig, to configure primary DL subcell.*
* 5G-L1-DL shall use the structure *superCellConfig ([11512175](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRnXyvvxEe-AqvopbP1qhQ)) to configure fronthaul duplication subcells. Besides the parameters in structure* superCellConfig ([11512175](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRnXyvvxEe-AqvopbP1qhQ))*, fronthaul duplication subcells have the same parameters as primary DL subcell.*\*\*
* The *subcellId in DlCell\_SetupReq is for primary subcell.*
* No restrictions for the supported Bandwidth. The actually supported bandwidth configuration refer to the associated SISO features.
* If FDD super cell setup success, 5G-L1-DL shall respond with parameter *status equal to* "EStatus5G\_Ok" and parameter *cause equal to* "0x00000000" in *DlCell\_SetupResp ([11512191](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRnXzvvxEe-AqvopbP1qhQ)).*\*\*\*\*
* If FDD super cell setup failure due to the parameters for setup primary subcell, 5G-L1-DL shall respond with parameter *status equal to* "EStatus5G\_NotOk" and proper *cause ([11512188](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRnXzfvxEe-AqvopbP1qhQ)) in* DlCell\_SetupResp ([11512191](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRnXzvvxEe-AqvopbP1qhQ)) as normal FDD subcell. All parameters of primary subcell and fronthaul duplication subcells shall be removed from L1.**\***\*
* **Note, the subcell type "A\*F" defined for fronthaul duplication subcell in [11094251](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kmaohvvSEe-AqvopbP1qhQ) is not visible in L1. ## 4.1.4 DL PRB pooling

### 4.1.4.1 Intra-subpool PRB pooling

#### 4.1.4.1.1 Concept

**4.1.4.1.1.0-1**  (ID: `11427136`)

Concept
L1 shall support baseband PRB pooling function which includes sharing resources needed for PRB computations (e.g. memory) among cells. PRB pooling takes place inside L1 subpool level and there is no sharing among L1 subpools. Cells share the PRB resources in L1 subpool level.

Note that PRB resources are granted independently for UL and DL.

* In DL a minimum number of PRB shall be granted to the cells for PDCCH channels as well as a minimum PRB for PDSCH traffic.
* In UL a  minimum number of PRB shall be granted to the cells for PUCCH channels as well as a minimum PRB for PUSCH traffic.

[CB007814-H] L1 shall also support this requirement in ABIO for FR1-FDD-eCPRI for 23R2-SR [End CB007814-H]

### 4.1.4.2 Inter-subpool PRB pooling

#### 4.1.4.2.1 Initial State

#### 4.1.4.2.1-1 Initial State

##### 4.1.4.2.1-1.1 Concept

**4.1.4.2.1-1.1.0-2**  (ID: `11427212`)

Concept
L1 DLPHY shall support dynamic resource allocation for PRB pooling functionality. The PRB pooling resources, which are shared among the subpools within L1 pool, are reconfigured by using two parameters(l1SpMaxNumStreamPrb, l1SpMaxNumLayerPrb) communicated to L1. These parameters are in terms of maximum capacity of stream PRBs and layer PRBs allocated for each subpool. In other words, each subpool receives its own capacity parameters. L1 DLPHY shall receive the reconfiguration parameters and shall respond to 5G-L2 (refer to [11513342](https://dn-prod.ext.net.nokia.com/rm/resources/BI_FRv6rPvxEe-AqvopbP1qhQ)).

L1 DLPHY implements this L1 inter-subpool pooling per L1 pool. When there are more than one L1 pool, L1 DLPHY can enable/disable inter-subpool pooling for each L1 pool independently in spite of which RAT/numerology was used for this L1 pool, and receiving dynamic BB reconfig from seperate BBRM if more than one L1 pool is enable inter-subpool pooling.

L1 DLPHY can support this inter-subpool pooling when there is more than one L1 subpool in the L1 pool. L1 DLPHY should disable Inter-subpool pooling if there is only one L1 subpool in this L1 pool.

- currentLayerPrbCapacityPerSp = min{currentLayerPrbCapacityPerSp , proposedLayerPrbCapacityPerSp}

In other words, BBRM calculates current capacities as the minimum of capacities between current and proposed capacities for stream and layer PRBs per subpools. Note that these values are not sent to L1.

a) If the duration between two consecutive DlPool\_BbResourceReconfReq messages is less than 2 SFN, L1 cannot handle the new reconfiguration.

b) If L1 receive DlPool\_BbResourceReconfReq message while the current DlPool\_BbResourceReconfReq message has not been answered yet.

3: The reconfiguration request is received by L1 while PRB**/Large BB pooling is disabled.**

4: Invalid L1 Pool id which is not known to L1.

5: Invalid L1 subpool id which is not known to L1.

6: The cause is unknown.

The calculated subpool capacities are sent to L1 through DlPool\_BbResourceReconfReq message in the same SFN (second SFN) as shown in the following figure.

The following information are transferred to L1:

• The calculated subpool capacity:

- l1SpMaxNumStreamPrb: Maximum number of allocated stream PRB per subpool

- l1SpMaxNumLayerPrb: Maximum number of allocated layer PRB per subpool

• The SFN in which L1 shall apply the new reconfigurations; this SFN is the second SFN of the next pooling interval (see the below figure).

In case of receiving DlPool\_BbResourceReconfResp with "success" status, L2 applies the new configuration, i.e., new  l1SPMaxNumStreamPrb and l1SPMaxNumLayerPrb per subpool.

In case or receiving DlPool\_BbResourceReconfResp with "failure" status, L2 keeps the current stream and layer PRB capacities per subpool.

Exceptional scenarios:

• If for any reason, L2 is not received DlPool\_BbResourceReconfResp message in the current pooling interval window, BBRM shall calculate the following PRB capacities and L2PS shall apply these values:

- currentStreamPrbCapacityPerSp = min{currentStreamPrbCapacityPerSp, proposedStreamPrbCapacityPerSp}

- currentLayerPrbCapacityPerSp = min{currentLayerPrbCapacityPerSp , proposedLayerPrbCapacityPerSp}

In other words, BBRM calculates current capacities as the minimum of capacities between current and proposed capacities for stream and layer PRBs per subpools. Note that these values are not sent to L1.

Once inter-subpool capacities are in place, BBRM calculates the L1 intra subpool capacities (PRBs among the subcells in the subpool) in the first SFN of the next pooling interval. L2PS uses the new L1 intra capacities in the second SFN of the next pooling interval (refer to [11212354](https://dn-prod.ext.net.nokia.com/rm/resources/BI_kxAu-fvSEe-AqvopbP1qhQ)).

3: The reconfiguration request is received by L1 while PRB/Large BB pooling is disabled.

4: Invalid L1 Pool id which is not known to L1.

5: Invalid L1 subpool id which is not known to L1.

#### 4.1.6.2.1 Definition

**4.1.6.2.1.0-1**  (ID: `11428058`)

Definition
The radio frame duration is fixed to 10 ms and each one consists of 10 sub-frames with 1ms duration each. Each sub-frame consists of a number of NR-slots which depends on the numerology used.

Each NR-slot consists of 14 OFDM symbols numbered from 0 to 13 whatever the numerology used.

| 30 | 2 | 20 | 0.5 |

Cyclic prefix (CP)
Each OFDM symbol has a cyclic prefix (CP). Only normal CP is supported.

 For other than the first OFDM symbol in every 0.5ms slot:

o  all OFDM symbols within 0.5ms have the same CP size.

 For the first OFDM symbol in 0.5ms slot:

o The CP is longer by 16 x Ts compared to other OFDM symbols, where Ts = 1/(15000 \* 2048) ~= 32.552 ns.

* In case of downsampling mode for FDD 20 MHz:

o It shall use the CP length as for 1536 iFFT.
